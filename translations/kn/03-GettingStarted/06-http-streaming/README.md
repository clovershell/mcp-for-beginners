# HTTPS ಸ್ಟ್ರೀಮಿಂಗ್ ಮಾದರಿ ಸಂದರ್ಭ ಪ್ರೋಟೋಕಾಲ್ (MCP) ಜೊತೆ

ಈ ಅಧ್ಯಾಯವು HTTPS ಬಳಸಿ ಮಾದರಿ ಸಂದರ್ಭ ಪ್ರೋಟೋಕಾಲ್ (MCP) ಜೊತೆ ಸುರಕ್ಷಿತ, ವಿಸ್ತರಿಸಬಹುದಾದ ಮತ್ತು ರಿಯಲ್ ಟೈಮ್ ಸ್ಟ್ರೀಮಿಂಗ್ ಅನ್ನು ಅನುಷ್ಠಾನಗೊಳಿಸುವ ಸಮಗ್ರ ಮಾರ್ಗದರ್ಶಿಯನ್ನು ನೀಡುತ್ತದೆ. ಇದರಲ್ಲಿ ಸ್ಟ್ರೀಮಿಂಗ್‌ಗಾಗಿ ಪ್ರೇರಣೆ, ಲಭ್ಯವಿರುವ ಸಾರಿಗೆ ವ್ಯವಸ್ಥೆಗಳು, MCP ನಲ್ಲಿ ಸ್ಟ್ರೀಮಬಲ್ HTTP ಅನ್ನು ಹೇಗೆ ಅನುಷ್ಠಾನಗೊಳಿಸುವುದು, ಸುರಕ್ಷತಾ ಉತ್ತಮ ಪ್ರಾಥಮಿಕತೆಗಳು, SSE ಯಿಂದ ಸ್ಥಳಾಂತರ ಮತ್ತು ನಿಮ್ಮ STEM ಸ್ಟ್ರೀಮಿಂಗ್ MCP ಅಪ್ಲಿಕೇಶನ್‌ಗಳನ್ನು ನಿರ್ಮಿಸುವ ನೈಪುಣ್ಯ ಮಾರ್ಗದರ್ಶಿ ಒಳಗೊಂಡಿದೆ.

> **ಮುಂದೆ ನೋಡೋಣ:** ಈ ಪಾಠವು Streamable HTTP ಅನ್ನು **MCP ನಿರ್ದಿಷ್ಟತೆ 2025-11-25** ಅಡಿ ವಿವರಿಸುತ್ತದೆ, ಅಲ್ಲಿ ಸೆಷನ್ 'initialize' ಸಂದರ್ಭದಲ್ಲಿ ಸ್ಥಾಪಿತವಾಗುತ್ತಿದ್ದು `Mcp-Session-Id` ಹೆಡರ್ ಮೂಲಕ ಪಿನ್ ಮಾಡಲ್ಪಡುತ್ತದೆ. `2026-07-28` ಬಿಡುಗಡೆ ಅಭ್ಯರ್ಥಿ ಸಂಪೂರ್ಣವಾಗಿ ಹ್ಯಾಂಡ್‌ಶೇಕ್ ಮತ್ತು ಸೆಷನ್ ಐಡಿ ಅನ್ನು ತೆಗೆದುಹಾಕಿದ್ದು, ಪ್ರತಿ ವಿನಂತಿಯೂ ಸ್ವಯಂ ಕಂಟೈನ್ಡ್ ಆಗಿದ್ದು ಯಾವುದೇ ಸೆರ್ವರ್ ಉದಾಹರಣೆಗೆ ರುಟಿಟವಾಗಬಹುದು ಯಾವುದೇ ಸ್ಥಿರ ಸೆಷನ್ ಇಲ್ಲದೆ. ವಿವರಗಳಿಗೆ [What’s Changing in MCP: The 2026-07-28 Release Candidate](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) ನೋಡಿ.

## MCP ನಲ್ಲಿ ಸಾರಿಗೆ ವ್ಯವಸ್ಥೆಗಳು ಮತ್ತು ಸ್ಟ್ರೀಮಿಂಗ್

ಈ ವಿಭಾಗವು MCP ನಲ್ಲಿ ಲಭ್ಯವಿರುವ ವಿಭಿನ್ನ ಸಾರಿಗೆ ವ್ಯವಸ್ಥೆಗಳನ್ನು ಮತ್ತು ಗ್ರಾಹಕರ ಮತ್ತು ಸೆರ್ವರ್‌ಗಳ ನಡುವೆ ರಿಯಲ್ ಟೈಮ್ ಸಂವಹನಕ್ಕೆ ಸ್ಟ್ರೀಮಿಂಗ್ ಸಾಮರ್ಥ್ಯಗಳನ್ನು ಸಕ್ರಿಯಗೊಳಿಸುವ ಅವರ ಪಾತ್ರವನ್ನು ಅನ್ವೇಷಿಸುತ್ತದೆ.

### ಸಾರಿಗೆ ವ್ಯವಸ್ಥೆ ಎಂದರೆ ಏನು?

ಒಂದು ಸಾರಿಗೆ ವ್ಯವಸ್ಥೆ ಗ್ರಾಹಕ ಮತ್ತು ಸೆರ್ವರ್ ನಡುವಿನ ಡೇಟಾ ವಿನಿಮಯವನ್ನು ನಿರ್ಧರಿಸುತ್ತದೆ. MCP ವಿವಿಧ ಪರಿಸರಗಳು ಮತ್ತು ಅಗತ್ಯಗಳಿಗೆ ಹೊಂದಿಕೊಳ್ಳಲು ಹಲವಾರು ಸಾರಿಗೆ ಪ್ರಕಾರಗಳನ್ನು ಬೆಂಬಲಿಸುತ್ತದೆ:

- **stdio**: ಸ್ಟ್ಯಾಂಡರ್ಡ್ ಇನ್‌ಪುಟ್/ಔಟ್‌ಪುಟ್, ಸ್ಥಳೀಯ ಮತ್ತು CLI ಆಧಾರಿತ ಉಪಕರಣಗಳಿಗೆ ಸೂಕ್ತ. ಸರಳ, ಆದರೆ ವೆಬ್ ಅಥವಾ ಕ್ಲೌಡ್‌ಗೆ ಸೂಕ್ತವಲ್ಲ.
- **SSE (ಸರ್ವರ್-ಸೆಂಟ್ ಇವೆಂಟ್ಸ್)**: ಸರ್ವರ್‌ಗಳು HTTP ಮೂಲಕ ಗ್ರಾಹಕರಿಗೆ ರಿಯಲ್ ಟೈಮ್ ಅಪ್‌ಡೇಟ್ಗಳನ್ನು ಪುಷ್ ಮಾಡಲು ಅನುಮತಿಸುತ್ತದೆ. ವೆಬ್ UI ಗಳಿಗೆ ಉತ್ತಮ, ಆದರೆ ವಿಸ್ತರಣಾ ಮತ್ತು ಮಿನುಗುಲ್ಲಿಕೆಯಲ್ಲಿ ಸೀಮಿತ. MCP ನಿರ್ದಿಷ್ಟತೆ 2025-06-18 ರಂದು, ಸ್ವತಂತ್ರ SSE ಸಾರಿಗೆ ನಿಷ್ಕ್ರಿಯಗೊಳಿಸಲಾಗಿದ್ದು "Streamable HTTP" ಸಾರಿಗೆಯ ಮೂಲಕ ಬದಲಾಯಿಸಲಾಗಿದೆ.
- **Streamable HTTP**: ಆಧುನಿಕ HTTP ಆಧಾರಿತ ಸ್ಟ್ರೀಮಿಂಗ್ ಸಾರಿಗೆ, ಸೂಚನೆಗಳನ್ನು ಮತ್ತು ಉತ್ತಮ ವಿಸ್ತಾರಣೆಯನ್ನು ಬೆಂಬಲಿಸುತ್ತದೆ. ಹೆಚ್ಚಿನ ಉತ್ಪಾದನೆ ಮತ್ತು ಕ್ಲೌಡ್ ಪರಿಸರಗಳಿಗೆ ಶಿಫಾರಸ್ಸು ಮಾಡಲಾಗಿದೆ.

### ಹೋಲಿಕೆ ಪಟ್ಟಿಕೆ

ಈ ಸಾರಿಗೆ ವ್ಯವಸ್ಥೆಗಳ ನಡುವಿನ ಭೇದಗಳನ್ನು ಅರ್ಥಮಾಡಿಕೊಳ್ಳಲು ಕೆಳಗಿನ ಹೋಲಿಕೆ ಪಟ್ಟಿಕೆಯನ್ನು ನೋಡಿ:

| ಸಾರಿಗೆ         | ರಿಯಲ್ ಟೈಮ್ ಅಪ್ಡೇಟ್ಗಳು | ಸ್ಟ್ರೀಯಮಿಂಗ್ | ವಿಸ್ತಾರಣೆ | ಬಳಕೆ ಪ್ರಕರಣ            |
|----------------|-------------------------|-------------|-----------|--------------------------|
| stdio          | ಇಲ್ಲ                     | ಇಲ್ಲ        | ಕಡಿಮೆ      | ಸ್ಥಳೀಯ CLI ಉಪಕರಣಗಳು     |
| SSE            | ಹೌದು                    | ಹೌದು       | ಮಧ್ಯಮ      | ವೆಬ್, ರಿಯಲ್ ಟೈಮ್ ಅಪ್ಡೇಟ್ಗಳು |
| Streamable HTTP| ಹೌದು                    | ಹೌದು       | ಉನ್ನತ     | ಕ್ಲೌಡ್, ಬಹು ಗ್ರಾಹಕರ     |

> **ಸಲಹೆ:** ಸರಿಯಾದ ಸಾರಿಗೆ ಆಯ್ಕೆ ಪ್ರದರ್ಶನ, ವಿಸ್ತಾರಣೆ ಮತ್ತು ಬಳಕೆದಾರ ಅನುಭವವನ್ನು ಪ್ರಭಾವಿಸುತ್ತದೆ. **Streamable HTTP** ಆಧುನಿಕ, ವಿಸ್ತಾರಣೆಯಾದ ಮತ್ತು ಕ್ಲೌಡ್-ಸಿದ್ಧ ಅಪ್ಲಿಕೇಶನ್‌ಗಾಗಿ ಶಿಫಾರಸು ಮಾಡಲಾಗಿದೆ.

ಹಿಂದಿನ ಅಧ್ಯಾಯಗಳಲ್ಲಿ ನಿಮಗೆ ತೋರಿಸಲಾದ stdio ಮತ್ತು SSE ಸಾರಿಗೆಗಳನ್ನು ಗಮನಿಸಿ ಮತ್ತು ಈ ಅಧ್ಯಾಯದಲ್ಲಿ ವರ್ಣಿಸಲಾದ streamable HTTP ಸಾರಿಗೆಯನ್ನು ಕಾಣಿರಿ.

## ಸ್ಟ್ರೀಮಿಂಗ್: ತತ್ವಗಳು ಮತ್ತು ಪ್ರೇರಣೆ

ಸ್ಟ್ರೀಮಿಂಗ್‌ನ ಮೂಲ ತತ್ವಗಳು ಮತ್ತು ಪ್ರೇರಣೆಗಳನ್ನು ಅರ್ಥಮಾಡಿಕೊಳ್ಳುವುದು ಪರಿಣಾಮಕಾರಿ ರಿಯಲ್ ಟೈಮ್ ಸಂವಹನ ವ್ಯವಸ್ಥೆಗಳನ್ನು ಅನುಷ್ಠಾನಗೊಳಿಸಲು ಅಗತ್ಯ.

**ಸ್ಟ್ರೀಮಿಂಗ್** ಎನ್ನುವುದು ನೆಟ್ಟ워크 ಪ್ರೋಗ್ರಾಮಿಂಗ್ ತಂತ್ರಜ್ಞಾನವಾಗಿದ್ದು, ಸಂಪೂರ್ಣ ಪ್ರತಿಕ್ರಿಯೆ ಸಿದ್ಧವಾಗಲು ಕಾಯುವುದರ ಬದಲು, ಡೇಟಾವನ್ನು ಸಣ್ಣ ಘಟಕಗಳಲ್ಲಿ ಅಥವಾ ಘಟನೆಗಳ ಸರಣಿಯಾಗಿ ಕಳುಹಿಸಬೇಕು ಮತ್ತು ಸ್ವೀಕರಿಸಬಹುದು. ಇದು ವಿಶೇಷವಾಗಿ ಉಪಯುಕ್ತ:

- ದೊಡ್ಡ ಫೈಲ್‌ಗಳು ಅಥವಾ ಡೇಟಾಸೆಟ್‌ಗಳು.
- ರಿಯಲ್ ಟೈಮ್ ಅಪ್‌ಡೇಟ್ಗಳು (ಹೆಸರು: ಚಾಟ್, ಪ್ರಾಗತಿಪಟ್ಟಿ).
- ಲಾಂಬಿಕಾಲ ಓಡುವ ಗಣನೆಗಳು, ಬಳಕೆದಾರನಿಗೆ ಮಾಹಿತಿ ನೀಡಲು.

ಸ್ಟ್ರೀಮಿಂಗ್ ಬಗ್ಗೆ ನಿಮಗೆ ತಿಳಿಯಬೇಕಾದ ಮುಖ್ಯ बातें ಇಲ್ಲಿವೆ:

- ಡೇಟಾ ಹಂತ ಹಂತವಾಗಿ ವಿತರಿಸಲಾಗುತ್ತದೆ, ಸಂಪೂರ್ಣವಾಗಿ ಒಂದೇ ಬಾರಿ ಅಲ್ಲ.
- ಗ್ರಾಹಕ ಡೇಟಾವನ್ನು ಆಗಲೇ ಬಂದಂತೆ ಪ್ರಕ್ರಿಯೆ ಮಾಡಬಹುದು.
- ಅಂದಾಜಿನ ವಿಳಂಬವನ್ನು ಕಡಿಮೆ ಮಾಡಿದಂತೆ ಬಳಕೆದಾರ ಅನುಭವವನ್ನು ಸುಧಾರಿಸುತ್ತದೆ.

### ಸ್ಟ್ರೀಮಿಂಗ್ ಬಳಸಬೇಕಾದ ಕಾರಣವೇನು?

ಸ್ಟ್ರೀಮಿಂಗ್ ಬಳಸುವ ಕಾರಣಗಳು ಹೀಗಿವೆ:

- ಬಳಕೆದಾರರು ತಕ್ಷಣ ಪ್ರತಿಕ್ರಿಯೆ ಪಡೆಯುತ್ತಾರೆ, ಕೇವಲ ಅಂತ್ಯದಲ್ಲಿ ಅಲ್ಲ.
- ರಿಯಲ್ ಟೈಮ್ ಅಪ್ಲಿಕೇಶನ್‌ಗಳು ಮತ್ತು ಸ್ಪಂದನಶೀಲ UI ಗಳನ್ನು ಸಕ್ರಿಯಗೊಳಿಸುತ್ತದೆ.
- ನೆಟ್‌ವರ್ಕ್ ಮತ್ತು ಗಣನಾ ಸಂಪನ್ಮೂಲಗಳ ಪರಿಣಾಮಕಾರಿಯಾದ ಬಳಕೆ.

### ಸಾಮಾನ್ಯ ಉದಾಹರಣೆ: HTTP ಸ್ಟ್ರೀಮಿಂಗ್ ಸರ್ವರ್ ಮತ್ತು ಗ್ರಾಹಕ

ಸ್ಟ್ರೀಮಿಂಗ್ ಅನ್ನು ಹೇಗೆ ಅನುಷ್ಠಾನಗೊಳಿಸಬಹುದೆಂಬುದರ ಸರಳ ಉದಾಹರಣೆ ಇಲ್ಲಿದೆ:

#### ಪೈಥಾನ್

**ಸರ್ವರ್ (ಪೈಥಾನ್, FastAPI ಮತ್ತು StreamingResponse ಬಳಸಿ):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**ಗ್ರಾಹಕ (ಪೈಥಾನ್, requests ಬಳಸಿ):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

ಈ ಉದಾಹರಣೆ ಸರ್ವರ್ ಗಳು ಸಂದೇಶಗಳ ಸರಣಿಯನ್ನು ಬರುತ್ತಿದ್ದಂತೆ ತಕ್ಷಣಕ್ಲೈಂಟ್ ಗೆ ಕಳುಹಿಸುವುದನ್ನು ತೋರಿಸುತ್ತದೆ, ಎಲ್ಲಾ ಸಂದೇಶಗಳಾದ ಬಳಿಕ ಕಾಯುವುದಿಲ್ಲ.

**ಇದು ಹೇಗೆ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ:**

- ಸರ್ವರ್ ಪ್ರತಿ ಸಂದೇಶವನ್ನು ಸಿದ್ಧವಾಗುತ್ತಿದ್ದಂತೆ ನೀಡುತ್ತದೆ.
- ಗ್ರಾಹಕ ಪ್ರತಿ ಭಾಗವನ್ನು ಬಂದಂತೆ ಸ್ವೀಕರಿಸಿ ಮುದ್ರಿಸುತ್ತದೆ.

**ಅಗತ್ಯತೆಗಳು:**

- ಸರ್ವರ್ ಸ್ಟ್ರೀಮಿಂಗ್ ಪ್ರತಿಕ್ರಿಯೆ ಬಳಸಿ (ಉದಾ., FastAPI ನಲ್ಲಿ `StreamingResponse`).
- ಗ್ರಾಹಕ ಪ್ರತಿಕ್ರಿಯೆಯನ್ನು ಸ್ಟ್ರೀಮ್ ಆಗಿ ಪ್ರಕ್ರಿಯೆಗೊಳಿಸಬೇಕು (`requests` ನಲ್ಲಿ `stream=True`).
- ಸಾಮಾನ್ಯವಾಗಿ ವಿಷಯ ಪ್ರಕಾರ `text/event-stream` ಅಥವಾ `application/octet-stream`.

#### ಜಾವಾ

**ಸರ್ವರ್ (Java, Spring Boot ಮತ್ತು Server-Sent Events ಬಳಸಿ):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**ಗ್ರಾಹಕ (Java, Spring WebFlux WebClient ಬಳಸಿ):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**ಜಾವಾ ಅನುಷ್ಠಾನ ಟಿಪ್ಪಣಿಗಳು:**

- Spring Boot ರಿಯಾಕ್ಟಿವ್ ಸ್ಟಾಕ್ `Flux` ಸ್ಟ್ರೀಮಿಂಗ್ ಗೆ ಉಪಯೋಗಿಸುತ್ತದೆ
- `ServerSentEvent` ಘಟನೆ ಮಾದರಿಯಲ್ಲಿ ರಚನೆಗೊಳ್ಳುವ ಸ್ಟ್ರೀಮಿಂಗ್ ಒದಗಿಸುತ್ತದೆ
- `WebClient` ಜೊತೆಗೆ `bodyToFlux()` ರಿಯಾಕ್ಟಿವ್ ಸ್ಟ್ರೀಮಿಂಗ್ ಸೇವೆಗೆ ಸಹಾಯ ಮಾಡುತ್ತದೆ
- `delayElements()` ಘಟನಗಳ ನಡುವೆ ಸಮಯ ತಪ್ಪಿಸುವಂತೆ ಅನುಕೂಲ ಮಾಡುತ್ತದೆ
- ಘಟನೆಗಳಿಗೆ `info`, `result` ಮುಂತಾದ ವಿಧಗಳು ನೀಡಬಹುದು ಎಂದರೆ ಕ್ಲೈಂಟ್ ಉತ್ತಮವಾಗಿ ಹ್ಯಾಂಡಲ್ ಮಾಡಬಹುದು

### ಹೋಲಿಕೆ: ಕ್ಲಾಸಿಕ್ ಸ್ಟ್ರೀಮಿಂಗ್ ವಿರುದ್ಧ MCP ಸ್ಟ್ರೀಮಿಂಗ್

ಸ್ಟ್ರೀಮಿಂಗ್ "ಕ್ಲಾಸಿಕ್" ರೀತಿಯಲ್ಲಿ ಹೇಗೆ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ ಮತ್ತು MCP ನಲ್ಲಿ ಹೇಗೆ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ ಎನ್ನುವುದರ ನಡುವಿನ ವ್ಯತ್ಯಾಸ ಕೆಳಗಿನಂತೆ ಹರಡಬಹುದು:

| ವೈಶಿಷ್ಟ್ಯ      | ಕ್ಲಾಸಿಕ್ HTTP ಸ್ಟ್ರೀಮಿಂಗ್      | MCP ಸ್ಟ್ರೀಮಿಂಗ್ (ಸೂಚನೆಗಳು)       |
|----------------|---------------------------|---------------------------------|
| ಮುಖ್ಯ ಪ್ರತಿಕ್ರಿಯೆ | ತುಂಡು ತುಂಡಾಗಿ             | ಒಂದರಲ್ಲೊಂದು, ಕೊನೆಯಲ್ಲಿ              |
| ಪ್ರಗತಿ ಅಪ್ಡೇಟ್ಗಳು| ಡೇಟಾ ತುಂಡುಗಳಾಗಿ ಕಳುಹಿಸಲಾಗಿದೆ| ಸೂಚನೆಗಳಾಗಿ ಕಳುಹಿಸಲಾಗುತ್ತದೆ        |
| ಗ್ರಾಹಕ ಅಗತ್ಯಗಳು | ಸ್ಟ್ರೀಮ್ ಪ್ರಕ್ರಿಯೆ ಮಾಡಬೇಕಾಗಿದೆ | ಸಂದೇಶ ಹ್ಯಾಂಡ್ಲರ್ ಅನುಷ್ಠಾನಗೊಳಿಸಬೇಕು|
| ಬಳಸುವ ಸಂದರ್ಭ | ದೊಡ್ಡ ಫೈಲ್‌ಗಳು, AI ಟೋಕೆನ್ ಸ್ಟ್ರೀಮ್‌ಗಳು | ಪ್ರಗತಿ, ಲಾಗ್‌ಗಳು, ರಿಯಲ್ ಟೈಮ್ ಪ್ರತಿಕ್ರಿಯೆ |

### ಮುಖ್ಯ ವ್ಯತ್ಯಾಸಗಳು ಗಮನಿಸಿದವು

ಜೊತೆಗೆ, ಕೆಲವು ಪ್ರಮುಖ ವ್ಯತ್ಯಾಸಗಳು ಇಲ್ಲಿವೆ:

- **ಸಂವಹನ ಮಾದರಿ:**
  - ಕ್ಲಾಸಿಕ್ HTTP ಸ್ಟ್ರೀಮಿಂಗ್: ಸರಳ ತುಂಡುಗಳ ವರ್ಗಾವಣೆ ಎನ್ಕೋಡಿಂಗ್ ಉಪಯೋಗಿಸಿ ಡೇಟಾವನ್ನು ತುಂಡುಗಳಲ್ಲಿ ಕಳುಹಿಸುತ್ತದೆ
  - MCP ಸ್ಟ್ರೀಮಿಂಗ್: JSON-RPC ಪ್ರೋಟೋಕಾಲ್ ಮೂಲಕ ರಚಿತ ಸೂಚನಾ ವ್ಯವಸ್ಥೆಯನ್ನು ಉಪಯೋಗಿಸುತ್ತದೆ

- **ಸಂದೇಶ ಸ್ವರೂಪ:**
  - ಕ್ಲಾಸಿಕ್ HTTP: ಸರಳ ಪಠ್ಯ ತುಂಡುಗಳು ಹೊಸ ಸಾಲುಗಳೊಂದಿಗೆ
  - MCP: ಮೆಟಾಡೇಟಾ ಜೊತೆಗೆ ರಚಿತ LoggingMessageNotification ವಸ್ತುಗಳು

- **ಗ್ರಾಹಕ ಅನುಷ್ಠಾನ:**
  - ಕ್ಲಾಸಿಕ್ HTTP: ಸರಳ ಸ್ಟ್ರೀಮ್ ಪ್ರತಿಕ್ರಿಯೆ ಪ್ರಕ್ರಿಯೆ ಗ್ರಾಹಕ
  - MCP: ವಿವಿಧ ಪ್ರಕಾರದ ಸಂದೇಶಗಳನ್ನು ಪ್ರಕ್ರಿಯೆಗೊಳಿಸುವ ಸಂದೇಶ ಹ್ಯಾಂಡ್ಲರ್ ಹೊಂದಿರುವ ಉನ್ನತ ಗ್ರಾಹಕ

- **ಪ್ರಗತಿ ಅಪ್ಡೇಟ್ಗಳು:**
  - ಕ್ಲಾಸಿಕ್ HTTP: ಪ್ರಗತಿ ಮುಖ್ಯ ಪ್ರತಿಕ್ರಿಯೆestream ಭಾಗವಾಗಿದೆ
  - MCP: ಪ್ರಗತಿ ಪ್ರತ್ಯೇಕ ಸೂಚನಾ ಸಂದೇಶಗಳಾಗಿ ಕಳುಹಿಸಲಾಗುತ್ತದೆ ಮತ್ತು ಮುಖ್ಯ ಪ್ರತಿಕ್ರಿಯೆ ಕೊನೆಯಲ್ಲಿ ಸಿಗುತ್ತದೆ

### ಶಿಫಾರಸುಗಳು

ಕ್ಲಾಸಿಕ್ ಸ್ಟ್ರೀಮಿಂಗ್ (`/stream`ದ ಮೂಲಕ) ಮತ್ತು MCP ಸ್ಟ್ರೀಮಿಂಗ್ ನಡುವೆ ಆಯ್ಕೆ ಮಾಡಲು ಕೆಲವು ಶಿಫಾರಸುಗಳು ಇಲ್ಲಿವೆ.

- **ಸರಳ ಸ್ಟ್ರೀಮಿಂಗ್ ಅಗತ್ಯಗಳಿಗೆ:** ಕ್ಲಾಸಿಕ್ HTTP ಸ್ಟ್ರೀಮಿಂಗ್ ಅನುಷ್ಠಾನಕ್ಕೆ ಸರಳ ಮತ್ತು ಮೂಲಭೂತ ಸ್ಟ್ರೀಮಿಂಗ್ ಅಗತ್ಯಗಳಿಗೆ ಸಮರ್ಪಕ.

- **ಸಂಕೀರ್ಣ, ಸಂವಹನಾತ್ಮಕ ಅಪ್ಲಿಕೇಶನ್‌ಗಳಿಗೆ:** MCP ಸ್ಟ್ರೀಮಿಂಗ್ ಹೆಚ್ಚಿನ ಮೆಟಾಡೇಟಾ ಮತ್ತು ಸೂಚನೆಗಳು ಮತ್ತು ಅಂತಿಮ ಫಲಿತಾಂಶಗಳ ನಡುವಿನ ಪ್ರತ್ಯೇಕಿಕೆಯೊಂದಿಗೆ ರಚಿತ ವಿಧಾನವನ್ನು ಒದಗಿಸುತ್ತದೆ.

- **ಕೃತಕ ಬುದ್ಧಿಮತ್ತೆ (AI) ಅಪ್ಲಿಕೇಶನ್‌ಗಳಿಗೆ:** MCP ಸಂಕೇತಣ ವ್ಯವಸ್ಥೆ ದೀರ್ಘಕಾಲೀನ AI ಕಾರ್ಯಗಳಿಗೆ ವಿಶೇಷವಾಗಿ ಉಪಯುಕ್ತ, ಬಳಕೆದಾರರಿಗೆ ಪ್ರಗತಿ ಕುರಿತು ಮಾಹಿತಿ ನೀಡುತ್ತದೆ.

## MCP ನಲ್ಲಿ ಸ್ಟ್ರೀಮಿಂಗ್

ಸರಿ, ನೀವು ಈಗಾಗಲೇ ಕ್ಲಾಸಿಕ್ ಸ್ಟ್ರೀಮಿಂಗ್ ಮತ್ತು MCP ಸ್ಟ್ರೀಮಿಂಗ್ ನಡುವಿನ ವ್ಯತ್ಯಾಸ ಮತ್ತು ಶಿಫಾರಸುಗಳನ್ನು ನೋಡಿದ್ದೀರಿ. ಈಗ MCPನಲ್ಲಿ ಸ್ಟ್ರೀಮಿಂಗ್ ಅನ್ನು ನೀವು ಹೇಗೆ ಉಪಯೋಗಿಸಬಹುದು ಎಂದು ವಿವರಿಸುತ್ತೇವೆ.

MCP ಫ್ರೇಮ್‌ವರ್ಕ್ ಒಳಗಿರುವ ಸ್ಟ್ರೀಮಿಂಗ್ ಕಾರ್ಯವಿಧಾನವನ್ನು ಅರ್ಥಮಾಡಿಕೊಳ್ಳುವುದು ದೀರ್ಘಕಾಲದ ಕಾರ್ಯಾಚರಣೆಗಳ ಸಂದರ್ಭದಲ್ಲಿ ಬಳಕೆದಾರರಿಗೆ ರಿಯಲ್ ಟೈಮ್ ಪ್ರತಿಕ್ರಿಯೆ ಒದಗಿಸುವ ಪ್ರತಿಕ್ರಿಯಾಶೀಲ ಅಪ್ಲಿಕೇಶನ್‌ಗಳನ್ನು ನಿರ್ಮಿಸುವಲ್ಲಿ ಅನಿವಾರ್ಯ.

MCPನಲ್ಲಿ, ಸ್ಟ್ರೀಮಿಂಗ್ ಎಂದರೆ ಮುಖ್ಯ ಪ್ರತಿಕ್ರಿಯೆಯನ್ನು ತುಂಡುಗೊಳಿಸುವುದಲ್ಲ, ಆದರೆ ಸಂಸ್ಕರಣೆ ನಡೆಯುತ್ತಿರುವಾಗ ಗ್ರಾಹಕಿಗೆ **ಸೂಚನೆಗಳು** ಕಳುಹಿಸುವುದು. ಈ ಸೂಚನೆಗಳಲ್ಲಿ ಪ್ರಗತಿ ಅಪ್ಡೇಟ್ಗಳು, ಲಾಗ್‌ಗಳು ಅಥವಾ ಇತರೆ ಘಟನೆಗಳು ಇರಬಹುದು.

### ಇದು ಹೇಗೆ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ

ಮುಖ್ಯ ಫಲಿತಾಂಶ ಇನ್ನೂ ಒಂದೇ ಪ್ರತಿಕ್ರಿಯೆಯಾಗಿ ಕಳುಹಿಸಲಾಗುತ್ತದೆ. ಆದಾಗ್ಯೂ, ಸಂಸ್ಕರಣೆಯ ಮಧ್ಯದಲ್ಲಿ ಪ್ರತ್ಯೇಕ ಸಂದೇಶಗಳಾಗಿ ಸೂಚನೆಗಳನ್ನು ಕಳುಹಿಸಿ ಗ್ರಾಹಕರನ್ನು ನೇರವಾಗಿ ಹಚ್ಚಲಾಗುತ್ತದೆ. ಗ್ರಾಹಕ ಈ ಸೂಚನೆಗಳನ್ನು ಹ್ಯಾಂಡಲ್ ಮಾಡಬೇಕಾಗುತ್ತದೆ ಮತ್ತು ತೋರಿಸಬಲ್ಲನು.

## ಸೂಚನೆ ಎಂದರೆ ಏನು?

ನಾವು "ಸೂಚನೆ" ಎಂದು ಹೇಳಿದರು, MCP ಸಂಧರ್ಭದಲ್ಲಿ ಅದರ ಅರ್ಥ ಏನು?

ಸೂಚನೆ ಒಂದು ಸಂದೇಶವಾಗಿದ್ದು, ಸೆರ್ವರ್‌ನಿಂದ ಗ್ರಾಹಕಿಗೆ ದೀರ್ಘಕಾಲದ ಕಾರ್ಯಾಚರಣೆಗಳ ಮಧ್ಯದಲ್ಲಿ ಪ್ರಗತಿ, ಸ್ಥಿತಿ ಅಥವಾ ಇತರೆ ಘಟನೆಗಳು ಏನಾಗುತ್ತಿವೆ ಎಂಬುದನ್ನು ತಿಳಿಸಲು ಕಳುಹಿಸಲಾಗುತ್ತದೆ. ಸೂಚನೆಗಳು ಪಾರದರ್ಶಕತೆ ಮತ್ತು ಬಳಕೆದಾರ ಅನುಭವವನ್ನು ಸುಧಾರಿಸುತ್ತವೆ.

ಉದಾಹರಣೆಗೆ, ಗ್ರಾಹಕವು ಪ್ರಾಥಮಿಕ ಹ್ಯಾಂಡ್‌ಶೇಕ್ ಸಫಲವಾದ ನಂತರ ಒಂದು ಸೂಚನೆ ಕಳುಹಿಸಬೇಕು ಎಂದು ನಿರೀಕ್ಷಿಸಲಾಗುತ್ತದೆ.

ಸೂಚನೆ JSON ಸಂದೇಶವಾಗಿ ಹೀಗೆ ಕಾಣಿಸುತ್ತದೆ:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

ಸೂಚನೆಗಳು MCP ನಲ್ಲಿ ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) ವಿಷಯದಲ್ಲಿ ಸೇರಿಕೊಂಡಿವೆ.

> **ನಿಷೇಧ ಸೂಚನೆ:** `2026-07-28` MCP ನಿರ್ದಿಷ್ಟತೆ ಬಿಡುಗಡೆ ಅರ್ಜಿದಾರನು Logging ಕಾಲ್ಪನಿಕೆಯನ್ನು stdio ಸಾರಿಗೆಗಳಿಗಾಗಿ `stderr` ಮತ್ತು ರಚನಾಕೃತ ದರ್ಶನಕ್ಕೆ OpenTelemetry ಗೆ ಪರಿವರ್ತನೆಯಾಗಿ ನಿಷೇಧಿಸಲಾಗಿದೆ. Logging ನುಡಿ `2025-11-25` ಮತ್ತು ಯಾವುದೇ ಅಧಿಕೃತ ನಿಷೇಧಕ್ಕೂ ವರ್ಷಕ್ಕಿಂತ ಕಡಿಮೆಯಲ್ಲದೆ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ. ವಿವರಗಳಿಗೆ [What’s Changing in MCP: The 2026-07-28 Release Candidate](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) ನೋಡಿ.

ಲಾಗಿಂಗ್ ಕಾರ್ಯಗತಗೊಳಿಸಲು, ಸೆರ್ವರ್ ಈ ಪದ್ದತಿಯಿಂದ ಫೀಚರ್ / ಸಾಮರ್ಥ್ಯವಾಗಿ ಸಕ್ರಿಯಗೊಳಿಸಬೇಕು:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> ಉಪಯೋಗಿಸುವ SDK ನಿರ್ಭರುತ್ತಿದ್ದು, ಲಾಗಿಂಗ್ ಸ್ವಯಂಚಾಲಿತವಾಗಿದೆ ಅಥವಾ ನಿಮ್ಮ ಸೆರ್ವರ್ ವಿನ್ಯಾಸದಲ್ಲಿ ಸ್ಪಷ್ಟವಾಗಿ ಸಕ್ರಿಯಗೊಳಿಸಬೇಕಾಗಬಹುದು.

ವಿವಿಧ ವಿಧಗಳ ಸೂಚನೆಗಳಿವೆ:

| ಮಟ್ಟ      | ವಿವರಣೆ                           | ಉದಾಹರಣೆಯ ಬಳಕೆ                    |
|-----------|---------------------------------|----------------------------------|
| debug     | ವಿವರವಾದ ಡಿಬಗ್ಗಿಂಗ್ ಮಾಹಿತಿ         | ಕಾರ್ಯಚರಣೆ ಪ್ರವೇಶ / ನಿರ್ಗಮಿಸಲು    |
| info      | ಸಾಮಾನ್ಯ ಮಾಹಿತಿಪೂರ್ಣ ಸಂದೇಶಗಳು    | ಕಾರ್ಯಾಚರಣೆ ಪ್ರಗತಿ ಅಪ್‌ಡೇಟ್ಗಳು   |
| notice    | ಸಾಮಾನ್ಯ ಆದರೂ ಗಮನಾರ್ಹ ಘಟನೆಗಳು   | ಸಂರಚನಾ ಬದಲಾವಣೆಗಳು               |
| warning   | ಎಚ್ಚರಿಕೆ ಸ್ಥಿತಿಗಳು                  | ನಿಷೇಧಿತ ವೈಶಿಷ್ಟ್ಯ ಬಳಕೆ            |
| error     | ದೋಷ ಸ್ಥಿತಿಗಳು                    | ಕಾರ್ಯಾಚರಣೆ ವಿಫಲತೆಗಳು           |
| critical  | ಪ್ರಮುಖ ಸ್ಥಿತಿಗಳು                  | ವ್ಯವಸ್ಥೆಯ ಘಟಕದ ವೈಫಲ್ಯಗಳು        |
| alert     | ತಕ್ಷಣದ ಕ್ರಮ ಕೈಗೊಳ್ಳಬೇಕು           | ಡೇಟಾ ಶುಚಿತ್ವಕ್ಕೆ ಧಗೆ            |
| emergency | ವ್ಯವಸ್ಥೆ ಬಳಕೆ ಸಾಧ್ಯವಿಲ್ಲ            | ಸಂಪೂರ್ಣ ವ್ಯವಸ್ಥೆಯ ವಿಫಲತೆ          |

## MCP ನಲ್ಲಿ ಸೂಚನೆಗಳನ್ನು ಅನುಷ್ಠಾನಗೊಳಿಸುವುದು

MCPನಲ್ಲಿ ಸೂಚನೆಗಳನ್ನು ಅನುಷ್ಠಾನಗೊಳಿಸಲು, ಕಾರ್ಯಾಚರಣೆಗಳನ್ನು ನಿರ್ವಹಿಸುವಾಗ ರಿಯಲ್ ಟೈಂ ಅಪ್ಡೇಟ್ಗಳನ್ನು ಹ್ಯಾಂಡಲ್ ಮಾಡಲು ಸರ್ವರ್ ಮತ್ತು ಗ್ರಾಹಕ ಪರಿಗಳನ್ನೂ ಸಜ್ಜುಗೊಳಿಸಬೇಕು. ಇದು ನಿಮ್ಮ ಅಪ್ಲಿಕೇಶನ್‌ಗಳಿಗೆ ಬಳಕೆದಾರರಿಗೆ ತಕ್ಷಣದ ಪ್ರತಿಕ್ರಿಯೆ ನೀಡಲು ಸಹಾಯ ಮಾಡುತ್ತದೆ.

### ಸರ್ವರ್ ಕಡೆ: ಸೂಚನೆ ಕಳುಹಿಸುವುದು

ಸರ್ವರ್ ಭಾಗದಿಂದ ಪ್ರಾರಂಭಿಸೋಣ. MCPನಲ್ಲಿ, ನೀವು ವಿನಂತಿಗಳನ್ನು ಸಂಸ್ಕರಿಸುವಾಗ ಸೂಚನೆಗಳನ್ನು ಕಳುಹಿಸಬಹುದಾದ ಉಪಕರಣಗಳನ್ನು ನಿಶ್ಚಿತಗೊಳಿಸುತ್ತೀರಿ. ಸರ್ವರ್ ಸಾಮಾನ್ಯವಾಗಿ `ctx` ಸಂಕಲ್ಪವನ್ನು ಉಪಯೋಗಿಸಿ ಗ್ರಾಹಕನಿಗೆ ಸಂದೇಶಗಳನ್ನು ಕಳುಹಿಸುತ್ತದೆ.

#### ಪೈಥಾನ್

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

ಹಿಂದಿನ ಉದಾಹರಣೆಯಲ್ಲಿ, `process_files` ಉಪಕರಣವು ಪ್ರತಿ ಫೈಲ್ ಪ್ರಕ್ರಿಯೆಗೊಳಿಸುತ್ತಿದ್ದಂತೆ ಮೂರು ಸೂಚನೆಗಳನ್ನು ಗ್ರಾಹಕಿಗೆ ಕಳುಹಿಸುತ್ತದೆ. `ctx.info()` ವಿಧಾನವು ಜಾಹಿರಾತು ಸಂದೇಶಗಳನ್ನು ಕಳುಹಿಸಲು ಉಪಯೋಗಿಸಲಾಗಿದೆ.

ಮೇಲ್ವಿಚಾರಿಸಲು, ಸರ್ವರ್ ಸ್ಟ್ರೀಮಿಂಗ್ ಸಾರಿಗೆಯನ್ನು (ಉದಾ: `streamable-http`) ಬಳಸಿ, ಮತ್ತು ಗ್ರಾಹಕ ಸಂದೇಶ ಕಳುಹಿಸಲು ಹ್ಯಾಂಡ್ಲರ್ ಹೊಂದಿರಬೇಕು. ಇದನ್ನು ಮಾಡಲು ಸರಿಯಾಗಿ ಸರ್ವರ್ ಅನ್ನು `streamable-http` ಸಾರಿಗೆಯನ್ನು ಉಪಯೋಗಿಸುವಂತೆ ಸಿದ್ಧಪಡಿಸುವುದು ಈ ಕೆಳಗಿನಂತೆ:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

ಈ .NET ಉದಾಹರಣೆಯಲ್ಲಿ, `ProcessFiles` ಉಪಕರಣವು `Tool` ಅಟ್ರಿಬ್ಯೂಟ್ ಬಳಸಿ ವಿನ್ಯಾಸಗೊಳ್ಳಿದ್ದು, ಪ್ರತಿ ಫೈಲ್ ಪ್ರಕ್ರಿಯೆಗೊಳಿಸುವಾಗ ಮೂರು ಸೂಚನೆಗಳನ್ನು ಗ್ರಾಹಕಿಗೆ ಕಳುಹಿಸುತ್ತದೆ. `ctx.Info()` ವಿಧಾನವು ಜಾಹಿರಾತು ಸಂದೇಶಗಳನ್ನು ಕಳುಹಿಸಲು ಬಳಕೆ ಮಾಡಲಾಗಿದೆ.

ನಿಮ್ಮ .NET MCP ಸರ್ವರ್‌ನಲ್ಲಿ ಸೂಚನೆಗಳನ್ನು ಸಕ್ರಿಯಗೊಳಿಸಲು, ಸ್ಟ್ರೀಮಿಂಗ್ ಸಾರಿಗೆಯನ್ನು ಉಪಯೋಗಿಸುತ್ತಿದ್ದೀರಾ ಎಂದು ಖಚಿತಪಡಿಸಿಕೊಳ್ಳಿ:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### ಗ್ರಾಹಕ ಕಡೆ: ಸೂಚನೆಗಳನ್ನು ಸ್ವೀಕಾರ ಮಾಡುವುದು

ಗ್ರಾಹಕವು ಸಂದೇಶ ಹ್ಯಾಂಡ್ಲರ್ ಅನ್ನು ಅನುಷ್ಠಾನಗೊಳಿಸಿ, ಬರುವ ಸೂಚನೆಗಳನ್ನು ಹ್ಯಾಂಡಲ್ ಮಾಡಿ ತೋರಿಸಬೇಕು.

#### ಪೈಥಾನ್

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

ಹಿಂದಿನ ಕೋಡ್‌ನಲ್ಲಿ, `message_handler` ಕಾರ್ಯವು ಬರುವ ಸಂದೇಶವು ಸೂಚನೆಯಾ ಎಂಬುದನ್ನು ಪರಿಶೀಲಿಸುತ್ತದೆ. ಆಗ ಆಗಲಿ, ಅದು ಸೂಚನೆಯನ್ನು ಮುದ್ರಿಸುತ್ತದೆ; ಇಲ್ಲದಿದ್ದರೆ, ಸಾಮಾನ್ಯ ಸರ್ವರ್ ಸಂದೇಶವಾಗಿ ಪ್ರಕ್ರಿಯೆಗೊಳಿಸುತ್ತದೆ. ಜೊತೆಗೆ `ClientSession` ಅನ್ನು `message_handler` ಜೊತೆ ಆರಂಭಿಸುವುದನ್ನು ಗಮನಿಸಿ.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

ಈ .NET ಉದಾಹರಣೆಯಲ್ಲಿ, `MessageHandler` ಕಾರ್ಯವು ಬರುವ ಸಂದೇಶ ಸೂಚನೆಯಾ ಎಂಬುದು ಪರಿಶೀಲಿಸಿ, ಆಗ ಇದ್ದರೆ ಸೂಚನೆಯನ್ನು ಮುದ್ರಿಸುತ್ತದೆ, ಇಲ್ಲವಾದರೆ ಸಾಮಾನ್ಯ ಸರ್ವರ್ ಸಂದೇಶವಾಗಿಯೇ ಪ್ರಕ್ರಿಯೆ ಮಾಡುತ್ತದೆ. `ClientSession` ಅನ್ನು `ClientSessionOptions` ಮೂಲಕ ಸಂದೇಶ ಹ್ಯಾಂಡ್ಲರ್ ಸಹಿತ ಆರಂಭಿಸಲಾಗಿದೆ.

ಸೂಚನೆಗಳನ್ನು ಸಕ್ರಿಯಗೊಳಿಸಲು, ಸರ್ವರ್ ಸ್ಟ್ರೀಮಿಂಗ್ ಸಾರಿಗೆ (ಉದಾ., `streamable-http`) ಉಪಯೋಗಿಸಬೇಕು ಮತ್ತು ಗ್ರಾಹಕಿಂದ 메시 ಹ್ಯಾಂಡ್ಲರ್ ಹೊಂದಿರಬೇಕು.

## ಪ್ರಗತಿ ಸೂಚನೆಗಳು ಮತ್ತು ದೃಶ್ಯಗಳು

ಈ ವಿಭಾಗವು MCP ನಲ್ಲಿ ಪ್ರಗತಿ ಸೂಚನೆಗಳ ಕಲ್ಪನೆ, ಅವುಗಳ ಮಹತ್ವ ಮತ್ತು Streamable HTTP ಬಳಸಿ ಅವುಗಳನ್ನು ಹೇಗೆ ಅನುಷ್ಠಾನಗೊಳಿಸುವುದು ಎಂಬುದನ್ನು ವಿವರಿಸುತ್ತದೆ. ನಿಮ್ಮ ಅರ್ಥದ ದೃಢೀಕರಣಕ್ಕೆ ಪ್ರಾಯೋಗಿಕ—theಯ ಆಸೈನ್ಮೆಂಟ್ ದೊರೆಯುತ್ತದೆ.

ಪ್ರಗತಿ ಸೂಚನೆಗಳು ದೀರ್ಘಕಾಲೀನ ಕಾರ್ಯಾಚರಣೆಗಳ ವೇಳೆ ಸರ್ವರ್‌ನಿಂದ ಗ್ರಾಹಕಿಗೆ ಬರುವ ನೇರ-ಕಾಲ ಸಂದೇಶಗಳು. ಸಂಪೂರ್ಣ ಪ್ರಕ್ರಿಯೆಯ ಕೊನೆಯನ್ನು ಕಾಯದೆ, ಸರ್ವರ್ ಪ್ರಸ್ತುತ ಸ್ಥಿತಿಯನ್ನು ಗ್ರಾಹಕರಿಗೆ ತಲುಪಿಸುತ್ತದೆ. ಇದು ಪಾರದರ್ಶಕತೆ, ಬಳಕೆದಾರ ಅನುಭವ ಮತ್ತು ಡಿಬಗ್ ಸೌಲಭ್ಯತೆಯನ್ನು ಸುಧಾರಿಸುತ್ತದೆ.

**ಉದಾಹರಣೆ:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### ಪ್ರಗತಿ ಸೂಚನೆಗಳನ್ನು ಏಕೆ ಬಳಸಬೇಕು?

ಪ್ರಗತಿ ಸೂಚನೆಗಳು ಹಲವಾರು ಕಾರಣಗಳಿಂದ ಅಗತ್ಯವಿವೆ:

- **ಉತ್ತಮ ಬಳಕೆದಾರ ಅನುಭವ:** ಕಾರ್ಯ ಮುಂದುವರಿದಂತೆ ಬಳಕೆದಾರರು ಅಪ್ಡೇಟ್ಗಳನ್ನು ತಕ್ಷಣ ನೋಡುವರು, ಕೇವಲ ಅಂತ್ಯದಲ್ಲಿ ಅಲ್ಲ.
- **ರಿಯಲ್ ಟೈಮ್ ಪ್ರತಿಕ್ರಿಯೆ:** ಗ್ರಾಹಕರು ಪ್ರಗತಿ ಪಟ್ಟಿಗಳು ಅಥವಾ ಲಾಗ್‌ಗಳನ್ನು ತೋರಿಸಬಹುದು, ಅಪ್ಲಿಕೇಶನ್ ಸ್ಪಂದನಶೀಲವಾಗಿರುತ್ತದೆ.
- **ಡಿಬಗ್ಗಿಂಗ್ ಮತ್ತು ಮೇಲ್ವಿಚಾರಣೆ ಸುಲಭ:** ಅಭಿವೃದ್ಧಿಪಡಕರು ಮತ್ತು ಬಳಕೆದಾರರು ಪ್ರಕ್ರಿಯೆ ನಾಪತ್ತೆಯಾದ ಅಥವಾ ನಿಧಾನವಾಗಿರುವ ಸ್ಥಳವನ್ನು ಕಾಣಬಹುದು.

### ಪ್ರಗತಿ ಸೂಚನೆಗಳನ್ನು ಹೇಗೆ ಅನುಷ್ಠಾನಗೊಳಿಸಬೇಕು

MCPನಲ್ಲಿ ನೀವು ಪ್ರಗತಿ ಸೂಚನೆಗಳನ್ನು ಹೀಗೆ ಅನುಷ್ಠಾನಗೊಳಿಸಬಹುದು:

- **ಸರ್ವರ್ ಮೇಲೆ:** ಪ್ರತಿ ಐಟಂ ಪ್ರಕ್ರಿಯೆಗೊಳಿಸಲಾಗುತ್ತಿದ್ದು `ctx.info()` ಅಥವಾ `ctx.log()` ಮೂಲಕ ಸೂಚನೆಗಳನ್ನು ಕಳುಹಿಸುತ್ತದೆ. ಇದು ಮುಖ್ಯ ಫಲಿತಾಂಶ ಸಿದ್ಧವಾಗುವುದಕ್ಕೆ ಮೊದಲು ಸಂದೇಶ ಕಳುಹಿಸುತ್ತದೆ.
- **ಗ್ರಾಹಕ ಮೇಲೆ:** ಬರುವ ಸೂಚನೆಗಳನ್ನು ಕೇಳಿ ತೋರಿಸುವ ಸಂದೇಶ ಹ್ಯಾಂಡ್ಲರ್ ಅನ್ನು ಅನುಷ್ಠಾನಗೊಳಿಸಿ. ಈ ಹ್ಯಾಂಡ್ಲರ್ ಸೂಚನೆಗಳು ಮತ್ತು ಅಂತಿಮ ಫಲಿತಾಂಶದ ನಡುವೆ ಭೇದವನ್ನು ಮಾಡುತ್ತದೆ.

**ಸರ್ವರ್ ಉದಾಹರಣೆ:**


#### ಪೈಥಾನ್

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

#### ಪೈಥಾನ್

```python

        print("SERVER MESSAGE:", message)
```

## ಭದ್ರತಾ ವಿಚಾರಣೆಗಳು

HTTP ಆಧಾರಿತ ಸಾರಿಗೆಗಳೊಂದಿಗೆ MCP ಸರ್ವರ್‌ಗಳನ್ನು ಜಾರಿಗೆ ತರುವಾಗ, ಭದ್ರತೆ ಬಹುಮುಖ್ಯವಾಗಿದ್ದು, ಇದು ಹಲವಾರು ದಾಳಿ ಮಾರ್ಗಗಳು ಮತ್ತು ರಕ್ಷಣಾ ವ್ಯವಸ್ಥೆಗಳ ಮೇಲೆ ಎಚ್ಚರಿಕೆಯಿಂದ ಗಮನ ಹರಿಸುವ ಅಗತ್ಯವಿದೆ.

### ಅವಲೋಕನ

HTTP ಮೂಲಕ MCP ಸರ್ವರ್‍ಗಳನ್ನು ಹೊರತಾಗಿಸುವಾಗ ಭದ್ರತೆ ಅಂತಿಮವಾಗಿ ಮುಖ್ಯವಾಗಿದೆ. ಸ್ಟ್ರೀಮಬಲ್ HTTP ಹೊಸ ದಾಳಿ ಭಾಗಗಳನ್ನು ಪರಿಚಯಿಸುತ್ತದೆ ಮತ್ತು ಜಾಗರೂಕ ಕಾನ್ಫಿಗರೇಷನ್ ಅಗತ್ಯವಿದೆ.

### ಪ್ರಮುಖ ಅಂಶಗಳು

- **ಮೂಲ ಶೀರ್ಷಿಕೆ ಪರಿಶೀಲನೆ**: DNS ಮರುಬಂಧನ ದಾಳಿಗಳನ್ನು ತಡೆಯಲು ಎల్లಾ ಸಮಯವೂ `Origin` ಹೆಡರ್ ಪರಿಶೀಲಿಸಲಿ.
- **ಲೋಕಲ್ಹೋಸ್ಟ್ ಬಂಧನ**: ಸ್ಥಳೀಯ ಅಭಿವೃದ್ಧಿಗಾಗಿ ಸರ್ವರ್‌ಗಳನ್ನು `localhost` ಗೆ ಬಂಧಿಸಿ ಸಾರ್ವಜನಿಕ ಇಂಟರ್ನೆಟ್‌ಗೆ ಬಹಿರಂಗವಾಗದಂತೆ ಮಾಡಲಿ.
- **ಪ್ರಾಮಾಣೀಕರಣ**: ಉತ್ಪಾದನಾ ನಿಯೋಜನಗಳಿಗಾಗಿ ಪ್ರಾಮಾಣೀಕರಣ (ಉದಾ., API ಕೀಗಳು, OAuth) ಜಾರಿಗೆ ತರುವುದನ್ನು ಮಾಡಿರಿ.
- **CORS**: ಪ್ರವೇಶವನ್ನು ನಿಯಂತ್ರಿಸಲು ಕ್ರಾಸ್-ಒರಿಜಿನ್ ಸಂಪನ್ಮೂಲ ಹಂಚಿಕೆ (CORS) ನಿಯಮಾವಳಿಗಳನ್ನು ಕಾನ್ಫಿಗರ್ ಮಾಡಿ.
- **HTTPS**: ಸಂಚಾರವನ್ನು ಎನ್‌ಕ್ರಿಪ್ಟ್ ಮಾಡಲು ಉತ್ಪಾದನೆಯಲ್ಲಿ HTTPS ಬಳಸಿರಿ.

### ಉತ್ತಮ ಅಭ್ಯಾಸಗಳು

- ಪರಿಶೀಲನೆ ಇಲ್ಲದೆ ಬಂದಿರುವ ವಿನಂತಿಗಳನ್ನು ಎಂದಿಗೂ ನಂಬಬೇಡಿ.
- ಎಲ್ಲಾ ಪ್ರವೇಶ ಮತ್ತು ದೋಷಗಳನ್ನು ಲಾಗ್ ಮಾಡಿರಿ ಮತ್ತು ನಿಗದಿಗೊಳಿಸಿ.
- ಭದ್ರತಾ ದುರ್ಬಲತೆಗಳನ್ನು ಸರಿಪಡಿಸಲು ನಿಯಮಿತವಾಗಿ ಅವಲಂಬನೆಗಳನ್ನು ನವೀಕರಿಸಿ.

### ಸವಾಲುಗಳು

- ಸುಲಭ ಅಭಿವೃದ್ಧಿಯೊಂದಿಗೆ ಭದ್ರತೆಯನ್ನು ಸಮತೋಲನಗೊಳಿಸುವುದು
- ವಿವಿಧ ಗ್ರಾಹಕ ಪರಿಸರಗಳೊಂದಿಗೆ ಹೊಂದಿಕೆ ನೋಡಿಕೊಳ್ಳುವುದು

## SSE ಯಿಂದ ಸ್ಟ್ರೀಮಬಲ್ HTTP ಗೆ ಅಪ್‌ಗ್ರೇಡ್

ಇದೀಗ ಸರ್ವರ್-ಸೆಂಟ್ ಇವೆಂಟ್ಸ್ (SSE) ಬಳಸಿ ನಡೆಯುತ್ತಿರುವ ಅಪ್ಲಿಕೇಶನ್‌ಗಳಿಗೆ ಸ್ಟ್ರೀಮಬಲ್ HTTP ಗೆ ವರ್ಗಾವಣೆ ಮಾಡಲು ನಿಮ್ಮ MCP ಜಾರಿಗೆ ಹೆಚ್ಚು ಶಕ್ತಿಯುತ ಮತ್ತು ದೀರ್ಘಕಾಲೀನ ಸ್ಥಾಯಿತ್ವದ ಮಾರ್ಗಗಳನ್ನು ಒದಗಿಸುತ್ತದೆ.

### ಏಕೆ ಅಪ್‌ಗ್ರೇಡ್ ಮಾಡಬೇಕು?

SSE ಯಿಂದ ಸ್ಟ್ರೀಮಬಲ್ HTTP ಗೆ ಅಪ್‌ಗ್ರೇಡ್ ಮಾಡಬೇಕಾದ ಎರಡು ಪ್ರಮುಖ ಕಾರಣಗಳಿವೆ:

- ಸ್ಟ್ರೀಮಬಲ್ HTTP SSE ಗಿಂತ ಉತ್ತಮ ಪ್ರಮಾಣಸ್ಪರ್ಧಾತ್ಮಕತೆ, ಹೊಂದಾಣಿಕೆ ಮತ್ತು ಶ್ರೀಮಂತ ನೋಟಿಫಿಕೇಶನ್ ಬೆಂಬಲವನ್ನು ನೀಡುತ್ತದೆ.
- ಇದು ಹೊಸ MCP ಅಪ್ಲಿಕೇಶನ್‌ಗಳಿಗಾಗಿ ಶಿಫಾರಸು ಮಾಡಲ್ಪಟ್ಟ ಸಾರಿಗೆ ವಿಧಾನವಾಗಿದೆ.

### ವರ್ಗಾವಣೆ ಪದಗಳು

ನಿಮ್ಮ MCP ಅಪ್ಲಿಕೇಶನ್‌ಗಳಲ್ಲಿ SSE ಯಿಂದ ಸ್ಟ್ರೀಮಬಲ್ HTTP ಗೆ ಹೇಗೆ ವರ್ಗಾಯಿಸಬಹುದು:

- `mcp.run()`ನಲ್ಲಿ `transport="streamable-http"` ಬಳಸಲು ಸರ್ವರ್ ಕೋಡ್ ನವೀಕರಿಸಿ.
- SSE ಗ್ರಾಹಕರ ಬದಲಿಗೆ `streamablehttp_client` ಬಳಸಿ ಗ್ರಾಹಕ ಕೋಡ್ ನವೀಕರಿಸಿ.
- ಸೂಚನೆಗಳನ್ನು ಪ್ರಕ್ರಿಯೆ ಮಾಡಲು ಗ್ರಾಹಕದಲ್ಲಿ ಸಂದೇಶ ನಿರ್ವಹಣೆಯೂ ಮಾಡುವಂತೆ ಜಾರಿಗೆ ತರುವಿರಿ.
- ಇತರೆ ಉಪಕರಣಗಳು ಮತ್ತು ವರ್ಕ್‌ಫ್ಲೋಗಳೊಂದಿಗೆ ಹೊಂದಾಣಿಕೆಗಾಗಿ ಪರೀಕ್ಷಿಸಿ.

### ಹೊಂದಾಣಿಕೆ ಪಾಲನೆ

ವರ್ಗಾವಣೆಯ ಸಮಯದಲ್ಲಿ ಉಳಿದ SSE ಗ್ರಾಹಕರೊಂದಿಗೆ ಹೊಂದಾಣಿಕೆಯನ್ನು ಉಳಿಸುವುದು ಶಿಫಾರಸು ಮಾಡಲ್ಪಡುತ್ತದೆ. ಕೆಲವು ತಂತ್ರಗಳು ಇಲ್ಲಿವೆ:

- ಬೇರೆ ಬೇರೆ ಎಂಡ್‌ಪಾಯಿಂಟ್‌ಗಳಲ್ಲಿ ಇಬ್ಬರೂ SSE ಮತ್ತು ಸ್ಟ್ರೀಮಬಲ್ HTTP ಅನ್ನು ನಡೆಸಿ ಬೆಂಬಲಿಸಬಹುದು.
- ಗ್ರಾಹಕರನ್ನು تدريجيವಾಗಿ ಹೊಸ ಸಾರಿಗೆಗೆ ವರ್ಗಾಯಿಸಿ.

### ಸವಾಲುಗಳು

ವರ್ಗಾವಣೆಯ ಸಂದರ್ಭದಲ್ಲಿ ಈ ಸವಾಲುಗಳನ್ನು ಎದುರಿಸಬೇಕು:

- ಎಲ್ಲಾ ಗ್ರಾಹಕರೂ ನವೀಕರಿಸಲ್ಪಟ್ಟಿದೆಯೆ ಎಂದು ಖಾತ್ರಿ ಮಾಡಿಕೊಳ್ಳುವುದು
- ನೋಟಿಫಿಕೇಶನ್ ವಿತರಣೆಯಲ್ಲಿ ಇರುವ ವ್ಯತ್ಯಾಸಗಳನ್ನು ನಿರ್ವಹಿಸುವುದು

## ಭದ್ರತಾ ವಿಚಾರಣೆಗಳು

HTTP ಆಧಾರಿತ ಸಾರಿಗೆಗಳಾದ ಸ್ಟ್ರೀಮಬಲ್ HTTP ನಂತಹವುವನ್ನು MCP ನಲ್ಲಿಅಳವಡಿಸುವಾಗ ಭದ್ರತೆ ಅತಿ ಪ್ರಾಥಮಿಕ ವಿಷಯವಾಗಿರಬೇಕು.

MCP ಸರ್ವರ್‌ಗಳನ್ನು HTTP ಆಧಾರಿತ ಸಾರಿಗೆಗಳೊಂದಿಗೆ ಜಾರಿಗೆ ತರಲು ಭದ್ರತೆ ಬಹುಮುಖ್ಯವಾಗಿದ್ದು, ಹಲವಾರು ದಾಳಿ ಮಾರ್ಗಗಳು ಮತ್ತು ರಕ್ಷಣಾ ಕ್ರಮಗಳಿಗೆ ಜಾಗರೂಕತೆ ಅಗತ್ಯವಿದೆ.

### ಅವಲೋಕನ

HTTP ಮುಖಾಂತರ MCP ಸರ್ವರ್‌ಗಳನ್ನು ಹೊರತಾಗಿಸುವಾಗ ಭದ್ರತೆ ಮಹತ್ವವುಳ್ಳುದು. ಸ್ಟ್ರೀಮಬಲ್ HTTP ಹೊಸ ದಾಳಿ ಭಾಗಗಳ ಪರಿಚಯ ಮತ್ತು ಜಾಗರೂಕ ಕಾನ್ಫಿಗರೆಶನ್ ಅನ್ನು ಬೇಕಾಗಿಸುತ್ತದೆ.

ಇಲ್ಲಿ ಕೆಲವು ಪ್ರಮುಖ ಭದ್ರತಾ ಪರಿಗಣನೆಗಳಿವೆ:

- **ಮೂಲ ಶೀರ್ಷಿಕೆ ಪರಿಶೀಲನೆ**: DNS ಮರುಬಂಧನ ದಾಳಿಗಳನ್ನು ತಡೆಯಲು ಯಾವಾಗಲೂ `Origin` ಹೆಡರ್ ಪರಿಶೀಲಿಸಬೇಕು.
- **ಲೋಕಲ್ಹೋಸ್ಟ್ ಬಂಧನ**: ಸ್ಥಳೀಯ ಅಭಿವೃದ್ಧಿಗಾಗಿ ಸರ್ವರ್‌ಗಳನ್ನು `localhost` ಗೆ ಬಂಧಿಸಿ ಸಾರ್ವಜನಿಕ ಇಂಟರ್ನೆಟ್‌ಗೆ ಬಹಿರಂಗವಾಗದಂತೆ ಮಾಡಿರಿ.
- **ಪ್ರಾಮಾಣೀಕರಣ**: ಉತ್ಪಾದನಾ ನಿಯೋಜನಗಳಿಗಾಗಿ ಪ್ರಾಮಾಣೀಕರಣ (API ಕೀಗಳು, OAuth ಇತ್ಯಾದಿ) ಜಾರಿಗೆ ತರುವಿರಿ.
- **CORS**: ಪ್ರವೇಶವನ್ನು ನಿಯಂತ್ರಿಸಲು ಕ್ರಾಸ್-ಒರಿಜಿನ್ ಸಂಪನ್ಮೂಲ ಹಂಚಿಕೆ (CORS) ನಿಯಮಾವಳಿಗಳನ್ನು ಕಾನ್ಫಿಗರ್ ಮಾಡಿ.
- **HTTPS**: ಸಂಚಾರವನ್ನು ಎನ್‌ಕ್ರಿಪ್ಟ್ ಮಾಡಲು ಉತ್ಪಾದನೆಯಲ್ಲಿ HTTPS ಬಳಸಿ.

### ಉತ್ತಮ ಅಭ್ಯಾಸಗಳು

ನಿಮ್ಮ MCP ಸ್ಟ್ರೀಮಿಂಗ್ ಸರ್ವರಿನಲ್ಲಿ ಭದ್ರತೆ ಜಾರಿಗೆ ತರುವಾಗ ಇವು ಕೆಲವು ಉತ್ತಮ ಅಭ್ಯಾಸಗಳು:

- ಪರಿಶೀಲನೆ ಇಲ್ಲದೆ ಬಂದಿರುವ ವಿನಂತಿಗಳನ್ನು ಎಂದಿಗೂ ನಂಬಬೇಡಿ.
- ಎಲ್ಲಾ ಪ್ರವೇಶ ಹಾಗೂ ದೋಷಗಳನ್ನು ಲಾಗ್ ಮಾಡಿ ಮತ್ತು ನಿಗದಿಗೊಳಿಸಿ.
- ಭದ್ರತಾ ದುರ್ಬಲತೆಗಳನ್ನು ಸರಿಪಡಿಸಲು ನಿಯಮಿತವಾಗಿ ಅವಲಂಬನೆಗಳನ್ನು ನವೀಕರಿಸಿ.

### ಸವಾಲುಗಳು

MCP ಸ್ಟ್ರೀಮಿಂಗ್ ಸರ್ವರ್‌ಗಳಲ್ಲಿ ಭದ್ರತೆ ಜಾರಿಗೆ ತರುವಾಗ ಕೆಲವು ಸವಾಲುಗಳನ್ನು ಎದುರಿಸಬೇಕಾಗುತ್ತದೆ:

- ಸುಲಭ ಅಭಿವೃದ್ಧಿಯೊಂದಿಗೆ ಭದ್ರತೆಯನ್ನು ಸಮತೋಲನಗೊಳಿಸುವುದು
- ವಿವಿಧ ಗ್ರಾಹಕ ಪರಿಸರಗಳೊಂದಿಗೆ ಹೊಂದಿಕೆ ನೋಡಿಕೊಳ್ಳುವುದು

### ಸಹಾಯ: ನಿಮ್ಮ ಸ್ವಂತ ಸ್ಟ್ರೀಮಿಂಗ್ MCP ಅಪ್ಲಿಕೇಶನ್ ನಿರ್ಮಿಸಿ

**ಪರಿಸ್ಥಿತಿ:**
ಐಟಂಗಳ (ಉದಾ., ಫೈಲ್‌ಗಳು ಅಥವಾ ಡಾಕ್ಯುಮೆಂಟ್‌ಗಳು) ಪಟ್ಟಿಯನ್ನು ಪ್ರಕ್ರಿಯೆ ಮಾಡಿ ಪ್ರತಿ ಐಟಂಗಾಗಿ ನೋಟಿಫಿಕೇಶನ್ ಕಳುಹಿಸುವ MCP ಸರ್ವರ್ ಮತ್ತು ಗ್ರಾಹಕರನ್ನು ನಿರ್ಮಿಸಿ. ನೋಟಿಫಿಕೇಶನ್ ಬರುವಂತೆ ಗ್ರಾಹಕ ಪ್ರತಿ ನೋಟಿಫಿಕೇಶನನ್ನು ಪ್ರದರ್ಶಿಸಬೇಕು.

**ಹಂತಗಳು:**

1. ಐಟಂ ಪಟ್ಟಿಯನ್ನು ಪ್ರಕ್ರಿಯೆಯಾಗಿ ನೋಟಿಫಿಕೇಶನ್ ಕಳುಹಿಸುವ ಸರ್ವರ್ ಉಪಕರಣವನ್ನು ಜಾರಿಗೆ ತರುವುದು.
2. ನೋಟಿಫಿಕೇಶನ್‌ಗಳನ್ನು ರಿಯಲ್ ಟೈಮ್‌ನಲ್ಲಿ ಪ್ರದರ್ಶಿಸಲು ಸಂದೇಶ ನಿರ್ವಹಣೆಯೊಂದಿಗೆ ಗ್ರಾಹಕವನ್ನು ಜಾರಿಗೆ ತರುವುದು.
3. ಸರ್ವರ್ ಮತ್ತು ಗ್ರಾಹಕ ಎರಡನ್ನೂ ನಡೆಸಿ ನಿಮ್ಮ ಜಾರಿಗೆ ಪರಿಶೀಲಿಸಿ ಮತ್ತು ನೋಟಿಫಿಕೇಶನ್‌ಗಳನ್ನು ಗಮನಿಸಿರಿ.
