# মডেল কনটেক্সট প্রোটোকল (MCP) সহ HTTPS স্ট্রিমিং

এই অধ্যায়ে HTTPS ব্যবহার করে মডেল কনটেক্সট প্রোটোকল (MCP) সহ নিরাপদ, স্কেলযোগ্য এবং রিয়েল-টাইম স্ট্রিমিং বাস্তবায়নের একটি ব্যাপক গাইড প্রদান করা হয়েছে। এটি স্ট্রিমিংয়ের প্রেরণা, উপলব্ধ ট্রান্সপোর্ট মেকানিজম, MCP-তে স্ট্রিমযোগ্য HTTP কীভাবে বাস্তবায়ন করবেন, সুরক্ষা সেরা প্র্যাকটিস, SSE থেকে মাইগ্রেশন এবং নিজস্ব স্ট্রিমিং MCP অ্যাপ্লিকেশন তৈরি করার জন্য ব্যবহারিক দিকনির্দেশনা কভার করে।

> **আগামীকে লক্ষ্য করে:** এই পাঠে **MCP স্পেসিফিকেশন 2025-11-25** অনুসারে স্ট্রিমযোগ্য HTTP বিবৃত হয়েছে, যেখানে `initialize` এর সময় একটি সেশন প্রতিষ্ঠিত হয় এবং `Mcp-Session-Id` হেডারের মাধ্যমে পিন করা হয়। `2026-07-28` রিলিজ ক্যান্ডিডেট হ্যান্ডশেক এবং সেশন আইডি সম্পূর্ণরূপে অপসারণ করে, প্রতিটি অনুরোধকে স্ব-নির্ভর এবং যে কোনো সার্ভার ইনস্ট্যান্সে রুটযোগ্য করে তোলে যেখানেই স্টিকি সেশন নেই। বিস্তারিত জানতে দেখুন [What's Changing in MCP: The 2026-07-28 Release Candidate](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md)।

## MCP-তে ট্রান্সপোর্ট মেকানিজম এবং স্ট্রিমিং

এই ধাপে MCP-তে উপলব্ধ বিভিন্ন ট্রান্সপোর্ট মেকানিজম এবং ক্লায়েন্ট ও সার্ভারের মধ্যে রিয়েল-টাইম যোগাযোগের জন্য স্ট্রিমিং সক্ষম করার তাদের ভূমিকা আলোচনা করা হয়েছে।

### ট্রান্সপোর্ট মেকানিজম কী?

একটি ট্রান্সপোর্ট মেকানিজম নির্ধারণ করে ডেটা কীভাবে ক্লায়েন্ট এবং সার্ভারের মধ্যে বিনিময় হয়। MCP বিভিন্ন পরিবেশ ও প্রয়োজন অনুযায়ী একাধিক ট্রান্সপোর্ট টাইপ সমর্থন করে:

- **stdio**: স্ট্যান্ডার্ড ইনপুট/আউটপুট, স্থানীয় এবং CLI-ভিত্তিক টুলগুলোর জন্য উপযুক্ত। সাধারণ কিন্তু ওয়েব বা ক্লাউডের জন্য উপযুক্ত নয়।
- **SSE (সার্ভার-সেন্ট ইভেন্টস)**: সার্ভারকে HTTP এর মাধ্যমে ক্লায়েন্টদের রিয়েল-টাইম আপডেট পাঠানোর অনুমতি দেয়। ওয়েব UI এর জন্য ভাল, কিন্তু স্কেলযোগ্যতা এবং নমনীয়তায় সীমাবদ্ধ। MCP স্পেসিফিকেশন 2025-06-18 থেকে, স্বতন্ত্র SSE (সার্ভার-সেন্ট ইভেন্টস) ট্রান্সপোর্টটি বাতিল হয়েছে এবং "স্ট্রিমযোগ্য HTTP" ট্রান্সপোর্ট দ্বারা প্রতিস্থাপিত হয়েছে।
- **স্ট্রিমযোগ্য HTTP**: আধুনিক HTTP-ভিত্তি স্ট্রিমিং ট্রান্সপোর্ট, যা বিজ্ঞপ্তি এবং উন্নত স্কেলযোগ্যতা সমর্থন করে। অধিকাংশ প্রোডাকশন এবং ক্লাউড পরিস্থিতির জন্য সুপারিশকৃত।

### তুলনামূলক টেবিল

নিম্নের তুলনামূলক টেবিলটি দেখে এই ট্রান্সপোর্ট মেকানিজমগুলির মধ্যে পার্থক্য বুঝুন:

| ট্রান্সপোর্ট       | রিয়েল-টাইম আপডেট | স্ট্রিমিং | স্কেলযোগ্যতা | ব্যবহারের ক্ষেত্র          |
|--------------------|--------------------|-----------|-------------|----------------------------|
| stdio              | না                 | না        | কম          | স্থানীয় CLI টুল            |
| SSE                | হ্যাঁ               | হ্যাঁ     | মাঝারি      | ওয়েব, রিয়েল-টাইম আপডেট   |
| স্ট্রিমযোগ্য HTTP  | হ্যাঁ               | হ্যাঁ     | উচ্চ        | ক্লাউড, মাল্টি-ক্লায়েন্ট   |

> **টিপ:** সঠিক ট্রান্সপোর্ট নির্বাচন পারফরম্যান্স, স্কেলযোগ্যতা এবং ব্যবহারকারীর অভিজ্ঞতায় প্রভাব ফেলে। আধুনিক, স্কেলযোগ্য এবং ক্লাউড-রেডি অ্যাপ্লিকেশনের জন্য **স্ট্রিমযোগ্য HTTP** সুপারিশকৃত।

মনে করুন পূর্বের অধ্যায়গুলিতে stdio এবং SSE ট্রান্সপোর্ট দেখানো হয়েছে এবং এই অধ্যায়ে স্ট্রিমযোগ্য HTTP ট্রান্সপোর্টটি আলোচনা করা হয়েছে।

## স্ট্রিমিং: ধারণা এবং প্রেরণা

স্ট্রিমিংয়ের মৌলিক ধারণা ও প্রেরণা বোঝা কার্যকর রিয়েল-টাইম যোগাযোগ ব্যবস্থা তৈরি করার জন্য অপরিহার্য।

**স্ট্রিমিং** হল একটি নেটওয়ার্ক প্রোগ্রামিং কৌশল যা ডেটা ছোট, পরিচালনাযোগ্য অংশ বা ইভেন্টের অনুক্রমে প্রেরণ ও গ্রহন করতে দেয়, পূর্ণ প্রতিক্রিয়া প্রস্তুতির জন্য অপেক্ষা না করে। এটি বিশেষত উপকারী:

- বড় ফাইল বা ডেটাসেটের জন্য।
- রিয়েল-টাইম আপডেটের জন্য (যেমন চ্যাট, প্রগ্রেস বার)।
- দীর্ঘমেয়াদি গণনায় যেখানে ব্যবহারকারীকে অবহিত রাখার দরকার।

স্ট্রিমিং সম্পর্কে উচ্চ পর্যায়ে যা জানা জরুরি:

- ডেটা ক্রমানুসারে সরবরাহিত হয়, একযোগে নয়।
- ক্লায়েন্ট ডেটা আগমনের সাথে সাথে প্রসেস করতে পারে।
- অনুভূত বিলম্ব কমায় এবং ব্যবহারকারীর অভিজ্ঞতা উন্নত করে।

### কেন স্ট্রিমিং ব্যবহার করবেন?

স্ট্রিমিং ব্যবহারের কারণসমূহ নিম্নরূপ:

- ব্যবহারকারীরা শেষ পর্যন্ত নয়, সঙ্গে সঙ্গেই প্রতিক্রিয়া পায়
- রিয়েল-টাইম অ্যাপ্লিকেশন ও প্রতিক্রিয়াশীল UI সক্ষম করে
- নেটওয়ার্ক এবং কম্পিউট রিসোর্সের আরও দক্ষ ব্যবহার করে

### সহজ উদাহরণ: HTTP স্ট্রিমিং সার্ভার ও ক্লায়েন্ট

এখানে কিভাবে স্ট্রিমিং বাস্তবায়ন করা যায় তার একটি সহজ উদাহরণ:

#### পাইথন

**সার্ভার (পাইথন, FastAPI ও StreamingResponse ব্যবহার করে):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**ক্লায়েন্ট (পাইথন, requests ব্যবহার করে):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

এই উদাহরণটি সার্ভার কতগুলো মেসেজ ক্লায়েন্টকে পাঠাচ্ছে যখন সেগুলো তৈরি হয়, সম্পূর্ণ মেসেজ কেনা পর্যন্ত অপেক্ষা করছে না বোঝায়।

**এটি কীভাবে কাজ করে:**

- সার্ভার প্রতিটি মেসেজ প্রস্তুত হওয়ার সাথে সাথে প্রেরণ করে।
- ক্লায়েন্ট আগমনের সাথে সাথে প্রতিটি অংশ গ্রহণ করে এবং মুদ্রণ করে।

**প্রয়োজনীয়তা:**

- সার্ভারকে স্ট্রিমিং রেসপন্স ব্যবহার করতে হবে (যেমন FastAPI তে `StreamingResponse`)।
- ক্লায়েন্টকে রেসপন্স স্ট্রিম হিসেবে প্রসেস করতে হবে (`stream=True` requests এ)।
- Content-Type সাধারণত `text/event-stream` বা `application/octet-stream`।

#### জাভা

**সার্ভার (জাভা, Spring Boot ও Server-Sent Events ব্যবহার করে):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**ক্লায়েন্ট (জাভা, Spring WebFlux WebClient ব্যবহার করে):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**জাভা বাস্তবায়নের নোটস:**

- Spring Boot এর রিয়েক্টিভ স্ট্যাক `Flux` স্ট্রিমিং এর জন্য ব্যবহার করে
- `ServerSentEvent` ইভেন্ট টাইপসহ গঠনমূলক ইভেন্ট স্ট্রিমিং প্রদান করে
- `WebClient` এর `bodyToFlux()` রিয়েক্টিভ স্ট্রিমিং গ্রাস সক্ষম করে
- `delayElements()` ইভেন্টের মাঝে প্রসেসিং সময় সিমুলেট করে
- ইভেন্টগুলোর টাইপ থাকতে পারে (`info`, `result`), ক্লায়েন্ট হ্যান্ডলিং উন্নত করতে

### তুলনা: ক্লাসিক স্ট্রিমিং বনাম MCP স্ট্রিমিং

"ক্লাসিক্যাল" স্ট্রিমিং এবং MCP-তে স্ট্রিমিং কীভাবে কাজ করে তার মধ্যে পার্থক্য নিচের মত উপস্থাপন করা যায়:

| বৈশিষ্ট্য                | ক্লাসিক HTTP স্ট্রিমিং       | MCP স্ট্রিমিং (বিজ্ঞপ্তি)       |
|-------------------------|------------------------------|---------------------------------|
| প্রধান রেসপন্স            | অংশভাগ                      | একক, শেষে                       |
| অগ্রগতি আপডেট          | ডেটা অংশ হিসেবে প্রেরণ      | বিজ্ঞপ্তি হিসেবে প্রেরণ         |
| ক্লায়েন্টের প্রয়োজনীয়তা | স্ট্রিম প্রক্রিয়া করতে হবে  | মেসেজ হ্যান্ডলারবাস্তবায়ন করতে হবে |
| ব্যবহারের ক্ষেত্র         | বড় ফাইল, AI টোকেন স্ট্রীম   | অগ্রগতি, লগ, রিয়েল-টাইম ফিডব্যাক |

### প্রধান পার্থক্যসমূহ

পাশাপাশি, কিছু প্রধান পার্থক্য হলো:

- **যোগাযোগ প্যাটার্ন:**
  - ক্লাসিক HTTP স্ট্রিমিং: ডেটা ভাগে ভাগে পাঠাতে সাধারণ chunked ট্রান্সফার এংকোডিং ব্যবহার করে
  - MCP স্ট্রিমিং: JSON-RPC প্রোটোকলের গঠনমূলক বিজ্ঞপ্তি সিস্টেম ব্যবহার করে

- **মেসেজ ফরম্যাট:**
  - ক্লাসিক HTTP: নতুন লাইন সহ সাধারণ টেক্সট অংশ
  - MCP: মেটাডেটাসহ গঠনমূলক LoggingMessageNotification অবজেক্ট

- **ক্লায়েন্ট বাস্তবায়ন:**
  - ক্লাসিক HTTP: স্ট্রিমিং রেসপন্স প্রক্রিয়াজাত করে এমন সাধারণ ক্লায়েন্ট
  - MCP: মেসেজ হ্যান্ডলারসহ ক্লায়েন্ট যা বিভিন্ন ধরনের মেসেজ প্রসেস করে

- **অগ্রগতি আপডেট:**
  - ক্লাসিক HTTP: অগ্রগতি মূল রেসপন্স স্ট্রিমের অংশ
  - MCP: অগ্রগতি আলাদা বিজ্ঞপ্তি মেসেজের মাধ্যমে প্রেরণ হয় যখন প্রধান রেসপন্স শেষে আসে

### সুপারিশসমূহ

ক্লাসিক্যাল স্ট্রিমিং (উপরে `/stream` ব্যবহার করে দেখানো হয়েছে) এবং MCP-র মাধ্যমে স্ট্রিমিং-এর মধ্যে নির্বাচনের ক্ষেত্রে আমরা কিছু জিনিস সুপারিশ করি।

- **সরল স্ট্রিমিংয়ের জন্য:** ক্লাসিক HTTP স্ট্রিমিং বাস্তবায়ন সহজ এবং মৌলিক স্ট্রিমিংয়ের জন্য যথেষ্ট।

- **জটিল, ইন্টারেক্টিভ অ্যাপ্লিকেশনের জন্য:** MCP স্ট্রিমিং অধিক জমাটবদ্ধ পদ্ধতি দেয়, সমৃদ্ধ মেটাডেটা এবং বিজ্ঞপ্তি ও চূড়ান্ত ফলাফলের মধ্যে ভিন্নতা সহ।

- **AI অ্যাপ্লিকেশনের জন্য:** MCP এর বিজ্ঞপ্তি সিস্টেম দীর্ঘমেয়াদি AI টাস্কের জন্য খুবই কার্যকর যেখানে ব্যবহারকারীকে অগ্রগতি জানাতে হয়।

## MCP-তে স্ট্রিমিং

তাহলে, আপনি এখনো মাসে কিছু সুপারিশ এবং তুলনা দেখেছেন ক্লাসিক্যাল স্ট্রিমিং ও MCP স্ট্রিমিংয়ের মধ্যে পার্থক্য নিয়ে। চলুন বিস্তারিতভাবে জেনে নেই আপনি MCP-তে স্ট্রিমিং কিভাবে ব্যবহার করতে পারেন।

MCP ফ্রেমওয়ার্কের মধ্যে স্ট্রিমিং কীভাবে কাজ করে তা বোঝা প্রতিক্রিয়াশীল অ্যাপ্লিকেশন তৈরির জন্য অপরিহার্য, যা দীর্ঘমেয়াদি কাজের সময় ব্যবহারকারীদের রিয়েল-টাইম ফিডব্যাক প্রদান করে।

MCP-তে স্ট্রিমিং মানে মূল রেসপন্স অংশে পাঠানো নয়, বরং যখন একটি টুল অনুরোধ প্রক্রিয়া করছে তখন ক্লায়েন্টকে **বিজ্ঞপ্তি** প্রেরণ করা। এই বিজ্ঞপ্তিগুলোতে অগ্রগতি আপডেট, লগ বা অন্যান্য ইভেন্ট থাকতে পারে।

### এটি কীভাবে কাজ করে

মূল ফলাফল এখনও একক রেসপন্স হিসেবে পাঠানো হয়। তবে প্রক্রিয়াকালীন আলাদা মেসেজ হিসেবে বিজ্ঞপ্তি পাঠানো হয়, যা ক্লায়েন্টকে রিয়েল-টাইমে আপডেট করে। ক্লায়েন্টকে এই বিজ্ঞপ্তি গ্রহণ ও প্রদর্শন করার ক্ষমতা থাকতে হবে।

## বিজ্ঞপ্তি কী?

আমরা "বিজ্ঞপ্তি" বললাম, MCP প্রসঙ্গে এর মানে কী?

বিজ্ঞপ্তি হলো সার্ভার থেকে ক্লায়েন্টকে পাঠানো একটি মেসেজ যা দীর্ঘমেয়াদি অপারেশনের অগ্রগতি, অবস্থা, বা অন্যান্য ইভেন্ট সম্পর্কে তথ্য দেয়। বিজ্ঞপ্তি স্বচ্ছতা এবং ব্যবহারকারীর অভিজ্ঞতা উন্নত করে।

উদাহরণস্বরূপ, ক্লায়েন্ট সার্ভারের সঙ্গে প্রাথমিক হ্যান্ডশেকের পর বিজ্ঞপ্তি পাঠাবে।

একটি বিজ্ঞপ্তি JSON মেসেজ হিসেবে দেখতে এরূপ:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

বিজ্ঞপ্তি MCP-এর একটি টপিকের অন্তর্গত যা ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) নামে পরিচিত।

> **বাতিলের নোটিশ:** `2026-07-28` MCP স্পেসিফিকেশনের রিলিজ ক্যান্ডিডেট Logging প্রিমিটিভকে বাতিল ঘোষণা করেছে এবং stdio ট্রান্সপোর্টের জন্য `stderr` এবং গঠনমূলক অবজারভেবিলিটির জন্য OpenTelemetry ব্যবহারের পক্ষে সুপারিশ করেছে। Logging `2025-11-25` তে এবং আনুষ্ঠানিক বাতিলের অন্তত এক বছর পর্যন্ত কাজ করবে। বিস্তারিত জানতে দেখুন [What's Changing in MCP: The 2026-07-28 Release Candidate](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md)।

লগিং চালু করতে, সার্ভারকে একটি ফিচার/ক্ষমতা হিসেবে সক্ষম করতে হবে যেমন:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> ব্যবহৃত SDK অনুযায়ী, লগিং ডিফল্টরূপে সক্ষম থাকতে পারে, অথবা Explicit আপনার সার্ভার কনফিগারেশনে সক্ষম করতে হতে পারে।

বিভিন্ন ধরনের বিজ্ঞপ্তি রয়েছে:

| স্তর       | বর্ণনা                        | উদাহরণ ব্যবহারের ক্ষেত্র          |
|-----------|------------------------------|---------------------------------|
| debug     | বিস্তারিত ডিবাগিং তথ্য        | ফাংশনের এন্ট্রি/এক্সিট পয়েন্ট   |
| info      | সাধারণ তথ্যবহুল মেসেজ       | অপারেশনের অগ্রগতি আপডেট         |
| notice    | স্বাভাবিক কিন্তু উল্লেখযোগ্য ইভেন্ট | কনফিগারেশন পরিবর্তন          |
| warning   | সতর্কতামূলক অবস্থা          | অব্যবহৃত ফিচারের ব্যবহার         |
| error     | ত্রুটির অবস্থা                | অপারেশন ব্যর্থতা                 |
| critical  | গুরত্বপূর্ণ অবস্থা            | সিস্টেম উপাদানের ব্যর্থতা         |
| alert     | অবিলম্বে ব্যবস্থা নেওয়া দরকার | ডেটা করাপশন সনাক্ত করা          |
| emergency | সিস্টেম ব্যবহার অযোগ্য         | সম্পূর্ণ সিস্টেম ব্যর্থতা          |

## MCP-তে বিজ্ঞপ্তি বাস্তবায়ন

MCP-তে বিজ্ঞপ্তি বাস্তবায়নের জন্য আপনাকে সার্ভার এবং ক্লায়েন্ট উভয় দিকেই রিয়েল-টাইম আপডেট পরিচালনা করার ব্যবস্থা করতে হবে। এটি আপনার অ্যাপ্লিকেশানকে দীর্ঘমেয়াদি অপারেশন চলাকালে ব্যবহারকারীকে অবিলম্বে প্রতিক্রিয়া প্রদানের সুযোগ দেয়।

### সার্ভার-পার্শ্ব: বিজ্ঞপ্তি প্রেরণ

শুরু করা যাক সার্ভার পার্শ্ব থেকে। MCP-তে আপনি এমন টুল ডিফাইন করেন যা অনুরোধ প্রক্রিয়াকালীন বিজ্ঞপ্তি পাঠাতে পারে। সার্ভার সাধারণত `ctx` অবজেক্ট ব্যবহার করে ক্লায়েন্টকে মেসেজ পাঠায়।

#### পাইথন

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

পূর্বের উদাহরণে, `process_files` টুল প্রতিটি ফাইল প্রক্রিয়া করার সময় তিনটি বিজ্ঞপ্তি ক্লায়েন্টকে পাঠায়। `ctx.info()` পদ্ধতি তথ্যবহুল মেসেজ পাঠাতে ব্যবহৃত হয়েছে।

এছাড়াও, বিজ্ঞপ্তি সক্ষম করতে নিশ্চিত হোন আপনার সার্ভার স্ট্রিমিং ট্রান্সপোর্ট (যেমন `streamable-http`) ব্যবহার করছে এবং ক্লায়েন্ট একটি মেসেজ হ্যান্ডলার বাস্তবায়ন করেছে যা বিজ্ঞপ্তি প্রক্রিয়া করে। নিচে দেখানো হলো কিভাবে সার্ভার `streamable-http` ট্রান্সপোর্ট ব্যবহার করার জন্য সেটআপ করবেন:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

এই .NET উদাহরণে, `ProcessFiles` টুল `Tool` অ্যাট্রিবিউট দিয়ে চিহ্নিত এবং প্রতিটি ফাইল প্রক্রিয়া করতে গিয়ে তিনটি বিজ্ঞপ্তি ক্লায়েন্টকে পাঠায়। `ctx.Info()` পদ্ধতি তথ্যবহুল মেসেজ পাঠাতে ব্যবহৃত হয়েছে।

.NET MCP সার্ভারে বিজ্ঞপ্তি সক্ষম করতে নিশ্চিত করুন আপনি স্ট্রিমিং ট্রান্সপোর্ট ব্যবহার করছেন:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### ক্লায়েন্ট-পার্শ্ব: বিজ্ঞপ্তি গ্রহণ

ক্লায়েন্টকে অবশ্যই একটি মেসেজ হ্যান্ডলার বাস্তবায়ন করতে হবে যা বিজ্ঞপ্তি আগমনের সাথে তাদের প্রসেস করে প্রদর্শন করতে পারে।

#### পাইথন

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

আগের কোডে, `message_handler` ফাংশন চেক করে আগত মেসেজটি বিজ্ঞপ্তি কিনা। যদি হয়, এটি বিজ্ঞপ্তি প্রিন্ট করে; না হলে এটি সাধারণ সার্ভার মেসেজ হিসেবে প্রক্রিয়া করে। পাশাপাশি লক্ষ্য করুন কীভাবে `ClientSession` `message_handler` দিয়ে ইনিশিয়ালাইজ করা হয়েছে যাতে আগত বিজ্ঞপ্তি পরিচালনা করা যায়।

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

এই .NET উদাহরণে, `MessageHandler` ফাংশন আগত মেসেজ বিজ্ঞপ্তি কিনা পরীক্ষা করে। যদি হয়, বিজ্ঞপ্তি প্রিন্ট করে; নয়তো সাধারণ সার্ভার মেসেজ হিসেবে প্রক্রিয়া করে। `ClientSession` মেসেজ হ্যান্ডলার `ClientSessionOptions` দ্বারা ইনিশিয়ালাইজ করা হয়েছে।

বিজ্ঞপ্তি সক্ষম করতে নিশ্চিত করুন আপনার সার্ভার স্ট্রিমিং ট্রান্সপোর্ট (যেমন `streamable-http`) ব্যবহার করছে এবং ক্লায়েন্ট একটি মেসেজ হ্যান্ডলার বাস্তবায়ন করেছে যা বিজ্ঞপ্তি প্রক্রিয়া করে।

## অগ্রগতি বিজ্ঞপ্তি ও পরিস্থিতি

এই অংশে MCP তে অগ্রগতি বিজ্ঞপ্তির ধারণা, এর গুরুত্ব এবং স্ট্রিমযোগ্য HTTP ব্যবহার করে কিভাবে বাস্তবায়ন করবেন তা ব্যাখ্যা করা হয়েছে। পাশাপাশি একটি ব্যবহারিক অ্যাসাইনমেন্টও রয়েছে আপনার বোঝাপড়া দৃঢ় করার জন্য।

অগ্রগতি বিজ্ঞপ্তি হলো সার্ভার থেকে ক্লায়েন্টকে দীর্ঘমেয়াদি অপারেশনের চলাকালে পাঠানো রিয়েল-টাইম মেসেজ। সম্পূর্ণ প্রক্রিয়া শেষ হওয়ার জন্য অপেক্ষা না করে সার্ভার ক্লায়েন্টকে বর্তমান অবস্থা সম্পর্কে আপডেট রাখে। এটি স্বচ্ছতা, ব্যবহারকারীর অভিজ্ঞতা এবং ডিবাগিং সহজ করে তোলে।

**উদাহরণ:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### কেন অগ্রগতি বিজ্ঞপ্তি ব্যবহার করবেন?

অগ্রগতি বিজ্ঞপ্তি কিছু কারণের জন্য অপরিহার্য:

- **উন্নত ব্যবহারকারীর অভিজ্ঞতা:** ব্যবহারকারীরা কাজের অগ্রগতির সময় আপডেট দেখে, শুধু শেষেই নয়।
- **রিয়েল-টাইম প্রতিক্রিয়া:** ক্লায়েন্টরা প্রগ্রেস বার বা লগ দেখাতে পারে, অ্যাপ রেসপন্সিভ লাগে।
- **সহজ ডিবাগিং ও মনিটরিং:** ডেভেলপার এবং ব্যবহারকারীরা দেখতে পারে কোন প্রক্রিয়া ধীর বা আটকে আছে।

### কিভাবে অগ্রগতি বিজ্ঞপ্তি বাস্তবায়ন করবেন

এখানে MCP তে কিভাবে অগ্রগতি বিজ্ঞপ্তি বাস্তবায়ন করবেন:

- **সার্ভারে:** প্রতিটি আইটেম প্রক্রিয়াকালে `ctx.info()` বা `ctx.log()` দিয়ে বিজ্ঞপ্তি পাঠান। এটি প্রধান ফলাফল প্রস্তুত হওয়ার আগে ক্লায়েন্টকে মেসেজ পাঠায়।
- **ক্লায়েন্টে:** এমন একটি মেসেজ হ্যান্ডলার বাস্তবায়ন করুন যা আগত বিজ্ঞপ্তি শুনবে ও প্রদর্শন করবে। এই হ্যান্ডলার বিজ্ঞপ্তি এবং চূড়ান্ত ফলাফলের মধ্যে পার্থক্য করে।

**সার্ভারের উদাহরণ:**


#### পাইথন

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**ক্লায়েন্ট উদাহরণ:**

#### পাইথন

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## নিরাপত্তা বিবেচনা

HTTP-ভিত্তিক ট্রান্সপোর্ট দিয়ে MCP সার্ভার তৈরি করার সময়, নিরাপত্তা একটি অত্যন্ত গুরুত্বপূর্ণ বিষয় যা একাধিক আক্রমণ পথ এবং সুরক্ষা প্রক্রিয়াগুলোর প্রতি সতর্ক দৃষ্টি প্রয়োজন।

### ওভারভিউ

MCP সার্ভার HTTP এর মাধ্যমে প্রকাশ করার সময় নিরাপত্তা অপরিহার্য। স্ট্রিমেবল HTTP নতুন আক্রমণ পৃষ্ঠ তৈরি করে এবং সাবধানে কনফিগারেশন প্রয়োজন।

### মূল বিষয়সমূহ

- **অরিজিন হেডার ভ্যালিডেশন**: DNS পুন:বন্ধন আক্রমণ প্রতিরোধের জন্য সর্বদা `Origin` হেডার যাচাই করুন।
- **লোকালহোস্ট বাইন্ডিং**: স্থানীয় ডেভেলপমেন্টের জন্য, সার্ভারগুলোকে `localhost` এ বাইন্ড করুন যাতে পাবলিক ইন্টারনেটে প্রকাশ পায় না।
- **প্রমাণীকরণ**: প্রোডাকশন ডিপ্লয়মেন্টের জন্য প্রমাণীকরণ (যেমন, API কী, OAuth) বাস্তবায়ন করুন।
- **CORS**: অ্যাক্সেস সীমাবদ্ধ করার জন্য Cross-Origin Resource Sharing (CORS) নীতিমালা কনফিগার করুন।
- **HTTPS**: ট্রাফিক এনক্রিপ্ট করার জন্য প্রোডাকশনে HTTPS ব্যবহার করুন।

### সর্বোত্তম অনুশীলন

- যাচাই ছাড়া আগত অনুরোধের প্রতি কখনো বিশ্বাস করবেন না।
- সমস্ত অ্যাক্সেস এবং ত্রুটি লগ ও মনিটর করুন।
- নিরাপত্তা দুর্বলতাগুলো ঠিক করার জন্য ডিপেন্ডেন্সিগুলো নিয়মিত আপডেট করুন।

### চ্যালেঞ্জসমূহ

- উন্নয়নের সহজতার সঙ্গে নিরাপত্তার ভারসাম্য রক্ষা করা
- বিভিন্ন ক্লায়েন্ট পরিবেশগুলোর সাথে সঙ্গতি নিশ্চিত করা

## SSE থেকে স্ট্রিমেবল HTTP-তে উন্নীতকরণ

বর্তমানে Server-Sent Events (SSE) ব্যবহার করছে এমন অ্যাপ্লিকেশনগুলোর জন্য, MCP বাস্তবায়নের উন্নতি এবং দীর্ঘমেয়াদী স্থিতিশীলতার জন্য স্ট্রিমেবল HTTP-তে স্থানান্তর করা বুদ্ধিমানের কাজ।

### কেন উন্নীত করা?

SSE থেকে স্ট্রিমেবল HTTP তে উন্নীত হওয়ার দুটি প্রলোভনীয় কারণ রয়েছে:

- স্ট্রিমেবল HTTP SSE থেকে উন্নত স্কেলেবিলিটি, সামঞ্জস্য এবং সমৃদ্ধ বিজ্ঞপ্তি সহায়তা প্রদান করে।
- এটি নতুন MCP অ্যাপ্লিকেশনগুলোর জন্য সুপারিশকৃত ট্রান্সপোর্ট।

### স্থানান্তর ধাপসমূহ

আপনার MCP অ্যাপ্লিকেশনগুলোতে SSE থেকে স্ট্রিমেবল HTTP-তে স্থানান্তর করার প্রক্রিয়া নিম্নরূপ:

- **সার্ভার কোড আপডেট করুন** `mcp.run()` এ `transport="streamable-http"` ব্যবহার করতে।
- **ক্লায়েন্ট কোড আপডেট করুন** SSE ক্লায়েন্টের পরিবর্তে `streamablehttp_client` ব্যবহার করতে।
- **ক্লায়েন্টে একটি মেসেজ হ্যান্ডলার বাস্তবায়ন করুন** যা বিজ্ঞপ্তিগুলো প্রক্রিয়াজাত করবে।
- **বিদ্যমান টুল ও ওয়ার্কফ্লো সঙ্গে সামঞ্জস্যের পরীক্ষা করুন**।

### সামঞ্জস্য রক্ষা

স্থানান্তর প্রক্রিয়ায় বিদ্যমান SSE ক্লায়েন্টদের সাথে সামঞ্জস্য রক্ষা করাই পরামর্শযোগ্য। কিছু কৌশল:

- আপনি SSE এবং স্ট্রিমেবল HTTP দুইটোকেই বিভিন্ন এন্ডপয়েন্টে চালিয়ে সমর্থন করতে পারেন।
- ধাপে ধাপে ক্লায়েন্টদের নতুন ট্রান্সপোর্টে স্থানান্তর করুন।

### চ্যালেঞ্জসমূহ

স্থানান্তরের সময় নিম্নলিখিত চ্যালেঞ্জগুলোর মোকাবিলা করুন:

- নিশ্চিত করুন সকল ক্লায়েন্ট আপডেট হয়েছে
- বিজ্ঞপ্তি ডেলিভারির পার্থক্যগুলো পরিচালনা করা

## নিরাপত্তা বিবেচনা

MCP সার্ভার বাস্তবায়নের সময় নিরাপত্তা সর্বোচ্চ অগ্রাধিকার হওয়া উচিত, বিশেষ করে Streamable HTTP-এর মতো HTTP-ভিত্তিক ট্রান্সপোর্ট ব্যবহার করার ক্ষেত্রে।

HTTP-ভিত্তিক ট্রান্সপোর্ট দিয়ে MCP সার্ভার তৈরি করার সময়, নিরাপত্তা একটি অত্যন্ত গুরুত্বপূর্ণ বিষয় যা একাধিক আক্রমণ পথ এবং সুরক্ষা প্রক্রিয়াগুলোর প্রতি সতর্ক দৃষ্টি প্রয়োজন।

### ওভারভিউ

MCP সার্ভার HTTP এর মাধ্যমে প্রকাশ করার সময় নিরাপত্তা অপরিহার্য। স্ট্রিমেবল HTTP নতুন আক্রমণ পৃষ্ঠ তৈরি করে এবং সাবধানে কনফিগারেশন প্রয়োজন।

এখানে কিছু মূল নিরাপত্তা বিবেচনা দেওয়া হলো:

- **অরিজিন হেডার ভ্যালিডেশন**: DNS পুন:বন্ধন আক্রমণ প্রতিরোধের জন্য সর্বদা `Origin` হেডার যাচাই করুন।
- **লোকালহোস্ট বাইন্ডিং**: স্থানীয় ডেভেলপমেন্টের জন্য, সার্ভারগুলোকে `localhost` এ বাইন্ড করুন যাতে পাবলিক ইন্টারনেটে প্রকাশ পায় না।
- **প্রমাণীকরণ**: প্রোডাকশন ডিপ্লয়মেন্টের জন্য প্রমাণীকরণ (যেমন, API কী, OAuth) বাস্তবায়ন করুন।
- **CORS**: অ্যাক্সেস সীমাবদ্ধ করার জন্য Cross-Origin Resource Sharing (CORS) নীতিমালা কনফিগার করুন।
- **HTTPS**: ট্রাফিক এনক্রিপ্ট করার জন্য প্রোডাকশনে HTTPS ব্যবহার করুন।

### সর্বোত্তম অনুশীলন

এছাড়াও, আপনার MCP স্ট্রিমিং সার্ভারে নিরাপত্তা বাস্তবায়নের সময় অনুসরণ করার জন্য কিছু সর্বোত্তম অনুশীলন এখানে দেওয়া হলো:

- যাচাই ছাড়া আগত অনুরোধের প্রতি কখনো বিশ্বাস করবেন না।
- সমস্ত অ্যাক্সেস এবং ত্রুটি লগ ও মনিটর করুন।
- নিরাপত্তা দুর্বলতাগুলো ঠিক করার জন্য ডিপেন্ডেন্সিগুলো নিয়মিত আপডেট করুন।

### চ্যালেঞ্জসমূহ

MCP স্ট্রিমিং সার্ভারে নিরাপত্তা বাস্তবায়নের সময় আপনি কিছু চ্যালেঞ্জের মুখোমুখি হবেন:

- উন্নয়নের সহজতার সঙ্গে নিরাপত্তার ভারসাম্য রক্ষা করা
- বিভিন্ন ক্লায়েন্ট পরিবেশগুলোর সাথে সঙ্গতি নিশ্চিত করা

### অ্যাসাইনমেন্ট: নিজস্ব স্ট্রিমিং MCP অ্যাপ তৈরি করুন

**পরিস্থিতি:**
একটি MCP সার্ভার এবং ক্লায়েন্ট তৈরি করুন যেখানে সার্ভার একটি আইটেম তালিকা (যেমন, ফাইল বা নথি) প্রক্রিয়া করে এবং প্রক্রিয়াজাত প্রতিটি আইটেমের জন্য একটি বিজ্ঞপ্তি পাঠায়। ক্লায়েন্ট আসা প্রতিটি বিজ্ঞপ্তি প্রদর্শন করবে।

**ধাপসমূহ:**

1. একটি সার্ভার টুল বাস্তবায়ন করুন যা তালিকা প্রক্রিয়া করে এবং প্রতিটি আইটেমের জন্য বিজ্ঞপ্তি প্রেরণ করে।
2. একটি ক্লায়েন্ট বাস্তবায়ন করুন যার একটি মেসেজ হ্যান্ডলার থাকবে রিয়েল টাইমে বিজ্ঞপ্তি প্রদর্শনের জন্য।
3. আপনার বাস্তবায়ন পরীক্ষা করুন সার্ভার এবং ক্লায়েন্ট উভয় চালিয়ে এবং বিজ্ঞপ্তিগুলো পর্যবেক্ষণ করুন।

## আরও পড়াশোনা ও পরবর্তী ধাপ

MCP স্ট্রিমিং নিয়ে আপনার যাত্রা চালিয়ে যেতে এবং আপনার জ্ঞান বিস্তার করতে, এই অংশে অতিরিক্ত সম্পদ এবং উন্নত অ্যাপ্লিকেশন তৈরির জন্য পরবর্তী ধাপের সুপারিশ রয়েছে।

- রিয়েল-টাইম অ্যানালিটিক্স, চ্যাট বা সহযোগিতামূলক সম্পাদনার জন্য স্ট্রিমিং ব্যবহার করে আরও উন্নত MCP টুলস তৈরি করার চেষ্টা করুন।
- MCP স্ট্রিমিংকে ফ্রন্টএন্ড ফ্রেমওয়ার্কস (React, Vue, ইত্যাদি) এর সাথে ইন্টিগ্রেট করে লাইভ UI আপডেটস অন্বেষণ করুন।
