# मॉडेल कॉन्टेक्स्ट प्रोटोकॉल (MCP) सह HTTPS स्ट्रीमिंग

हा अध्याय HTTPS वापरून मॉडेल कॉन्टेक्स्ट प्रोटोकॉल (MCP) सह सुरक्षित, स्केलेबल आणि रिअल-टाइम स्ट्रीमिंग कसे अंमलात आणायचे यासाठी सविस्तर मार्गदर्शन प्रदान करतो. यात स्ट्रीमिंगसाठी प्रेरणा, उपलब्ध ट्रान्सपोर्ट मेकॅनिझम, MCP मध्ये स्ट्रीम करू शकणाऱ्या HTTP ची अंमलबजावणी कशी करावी, सुरक्षा सर्वोत्तम पद्धती, SSE वरून स्थलांतर, आणि स्वतःचे स्ट्रीमिंग MCP अनुप्रयोग तयार करण्यासाठी व्यवहार्य मार्गदर्शन यांचा समावेश आहे.

> **आगामी दृष्टिकोन:** हा धडा **MCP स्पेसिफिकेशन 2025-11-25** अंतर्गत Streamable HTTP वर्णन करतो, जिथे `initialize` दरम्यान सत्र स्थापन केले जाते आणि `Mcp-Session-Id` हेडर वापरून पिन केले जाते. `2026-07-28` रिलीज कॅंडिडेट संपूर्ण हँडशेक आणि सत्र आयडी काढून टाकतो, ज्यामुळे प्रत्येक विनंती स्वयं-संलग्न आणि कोणत्याही सर्व्हर उदाहरणाला राऊटेबल होते, स्टिकी सत्रांशिवाय. तपशीलांसाठी [MCP मध्ये काय बदलते: 2026-07-28 रिलीज कॅंडिडेट](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) पहा.

## MCP मधील ट्रान्सपोर्ट मेकॅनिझम आणि स्ट्रीमिंग

हा विभाग MCP मध्ये उपलब्ध विविध ट्रान्सपोर्ट मेकॅनिझम आणि क्लायंट व सर्व्हर यांच्यात रिअल-टाइम संवादासाठी स्ट्रीमिंग क्षमतांसाठी त्यांची भूमिका तपासतो.

### ट्रान्सपोर्ट मेकॅनिझम म्हणजे काय?

ट्रान्सपोर्ट मेकॅनिझम म्हणजे क्लायंट आणि सर्व्हर यांच्यात डेटा कसा देवाणघेवाण होतो हे ठरवणारी पद्धत. MCP विविध वातावरण आणि गरजांसाठी अनेक प्रकारच्या ट्रान्सपोर्टना समर्थन देते:

- **stdio**: स्टँडर्ड इनपुट/आउटपुट, स्थानिक आणि CLI-आधारित साधनांसाठी योग्य. सोपे परंतु वेब किंवा क्लाउडसाठी योग्य नाही.
- **SSE (सर्व्हर-सेंट इव्हेंट्स)**: सर्व्हर्सना HTTP वरून रिअल-टाइम अपडेट्स क्लायंटकडे पाठवण्याची परवानगी देते. वेब UI साठी चांगले, पण स्केलेबिलिटी आणि लवचीकतेत मर्यादित. MCP स्पेसिफिकेशन 2025-06-18 पासून, स्वतंत्र SSE (सर्व्हर-सेंट इव्हेंट्स) ट्रान्सपोर्ट अप्रचलित झाला आणि "Streamable HTTP" ट्रान्सपोर्टने त्याची जागा घेतली आहे.
- **Streamable HTTP**: आधुनिक HTTP-आधारित स्ट्रीमिंग ट्रान्सपोर्ट, सूचना पाठवण्याला आणि उत्तम स्केलेबिलिटीला समर्थन देतो. बहुतेक उत्पादन आणि क्लाउड परिस्थितीसाठी शिफारसीय.

### तुलना सारणी

या ट्रान्सपोर्ट मेकॅनिझममधील फरक समजून घेण्यासाठी खालील तुलना सारणी पहा:

| ट्रान्सपोर्ट       | रिअल-टाइम अपडेट्स | स्ट्रीमिंग | स्केलेबिलिटी | वापर प्रकरण               |
|-------------------|--------------------|-----------|---------------|--------------------------|
| stdio             | नाही               | नाही      | कमी           | स्थानिक CLI साधने        |
| SSE               | होय                | होय       | मध्यम         | वेब, रिअल-टाइम अपडेट्स  |
| Streamable HTTP   | होय                | होय       | जास्त         | क्लाउड, बहु-क्लायंट       |

> **टिप:** योग्य ट्रान्सपोर्ट निवडणे कार्यक्षमता, स्केलेबिलिटी, आणि वापरकर्ता अनुभवावर परिणाम करते. आधुनिक, स्केलेबल, आणि क्लाउड-तयार अनुप्रयोगांसाठी **Streamable HTTP** शिफारस केली जाते.

तुम्हाला पूर्वीच्या अध्यायांमध्ये stdio आणि SSE ट्रान्सपोर्ट दाखवले गेले होते आणि या अध्यायात आढळणारा ट्रान्सपोर्ट म्हणजे streamable HTTP आहे.

## स्ट्रीमिंग: संकल्पना आणि प्रेरणा

प्रभावी रिअल-टाइम संवाद प्रणाली अंमलात आणण्यासाठी स्ट्रीमिंगच्या मूलभूत संकल्पना आणि प्रेरणा समजणे आवश्यक आहे.

**स्ट्रीमिंग** ही नेटवर्क प्रोग्रामिंगमधील एक तंत्र आहे ज्याद्वारे पूर्ण प्रतिसाद तयार होण्याची प्रतिक्षा न करता डेटा लहान, व्यवस्थापनीय भागांमध्ये किंवा इव्हेंटच्या अनुक्रमणिकेप्रमाणे पाठवला आणि प्राप्त केला जातो. हे विशेषतः उपयुक्त आहे:

- मोठ्या फायली किंवा डेटासेटसाठी.
- रिअल-टाइम अपडेट्ससाठी (उदा. चॅट, प्रगती पट्टे).
- दीर्घकालीन गणनांसाठी ज्यात वापरकर्त्याला माहिती देऊ इच्छितात.

उच्च स्तरावर स्ट्रीमिंगबद्दल तुम्हाला काय माहित असावे:

- डेटा क्रमिकपणे वितरित केला जातो, एकावेळी सर्व डेटा नाही.
- क्लायंट डेटाचे आगमन होताच ते प्रक्रिया करू शकतो.
- अनुमानित उशीर कमी होतो आणि वापरकर्ता अनुभव सुधारतो.

### स्ट्रीमिंग वापरण्याचे कारण काय?

स्ट्रीमिंग वापरण्याची कारणे खालीलप्रमाणे आहेत:

- वापरकर्त्यांना लगेच फीडबॅक मिळतो, केवळ शेवटी नाही.
- रिअल-टाइम अनुप्रयोग आणि प्रतिसादक्षम UI सारख्या सुविधा सक्षम करतो.
- नेटवर्क आणि संगणन संसाधनांचा अधिक कार्यक्षम वापर.

### सोपा उदाहरण: HTTP स्ट्रीमिंग सर्व्हर आणि क्लायंट

स्ट्रीमिंग कसे अंमलात आणता येते याचे एक सोपे उदाहरण येथे आहे:

#### पायथन

**सर्व्हर (पायथन, FastAPI आणि StreamingResponse वापरून):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**क्लायंट (पायथन, requests वापरून):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

हे उदाहरण सर्व्हरकडून उपलब्ध होताच संदेशांची मालिका क्लायंटकडे पाठवली जाते, केवळ सर्व संदेश तयार होईपर्यंत प्रतीक्षा करत नाही.

**हे कसे कार्यान्वित होते:**

- सर्व्हर प्रत्येक संदेश तयार होईपर्यंत थांबत नाही आणि तो यील्ड करतो.
- क्लायंट प्रत्येक भाग घेऊन तो प्रदर्शित करतो.

**अटी:**

- सर्व्हरला स्ट्रीमिंग प्रतिसाद वापरावा लागतो (उदा. FastAPI मध्ये `StreamingResponse`).
- क्लायंटला प्रतिसाद स्ट्रिम म्हणून प्रक्रिया करावयाची आहे (`stream=True` requests मध्ये).
- Content-Type सहसा `text/event-stream` किंवा `application/octet-stream` असते.

#### जावा

**सर्व्हर (जावा, Spring Boot आणि Server-Sent Events वापरून):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**क्लायंट (जावा, Spring WebFlux WebClient वापरून):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**जावा अंमलबजावणी नोट्स:**

- Spring Boot चा प्रतिसाद उत्साही स्टॅक `Flux` सह स्ट्रीमिंग साठी वापरतो
- `ServerSentEvent` इव्हेंट प्रकारांसह संरचित इव्हेंट स्ट्रीमिंग प्रदान करतो
- `WebClient` चा `bodyToFlux()` प्रतिसाद उत्साही स्ट्रीमिंग वापर सक्षम करतो
- `delayElements()` इव्हेंट्समधील प्रक्रिया वेळ अनुकरण करतो
- इव्हेंट्सना उत्तम क्लायंट हाताळणीसाठी प्रकार (`info`, `result`) असू शकतात

### तुलना: पारंपारिक स्ट्रीमिंग वि. MCP स्ट्रीमिंग

स्ट्रीमिंग पारंपारिक पद्धतीने कशी काम करते आणि MCP मध्ये कशी करते यातील फरक अशा प्रकारे दाखवता येऊ शकतो:

| वैशिष्ट्य             | पारंपारिक HTTP स्ट्रीमिंग      | MCP स्ट्रीमिंग (सूचना)            |
|-----------------------|-------------------------------|-----------------------------------|
| मुख्य प्रतिसाद        | चंक्ड                         | एकदाच, शेवटी                   |
| प्रगती अपडेट्स       | डेटा चंक म्हणून पाठवले जाते   | सूचनांद्वारे पाठवले जाते          |
| क्लायंट आवश्यकता     | स्ट्रीम प्रक्रिया करणे आवश्यक | संदेश हँडलर अंमलात आणणे आवश्यक  |
| वापर प्रकरण          | मोठ्या फायली, AI टोकन स्ट्रीम  | प्रगती, लॉग, रिअल-टाइम फीडबॅक   |

### वैशिष्ट्याने पाहिलेले मुख्य फरक

शिवाय, इथे काही मुख्य फरक आहेत:

- **संवाद नमुना:**
  - पारंपारिक HTTP स्ट्रीमिंग: सोपी चंक्ड ट्रान्सफर एनकोडिंग वापरून डेटा चंका पाठवतो
  - MCP स्ट्रीमिंग: JSON-RPC प्रोटोकॉलसह संरचित सूचना प्रणाली वापरतो

- **संदेश स्वरूप:**
  - पारंपारिक HTTP: नवीन ओळींसह प्लेन टेक्स्ट चंका
  - MCP: मेटाडेटा असलेले संरचित LoggingMessageNotification ऑब्जेक्ट्स

- **क्लायंट अंमलबजावणी:**
  - पारंपारिक HTTP: साधा क्लायंट जो स्ट्रीमिंग प्रतिसाद प्रक्रिया करतो
  - MCP: अधिक प्रगत क्लायंट ज्याला विविध प्रकारचे संदेश हाताळण्यासाठी संदेश हँडलर आहे

- **प्रगती अपडेट्स:**
  - पारंपारिक HTTP: प्रगती मुख्य प्रतिसाद स्ट्रिमचा भाग असते
  - MCP: प्रगती वेगळ्या सूचना संदेशांद्वारे पाठवली जाते, आणि मुख्य प्रतिसाद शेवटी येतो

### शिफारसी

क्लासिक स्ट्रीमिंग (स्त्रोतांची झलक आपल्याला `/stream` वापरून दाखविली आहे) आणि MCP द्वारे स्ट्रीमिंग निवडताना काही शिफारसी आहेत.

- **सोप्या स्ट्रीमिंग गरजांसाठी:** पारंपारिक HTTP स्ट्रीमिंग सोपे अंमलात आणता येते आणि मूलभूत गरजांसाठी पुरेसे आहे.

- **संकुल, संवादात्मक अनुप्रयोगांसाठी:** MCP स्ट्रीमिंग अधिक संरचित दृष्टीकोन देते ज्यात समृद्ध मेटाडेटा आणि सूचना व अंतिम निकाल यांच्यात विभाजन आहे.

- **AI अनुप्रयोगांसाठी:** MCP ची सूचना प्रणाली दीर्घकालीन AI कार्यांसाठी उपयुक्त आहे जिथे प्रगतीबद्दल वापरकर्त्यांना सतत माहिती द्यायची असते.

## MCP मध्ये स्ट्रीमिंग

ठीक आहे, तुम्हाला क्लासिक स्ट्रीमिंग आणि MCP मधील स्ट्रीमिंग यातील तुलना आणि शिफारसी काही प्रमाणात पाहिल्या आहेत. आता तपशीलात जाऊया की तुम्ही MCP मध्ये स्ट्रीमिंग कसे वापरू शकता.

MCP फ्रेमवर्कमध्ये स्ट्रीमिंग कशी कार्यान्वित होते हे समजून घेणे महत्त्वाचे आहे जेणेकरून तुम्ही दीर्घकालीन ऑपरेशन दरम्यान वापरकर्त्यांना रिअल-टाइम फीडबॅक पुरवणारे प्रतिसादक्षम अनुप्रयोग तयार करू शकाल.

MCP मध्ये स्ट्रीमिंग म्हणजे मुख्य प्रतिसाद चंका मध्ये पाठवणे नव्हे, तर एखाद्या उपकरणाने विनंती प्रक्रिया करताना क्लायंटला सूचना पाठवणे आहे. या सूचनांमध्ये प्रगती अपडेट्स, लॉग्स किंवा इतर इव्हेंट्स असू शकतात.

### हे कसे कार्य करते

मुख्य निकाल तरीही एकाच प्रतिसादात पाठविला जातो. परंतु प्रक्रिया दरम्यान स्वतंत्र संदेशांमध्ये सूचना पाठवता येतात आणि त्याद्वारे क्लायंटला रिअल-टाइममध्ये अद्ययावत केले जाते. क्लायंटने या सूचनांना हाताळणे आणि प्रदर्शित करणे आवश्यक आहे.

## सूचना म्हणजे काय?

आपण "सूचना" म्हटले, तर ती MCP संदर्भात काय आहे?

सूचना म्हणजे एक संदेश जो सर्व्हरकडून क्लायंटला पाठविला जातो ज्यात प्रगती, स्थिती किंवा दीर्घकालीन ऑपरेशन दरम्यान इतर इव्हेंट्स विषयी माहिती दिली जाते. सूचना पारदर्शकता आणि वापरकर्ता अनुभव सुधारतात.

उदाहरणार्थ, सर्व्हरशी प्रारंभिक हँडशेक पूर्ण झाल्यावर क्लायंटने एक सूचना पाठवणे अपेक्षित आहे.

सूचना JSON संदेश म्हणून असे दिसते:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

सूचना MCP मधील ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) विषयाशी संबंधित आहेत.

> **अप्रचलन सूचना:** `2026-07-28` MCP स्पेसिफिकेशन रिलीज कॅंडिडेट लॉगिंग प्रिमिटिव्हला stdio ट्रान्सपोर्टसाठी `stderr` आणि संरचित निरीक्षणासाठी OpenTelemetry च्या फायद्यासाठी अप्रचलित घोषित करतो. लॉगिंग `2025-11-25` मध्ये आणि औपचारिक अप्रचलनानंतर कमीतकमी एका वर्षासाठी कार्यरत राहील. तपशीलांसाठी [MCP मध्ये काय बदलते: 2026-07-28 रिलीज कॅंडिडेट](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) पहा.

लॉगिंग कार्यान्वित करण्यासाठी, सर्व्हरला फिचर/क्षमता म्हणून खालीलप्रमाणे सक्षम करावे लागते:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> वापरल्या जाणाऱ्या SDK नुसार, लॉगिंग पूर्वनिर्धारितरित्या सक्षम असू शकते किंवा तुम्हाला सर्व्हर कॉन्फिगरेशनमध्ये ते स्पष्टपणे सक्षम करावे लागेल.

विविध प्रकारच्या सूचनाः

| पातळी         | वर्णन                          | उदाहरण वापर प्रकरण            |

## MCP मध्ये सूचना अंमलात आणणे

MCP मध्ये सूचना अंमलात आणण्यासाठी, तुम्हाला रिअल-टाइम अपडेट्स हाताळण्यासाठी सर्व्हर आणि क्लायंट दोन्ही बाजूंना सेट अप करावे लागेल. यामुळे तुमचा अनुप्रयोग दीर्घकालीन ऑपरेशनदरम्यान वापरकर्त्यांना तत्काळ फीडबॅक देऊ शकतो.

### सर्व्हर-बाजू: सूचना पाठवणे

आता सर्व्हर बाजूने सुरुवात करूया. MCP मध्ये, तुम्ही अशी उपकरणे परिभाषित करता जी विनंती प्रक्रिया करताना सूचना पाठवू शकतात. सर्व्हर संदर्भ ऑब्जेक्ट (साधारणतः `ctx`) वापरून क्लायंटला संदेश पाठवतो.

#### पायथन

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

मागील उदाहरणात, `process_files` टूल प्रत्येक फाइल प्रक्रिया करताना क्लायंटकडे तीन सूचना पाठवते. `ctx.info()` मेथड माहितीपूर्ण संदेश पाठवण्यासाठी वापरले जाते.

शिवाय, सूचना सक्षम करण्यासाठी तुमचा सर्व्हर स्ट्रीमिंग ट्रान्सपोर्ट (उदा. `streamable-http`) वापरत असेल आणि तुमचा क्लायंट सूचना प्रक्रिया करण्यासाठी संदेश हँडलर लागू करेल याची खात्री करा. तुमचा सर्व्हर `streamable-http` ट्रान्सपोर्ट वापरण्यासाठी असा सेट करू शकता:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

या .NET उदाहरणात, `ProcessFiles` टूल `Tool` अॅट्रिब्यूटने सजवलेले आहे आणि प्रत्येक फाइल प्रक्रिया करताना क्लायंटकडे तीन सूचना पाठवते. `ctx.Info()` मेथड माहितीपूर्ण संदेश पाठवण्यासाठी वापरले जाते.

तुमच्या .NET MCP सर्व्हरमध्ये सूचना सक्षम करण्यासाठी, स्ट्रीमिंग ट्रान्सपोर्ट वापरत आहात याची खात्री करा:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### क्लायंट-बाजू: सूचना प्राप्त करणे

क्लायंटने संदेश हँडलर लागू करणे आवश्यक आहे जे येणाऱ्या सूचना प्रक्रिया करेल आणि प्रदर्शित करेल.

#### पायथन

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

मागील कोडमध्ये, `message_handler` फंक्शन तपासते की येणारा संदेश सूचना आहे का. जर होय, तर सूचना छापते; अन्यथा तो सामान्य सर्व्हर संदेश म्हणून प्रक्रिया करते. तसेच पाहा की `ClientSession` मध्ये `message_handler` कसे इनिशियलाइज केला आहे जेणेकरून येणाऱ्या सूचनांना हाताळता येईल.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

या .NET उदाहरणात, `MessageHandler` फंक्शन तपासते की येणारा संदेश सूचना आहे का. जर होय, तर सूचना छापते; अन्यथा तो सामान्य सर्व्हर संदेश म्हणून प्रक्रिया करते. `ClientSessionOptions` वापरून `ClientSession` संदेश हँडलरसह इनिशियलाइज केला आहे.

सूचना सक्षम करण्यासाठी, तुमचा सर्व्हर स्ट्रीमिंग ट्रान्सपोर्ट (उदा. `streamable-http`) वापरत आहे आणि तुमचा क्लायंट सूचना प्रक्रिया करण्यासाठी संदेश हँडलर लागू करतो याची खात्री करा.

## प्रगती सूचना आणि परिस्थिती

हा विभाग MCP मधील प्रगती सूचना संकल्पना, त्यांचे महत्त्व, आणि स्ट्रीमिंग HTTP वापरून त्यांची अंमलबजावणी कशी करावी हे समजावून सांगतो. तुमच्या समजुतीसाठी एक व्यावहारिक असाइनमेंट देखील आहे.

प्रगती सूचना म्हणजे रिअल-टाइम संदेश जे दीर्घकालीन ऑपरेशन दरम्यान सर्व्हरकडून क्लायंटकडे पाठवले जातात. पूर्ण प्रक्रिया पूर्ण होण्याची प्रतिक्षा न करता, सर्व्हर क्लायंटला चालू स्थितीबाबत अद्यतनित ठेवतो. यामुळे पारदर्शकता, वापरकर्ता अनुभव सुधारतो आणि डिबगिंग सोपे होते.

**उदाहरण:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### प्रगती सूचना का वापराव्यात?

प्रगती सूचना कारणांसाठी अत्यावश्यक आहेत:

- **चांगला वापरकर्ता अनुभव:** वापरकर्त्यांना कामाची प्रगती वाटते, फक्त शेवटी नाही.
- **रिअल-टाइम फीडबॅक:** क्लायंट प्रगती पट्टी किंवा लॉग दर्शवू शकतो, ज्यामुळे अ‍ॅप प्रतिसादक्षम वाटतो.
- **सुलभ डिबगिंग आणि निरीक्षण:** विकसक आणि वापरकर्ते कुठे प्रक्रिया मंद आहे किंवा अडकली आहे ते पाहू शकतात.

### प्रगती सूचना कशा अंमलात आणायच्या

येथे MCP मध्ये प्रगती सूचना अंमलात आणण्याचा मार्ग आहे:

- **सर्व्हरवर:** प्रत्येक घटक प्रक्रिया होत असताना `ctx.info()` किंवा `ctx.log()` वापरून सूचना पाठवा. यामुळे मुख्य निकाल तयार होण्याआधी क्लायंटला संदेश पाठवला जातो.
- **क्लायंटवर:** एक संदेश हँडलर लागू करा जो येणाऱ्या सूचनांना ऐकतो आणि दर्शवतो. हा हँडलर सूचना आणि अंतिम निकाल यात तफावत करतो.

**सर्व्हर उदाहरण:**


#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**क्लायंट उदाहरण:**

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## सुरक्षा विचार

जेव्हा HTTP-आधारित ट्रान्सपोर्टसह MCP सर्व्हरची अंमलबजावणी केली जाते, तेव्हा सुरक्षा हा एक महत्त्वाचा विचार बनतो ज्यासाठी अनेक हल्ल्याच्या मार्ग आणि संरक्षण यंत्रणांकडे काळजीपूर्वक लक्ष द्यावे लागते.

### आढावा

HTTP वर MCP सर्व्हर उघडल्यावर सुरक्षा अत्यंत महत्त्वाची असते. स्ट्रीमेबल HTTP नवीन हल्ल्याचे मार्ग तयार करतो आणि काळजीपूर्वक संरचना आवश्यक असते.

### मुख्य मुद्दे

- **Origin Header वैधता**: DNS रीबाइंडिंग हल्ले टाळण्यासाठी नेहमी `Origin` हेडरची पडताळणी करा.
- **Localhost Binding**: स्थानिक विकासासाठी, सर्व्हर `localhost` ला बांधा ज्यामुळे ते सार्वजनिक इंटरनेटवर उघडणार नाहीत.
- **प्रमाणीकरण**: उत्पादनासाठी प्रमाणीकरण (उदा., API कीज, OAuth) लागू करा.
- **CORS**: प्रवेश मर्यादित करण्यासाठी Cross-Origin Resource Sharing (CORS) धोरणे संरचीत करा.
- **HTTPS**: ट्रॅफिक एन्क्रिप्ट करण्यासाठी उत्पादनात HTTPS वापरा.

### सर्वोत्तम पद्धती

- येणाऱ्या विनंत्यांवर कधीही पडताळणी न करता विश्वास ठेवू नका.
- सर्व प्रवेश आणि त्रुटी नोंदवा आणि देखरेख करा.
- सुरक्षेतील कमतरतांसाठी डिपेंडन्सी नियमितपणे अद्ययावत करा.

### आव्हाने

- विकासाच्या सोपेपणासह सुरक्षेचा समतोल राखणे
- विविध क्लायंट वातावरणांसोबत सुसंगतता सुनिश्चित करणे

## SSE वरून Streamable HTTP कडे अपग्रेड करणे

सध्या Server-Sent Events (SSE) वापरणाऱ्या अनुप्रयोगांसाठी, Streamable HTTP कडे स्थलांतर केल्यास वाढीव क्षमता आणि आपल्या MCP अंमलबजावणीसाठी दीर्घकालीन टिकावाची हमी प्राप्त होते.

### का अपग्रेड करावे?

SSE वरून Streamable HTTP कडे स्थलांतर करण्याची दोन ठोस कारणे आहेत:

- Streamable HTTP SSE पेक्षा अधिक स्केलेबिलिटी, सुसंगतता आणि श्रीमंत सूचनासाठी समर्थन देते.
- नवीन MCP अनुप्रयोगांसाठी हे शिफारस केलेले ट्रान्सपोर्ट आहे.

### स्थलांतर चरण

आपल्या MCP अनुप्रयोगांमध्ये SSE वरून Streamable HTTP कडे स्थलांतर कसे कराल:

- `mcp.run()` मध्ये `transport="streamable-http"` वापरून सर्व्हर कोड अद्यतनित करा.
- SSE क्लायंटऐवजी `streamablehttp_client` वापरण्यासाठी क्लायंट कोड अद्यतनित करा.
- क्लायंटमध्ये सूचना प्रक्रिया करण्यासाठी एक मेसेज हँडलर अंमलात आणा.
- विद्यमान साधने आणि कार्यप्रवाहांसह सुसंगततेसाठी चाचणी करा.

### सुसंगतता राखणे

स्थलांतर प्रक्रियेदरम्यान विद्यमान SSE क्लायंटशी सुसंगतता राखणे शिफारसीय आहे. काही धोरणे येथे आहेत:

- आपण वेगवेगळ्या एंडपॉईंटवर दोन्ही ट्रान्सपोर्ट चालवून SSE आणि Streamable HTTP दोन्ही समर्थित ठेऊ शकता.
- हळूहळू क्लायंट्स नवीन ट्रान्सपोर्टकडे स्थलांतरित करा.

### आव्हाने

स्थलांतरादरम्यान खालील आव्हाने हाताळणे आवश्यक आहे:

- सर्व क्लायंट्स अद्यतनित झाले आहेत याची खात्री करणे
- सूचनांच्या वितरणातील फरक हाताळणे

## सुरक्षा विचार

HTTP-आधारित ट्रान्सपोर्टसह, विशेषत: MCP मध्ये Streamable HTTP वापरताना, सुरक्षा सर्वोच्च प्राधान्य असावी.

जेव्हा HTTP-आधारित ट्रान्सपोर्टसह MCP सर्व्हर अंमलात आणले जातात, तेव्हा सुरक्षा हा एक महत्त्वाचा विचार बनतो ज्यासाठी अनेक हल्ल्याच्या मार्गाकडे आणि संरक्षण यंत्रणांकडे काळजीपूर्वक लक्ष द्यावे लागते.

### आढावा

HTTP वर MCP सर्व्हर उघडल्यावर सुरक्षा अत्यंत महत्त्वाची असते. स्ट्रीमेबल HTTP नवीन हल्ल्याचे मार्ग तयार करतो आणि काळजीपूर्वक संरचना आवश्यक असते.

येथे काही मुख्य सुरक्षा विचार आहेत:

- **Origin Header वैधता**: DNS रीबाइंडिंग हल्ले टाळण्यासाठी नेहमी `Origin` हेडरची पडताळणी करा.
- **Localhost Binding**: स्थानिक विकासासाठी, सर्व्हर `localhost` ला बांधा ज्यामुळे ते सार्वजनिक इंटरनेटवर उघडणार नाहीत.
- **प्रमाणीकरण**: उत्पादनासाठी प्रमाणीकरण (उदा., API कीज, OAuth) लागू करा.
- **CORS**: प्रवेश मर्यादित करण्यासाठी Cross-Origin Resource Sharing (CORS) धोरणे संरचीत करा.
- **HTTPS**: ट्रॅफिक एन्क्रिप्ट करण्यासाठी उत्पादनात HTTPS वापरा.

### सर्वोत्तम पद्धती

आपल्या MCP स्ट्रीमिंग सर्व्हरमध्ये सुरक्षा अंमलात आणताना खालील सर्वोत्तम पद्धतींचे पालन करा:

- येणाऱ्या विनंत्यांवर कधीही पडताळणी न करता विश्वास ठेवू नका.
- सर्व प्रवेश आणि त्रुटी नोंदवा आणि देखरेख करा.
- सुरक्षेतील कमतरतांसाठी डिपेंडन्सी नियमितपणे अद्ययावत करा.

### आव्हाने

MCP स्ट्रीमिंग सर्व्हरमध्ये सुरक्षा अंमलात आणताना काही आव्हानांना सामोरे जावे लागू शकते:

- विकासाच्या सोपेपणासह सुरक्षेचा समतोल राखणे
- विविध क्लायंट वातावरणांसोबत सुसंगतता सुनिश्चित करणे

### कार्य: तुमचा स्वतःचा स्ट्रीमिंग MCP अ‍ॅप तयार करा

**परिस्थिती:**
असा एक MCP सर्व्हर आणि क्लायंट तयार करा जिथे सर्व्हर आयटम्सची (उदा., फाइल्स किंवा दस्तऐवज) यादी प्रक्रिया करतो आणि प्रत्येक प्रक्रिया झालेल्या आयटमसाठी सूचना पाठवतो. क्लायंट प्रत्येक सूचना प्राप्त होताच त्याचे प्रदर्शन करेल.

**टप्पे:**

1. अशी सर्व्हर टूल अंमलात आणा जी यादी प्रक्रिया करते आणि प्रत्येक आयटमसाठी सूचना पाठवते.
2. मेसेज हँडलर असलेला क्लायंट अंमलात आणा जो रिअल टाइममध्ये सूचना दर्शवेल.
3. सर्व्हर आणि क्लायंट दोन्ही चालवून आपल्या अंमलबजावणीची चाचणी करा आणि सूचनांचे निरीक्षण करा.
