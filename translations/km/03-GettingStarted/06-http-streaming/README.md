# ការផ្សាយតាម HTTPS ជាមួយពិធីការបរិបទម៉ូដែល (MCP)

មេរៀននេះផ្តល់មគ្គុទេសក៍លម្អិតសម្រាប់ការអនុវត្តបច្ចេកវិទ្យាផ្សាយបន្តដែលមានសុវត្ថិភាព អាចពង្រីកបាន និងរាល់ពេលជាមួយពិធីការបរិបទម៉ូដែល (MCP) ដោយប្រើ HTTPS។ វារួមបញ្ចូលទាំងមូលហេតុផលសម្រាប់ការផ្សាយបន្ត ឧបករណ៍ដឹកជញ្ជូនដែលមាន សៀវភៅបរិស្ថាន HTTP ដែលអាចផ្សាយបាននៅ MCP បច្ចេកវិទ្យាសុវត្ថិភាពល្អបំផុត ការផ្លាស់ប្តូរពី SSE និងការណែនាំ ប្រព័ន្ធសម្រាប់បង្កើតកម្មវិធី MCP ផ្សាយបន្តរបស់អ្នកផ្ទាល់។

> **មើលទៅមុខ៖** មេរៀននេះពណ៌នាអំពី HTTP ដែលអាចផ្សាយបន្តនៅក្រោម **MCP Specification 2025-11-25** ដែលមានកំណត់សម័យសម្រង់នៅពេល `initialize` ហើយត្រូវបានភ្ជាប់ជាមួយក្បាល `Mcp-Session-Id`។ កំណែបញ្ចេញ `2026-07-28` នឹងដកនេះចេញពេញលេញមិនមានការចាប់ដៃគូ និងអត្តសញ្ញាណសម័យទាំងស្រុង ធ្វើឱ្យការស្នើរសុំគ្រប់លក្ខណៈជាតែមួយដែលអាចផ្ញើទៅមជ្ឍមណ្ឌលណាមួយដោយគ្មានសម័យភ្ជាប់មិនចេះផ្អែម។ មើល [មានអ្វីកំពុងផ្លាស់ប្ដូរនៅ MCP: កំណែបញ្ចេញ 2026-07-28](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) សម្រាប់ព័ត៌មានលម្អិត។

## ឧបករណ៍ដឹកជញ្ជូន និងការផ្សាយបន្តនៅ MCP

ផ្នែកនេះពិភាក្សាអំពីឧបករណ៍ដឹកជញ្ជូនផ្សេងៗដែលមានក្នុង MCP និងតួនាទីរបស់វានៅក្នុងការអនុញ្ញាតឱ្យមានសមត្ថភាពផ្សាយបន្តសម្រាប់ការទំនាក់ទំនងរាល់ពេលរវាងអ្នកប្រើនិងម៉ាស៊ីនបម្រើ។

### តើឧបករណ៍ដឹកជញ្ជូនគឺជាអ្វី?

ឧបករណ៍ដឹកជញ្ជូនកំណត់ពីរបៀបបម្លែងទិន្នន័យរវាងឧបករណ៍ដាក់សំណើ និងម៉ាស៊ីនបម្រើ។ MCP គាំទ្ររបៀបដឹកជញ្ជូនច្រើនដើម្បីឱ្យសមស្របនឹងបរិស្ថាន និងតម្រូវការផ្សេងៗ៖

- **stdio**: ពីរប្រភេទ input/output ស្តង់ដារ ដែលសមស្របសម្រាប់ឧបករណ៍នៅក្នុងមធ្យោបាយមូលដ្ឋាន និង CLI។ ងាយស្រួល ប៉ុន្តែមិនសមសម្រាប់វេប អេក្លោវឌ៍។
- **SSE (Server-Sent Events)**: អនុញ្ញាតឱ្យម៉ាស៊ីនបម្រើបញ្ចូនព័ត៌មានបច្ចុប្បន្នបន្តរាល់ពេលទៅឧបករណ៍ដាក់សំណើតាមរយៈ HTTP។ ល្អសម្រាប់វេប UI ប៉ុន្តែមានកំណត់លើសមត្ថភាពពង្រីក និងបត់បែន។ ផ្តើមពី MCP Specification 2025-06-18 ការដឹកជញ្ជូន SSE តែមួយបានបញ្ឈប់ និងប្តូរជាមួយ "Streamable HTTP" ដឹកជញ្ជូន។
- **Streamable HTTP**: ដឹកជញ្ជូនផ្សាយបន្តមូលដ្ឋាន HTTP សម័យថ្មី ដែលគាំទ្រជូនដំណឹង និងសមត្ថភាពពង្រីកល្អជាង។ បានណែនាំសម្រាប់ការផលិត និងបរិស្ថានអេក្លោវឌ៍។

### តារាងប្រៀបធៀប

សូមមើលតារាងប្រៀបធៀបខាងក្រោមដើម្បីយល់ពីភាពខុសគ្នារវាងឧបករណ៍ដឹកជញ្ជូនទាំងនេះ៖

| ដឹកជញ្ជូន      | បច្ចុប្បន្នភាពជារាល់ពេល | ផ្សាយបន្ត | សមត្ថភាពពង្រីក  | ករណីប្រើប្រាស់          |
|-----------------|---------------------|------------|------------------|-------------------------|
| stdio           | មិន                   | មិន         | ទាប               | ឧបករណ៍ CLI មូលដ្ឋាន   |
| SSE             | មាន                   | មាន         | មធ្យម             | វេប, បច្ចុប្បន្នភាពពេលពិត |
| Streamable HTTP | មាន                   | មាន         | ខ្ពស់              | អេក្លោវឌ៍, អតិថិជនច្រើន  |

> **ប្រាប់ផ្លាប់៖** ការជ្រើសរើសដឹកជញ្ជូនត្រឹមត្រូវប៉ះពាល់ដល់ការប្រតិបត្តិការសមត្ថភាពពង្រីក និងបទពិសោធន៍អ្នកប្រើ។ **Streamable HTTP** ត្រូវបានណែនាំសម្រាប់កម្មវិធីទំនើប មានសមត្ថភាពពង្រីក និងភ្ជាប់បរិស្ថានអេក្លោវឌ៍។

សូមចំណាំពីការដឹកជញ្ជូន stdio និង SSE ដែលបានបង្ហាញក្នុងមេរៀនមុន និងពីរបៀបដែល Streamable HTTP ជាដឹកជញ្ជូនដែលពិភាក្សា នៅមេរៀននេះ។

## ការផ្សាយបន្ត៖ គំនិត និងមូលហេតុ

ការយល់ដឹងអំពីគំនិតមូលដ្ឋាន និងមូលហេតុក្រោយផ្នែកផ្សាយបន្តគឺសំខាន់សម្រាប់អនុវត្តប្រព័ន្ធទំនាក់ទំនងរាល់ពេលយ៉ាងមានប្រសិទ្ធិភាព។

**ការផ្សាយបន្ត** គឺជាបច្ចេកទេសកម្មវិធីបណ្តាញដែលឲ្យអាចផ្ញើ និងទទួលទិន្នន័យជាឈុតតូចៗ ឬជាលំដាប់ព្រឹត្តិការណ៍មួយ ដោយមិនចាំរង់ចាំពេលឆ្លើយតបពេញលេញ។ វាដោយសារ:

- ឯកសារទូលំទូលាយ ឬដាតាច្រើន។
- បច្ចុប្បន្នភាពរាល់ពេល (ឧ. ការជជែក កម្មវិធីបង្ហាញកំណត់រយៈពេល)។
- ការគណនា​រយៈពេលយូរ​ដែលចង់រក្សាព័ត៌មានឲ្យអ្នកប្រើបានដឹង។

អ្វីដែលអ្នកត្រូវដឹងអំពីការផ្សាយបន្តនៅកម្រិតខ្ពស់គឺ៖

- ទិន្នន័យត្រូវបានបញ្ជូនម្ដងៗ កុំសារ។
- អ្នកប្រើអាចដំណើរការទិន្នន័យនៅពេលវាចូល។
- កាត់បន្ថយការពន្យារពេលដែលមានអារម្មណ៍ និងបង្កើនបទពិសោធន៍អ្នកប្រើ។

### ហេតុអ្វីបានជាចង់ប្រើការផ្សាយបន្ត?

មូលហេតុដែលត្រូវប្រើការផ្សាយបន្តមានដូចជា៖

- អ្នកប្រើទទួលបានមតិបញ្ចាំងភ្លាមៗ មិនត្រឹមតែចុងបញ្ចប់ប៉ុណ្ណោះ។
- អនុញ្ញាតកម្មវិធីរាល់ពេល និង UI ប្រតិបត្តិការឆ្លើយតប។
- ប្រើប្រាស់ធនធានបណ្ដាញ និងកុំព្យូទ័រឱ្យមានប្រសិទ្ធភាពជាងមុន។

### ឧទាហរណ៍សាមញ្ញ៖ ម៉ាស៊ីនបម្រើ និងអតិថិជន HTTP ផ្សាយបន្ត

អ្នកអាចឃើញឧទាហរណ៍សាមញ្ញមួយអំពីរបៀបដែលការផ្សាយបន្តអាចត្រូវបានអនុវត្ត៖

#### Python

**ម៉ាស៊ីនបម្រើ (Python, ប្រើ FastAPI និង StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**អតិថិជន (Python, ប្រើ requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

ឧទាហរណ៍នេះបង្ហាញពីម៉ាស៊ីនបម្រើផ្ញើសារដែលមានជាប់ជាលំដាប់ទៅកាន់អតិថិជនពេលវាត្រូវបានរៀបចំមក។

**របៀបវាដំណើរការ:**

- ម៉ាស៊ីនបម្រើបញ្ចេញសារ១ៗនៅពេលវាត្រូវបានរៀបចំរួច។
- អតិថិជនទទួល និងបោះពុម្ពសារនីមួយៗនៅពេលវាចូល។

**តម្រូវការ:**

- ម៉ាស៊ីនបម្រើត្រូវប្រើប្រតិកម្មផ្សាយបន្ត (ឧ. `StreamingResponse` ក្នុង FastAPI)។
- អតិថិជនត្រូវដំណើរការប្រាក់ប្រតិកម្មជាប្រភេទផ្សាយបន្ត (`stream=True` នៅ requests)។
- ប្រភេទមាតិកាទូទៅជាទូទៅគឺ `text/event-stream` ឬ `application/octet-stream`។

#### Java

**ម៉ាស៊ីនបម្រើ (Java, ប្រើ Spring Boot និង Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**អតិថិជន (Java, ប្រើ Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**កំណត់សម្គាល់អំពីការអនុវត្ត Java:**

- ប្រើរចនាសម្ព័ន្ធ Spring Boot នៃ reactive stack ជាមួយ `Flux` សម្រាប់ផ្សាយបន្ត
- `ServerSentEvent` ផ្តល់នូវការផ្សាយព្រឹត្តិការណ៍ដែលមានរចនាសម្ព័ន្ធជាមួយប្រភេទព្រឹត្តិការណ៍
- `WebClient` ជាមួយ `bodyToFlux()` អនុញ្ញាតឲ្យអ្នកប្រើប្រាស់ reactive streaming
- `delayElements()` ធ្វើការសមមេត្តារវាងព្រឹត្តិការណ៍
- ព្រឹត្តិការណ៍អាចមានប្រភេទ (`info`, `result`) ដើម្បីគ្រប់គ្រងអតិថិជនបានល្អ

### តម្លៃប្រៀបធៀប៖ ការផ្សាយបន្តបែបបុរាណ និង MCP

ភាពខុសគ្នារវាងការផ្សាយបន្តបែប "បុរាណ" និងរបៀបដែលវាដំណើរការនៅ MCP អាចបង្ហាញដូចខាងក្រោម៖

| លក្ខណៈ              | ការផ្សាយបន្ត HTTP បែបបុរាណ | ការផ្សាយបន្ត MCP (ជូនដំណឹង)  |
|---------------------|-----------------------------|----------------------------------|
| ចម្លើយសំខាន់         | ចែកជាប្លុកប្រភេទ chunked   | ចម្លើយតែមួយនៅចុងការ            |
| បច្ចុប្បន្នភាពជំរុញ   | ផ្ញើជា chunked data        | ផ្ញើជាជូនដំណឹង                |
| តម្រូវការអតិថិជន      | ត្រូវដំណើរការផ្សាយបន្ត    | ត្រូវអនុវត្តកម្មវិធីដំណើរការសារ    |
| ករណីប្រើប្រាស់        | ឯកសារធំៗ សាច់រូបសញ្ញា AI| ការរីកចម្រើន កំណត់ហេតុ មតិកម្មពេលពិត |

### ភាពខុសគ្នាសំខាន់ដែលឃើញ

លើសពីនេះ មានភាពខុសគ្នាសំខាន់ខ្លះៗ៖

- **លំនាំទំនាក់ទំនង៖**
  - ការផ្សាយបន្ត HTTP បែបបុរាណ៖ ប្រើការចែកចាយ chunked transfer encoding ដើម្បីផ្ញើទិន្នន័យជាប្លុក
  - MCP៖ ប្រើប្រព័ន្ធជូនដំណឹងដែលមានរចនាសម្ព័ន្ធជាមួយពិធីការគណនា JSON-RPC

- **ទ្រង់ទ្រាយសារ៖**
  - HTTP បែបបុរាណ៖ សារ​សំលេងបែបអក្សរឆ្លើយតបជាពីរដង
  - MCP៖ វត្ថុ LoggingMessageNotification ដែលមាន ប្រ.metadata ។

- **ការអនុវត្តអតិថិជន៖**
  - HTTP បែបបុរាណ៖ អតិថិជនសាមញ្ញដែលដំណើរការឆ្លើយតបផ្សាយបន្ត
  - MCP៖ អតិថិជនមានភាពស្មុគស្មាញជាង ដោយមានអ្នកដំណើរការសារដើម្បីដំណើរការប្រភេទសារផ្សេងៗ

- **បច្ចុប្បន្នភាពកំណត់ត្រា៖**
  - HTTP បែបបុរាណ៖ កំណត់ត្រាជាផ្នែកនៃចម្លើយសំខាន់
  - MCP៖ កំណត់ត្រាត្រូវបានផ្ញើតាមសារជូនដំណឹងផ្សេងៗ ខណៈដែលចម្លើយសំខាន់មកនៅចុង។

### អនុសាសន៍

មានពីររឿងដែលយើងណែនាំនៅពេលជ្រើសរើសរវាងការអនុវត្តការផ្សាយបន្តបែបបុរាណ (ជា endpoint ដែលបានបង្ហាញខាងលើប្រើ `/stream`) និងការជ្រើសរើសផង MCP streaming។

- **សម្រាប់តម្រូវការផ្សាយបន្តសាមញ្ញ៖** ការផ្សាយបន្ត HTTP បែបបុរាណមានភាពសាមញ្ញក្នុងការអនុវត្ត និងគ្រប់គ្រាន់សម្រាប់តម្រូវការលើសីត។

- **សម្រាប់កម្មវិធីសំបូរសង្គម និងអន្តរកម្ម៖** MCP streaming ផ្តល់វិធីសាស្ត្រដែលមានរចនាសម្ព័ន្ធច្បាស់ជាមួយ metadata និងការផ្ដាច់ចេញរវាងការជូនដំណឹងនិងលទ្ធផលចុងក្រោយ។

- **សម្រាប់កម្មវិធី AI៖** ប្រព័ន្ធជូនដំណឹងរបស់ MCP មានប្រយោជន៍ជាពិសេសសម្រាប់ការងារយូរអង្វែង ដែលចង់ផ្តល់ព័ត៌មានកំណត់ដល់អ្នកប្រើ។

## ការផ្សាយបន្តនៅ MCP

យ៉ាងហោចណាស់ អ្នកបានឃើញសំណើណែនាំ និងការ​ប្រៀបធៀបរវាងការផ្សាយបន្តបែបបុរាណ និងការផ្សាយបន្តនៅ MCP។ ឥឡូវមករកមើលលម្អិតពីរបៀបដែលអ្នកអាចប្រើប្រាស់ការផ្សាយបន្តនៅ MCP។

ការយល់ដឹងពីរបៀបការផ្សាយបន្តដំណើរការនៅខ្ទង់ MCP គឺសំខាន់សម្រាប់បង្កើតកម្មវិធីដែលឆ្លើយតបយ៉ាងរហ័ស នឹងផ្តល់មតិបញ្ចាំងពេលពិតដល់អ្នកប្រើនៅពេលក្រុមហ៊ុនដំណើរការយូរ។

នៅក្នុង MCP ការផ្សាយបន្តមិនមែនជាបញ្ចូនចម្លើយសំខាន់ជាច្រើនប្លុកទេ ប៉ុន្តែជាការផ្ញើ **សារជូនដំណឹង** ទៅឲ្យអតិថិជន ខណៈដែលឧបករណ៍កំពុងដំណើរការសំណើ។ សារជូនដំណឹងទាំងនេះអាចរួមបញ្ចូលការអាប់ដេតកំណត់ត្រា កំណត់ហេតុ ឬព្រឹត្តិការណ៍ផ្សេងទៀត។

### របៀបវាដំណើរការ

លទ្ធផលសំខាន់នៅតែត្រូវផ្ញើជាចម្លើយតែមួយ។ ប៉ុន្តែសារជូនដំណឹងអាចផ្ញើបានជាសារ​ដាច់ដោយឡែកខណៈដែលកំពុងដំណើរការ ហើយធ្វើឲ្យអតិថិជនបានអាប់ដេតពិតនៅពេលនោះ។ អតិថិជនត្រូវតែមានសមត្ថភាពក្នុងការគ្រប់គ្រងនិងបង្ហាញសារជូនដំណឹងទាំងនេះ។

## តើសារជូនដំណឹងគឺជាអ្វី?

យើងបាននិយាយពី "សារជូនដំណឹង" តើវាមានអត្ថន័យដូចម្តេចនៅក្នុងបរិបទ MCP?

សារជូនដំណឹងគឺជាសារ​ដែលផ្ញើពីម៉ាស៊ីនបម្រើទៅអតិថិជន ដើម្បីជូនដំណឹងអំពី​កំណត់ត្រា ស្ថានភាព ឬព្រឹត្តិការណ៍ផ្សេងៗ​នៅពេលដំណើរការយូរ។ សារជូនដំណឹងបង្កើនភាពច្បាស់លាស់ និងបទពិសោធន៍អ្នកប្រើ។

ឧទាហរណ៍ អតិថិជនគួរតែផ្ញើសារជូនដំណឹងមួយពេលការចាប់ដៃគូដំបូងជាមួយម៉ាស៊ីនបម្រើបានធ្វើរួច។

សារជូនដំណឹងមើលទៅដូចជា​សារប្រភេទ JSON៖

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

សារជូនដំណឹងស្ថិតនៅក្នុងប្រធានបទនៅ MCP ដែលហៅថា ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging)។

> **សេចក្ដីជូនដំណឹងបញ្ឈប់ប្រើ៖** កំណែបញ្ចេញ MCP 2026-07-28 ប្រាប់ថា Logging primitive ត្រូវបានដកចេញជំនួសដោយ `stderr` សម្រាប់ដឹកជញ្ជូន stdio និង OpenTelemetry សម្រាប់ការគ្រប់គ្រងប្រព័ន្ធប្រើប្រាស់ជារចនាសម្ព័ន្ធ។ Logging នៅតែដំណើរការនៅក្នុង `2025-11-25` និងយ៉ាងហោចណាស់មួយឆ្នាំបន្ទាប់ពីការដកចេញផ្លូវការណ៍។ មើល [មានអ្វីកំពុងផ្លាស់ប្ដូរនៅ MCP: កំណែបញ្ចេញ 2026-07-28](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md)។

ដើម្បីអោយ logging ដំណើរការ ម៉ាស៊ីនបម្រើត្រូវបើកវាទៅជាគុណលក្ខណៈ/មុខងារ ដូចខាងក្រោម៖

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> បន្ទាប់ពីប្រើ SDK ផ្សេងៗ Logging អាចត្រូវបានបើកដាច់ដោយឡែក ឬអ្នកត្រូវបើកវាដោយដៃនៅក្នុងកំណត់រចនាសម្ព័ន្ធម៉ាស៊ីនបម្រើរបស់អ្នក។

| កម្រិត         | ការពិពណ៌នា                 | ករណីប្រើប្រាស់ឧទាហរណ៍         |
|---------------|-----------------------------|-----------------------------------|
| debug         | ព័ត៌មានលម្អិតសម្រាប់ដល់ភ្លើងការងារ | ចំណុចចូល/ចេញមុខងារ           |
| info          | សារព័ត៌មានទូទៅ             | កំណត់ត្រាការរីកចម្រើន              |
| notice        | ព្រឹត្តិការណ៍ធម្មតាហើយមានសារៈសំខាន់     | ការផ្លាស់ប្ដូរការកំណត់                |

| alert         | ត្រូវធ្វើសកម្មភាពភ្លាមៗ        | ការរុះរើទិន្នន័យ                     |
| emergency     | ប្រព័ន្ធមិនអាចប្រើប្រាស់បាន     | ការបរាជ័យប្រព័ន្ធពេញលេញ             |

## ការអនុវត្តសារជូនដំណឹងនៅ MCP

ដើម្បីអនុវត្តសារជូនដំណឹងនៅ MCP អ្នកត្រូវតែតំឡើងម៉ាស៊ីនបម្រើនិងអតិថិជនរួចហើយដើម្បីគ្រប់គ្រងការអាប់ដេតរាល់ពេល។ វាអនុញាតឲ្យកម្មវិធីរបស់អ្នកផ្តល់មតិបញ្ចាំងភ្លាមៗទៅអ្នកប្រើពេលមានប្រតិបត្តិការយូរ។

### ផ្នែកម៉ាស៊ីនបម្រើ៖ ផ្ញើសារជូនដំណឹង

ចាប់ផ្តើមពីផ្នែកម៉ាស៊ីនបម្រើ។ នៅ MCP អ្នកកំណត់ឧបករណ៍ដែលអាចផ្ញើជូនដំណឹង ខណៈកំពុងដំណើរការសំណើ។ ម៉ាស៊ីនបម្រើប្រើវត្ថុ context (ជាទូទៅគឺ `ctx`) ដើម្បីផ្ញើសារ ទៅអតិថិជន។

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

នៅក្នុងឧទាហរណ៍មុនមួយឧបករណ៍ `process_files` ផ្ញើសារជូនដំណឹងបីទៅអតិថិជន ខណៈដែលវាកំពុងដំណើរការឯកសារតែមួយៗ។ វិធីសាស្រ្ត `ctx.info()` ត្រូវបានប្រើសម្រាប់ផ្ញើសារព័ត៌មាន។

បន្ថែមទៀត ដើម្បីបើកសារជូនដំណឹង, សូមប្រាកដថាម៉ាស៊ីនបម្រើរបស់អ្នកប្រើដឹកជញ្ជូនផ្សាយបន្ត (ដូចជា `streamable-http`) និងអតិថិជនរបស់អ្នកអនុវត្តកម្មវិធីដំណើរការសារសម្រាប់គ្រប់គ្រងសារជូនដំណឹង។ វែរបៀបដូចខាងក្រោមដើម្បីតំឡើងម៉ាស៊ីនបម្រើប្រើដឹកជញ្ជូន `streamable-http`៖

```python
mcp.run(transport="streamable-http")
```

#### .NET


ដើម្បីបើកសារជូនដំណឹងនៅម៉ាស៊ីនបម្រើ MCP .NET របស់អ្នក សូមប្រាកដថាអ្នកប្រើដឹកជញ្ជូនផ្សាយបន្ត៖

### ផ្នែកអតិថិជន៖ ទទួលសារជូនដំណឹង

អតិថិជនត្រូវអនុវត្តកម្មវិធីដំណើរការសារដើម្បីដំណើរការនិងបង្ហាញសារជូនដំណឹងពេលវាចូល។

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

នៅក្នុងកូដមុនមួយ `message_handler` ត្រួតពិនិត្យប្រសិនបើសារចូលគឺជាសារជូនដំណឹង ប្រសិនបើបាទវាកាន់តែបោះពុម្ពសារជូនដំណឹង បើមិនអញ្ជឹង វាដំណើរការជាសារserverធម្មតា។ ក៏ចំណាំពីរបៀបដែល `ClientSession` ត្រូវបានបង្កើតជាមួយ `message_handler` ដើម្បីគ្រប់គ្រងសារជូនដំណឹងចូល។

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

នៅក្នុងឧទាហរណ៍ .NET នេះ `MessageHandler` ពិនិត្យប្រសិនបើសារចូលគឺជាសារជូនដំណឹង ប្រសិនបើបាទវាបោះពុម្ពសារជូនដំណឹង បើមិនបាទវាដំណើរការជាសារម៉ាស៊ីនបម្រើធម្មតា។ `ClientSession` ត្រូវបានបង្កើតជាមួយកម្មវិធីដំណើរការសារដោយរយៈ `ClientSessionOptions`។

ដើម្បីបើកសារជូនដំណឹង សូមប្រាកដថាម៉ាស៊ីនបម្រើប្រើដឹកជញ្ជូនផ្សាយបន្ត (ដូចជា `streamable-http`) និងអតិថិជនរបស់អ្នកអនុវត្តកម្មវិធីដំណើរការសារសម្រាប់គ្រប់គ្រងសារជូនដំណឹង។

## សារជូនដំណឹងកំណត់ត្រា និងស្ថានភាព

ផ្នែកនេះពន្យល់ពីគំនិតសារជូនដំណឹងកំណត់ត្រា នៅ MCP, ហេតុផលដែលវាមានសារៈសំខាន់ និងរបៀបអនុវត្តវា ដោយប្រើ Streamable HTTP។ អ្នកក៏នឹងឃើញការងារអនុវត្តសម្រាប់បំផ្លាញជំនាញរបស់អ្នកផងដែរ។

សារជូនដំណឹងកំណត់ត្រាជាសារពេលពិតដែលផ្ញើពីម៉ាស៊ីនបម្រើទៅអតិថិជនក្នុងពេលប្រតិបត្តិការយូរ។ ជំនួសមិនចាំរង់ចាំដំណើរការបញ្ចប់ទាំងមូល ម៉ាស៊ីនបម្រើនៅតែផ្ដល់អាប់ដេតអំពីស្ថានភាពបច្ចុប្បន្នទៅអតិថិជន។ វាបង្កើនភាពច្បាស់ និងបទពិសោធន៍អ្នកប្រើ និងធ្វើឲ្យការស្វែងរកកំហុសកាន់តែងាយស្រួល។

**ឧទាហរណ៍៖**

```text

"Processing document 2/10"
...
"Processing complete!"

```

### ហេតុអ្វីបានជាប្រើសារជូនដំណឹងកំណត់ត្រា?

សារជូនដំណឹងកំណត់ត្រាមានសារៈសំខាន់ច្រើនហេតុផល៖

- **បទពិសោធន៍អ្នកប្រើល្អជាងមុន៖** អ្នកប្រើឃើញការអាប់ដេតពេលការងាររត់ តាមពិតមិនត្រឹមតែចុងបញ្ចប់។
- **មតិកម្លាស់បម្រួលពេលពិត៖** អតិថិជនអាចបង្ហាញការតំណាងកំណត់ត្រា ឬកំណត់ហេតុ ធ្វើឲ្យកម្មវិធីមានការឆ្លើយតបលឿន។
- **ងាយស្រួលស្វែងរកកំហុស និងត្រួតពិនិត្យ៖** អ្នកអភិវឌ្ឍនិងអ្នកប្រើអាចមើលចំណុច ដែលប្រតិបត្តិការអាចយឺត ឬអាចជាប់។

### របៀបអនុវត្តសារជូនដំណឹងកំណត់ត្រា

វាទាំងនេះជារបៀបដែលអ្នកអាចអនុវត្តសារជូនដំណឹងកំណត់ត្រានៅ MCP៖

- **នៅម៉ាស៊ីនបម្រើ៖** ប្រើ `ctx.info()` ឬ `ctx.log()` ដើម្បីផ្ញើសារជូនដំណឹងពេលធាតុនីមួយៗត្រូវបានដំណើរការ។ វាបញ្ជូនសារទៅអតិថិជនមុនពេលលទ្ធផលសំខាន់រួចរាល់។
- **នៅអតិថិជន៖** អនុវត្តកម្មវិធីដំណើរការសារដែលស្តាប់ និងបង្ហាញសារជូនដំណឹងពេលវាចូល។ កម្មវិធីដំណើរការនេះចាត់ទុកផ្តាច់រវាងសារជូនដំណឹង និងលទ្ធផលចុងក្រោយ។

**ឧទាហរណ៍ម៉ាស៊ីនបម្រើ៖**

```python
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

## ការពិចារណាអំពីសុវត្ថិភាព

នៅពេលអនុវត្តម៉ាស៊ីនមេ MCP ជាមួយការដឹកជញ្ជូនផ្អែកលើ HTTP សុវត្ថិភាពក្លាយជាបំណងសំខាន់ ដើម្បីត្រូវបានអនុវត្តដោយប្រុងប្រយ័ត្នចំពោះវិថីប្រហារជាច្រើន និងយន្តការការពារ។

### ទិដ្ឋភាពទូទៅ

សុវត្ថិភាពគឺមានសារៈសំខាន់ពេលបង្ហាញម៉ាស៊ីនមេ MCP តាមរយៈ HTTP។ Streamable HTTP ផ្ដល់នូវផ្ទៃប្រហារថ្មី និងទាមទារការកំណត់ត្រឹមត្រូវយ៉ាងប្រុងប្រយ័ត្ន។

### ចំណុចសំខាន់

- **ការផ្ទៀងផ្ទាត់ចំណងជើង Origin**៖ តែងតែផ្ទៀងផ្ទាត់ចំណងជើង `Origin` ដើម្បីការពារការវាយប្រហារជាតំណមី DNS។
- **ការតំណបទៅ localhost**៖ សម្រាប់ការអភិវឌ្ឍន៍ក្នុងស្រុក ចងខ្សែម៉ាស៊ីនមេទៅ `localhost` ដើម្បីចៀសវាងការបង្ហាញទៅអ៊ីនធឺណេតសាធារណៈ។
- **ការផ្ទៀងផ្ទាត់**៖ អនុវត្តការផ្ទៀងផ្ទាត់ (ដូចជា API keys, OAuth) សម្រាប់ការដាក់ឲ្យដំណើរការផលិតកម្ម។
- **CORS**៖ កំណត់គោលការណ៍ការចែករំលែកឆ្លង-origin (CORS) ដើម្បីដាក់កំណត់ការចូលប្រើ។
- **HTTPS**៖ ប្រើ HTTPS ក្នុងផលិតកម្មដើម្បីរុងរឿងចរាចរណ៍។

### ហៅថាវិធានការល្អបំផុត

- កុំធ្លាប់ទុកចិត្តក្នុងសំណើដែលមិនបានផ្ទៀងផ្ទាត់។
- ចុះតាមដាន និងកត់ត្រាការចូលប្រើ និងកំហុសទាំងអស់។
- ទប់ស្កាត់ដោយធ្វើបច្ចុប្បន្នភាពប្រព័ន្ធជាប់ដោយរៀងរាល់ករណីដើម្បីជួសជុលច្រកសុវត្ថិភាព។

### បញ្ហា

- ការប្រកួតប្រជែងរវាងសុវត្ថិភាព និងភាពងាយស្រួលរបស់ការអភិវឌ្ឍ។
- ការធានាគោលលទ្ធភាពជាមួយបរិយាកាសអតិថិជនផ្សេងៗ។

## ការកែលម្អពី SSE ទៅ Streamable HTTP

សម្រាប់កម្មវិធីដែលកំពុងប្រើ Server-Sent Events (SSE) ការផ្ទេរទៅ Streamable HTTP ផ្ដល់ជំនួយសមត្ថភាពប្រសើរ និងពន្លឿនភាពរយៈពេលវែងសម្រាប់ការអនុវត្ត MCP របស់អ្នក។

### មូលហេតុអ្វីបានជាត្រូវកែលម្អ?

មានមូលហេតុពីរដែលគួរឱ្យចាប់អារម្មណ៍សម្រាប់កែលម្អពី SSE ទៅ Streamable HTTP៖

- Streamable HTTP ផ្ដល់នូវភាពអាចរីករាលដាលប្រសើរ ការគាំទ្រគោលលទ្ធភាព និងការចេញផ្សាយជាដំណឹងសម្បូរបែបជាង SSE។
- វាជាជម្រើសដឹកជញ្ជូនដែលណែនាំសម្រាប់កម្មវិធី MCP ថ្មីៗ។

### ជំហានក្នុងការផ្ទេរ

នេះគឺជារបៀបដែលអ្នកអាចផ្ទេរពី SSE ទៅ Streamable HTTP ក្នុងកម្មវិធី MCP របស់អ្នក៖

- **ធ្វើបច្ចុប្បន្នកម្មកូដម៉ាស៊ីនមេ** ដើម្បីប្រើ `transport="streamable-http"` ក្នុង `mcp.run()`។
- **ធ្វើបច្ចុប្បន្នកម្មកូដអតិថិជន** ដើម្បីប្រើ `streamablehttp_client` ជំនួស SSE client។
- **អនុវត្តកម្មវិធីដំណើរការសារ** ក្នុងអតិថិជនសម្រាប់ដំណើរការជាថ្មីនៃការជូនដំណឹង។
- **សាកល្បងរួមគ្នានឹងគ្រឿងចក្រ និងដំណើរការត្រៀមទទួល**។

### រក្សាគោលលទ្ធភាព

អ្នកត្រូវបានណែនាំឲ្យរក្សាគោលលទ្ធភាពជាមួយអតិថិជន SSE មានស្រាប់ក្នុងអំឡុងពេលផ្ទេរនេះ។ មានយុទ្ធសាស្ត្រតំណរភាពខាងក្រោម៖

- អ្នកអាចគាំទ្រទាំង SSE និង Streamable HTTP ដោយដំណើរការទាំងពីរនៅចំណុចបញ្ចប់ផ្សេងគ្នា។
- ផ្ទេរអតិថិជនទៅប្រព័ន្ធដឹកជញ្ជូនថ្មីយ៉ាងជាបន្តបន្ទាប់។

### បញ្ហា

ត្រូវប្រាកដថាអ្នកដោះស្រាយបញ្ហាខាងក្រោមក្នុងអំឡុងពេលផ្ទេរ៖

- ធ្វើឲ្យប្រាកដថាអតិថិជនទាំងអស់ ត្រូវបានធ្វើបច្ចុប្បន្នភាព។
- ដោះស្រាយភាពខុសគ្នាក្នុងការផ្ដល់ដំណឹង។

## ការពិចារណាអំពីសុវត្ថិភាព

សុវត្ថិភាពគួរតែងតែមានអាទិភាពខ្ពស់ខណៈពេលអនុវត្តម៉ាស៊ីនមេណាមួយ ជាពិសេសពេលប្រើប្រាស់ប្រព័ន្ធផ្សព្វផ្សាយដែលផ្អែកលើ HTTP ដូចជា Streamable HTTP ក្នុង MCP។

នៅពេលអនុវត្តម៉ាស៊ីនមេ MCP ជាមួយប្រព័ន្ធផ្សព្វផ្សាយ HTTP សុវត្ថិភាពក្លាយជាបំណងសំខាន់ដែលត្រូវតែប្រុងប្រយ័ត្នចំពោះការវាយប្រហារជាច្រើន និងយន្តការការពារ។

### ទិដ្ឋភាពទូទៅ

សុវត្ថិភាពគឺសំខាន់ណាស់ពេលបង្ហាញម៉ាស៊ីនមេ MCP តាមរយៈ HTTP។ Streamable HTTP ធ្វើឲ្យមានផ្ទៃប្រហារថ្មី និងទាមទារការកំណត់ត្រឹមត្រូវយ៉ាងប្រុងប្រយ័ត្ន។

នេះជាការពិចារណាសុវត្ថិភាពសំខាន់ៗ៖

- **ការផ្ទៀងផ្ទាត់ចំណងជើង Origin**៖ តែងតែផ្ទៀងផ្ទាត់ចំណងជើង `Origin` ដើម្បីការពារការវាយប្រហារជាតំណមី DNS។
- **ការតំណបទៅ localhost**៖ សម្រាប់ការអភិវឌ្ឍន៍ក្នុងស្រុក ចងខ្សែម៉ាស៊ីនមេទៅ `localhost` ដើម្បីចៀសវាងការបង្ហាញទៅអ៊ីនធឺណេតសាធារណៈ។
- **ការផ្ទៀងផ្ទាត់**៖ អនុវត្តការផ្ទៀងផ្ទាត់ (ដូចជា API keys, OAuth) សម្រាប់ការដាក់ឲ្យដំណើរការផលិតកម្ម។
- **CORS**៖ កំណត់គោលការណ៍ការចែករំលែកឆ្លង-origin (CORS) ដើម្បីដាក់កំណត់ការចូលប្រើ។
- **HTTPS**៖ ប្រើ HTTPS ក្នុងផលិតកម្មដើម្បីរុងរឿងចរាចរណ៍។

### ហៅថាវិធានការល្អបំផុត

បន្ថែមពីនេះទៀត មានវិធានការល្អបំផុតខាងក្រោមដែលគួរតែអនុវត្តពេលបង្កើតសុវត្ថិភាពនៅម៉ាស៊ីនមេ MCP របស់អ្នក៖

- កុំធ្លាប់ទុកចិត្តក្នុងសំណើដែលមិនបានផ្ទៀងផ្ទាត់។
- ចុះតាមដាន និងកត់ត្រាការចូលប្រើ និងកំហុសទាំងអស់។
- ទប់ស្កាត់ដោយធ្វើបច្ចុប្បន្នភាពប្រព័ន្ធជាប់ដោយរៀងរាល់ករណីដើម្បីជួសជុលច្រកសុវត្ថិភាព។

### បញ្ហា

អ្នកនឹងប្រឈមមុខនឹងបញ្ហាមួយចំនួនពេលអនុវត្តសុវត្ថិភាពនៅម៉ាស៊ីនមេស្ទ្រីម MCP៖

- ការប្រកួតប្រជែងរវាងសុវត្ថិភាព និងភាពងាយស្រួលរបស់ការអភិវឌ្ឍ។
- ការធានាគោលលទ្ធភាពជាមួយបរិយាកាសអតិថិជនផ្សេងៗ។

### បេសកកម្ម៖ សាងសង់កម្មវិធី MCP ស្ទ្រីមរបស់អ្នកផ្ទាល់

**ស្ថានការណ៍ ៖**
សង់ម៉ាស៊ីនមេ និងអតិថិជន MCP ដែលម៉ាស៊ីនមេដំណើរការបញ្ជីធាតុ (ឧទាហរណ៍ ឯកសារ ឬឯកសារ) ហើយផ្ញើសារជូនដំណឹងមួយសម្រាប់មុខមួយក្នុងចំណោមព័ត៌មានដែលបានដំណើរការ។ អតិថិជនគួរតែនៅបង្ហាញសារជូនដំណឹងនីមួយៗពេលវាមកដល់។

**ជំហានៈ**

1. អនុវត្តឧបករណ៍ម៉ាស៊ីនមេដែលដំណើរការបញ្ជីមួយ ហើយផ្ញើសារជូនដំណឹងសម្រាប់មុខមួយក្នុងចំណោមព័ត៌មាន។
2. អនុវត្តអតិថិជនដែលមានកម្មវិធីដំណើរការសារសម្រាប់បង្ហាញសារជូនដំណឹងក្នុងពេលពិត។
3. សាកល្បងការអនុវត្តរបស់អ្នកដោយបើកដំណើរការម៉ាស៊ីនមេ និងអតិថិជន ហើយមើលសារជូនដំណឹង។
