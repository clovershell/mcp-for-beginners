# סטרימינג HTTPS עם פרוטוקול הקשר מודל (MCP)

פרק זה מספק מדריך מקיף ליישום סטרימינג מאובטח, בר-קנה-מידה ובזמן אמת עם פרוטוקול הקשר מודל (MCP) באמצעות HTTPS. הוא עוסק במניעים לסטרימינג, מנגנוני ההעברה הזמינים, כיצד ליישם HTTP סטרימינג ב-MCP, שיטות עבודה מומלצות לאבטחה, מעבר מ-SSE והנחיות מעשיות לבניית יישומי סטרימינג MCP משלכם.

> **מבט לעתיד:** שיעור זה מתאר HTTP סטרימינג תחת **מפרט MCP 2025-11-25**, שבו מושב מוקם במהלך `initialize` ומוצמד עם כותרת `Mcp-Session-Id`. מועמד לשחרור `2026-07-28` מסיר לחלוטין את הידוק הידיים ואת מזהה המושב, מה שהופך כל בקשה לעצמאית וניתנת לניתוב לכל מופע שרת ללא מושבים דביקים. ראו [מה משתנה ב-MCP: מועמד לשחרור 2026-07-28](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) לפרטים.

## מנגנוני העברה וסטרימינג ב-MCP

חלק זה בוחן את מנגנוני ההעברה השונים הזמינים ב-MCP ותפקידם בהפעלת יכולות סטרימינג לתקשורת בזמן אמת בין לקוחות לשרתים.

### מהו מנגנון העברה?

מנגנון העברה מגדיר כיצד נתונים מוחלפים בין הלקוח לשרת. MCP תומך בסוגי העברה מרובים כדי להתאים לסביבות ודרישות שונות:

- **stdio**: קלט/פלט סטנדרטי, מתאים לכלים מקומיים ובסיסיים שורת פקודה. פשוט אך לא מתאים לאינטרנט או לענן.
- **SSE (אירועים שנשלחים מהשרת)**: מאפשר לשרתים לדחוף עדכונים בזמן אמת ללקוחות דרך HTTP. טוב לממשקי ווב, אך מוגבל בקנה מידה וגמישות. החל ממפרט MCP 2025-06-18, מנגנון SSE הסטנדרטי הוסר והוחלף ב-״HTTP סטרימינג״.
- **HTTP סטרימינג**: מנגנון סטרימינג מודרני מבוסס HTTP, תומך בהתראות וסקלאביליות טובה יותר. מומלץ למרבית תרחישי ייצור וענן.

### טבלת השוואה

עיינו בטבלה למטה כדי להבין את ההבדלים בין מנגנוני ההעברה:

| העברה             | עדכוני זמן אמת     | סטרימינג  | סקלאביליות | מקרי שימוש              |
|-------------------|------------------|-----------|-------------|-------------------------|
| stdio             | לא               | לא        | נמוכה      | כלים מקומיים שורת פקודה  |
| SSE               | כן               | כן        | בינונית     | ווב, עדכוני זמן אמת      |
| HTTP סטרימינג     | כן               | כן        | גבוהה      | ענן, לקוחות מרובים       |

> **טיפ:** בחירת מנגנון ההעברה משפיעה על ביצועים, סקלאביליות וחווית המשתמש. **HTTP סטרימינג** מומלץ ליישומים מודרניים, סקלאביליים ומוכנים לענן.

שימו לב למנגנוני ההעברה stdio ו-SSE שהוצגו בפרקים הקודמים ואיך HTTP סטרימינג הוא מנגנון ההעברה שמכוסה בפרק זה.

## סטרימינג: מושגים ומניעים

הבנת המושגים והמניעים הבסיסיים מאחורי סטרימינג חיונית ליישום מערכות תקשורת בזמן אמת יעילות.

**סטרימינג** היא טכניקה בתכנות רשת המאפשרת לשלוח ולקבל נתונים בחלקים קטנים וניתנים לניהול או כרצף של אירועים, במקום להמתין לתגובה מלאה. זה שימושי במיוחד ל:

- קבצים גדולים או מערכי נתונים.
- עדכוני זמן אמת (למשל, צ'אט, סרגלי התקדמות).
- חישובים ארוכי טווח שבהם רוצים להמשיך לעדכן את המשתמש.

הנה מה שצריך לדעת על סטרימינג ברמה גבוהה:

- הנתונים נמסרים בהדרגה, לא כולם בבת אחת.
- הלקוח יכול לעבד נתונים כשהם מגיעים.
- מפחית תחושת השיהוי ומשפר את חווית המשתמש.

### למה להשתמש בסטרימינג?

הסיבות לשימוש בסטרימינג הן:

- המשתמשים מקבלים משוב מידי, לא רק בסוף
- מאפשר יישומים בזמן אמת וממשקי משתמש רגישים
- ניצול יעיל יותר של משאבי רשת ומחשב

### דוגמה פשוטה: שרת ולקוח סטרימינג HTTP

הנה דוגמה פשוטה לאיך סטרימינג יכול להיות מיושם:

#### פייתון

**שרת (Python, שימוש ב-FastAPI ו-StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**לקוח (Python, שימוש ב-requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

דוגמה זו מראה שרת ששולח סדרת הודעות ללקוח כשהן זמינות, במקום להמתין שכל ההודעות יהיו מוכנות.

**איך זה עובד:**

- השרת מייצר כל הודעה כשהיא מוכנה.
- הלקוח מקבל ומדפיס כל חלק שמגיע.

**דרישות:**

- השרת חייב להשתמש בתגובה סטרימינג (למשל, `StreamingResponse` ב-FastAPI).
- הלקוח חייב לעבד את התגובה כסטרים (`stream=True` ב-requests).
- סוג התוכן בדרך כלל `text/event-stream` או `application/octet-stream`.

#### ג'אווה

**שרת (Java, שימוש ב-Spring Boot וב-SSE):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**לקוח (Java, שימוש ב-Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**הערות על יישום בג'אווה:**

- משתמש בערימת ריאקטיבית של Spring Boot עם `Flux` לסטרימינג
- `ServerSentEvent` מספק סטרימינג מאורגן עם סוגי אירועים
- `WebClient` עם `bodyToFlux()` מאפשר צריכת סטרימינג ריאקטיבית
- `delayElements()` מדמה זמן עיבוד בין אירועים
- לאירועים יכולים להיות סוגים (`info`, `result`) לטיפול טוב יותר בלקוח

### השוואה: סטרימינג קלאסי מול סטרימינג MCP

ההבדלים בין אופן פעולת סטרימינג "קלאסי" לבין ב-MCP ניתנים לתיאור כך:

| תכונה                | סטרימינג HTTP קלאסי          | סטרימינג MCP (התראות)           |
|----------------------|-----------------------------|---------------------------------|
| תגובה ראשית           | מפוצלת                      | יחידה, בסוף                      |
| עדכוני התקדמות        | נשלחים כחלקים של נתונים     | נשלחים כהתראות                  |
| דרישות לקוח          | חייב לעבד סטרים             | חייב ליישם מטפל בהודעות          |
| מקרי שימוש           | קבצים גדולים, זרמי טוקנים AI | התקדמות, יומנים, משוב בזמן אמת  |

### הבדלים מרכזיים שנצפו

בנוסף, הנה כמה הבדלים מרכזיים:

- **תבנית תקשורת:**
  - סטרימינג HTTP קלאסי: משתמש בקידוד חלקים פשוט כדי לשלוח נתונים בחתיכות
  - סטרימינג MCP: משתמש במערכת התראות עם פרוטוקול JSON-RPC

- **פורמט הודעה:**
  - HTTP קלאסי: טקסט פשוט בפסקאות עם שורות חדשות
  - MCP: אובייקטים של LoggingMessageNotification עם מטא-נתונים מובנים

- **מימוש לקוח:**
  - HTTP קלאסי: לקוח פשוט שמעבד תגובות סטרימינג
  - MCP: לקוח מורכב עם מטפל בהודעות עבור סוגי הודעות שונים

- **עדכוני התקדמות:**
  - HTTP קלאסי: ההתקדמות היא חלק מזרם התגובה הראשית
  - MCP: ההתקדמות נשלחת כהודעות התראה נפרדות בעוד התגובה הראשית מגיעה בסוף

### המלצות

יש כמה דברים שאנחנו ממליצים עליהם כשמדובר בבחירה בין יישום סטרימינג קלאסי (כפי שהראינו ממסלול `/stream`) לבין סטרימינג דרך MCP.

- **לצרכי סטרימינג פשוטים:** סטרימינג HTTP קלאסי פשוט יותר ליישום ומספיק לצרכי סטרימינג בסיסיים.

- **ליישומים מורכבים ואינטראקטיביים:** סטרימינג MCP מספק גישה מובנית יותר עם מטא-נתונים עשירים והפרדה בין התראות לתוצאות סופיות.

- **ליישומי AI:** מערכת ההתראות של MCP מועילה במיוחד למשימות AI ארוכות שבהן רוצים להמשיך לעדכן את המשתמשים בהתקדמות.

## סטרימינג ב-MCP

אוקיי, עד עכשיו ראיתם כמה המלצות והשוואות על ההבדל בין סטרימינג קלאסי לסטרימינג ב-MCP. בואו נצלול לפרטים בדיוק איך ניתן לנצל סטרימינג ב-MCP.

הבנת אופן פעולת הסטרימינג במסגרת MCP חיונית לבניית יישומים רגישים המעניקים משוב בזמן אמת למשתמשים במהלך פעולות ארוכות.

ב-MCP, סטרימינג אינו מתייחס לשליחת התגובה הראשית בחתיכות, אלא לשליחת **התראות** ללקוח בזמן שהכלי מעבד בקשה. התראות אלו יכולות לכלול עדכוני התקדמות, יומנים או אירועים אחרים.

### איך זה עובד

התוצאה הראשית עדיין נשלחת בתגובה יחידה. עם זאת, ניתן לשלוח התראות כהודעות נפרדות במהלך העיבוד וכך לעדכן את הלקוח בזמן אמת. הלקוח צריך להיות מסוגל לטפל ולהציג את ההתראות האלה.

## מהי התראה?

אמרנו "התראה", מה פירוש הדבר בהקשר של MCP?

התראה היא הודעה שנשלחת מהשרת ללקוח כדי ליידע על התקדמות, מצב או אירועים אחרים במהלך פעולה ארוכת-טווח. התראות משפרות את השקיפות וחווית המשתמש.

לדוגמה, הלקוח אמור לשלוח התראה ברגע שה הידוק הידיים הראשוני עם השרת בוצע.

התראה נראית כך כהודעת JSON:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

התראות שייכות לנושא ב-MCP הנקרא ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging).

> **הודעת הפסקת שימוש:** מועמד לשחרור מפרט MCP `2026-07-28` מסמן את ה-Logging כנדיר לטובת `stderr` עבור העברות stdio ו-OpenTelemetry לתצפית מובנית. ה-Logging ממשיך לפעול ב-`2025-11-25` ולפחות שנה לאחר הפסקת שימוש פורמלית. ראו [מה משתנה ב-MCP: מועמד לשחרור 2026-07-28](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md).

כדי להפעיל את הלוגינג, השרת צריך לאפשר זאת כיכולת/פיצ'ר כך:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> בהתאם ל-SDK שבו משתמשים, ייתכן שהלוגינג פעיל כברירת מחדל, או שתצטרכו לאפשר זאת במפורש בהגדרות השרת שלכם.

קיימים סוגים שונים של התראות:

| רמה       | תיאור                         | דוגמת שימוש                  |
|-----------|-------------------------------|-----------------------------|
| debug     | מידע מפורט לניפוי שגיאות       | נקודות כניסה/יציאה מפונקציה |
| info      | הודעות מידע כלליות             | עדכוני התקדמות בפעולה       |
| notice    | אירועים רגילים אך משמעותיים    | שינויים בתצורה              |
| warning   | תנאי אזהרה                    | שימוש בפיצ'ר מיושן          |
| error     | תנאי שגיאה                   | כשלונות בפעולה             |
| critical  | תנאים קריטיים                | כשלונות ברכיבי מערכת        |
| alert     | יש לנקוט פעולה מיידית          | זיהוי נזק בנתונים          |
| emergency | המערכת אינה שמישה            | כשלון מלא של המערכת         |

## יישום התראות ב-MCP

ליישם התראות ב-MCP, יש להגדיר הן בצד השרת והן בצד הלקוח טיפול בעדכונים בזמן אמת. זה מאפשר ליישום שלכם לספק משוב מיידי למשתמשים במהלך פעולות ארוכות.

### צד השרת: שליחת התראות

נתחיל בצד השרת. ב-MCP מגדירים כלים שיכולים לשלוח התראות בזמן עיבוד בקשות. השרת משתמש באובייקט ההקשר (בדרך כלל `ctx`) כדי לשלוח הודעות ללקוח.

#### פייתון

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

בדוגמה הקודמת, הכלי `process_files` שולח שלוש התראות ללקוח ככל שהוא מעבד כל קובץ. מתודת `ctx.info()` משמשת לשליחת הודעות מידע.

בנוסף, כדי להפעיל את ההתראות, ודאו שהשרת משתמש במנגנון סטרימינג (כגון `streamable-http`) ושהלקוח מיישם מטפל בהודעות לעיבוד התראות. כך ניתן להגדיר את השרת להשתמש במנגנון `streamable-http`:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

בדוגמה זו של .NET, הכלי `ProcessFiles` מסומן עם התכונה `Tool` ושולח שלוש התראות ללקוח תוך עיבוד כל קובץ. מתודת `ctx.Info()` משמשת לשליחת הודעות מידע.

כדי להפעיל התראות בשרת MCP מבוסס .NET, ודאו שאתם משתמשים במנגנון סטרימינג:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### צד הלקוח: קבלת התראות

על הלקוח ליישם מטפל בהודעות שיטפל ויציג התראות בזמן שהן מגיעות.

#### פייתון

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

בקוד הזה, פונקציית `message_handler` בודקת אם ההודעה הנכנסת היא התראה. אם כן, היא מדפיסה את ההתראה; אחרת היא מעבדת אותה כהודעה רגילה מהשרת. שימו לב גם ש-`ClientSession` מאתחלת עם `message_handler` כדי לטפל בהתראות הנכנסות.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

בדוגמה זו של .NET, פונקציית `MessageHandler` בודקת אם ההודעה הנכנסת היא התראה. אם כן, היא מדפיסה את ההתראה; אחרת היא מעבדת אותה כהודעה רגילה מהשרת. אובייקט `ClientSession` מאתחל עם מטפל ההודעות דרך `ClientSessionOptions`.

כדי להפעיל התראות, ודאו שהשרת משתמש במנגנון סטרימינג (כגון `streamable-http`) והלקוח מיישם מטפל להודעות להליך התראות.

## התראות התקדמות ותרחישים

חלק זה מסביר את מושג התראות ההתקדמות ב-MCP, מדוע הן חשובות, וכיצד ליישם אותן באמצעות HTTP סטרימינג. תמצאו גם משימת תרגול לחיזוק ההבנה שלכם.

התראות התקדמות הן הודעות בזמן אמת שנשלחות מהשרת ללקוח במהלך פעולות ארוכות. במקום להמתין לסיום מלא של התהליך, השרת מעדכן את הלקוח על המצב הנוכחי. זה משפר את השקיפות, חווית המשתמש ומקל על איתור תקלות.

**דוגמה:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### מדוע להשתמש בהתראות התקדמות?

התראות התקדמות חיוניות מכמה סיבות:

- **חוויית משתמש משופרת:** המשתמשים רואים עדכונים כמו שהעבודה מתקדמת, לא רק בסוף.
- **משוב בזמן אמת:** הלקוחות יכולים להציג סרגלי התקדמות או יומנים, מה שגורם לאפליקציה להרגיש רספונסיבית.
- **איתור ולוגינג קל יותר:** מפתחים ומשתמשים יכולים לראות איפה התהליך איטי או תקוע.

### כיצד ליישם התראות התקדמות

כך ניתן ליישם התראות התקדמות ב-MCP:

- **בשרת:** השתמשו ב-`ctx.info()` או `ctx.log()` לשליחת התראות כאשר כל פריט מעובד. זאת שולחת הודעה ללקוח לפני שהתוצאה הראשית מוכנה.
- **בלקוח:** יישמו מטפל הודעות שמאזין ומציג התראות כמו שהן מגיעות. מטפל זה מבדיל בין התראות לתוצאה הסופית.

**דוגמה שרת:**


#### פייתון

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

### סקירה כללית

אבטחה היא קריטית כאשר חושפים שרתי MCP דרך HTTP. HTTP ניתנת להזרים מציגה משטחים חדשים להתקפה ודורשת תצורה מוקפדת.

### נקודות מפתח

- **אימות כותרת Origin**: תמיד אמת את כותרת `Origin` כדי למנוע התקפות DNS rebinding.
- **קישור ל-localhost**: עבור פיתוח מקומי, קשר את השרתים ל-`localhost` כדי למנוע חשיפתם לאינטרנט הציבורי.
- **אימות**: יישם אימות (כגון מפתחות API, OAuth) לפריסות ייצור.
- **CORS**: קבע מדיניות CORS להגבלת הגישה.
- **HTTPS**: השתמש בפרוטוקול HTTPS בייצור כדי להצפין תעבורה.

### נוהגים מומלצים

- אל תבטח מעולם בבקשות נכנסות ללא אימות.
- תעד ונטר כל גישה ושגיאות.
- עדכן באופן קבוע את התלויות כדי לתקן נקודות תורפה באבטחה.

- הבטחת תאימות לסביבות לקוח שונות

## שדרוג מ-SSE ל-Streamable HTTP

עבור אפליקציות שמשתמשות כיום באירועים שנשלחים מהשרת (SSE), המעבר ל-Streamable HTTP מספק יכולות משופרות וקיימות טובה יותר לאורך זמן למימושי MCP שלך.

### מדוע לשדרג?

ישנם שני סיבות משכנעות לשדרג מ-SSE ל-Streamable HTTP:

- Streamable HTTP מציע קנה מידה טוב יותר, תאימות ותמיכה עשירה יותר בהתראות מאשר SSE.
- זו התחבורה המומלצת לאפליקציות MCP חדשות.

### שלבי ההגירה

כך אפשר לעבור מ-SSE ל-Streamable HTTP באפליקציות MCP שלך:

- **עדכן את קוד השרת** לשימוש ב-`transport="streamable-http"` ב-`mcp.run()`.
- **עדכן את קוד הלקוח** לשימוש ב-`streamablehttp_client` במקום בלקוח SSE.
- **ממש הודע טיפול** בלקוח לעיבוד ההתראות.
- **בדוק את התאימות** עם הכלים וזרימות העבודה הקיימות.

### שמירת תאימות

מומלץ לשמור על תאימות עם לקוחות SSE קיימים במהלך תהליך ההגירה. הנה כמה אסטרטגיות:

- אפשר לתמוך גם ב-SSE וגם ב-Streamable HTTP על ידי הפעלת שתי התחבורה בנקודות קצה שונות.
- העבר את הלקוחות בהדרגה לתחבורה החדשה.

### אתגרים

דאג להתמודד עם האתגרים הבאים במהלך ההגירה:

- הבטחת שכל הלקוחות יעודכנו
- טיפול בהבדלים באספקת ההתראות
