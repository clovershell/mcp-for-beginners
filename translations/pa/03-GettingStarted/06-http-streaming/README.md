# HTTPS ਮਾਡਲ ਕਾਂਟੈਕਸਟ ਪ੍ਰੋਟੋਕੋਲ (MCP) ਨਾਲ ਸਟ੍ਰੀਮਿੰਗ

ਇਸ अध्याय ਵਿੱਚ HTTPS ਦੀ ਵਰਤੋਂ ਨਾਲ ਮਾਡਲ ਕਾਂਟੈਕਸਟ ਪ੍ਰੋਟੋਕੋਲ (MCP) ਨਾਲ ਸੁਰੱਖਿਅਤ, ਸਕੇਲੇਬਲ ਅਤੇ ਰੀਅਲ-ਟਾਈਮ ਸਟ੍ਰੀਮਿੰਗ ਲਾਗੂ ਕਰਨ ਲਈ ਇਕ ਵਿਆਪਕ ਗਾਈਡ ਦਿੱਤੀ ਗਈ ਹੈ। ਇਸ ਵਿਚ ਸਟ੍ਰੀਮਿੰਗ ਦਾ ਉਦੇਸ਼, ਉਪਲਬਧ ਟਰਾਂਸਪੋਰਟ ਮਕੈਨਿਜ਼ਮ, MCP ਵਿੱਚ ਸਟ੍ਰੀਮ ਕਰਨਯੋਗ HTTP ਨੂੰ ਕਿਵੇਂ ਲਾਗੂ ਕਰਨਾ, ਸੁਰੱਖਿਆ ਦੀਆਂ ਸਰੋਤਾਂ, SSE ਤੋਂ ਮਾਈਗ੍ਰੇਸ਼ਨ ਅਤੇ ਆਪਣੇ ਸਟ੍ਰੀਮਿੰਗ MCP ਐਪਲੀਕੇਸ਼ਨਾਂ ਬਣਾਉਣ ਲਈ ਪ੍ਰਯੋਗਿਕ ਦਿਸ਼ਾ-ਨਿਰਦੇਸ਼ ਆਏ ਹਨ।

> **ਅੱਗੇ ਦੇਖਦੇ ਹੋਏ:** ਇਹ ਪਾਠ **MCP ਸਪੇਸਿਫਿਕੇਸ਼ਨ 2025-11-25** ਦੇ ਤਹਿਤ ਸਟ੍ਰੀਮ ਕਰਨਯੋਗ HTTP ਦਾ ਵਰਣਨ ਕਰਦਾ ਹੈ, ਜਿੱਥੇ `initialize` ਦੌਰਾਨ ਸੈਸ਼ਨ ਬਣਦਾ ਹੈ ਅਤੇ `Mcp-Session-Id` ਹੈਡਰ ਨਾਲ ਪਿੰਨ ਕੀਤਾ ਜਾਂਦਾ ਹੈ। `2026-07-28` ਰਿਲੀਜ਼ ਉਮੀਦਵਾਰ ਹੱਥ-ਮਿਲਾਉਣ ਅਤੇ ਸੈਸ਼ਨ ID ਨੂੰ ਪੂਰੀ ਤਰ੍ਹਾਂ ਹਟਾ ਦਿੰਦਾ ਹੈ, ਜਿਸ ਨਾਲ ਹਰ ਬੇਨਤੀ ਸੁਤੰਤਰ ਹੋ ਜਾਦੀ ਹੈ ਅਤੇ ਕਿਸੇ ਵੀ ਸਰਵਰ ਇੰਸਟੈਂਸ ਨੂੰ ਬਿਨਾਂ ਸਟਿੱਕੀ ਸੈਸ਼ਨਾਂ ਦੇ ਰੂਟ ਕੀਤਾ ਜਾ ਸਕਦਾ ਹੈ। ਵਿਸਥਾਰ ਲਈ ਵੇਖੋ [What's Changing in MCP: The 2026-07-28 Release Candidate](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md)।

## MCP ਵਿੱਚ ਟਰਾਂਸਪੋਰਟ ਮਕੈਨਿਜ਼ਮ ਅਤੇ ਸਟ੍ਰੀਮਿੰਗ

ਇਹ ਭਾਗ MCP ਵਿੱਚ ਉਪਲਬਧ ਵੱਖ-ਵੱਖ ਟਰਾਂਸਪੋਰਟ ਮਕੈਨਿਜ਼ਮਾਂ ਦੀ ਜਾਂਚ ਕਰਦਾ ਹੈ ਅਤੇ ਇਹ ਸਟ੍ਰੀਮਿੰਗ ਸਮਰੱਥਾਵਾਂ ਨੂੰ ਰੀਅਲ-ਟਾਈਮ ਸਾਂਝਾ ਕਰਨ ਲਈ ਕਿਵੇਂ ਯੋਗ ਬਣਾਉਂਦੀਆਂ ਹਨ।

### ਟਰਾਂਸਪੋਰਟ ਮਕੈਨਿਜ਼ਮ ਕੀ ਹੈ?

ਟਰਾਂਸਪੋਰਟ ਮਕੈਨਿਜ਼ਮ ਇਸ ਗੱਲ ਨੂੰ ਵਿਆਖਿਆ ਕਰਦਾ ਹੈ ਕਿ ਕਿਵੇਂ ਕਲਾਇੰਟ ਅਤੇ ਸਰਵਰ ਵਿਚਕਾਰ ਡਾਟਾ ਦਾ ਅਦਾਂ-ਪ੍ਰਦਾਂ ਹੁੰਦਾ ਹੈ। MCP ਵੱਖ-ਵੱਖ ਟਰਾਂਸਪੋਰਟ ਕਿਸਮਾਂ ਨੂੰ ਸਮਰਥਨ ਦਿੰਦਾ ਹੈ ਜੋ ਵੱਖ-ਵੱਖ ਮਾਹੌਲ ਅਤੇ ਲੋੜਾਂ ਲਈ ਉਚਿਤ ਹਨ:

- **stdio**: ਸਧਾਰਣ ਇਨਪੁਟ/ਆਉਟਪੁਟ, ਸਥਾਨਕ ਅਤੇ CLI-ਆਧਾਰਿਤ ਟੂਲਾਂ ਲਈ ਉਚਿਤ। ਸਾਦਾ ਪਰ ਵੈੱਬ ਜਾਂ ਕਲਾਉਡ ਲਈ ਢੁਕਵਾਂ ਨਹੀਂ।
- **SSE (ਸਰਵਰ-ਪਠਿਤ ਘਟਨਾਵਾਂ)**: HTTP ਰਾਹੀਂ ਕਲਾਇੰਟਾਂ ਨੂੰ ਰੀਅਲ-ਟਾਈਮ ਅਪਡੇਟ ਭੇਜਣ ਦੀ ਆਗਿਆ ਦਿੰਦਾ ਹੈ। ਵੈੱਬ UI ਲਈ ਵਧੀਆ, ਪਰ ਯੋਗਤਾ ਅਤੇ ਲਚਕੀਲਾਪਣ ਵਿਚ ਸੀਮਿਤ। MCP ਸਪੇਸਿਫਿਕੇਸ਼ਨ 2025-06-18 ਦੇ ਅਨੁਸਾਰ, ਖੁਦਮੁਖਤਾਰ SSE ਟਰਾਂਸਪੋਰਟ ਨੂੰ ਹਟਾਇਆ ਗਿਆ ਅਤੇ "ਸਟ੍ਰੀਮ ਕਰਨਯੋਗ HTTP" ਨਾਲ ਬਦਲ ਦਿੱਤਾ ਗਿਆ।
- **ਸਟ੍ਰੀਮ ਕਰਨਯੋਗ HTTP**: ਆਧੁਨਿਕ HTTP-ਆਧਾਰਿਤ ਸਟ੍ਰੀਮਿੰਗ ਟਰਾਂਸਪੋਰਟ, ਸੂਚਨਾਵਾਂ ਅਤੇ ਵਧੀਆ ਸਕੇਲੇਬਿਲਟੀ ਦਾ ਸਮਰਥਨ ਕਰਦਾ ਹੈ। ਜ਼ਿਆਦਾਤਰ ਉਤਪਾਦਨ ਅਤੇ ਕਲਾਉਡ ਮਾਮਲਿਆਂ ਲਈ ਸਿਫਾਰਸ਼ੀ।

### ਤੁਲਨਾਤਮਕ ਸੂਚੀ

ਹੇਠਾਂ ਦਿੱਤੀ ਤੁਲਨਾਤਮਕ ਸੂਚੀ ਦੇਖੋ ਤਾਂ ਕਿ ਇਨ੍ਹਾਂ ਟਰਾਂਸਪੋਰਟ ਮਕੈਨਿਜ਼ਮਾਂ ਵਿਚ ਅੰਤਰ ਸਮਝ ਸਕੋ:

| ਟਰਾਂਸਪੋਰਟ        | ਰੀਅਲ-ਟਾਈਮ ਅਪਡੇਟਸ | ਸਟ੍ਰੀਮਿੰਗ | ਸਕੇਲੇਬਿਲਟੀ | ਵਰਤੋਂ ਦਾ ਮਾਮਲਾ          |
|-------------------|-------------------|-----------|-------------|--------------------------|
| stdio             | ਨਹੀਂ              | ਨਹੀਂ      | ਘੱਟ        | ਸਥਾਨਕ CLI ਟੂਲ          |
| SSE               | ਹਾਂ               | ਹਾਂ       | ਮਧਯਮ      | ਵੈੱਬ, ਰੀਅਲ-ਟਾਈਮ ਅਪਡੇਟ  |
| ਸਟ੍ਰੀਮ ਕਰਨਯੋਗ HTTP | ਹਾਂ               | ਹਾਂ       | ਉੱਚ        | ਕਲਾਉਡ, ਬਹੁ-ਕਲਾਇੰਟ      |

> **ਟਿੱਪ:** ਸਹੀ ਟਰਾਂਸਪੋਰਟ ਦੀ ਚੋਣ ਕਾਰਗੁਜ਼ਾਰੀ, ਸਕੇਲੇਬਿਲਟੀ ਅਤੇ ਉਪਭੋਗਤਾ ਅਨੁਭਵ 'ਤੇ ਪ੍ਰਭਾਵ ਪਾਉਂਦੀ ਹੈ। ਆਧੁਨਿਕ, ਸਕੇਲੇਬਲ ਅਤੇ ਕਲਾਉਡ-ਤਿਆਰ ਐਪਲੀਕੇਸ਼ਨਾਂ ਲਈ **ਸਟ੍ਰੀਮ ਕਰਨਯੋਗ HTTP** ਦੀ ਸਿਫਾਰਸ਼ ਕੀਤੀ ਗਈ ਹੈ।

ਜਿਹੜੇ ਟਰਾਂਸਪੋਰਟ stdio ਅਤੇ SSE ਪਿਛਲੇ ਅਧਿਆਿਆਂ ਵਿੱਚ ਦਿਖਾਏ ਗਏ ਸਨ ਅਤੇ ਜਿਸ ਟਰਾਂਸਪੋਰਟ ਨੂੰ ਇਸ ਅਧਿਆਇ ਵਿੱਚ ਕਵਰ ਕੀਤਾ ਗਿਆ ਹੈ ਉਹ ਸਟ੍ਰੀਮ ਕਰਨਯੋਗ HTTP ਹੈ, ਜਾਣੋ।

## ਸਟ੍ਰੀਮਿੰਗ: ਧਾਰਣਾ ਅਤੇ ਪ੍ਰੇਰਣਾ

ਸਟ੍ਰੀਮਿੰਗ ਦੇ ਮੁੱਖ ਧਾਰਣਾਵਾਂ ਅਤੇ ਪ੍ਰੇਰਣਾਵਾਂ ਨੂੰ ਸਮਝਣਾ ਪ੍ਰਭਾਵਸ਼ਾਲੀ ਰੀਅਲ-ਟਾਈਮ ਸੰਚਾਰ ਪ੍ਰਣਾਲੀਆਂ ਲਾਗੂ ਕਰਨ ਲਈ ਲਾਜ਼ਮੀ ਹੈ।

**ਸਟ੍ਰੀਮਿੰਗ** ਨੈੱਟਵਰਕ ਪ੍ਰੋਗ੍ਰਾਮਿੰਗ ਦੀ ਇੱਕ ਤਕਨਾਲੋਜੀ ਹੈ ਜੋ ਡਾਟਾ ਨੂੰ ਛੋਟੇ, ਸੰਭਾਲ ਯੋਗ ਹਿੱਸਿਆਂ ਵਿੱਚ ਜਾਂ ਘਟਨਾਵਾਂ ਦੀ ਲੜੀ ਵਜੋਂ ਭੇਜਣ ਅਤੇ ਪ੍ਰਾਪਤ ਕਰਨ ਦੀ ਆਗਿਆ ਦਿੰਦੀ ਹੈ, ਬਜ਼ਾਏ ਇਹਦੇ ਕਿ ਪੂਰੇ ਜਵਾਬ ਦੇ ਤਈਆਰ ਹੋਣ ਦੀ ਉਡੀਕ ਕੀਤੀ ਜਾਵੇ। ਇਹ ਖਾਸ ਕਰਕੇ ਫਾਇਦੇਮੰਦ ਹੈ:

- ਵੱਡੇ ਫਾਈਲਾਂ ਜਾਂ ਡੇਟਾਸੈੱਟ
- ਰੀਅਲ-ਟਾਈਮ ਅਪਡੇਟ (ਉਦਾਹਰਨ ਲਈ, ਚੈਟ, ਪ੍ਰਗਤੀ ਬਾਰ)
- ਲੰਬੇ ਸਮੇਂ ਲੱਗਣ ਵਾਲੀਆਂ ਗਣਨਾਵਾਂ ਜਿੱਥੇ ਤੁਸੀਂ ਉਪਭੋਗਤਾ ਨੂੰ ਜਾਣੂ ਕਰਵਾਉਣਾ ਚਾਹੁੰਦੇ ਹੋ

ਸਟ੍ਰੀਮਿੰਗ ਬਾਰੇ ਤੁਹਾਨੂੰ ਉੱਚ-ਸਤਰ 'ਤੇ ਜਾਣਨਯੋਗ ਗੱਲਾਂ:

- ਡਾਟਾ ਲਗਾਤਾਰ ਦਿੱਤਾ ਜਾਂਦਾ ਹੈ, ਇੱਕ ਵਾਰੀ ਨਹੀਂ।
- ਕਲਾਇੰਟ ਆਉਂਦਾ ਡਾਟਾ ਪ੍ਰਕਿਰਿਆ ਕਰ ਸਕਦਾ ਹੈ।
- ਫ਼ੈਲਾਵ ਦਰਸ਼ਣ ਨੂੰ ਘਟਾਉਂਦਾ ਹੈ ਅਤੇ ਉਪਭੋਗਤਾ ਅਨੁਭਵ ਵਿਚ ਸੁਧਾਰ ਕਰਦਾ ਹੈ।

### ਸਟ੍ਰੀਮਿੰਗ ਕਿਉਂ ਵਰਤੋਂ ਕਰੀਏ?

ਸਟ੍ਰੀਮਿੰਗ ਵਰਤਣ ਦੇ ਕਾਰਣ ਇਨ੍ਹਾਂ ਹਨ:

- ਉਪਭੋਗਤਾ ਤੇਜ਼ ਫੀਡਬੈਕ ਪ੍ਰਾਪਤ ਕਰਦੇ ਹਨ, ਸਿਰਫ ਅੰਤ ਵਿੱਚ ਨਹੀਂ
- ਰੀਅਲ-ਟਾਈਮ ਐਪਲੀਕੇਸ਼ਨਾਂ ਅਤੇ ਤੁਰੰਤ ਜਵਾਬ ਦੇਣ ਵਾਲੀਆਂ UI ਨੂੰ ਯੋਗ ਬਣਾਉਂਦਾ ਹੈ
- ਨੈੱਟਵਰਕ ਅਤੇ ਕਮਿਊੱਟ ਖੇਤਰਾਂ ਦਾ ਜ਼ਿਆਦਾ ਕੁਸ਼ਲ ਵਰਤੋਂ

### ਸਾਦਾ ਉਦਾਹਰਨ: HTTP ਸਟ੍ਰੀਮਿੰਗ ਸਰਵਰ ਅਤੇ ਕਲਾਇੰਟ

ਹੇਠਾਂ ਇੱਕ ਸਾਦਾਰਨ ਉਦਾਹਰਨ ਹੈ ਕਿ ਕਿਵੇਂ ਸਟ੍ਰੀਮਿੰਗ ਲਾਗੂ ਕੀਤਾ ਜਾ ਸਕਦਾ ਹੈ:

#### ਪਾਇਥਨ

**ਸਰਵਰ (ਪਾਇਥਨ, FastAPI ਅਤੇ StreamingResponse ਦੀ ਵਰਤੋਂ ਨਾਲ):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**ਕਲਾਇੰਟ (ਪਾਇਥਨ, requests ਦੀ ਵਰਤੋਂ ਨਾਲ):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

ਇਹ ਉਦਾਹਰਨ ਦਰਸਾਉਂਦੀ ਹੈ ਕਿ ਸਰਵਰ ਕਿਵੇਂ ਕਈ ਸੁਨੇਹੇ ਕਲਾਇੰਟ ਨੂੰ ਭੇਜਦਾ ਹੈ ਜਿਵੇਂ ਹੀ ਉਹ ਉਪਲਬਧ ਹੁੰਦੇ ਹਨ, ਬਦਲੇ ਵਿਚ ਸਾਰੇ ਸੁਨੇਹੇ ਤਈਆਰ ਹੋਣ ਦੀ ਉਡੀਕ ਕਰਨ ਦੀ ਬਜਾਇ।

**ਇਹ ਕਿਵੇਂ ਕੰਮ ਕਰਦਾ ਹੈ:**

- ਸਰਵਰ ਹਰ ਸੁਨੇਹਾ ਜਿੰਨਾ ਤੇਆਰ ਹੁੰਦਾ ਹੈ ਉਦੋਂYield ਕਰਦਾ ਹੈ।
- ਕਲਾਇੰਟ ਹਰ ਹਿੱਸਾ ਪ੍ਰਾਪਤ ਕਰਦਾ ਹੈ ਅਤੇ ਛਪਵਾਉਂਦਾ ਹੈ ਜਿਵੇਂ ਹੀ ਉਹ ਆਉਂਦਾ ਹੈ।

**ਲੋੜੀਂਦਾ:** 

- ਸਰਵਰ ਨੂੰ ਸਟ੍ਰੀਮਿੰਗ ਜਵਾਬ ਵਰਤਣਾ ਜ਼ਰੂਰੀ ਹੈ (ਜਿਵੇਂ `StreamingResponse` FastAPI ਵਿੱਚ).
- ਕਲਾਇੰਟ ਜਵਾਬ ਨੂੰ ਸਟ੍ਰੀਮ ਵਜੋਂ ਪ੍ਰਕਿਰਿਆ ਕਰੇ (`stream=True` requests ਵਿੱਚ).
- ਸਮੱਗਰੀ ਕਿਸਮ ਆਮ ਤੌਰ ਤੇ `text/event-stream` ਜਾਂ `application/octet-stream` ਹੁੰਦੀ ਹੈ।

#### ਜਾਵਾ

**ਸਰਵਰ (ਜਾਵਾ, Spring Boot ਅਤੇ Server-Sent Events ਵਰਤ ਕੇ):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**ਕਲਾਇੰਟ (ਜਾਵਾ, Spring WebFlux WebClient ਵਰਤ ਕੇ):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**ਜਾਵਾ ਤਿਆਰੀ ਨੋਟਸ:**

- Spring Boot ਦੇ ਪ੍ਰਤੀਕਿਰਿਆਸ਼ੀਲ ਸਟੈਕ ਨਾਲ `Flux` ਸਟ੍ਰੀਮਿੰਗ ਲਈ ਵਰਤੀ ਜਾਂਦੀ ਹੈ
- `ServerSentEvent` ਘਟਨਾ ਪ੍ਰਕਾਰਾਂ ਨਾਲ ਸੰਰਚਿਤ ਘਟਨਾ ਸਟ੍ਰੀਮਿੰਗ ਦਿੰਦਾ ਹੈ
- `WebClient` ਦਾ `bodyToFlux()` ਪ੍ਰਤੀਕਿਰਿਆਸ਼ੀਲ ਸਟ੍ਰੀਮਿੰਗ ਖਪਤ ਯੋਗ ਬਣਾਉਂਦਾ ਹੈ
- `delayElements()` ਘਟਨਾਵਾਂ ਵਿਚਕਾਰ ਪ੍ਰਕਿਰਿਆ ਸਮਾਂ ਨਕਲ ਕਰਦਾ ਹੈ
- ਘਟਨਾਵਾਂ ਨੂੰ ਵਿੱਚ ਪ੍ਰਕਾਰ ਹੋ ਸਕਦੇ ਹਨ (`info`, `result`) ਉੱਚ-ਗੁਣਵੱਤਾ ਕਲਾਇੰਟ ਸੰਭਾਲ ਲਈ

### ਤੁਲਨਾ: ਕਲਾਸਿਕ ਸਟ੍ਰੀਮਿੰਗ ਬਨਾਮ MCP ਸਟ੍ਰੀਮਿੰਗ

ਇਹ ਦਰਸਾਉਂਦਾ ਹੈ ਕਿ ਕਲਾਸਿਕ ਸਟ੍ਰੀਮਿੰਗ ਅਤੇ MCP ਸਟ੍ਰੀਮਿੰਗ ਵਿਚਾਰ ਵਾਲੀ ਸਰਵਰ ਵਿਧੀ ਕਿਵੇਂ ਵੱਖ-ਵੱਖ ਹੈ:

| ਫੀਚਰ                | ਕਲਾਸਿਕ HTTP ਸਟ੍ਰੀਮਿੰਗ        | MCP ਸਟ੍ਰੀਮਿੰਗ (ਸੂਚਨਾਵਾਂ)    |
|------------------------|------------------------------|-------------------------------|
| ਮੁੱਖ ਜਵਾਬ              | ਹਿੱਸਿਆਂ ਵਿੱਚ                  | ਇਕੱਲਾ, ਅੰਤ ਵਿੱਚ               |
| ਪ੍ਰਗਤੀ ਅਪਡੇਟ           | ਡਾਟਾ ਹਿੱਸਿਆਂ ਵਜੋਂ ਭੇਜੇ ਜਾਂਦੇ ਹਨ | ਸੂਚਨਾਵਾਂ ਵਜੋਂ ਭੇਜੇ ਜਾਂਦੇ ਹਨ |
| ਕਲਾਇੰਟ ਲੋੜੀਂਦਾ        | ਸਟ੍ਰੀਮ ਪ੍ਰਕਿਰਿਆ ਕਰਨੀ ਚਾਹੀਦੀ ਹੈ | ਸੁਨੇਹਾ ਹੈਂਡਲਰ ਲਾਗੂ ਕਰਨਾ ਚਾਹੀਦਾ ਹੈ |
| ਵਰਤੋਂ ਦਾ ਮਾਮਲਾ         | ਵੱਡੇ ਫਾਇਲ, AI ਟੋਕਨ ਸਟ੍ਰੀਮ  | ਪ੍ਰਗਤੀ, ਲਾਗ, ਰੀਅਲ-ਟਾਈਮ ਫੀਡਬੈਕ |

### ਮੁੱਖ ਅੰਤਰ

ਹੋਰ, ਕੁਝ ਮੁੱਖ ਅੰਤਰ ਇਨ੍ਹਾਂ ਹਨ:

- **ਸੰਚਾਰ ਪੈਟਰਨ:**
  - ਕਲਾਸਿਕ HTTP ਸਟ੍ਰੀਮਿੰਗ: ਸਧਾਰਣ ਹਿੱਸਿਆਂ ਦੀ ਟਰਾਂਸਫਰ ਇੰਕੋਡਿੰਗ ਨਾਲ ਡਾਟਾ ਭੇਜਦਾ ਹੈ
  - MCP ਸਟ੍ਰੀਮਿੰਗ: JSON-RPC ਪ੍ਰੋਟੋਕੋਲ ਨਾਲ ਸੰਰਚਿਤ ਸੂਚਨਾ ਪ੍ਰਣਾਲੀ ਵਰਤਦਾ ਹੈ

- **ਸੁਨੇਹਾ ਰੂਪ:**
  - ਕਲਾਸਿਕ HTTP: ਸਧਾਰਣ ਟੈਕਸਟ ਹਿੱਸੇ ਨਵੀਂ ਲਾਈਨਾਂ ਨਾਲ
  - MCP: ਵਿਆਖਿਆਤਮਕ LoggingMessageNotification ਔਬਜੈਕਟ ਜ਼ਰੂਰੀ ਮੈਟਾਡੇਟਾ ਦੇ ਨਾਲ

- **ਕਲਾਇੰਟ ਲਾਗੂ ਕਰਨ:**
  - ਕਲਾਸਿਕ HTTP: ਇੱਕ ਸਧਾਰਨ ਕਲਾਇੰਟ ਜੋ ਸਟ੍ਰੀਮ ਜਵਾਬ ਪ੍ਰਕਿਰਿਆ ਕਰਦਾ ਹੈ
  - MCP: ਇੱਕ ਹੋਰ ਪ੍ਰਗਟ ਮੈਂਬਰ ਕਲਾਇੰਟ ਜਿਸ ਨੂੰ ਵੱਖ-ਵੱਖ ਕਿਸਮਾਂ ਦੇ ਸੁਨੇਹੇ ਸੰਭਾਲਣ ਲਈ ਸੁਨੇਹਾ ਹੈਂਡਲਰ ਹੈ

- **ਪ੍ਰਗਤੀ ਅਪਡੇਟਸ:**
  - ਕਲਾਸਿਕ HTTP: ਪ੍ਰਗਤੀ ਮੁੱਖ ਜਵਾਬ ਸਟ੍ਰੀਮ ਦਾ ਹਿੱਸਾ ਹੁੰਦੀ ਹੈ
  - MCP: ਪ੍ਰਗਤੀ ਵੱਖ-ਵੱਖ ਸੂਚਨਾ ਸੁਨੇਹਿਆਂ ਰਾਹੀਂ ਭੇਜੀ ਜਾਂਦੀ ਹੈ ਜਦਕਿ ਮੁੱਖ ਜਵਾਬ ਅੰਤ ਵਿੱਚ ਆਉਂਦਾ ਹੈ

### ਸਿਫਾਰਸ਼ਾਂ

ਇੱਥੇ ਕੁਝ ਸਿਫਾਰਸ਼ਾਂ ਹਨ ਜਦੋਂ ਤੁਸੀਂ ਕਲਾਸਿਕ ਸਟ੍ਰੀਮਿੰਗ (`/stream` ਵਰਗਾ ਏਂਡਪੋਇੰਟ) ਨੂੰ MCP ਸਟ੍ਰੀਮਿੰਗ ਨਾਲ ਮੁਕਾਬਲਾ ਕਰਦੇ ਹੋ।

- **ਸਧਾਰਨ ਸਟ੍ਰੀਮਿੰਗ ਲਈ:** ਕਲਾਸਿਕ HTTP ਸਟ੍ਰੀਮਿੰਗ ਅਸਾਨ਼ ਹੈ ਅਤੇ ਮੂਲ ਸਟ੍ਰੀਮਿੰਗ ਲੋੜਾਂ ਲਈ ਕਾਫ਼ੀ ਹੈ।

- **ਜਟਿਲ, ਇੰਟਰਐਕਟਿਵ ਐਪਲੀਕੇਸ਼ਨਾਂ ਲਈ:** MCP ਸਟ੍ਰੀਮਿੰਗ ਇੱਕ ਸੂਚਿਤ ਦ੍ਰਿਸ਼ਟੀਕੋਣ ਦਿੰਦਾ ਹੈ ਜਿਸ ਵਿੱਚ ਤਜ਼ੀਨਮੈਟਿਕ ਮੈਟਾਡੇਟਾ ਅਤੇ ਸੂਚਨਾਵਾਂ ਅਤੇ ਅੰਤਿਮ ਨਤੀਜੇ ਵਿਚਕਾਰ ਵੱਖ-ਵੱਖ ਕਰਦਾ ਹੈ।

- **AI ਐਪਲੀਕੇਸ਼ਨਾਂ ਲਈ:** MCP ਦੀ ਸੂਚਨਾ ਪ੍ਰਣਾਲੀ ਲੰਬੇ ਸਮੇਂ ਚੱਲਣ ਵਾਲੇ AI ਕਾਰਜਾਂ ਲਈ ਖਾਸ ਤੌਰ 'ਤੇ ਲਾਭਕਾਰੀ ਹੈ ਜਿੱਥੇ ਤੁਸੀਂ ਵਰਤੋਂਕਾਰਾਂ ਨੂੰ ਪ੍ਰਗਤੀ ਦਾ ਜਾਣੂ ਕਰਵਾਉਣਾ ਚਾਹੁੰਦੇ ਹੋ।

## MCP ਵਿੱਚ ਸਟ੍ਰੀਮਿੰਗ

ਠੀਕ ਹੈ, ਤੁਸੀਂ ਹੁਣ ਤੱਕ ਕੁਝ ਸਿਫਾਰਸ਼ਾਂ ਅਤੇ ਤੁਲਨਾਵਾਂ ਦੇਖੀਆਂ ਹਨ ਕਿ ਕਲਾਸਿਕ ਸਟ੍ਰੀਮਿੰਗ ਅਤੇ MCP ਸਟ੍ਰੀਮਿੰਗ ਵਿੱਚ ਅੰਤਰ ਕਿਵੇਂ ਹੈ। ਹੁਣ ਦੇਖਦੇ ਹਾਂ ਕਿ ਤੁਸੀਂ MCP ਵਿੱਚ ਸਟ੍ਰੀਮਿੰਗ ਦਾ ਕਿਵੇਂ ਲਾਭ ਉਠਾ ਸਕਦੇ ਹੋ।

MCP ਫਰੇਮਵਰਕ ਵਿੱਚ ਸਟ੍ਰੀਮਿੰਗ ਕਿਵੇਂ ਕੰਮ ਕਰਦੀ ਹੈ ਇਹ ਜਾਣਣਾ ਜਰੂਰੀ ਹੈ ਤਾਂ ਜੋ ਤੁਸੀਂ ਲੰਬੇ ਸਮੇਂ ਚੱਲਣ ਵਾਲੇ ਕਾਰਜਾਂ ਦੌਰਾਨ ਉਪਭੋਗਤਾਵਾਂ ਨੂੰ ਤੁਰੰਤ ਫੀਡਬੈਕ ਦੇਣ ਵਾਲੀਆਂ ਜਵਾਬਦਾਰ ਐਪਲੀਕੇਸ਼ਨਾਂ ਬਣਾ ਸਕੋ।

MCP ਵਿੱਚ, ਸਟ੍ਰੀਮਿੰਗ ਮੁੱਖ ਜਵਾਬ ਨੂੰ ਹਿੱਸਿਆਂ ਵਿੱਚ ਭੇਜਣ ਬਾਰੇ ਨਹੀਂ ਹੈ, ਪਰ ਸੰਦ ਪ੍ਰਕਿਰਿਆ ਕਰਦਾ ਸਮੇਂ ਕਲਾਇੰਟ ਨੂੰ **ਸੂਚਨਾਵਾਂ** ਭੇਜਣ ਬਾਰੇ ਹੈ। ਇਹ ਸੂਚਨਾਵਾਂ ਪ੍ਰਗਤੀ ਅਪਡੇਟ, ਲਾਗ ਜਾ ਹੋਰ ਘਟਨਾਵਾਂ ਸ਼ਾਮਲ ਕਰ ਸਕਦੀਆਂ ਹਨ।

### ਇਹ ਕਿਵੇਂ ਕੰਮ ਕਰਦਾ ਹੈ

ਮੁੱਖ ਨਤੀਜਾ ਫਿਰ ਵੀ ਇਕੱਲੇ ਜਵਾਬ ਵਜੋਂ ਭੇਜਿਆ ਜਾਂਦਾ ਹੈ। ਪਰ ਸੂਚਨਾਵਾਂ ਪ੍ਰਕਿਰਿਆ ਦੌਰਾਨ ਵੱਖ-ਵੱਖ ਸੁਨੇਹਿਆਂ ਵਜੋਂ ਭੇਜੀਆਂ ਜਾ ਸਕਦੀਆਂ ਹਨ ਅਤੇ ਇਸ ਤਰ੍ਹਾਂ ਕਲਾਇੰਟ ਨੂੰ ਰੀਅਲ-ਟਾਈਮ ਵਿੱਚ ਅਪਡੇਟ ਕੀਤਾ ਜਾਂਦਾ ਹੈ। ਕਲਾਇੰਟ ਨੂੰ ਇਹ ਸੂਚਨਾਵਾਂ ਸੰਭਾਲਣ ਅਤੇ ਦਿਖਾਉਣ ਯੋਗ ਹੋਣਾ ਚਾਹੀਦਾ ਹੈ।

## ਸੂਚਨਾ ਕੀ ਹੈ?

ਅਸੀਂ ਕਿਹਾ "ਸੂਚਨਾ", MCP ਸੰਦਰਭ ਵਿੱਚ ਇਸਦਾ ਕੀ ਅਰਥ ਹੈ?

ਸੂਚਨਾ ਉਹ ਸੁਨੇਹਾ ਹੈ ਜੋ ਸਰਵਰ ਵੱਲੋਂ ਕਲਾਇੰਟ ਨੂੰ ਭੇਜਿਆ ਜਾਂਦਾ ਹੈ ਤਾਂ ਜੋ ਪ੍ਰਗਤੀ, ਸਥਿਤੀ ਜਾਂ ਲੰਬੇ ਸਮੇਂ ਚੱਲਣ ਵਾਲੇ ਕਾਰਜ ਦੌਰਾਨ ਹੋਰ ਘਟਨਾਵਾਂ ਦੀ ਜਾਣਕਾਰੀ ਦਿੱਤੀ ਜਾ ਸਕੇ। ਸੂਚਨਾਵਾਂ ਪਾਰਦਰਸ਼ਤਾ ਅਤੇ ਉਪਭੋਗਤਾ ਅਨੁਭਵ ਨੂੰ ਬਿਹਤਰ ਬਣਾਉਂਦੀਆਂ ਹਨ।

ਉਦਾਹਰਨ ਲਈ, ਇੱਕ ਕਲਾਇੰਟ ਸੂਚਨਾ ਭੇਜਦਾ ਹੈ ਜਦੋਂ ਸਰਵਰ ਨਾਲ ਮੁੱਢਲਾ ਹੱਥ-ਮਿਲਾਉਣ ਸਮਾਪਤ ਹੁੰਦਾ ਹੈ।

ਇੱਕ ਸੂਚਨਾ ਜਾਣਕਾਰੀ ਦੀ ਤਰ੍ਹਾਂ ਇੱਕ JSON ਸੁਨੇਹੇ ਵਾਂਗ ਦਿੱਤੀ ਜਾਂਦੀ ਹੈ:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

ਸੂਚਨਾਵਾਂ MCP ਵਿੱਚ ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) ਨਾਮਕ ਵਿਸ਼ੇ ਤਹਿਤ ਆਉਂਦੀਆਂ ਹਨ।

> **ਮਿਆਦ ਉਤਾਰਨ ਸੂਚਨਾ:** `2026-07-28` MCP ਵਿਸ਼ੇਸ਼ ਰਿਲੀਜ਼ ਉਮੀਦਵਾਰ ਵਿੱਚ Logging ਪ੍ਰ primitਵ ਤੋੜ ਦਿੱਤੀ ਗਈ ਹੈ ਅਤੇ stdio ਟਰਾਂਸਪੋਰਟਾਂ ਲਈ `stderr` ਅਤੇ ਸੰਰਚਿਤ ਨਿਗਰਾਨੀ ਲਈ OpenTelemetry ਨੂੰ ਵਰਤੀ ਜਾ ਰਹੀ ਹੈ। Logging `2025-11-25` ਵਿੱਚ ਅਤੇ ਕਿਸੇ ਵੀ ਅਧਿਕਾਰਿਕ ਮਿਆਦ ਉਤਾਰਨ ਤੋਂ ਇੱਕ ਸਾਲ ਤੋਂ ਵੱਧ ਸਮੇਂ ਲਈ ਕੰਮ ਕਰਦੀ ਰਹੇਗੀ। ਵੇਖੋ [What's Changing in MCP: The 2026-07-28 Release Candidate](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) ਵਿਸਥਾਰ ਲਈ।

ਲਾਗਿੰਗ ਨੂੰ ਕੰਮ ਵਿੱਚ ਲਿਆਉਣ ਲਈ, ਸਰਵਰ ਨੂੰ ਇਸਨੂੰ ਫੀਚਰ/ਸਮਰੱਥਾ ਵਜੋਂ ਯੋਗ ਕਰਨਾ ਚਾਹੀਦਾ ਹੈ ਜਿਵੇਂ:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> ਵਰਤੇ ਗਏ SDK ਦੇ ਅਨੁਸਾਰ, ਲਾਗਿੰਗ ਡਿਫ਼ਾਲਟ ਰੂਪ ਵਿੱਚ ਯੋਗ ਹੋ ਸਕਦੀ ਹੈ ਜਾਂ ਤੁਹਾਨੂੰ ਆਪਣੀ ਸਰਵਰ ਕੰਫਿਗਰੇਸ਼ਨ ਵਿੱਚ ਇਸ ਦੀ ਸਪਸ਼ਟ ਤੌਰ 'ਤੇ ਸੈੱਟਿੰਗ ਕਰਨ ਦੀ ਲੋੜ ਹੋ ਸਕਦੀ ਹੈ।

ਸੂਚਨਾਵਾਂ ਦੇ ਵੱਖ-ਵੱਖ ਕਿਸਮਾਂ ਹਨ:

| ਪੱਧਰ       | ਵਰਣਨ                     | ਉਦਾਹਰਨ ਵਰਤੋਂ ਦਾ ਮਾਮਲਾ       |
|-------------|----------------------------|------------------------------|
| debug       | ਵਿਸਥਾਰਪੂਰਵਕ ਡੀਬੱਗਿੰਗ ਜਾਣਕਾਰੀ | ਫੰਕਸ਼ਨ ਪ੍ਰਵੇਸ਼/ਬਾਹਰ ਨਿਕਲਣਾ    |
| info        | ਆਮ ਜਾਣਕਾਰੀ ਸੁਨੇਹੇ          | ਕਾਰਜ ਦੀ ਪ੍ਰਗਤੀ ਅਪਡੇਟ          |
| notice      | ਨਾਰਮਲ ਪਰ ਮਹੱਤਵਪੂਰਨ ਘਟਨਾਵਾਂ | ਸੰਰਚਨਾ ਬਦਲਾਵ               |
| warning     | ਚੇਤਾਵਨੀ ਦੀਆਂ ਸਥਿਤੀਆਂ      | ਅਪਰੇਟਿੰਗ ਫੀਚਰ ਦੀ ਵਰਤੋਂ      |
| error       | ਗਲਤੀ ਦੀਆਂ ਸਥਿਤੀਆਂ          | ਕਾਰਜ ਦੀ ਅਸਫਲਤਾਵਾਂ           |
| critical    | ਗੰਭੀਰ ਸਥਿਤੀਆਂ            | ਪ੍ਰਣਾਲੀ ਘਟਕ ਦੀਆਂ ਅਸਫਲਤਾਵਾਂ  |
| alert       | ਤੁਰੰਤ ਕਾਰਵਾਈ ਲੋੜੀਂਦੀ ਹੈ    | ਡਾਟਾ ਵਿਗੜਨ ਦਾ ਪਤਾ ਲੱਗਣਾ     |
| emergency   | ਸਿਸਟਮ ਅਣਉਪਯੋਗ ਹੈ           | ਪੂਰਾ ਸਿਸਟਮ ਫੇਲ              |

## MCP ਵਿੱਚ ਸੂਚਨਾਵਾਂ ਲਾਗੂ ਕਰਨਾ

MCP ਵਿੱਚ ਸੂਚਨਾਵਾਂ ਲਾਗੂ ਕਰਨ ਲਈ ਤੁਹਾਨੂੰ ਦੋਹਾਂ ਸਰਵਰ ਅਤੇ ਕਲਾਇੰਟ ਪਾਸਿਆਂ ਨੂੰ ਰੀਅਲ-ਟਾਈਮ ਅਪਡੇਟ ਸੰਭਾਲਣ ਯੋਗ ਬਣਾਉਣਾ ਪੈਂਦਾ ਹੈ। ਇਹ ਤੁਹਾਡੇ ਐਪਲੀਕੇਸ਼ਨ ਨੂੰ ਲੰਬੇ ਸਮੇਂ ਚੱਲਣ ਵਾਲੇ ਕਾਰਜ ਦੌਰਾਨ ਤੁਰੰਤ ਪ੍ਰਤੀਕਿਰਿਆ ਦਿੰਦਾ ਹੈ।

### ਸਰਵਰ-ਪਾਸਾ: ਸੂਚਨਾਵਾਂ ਭੇਜਣਾ

ਆਓ ਸਰਵਰ ਪਾਸੇ ਤੋਂ ਸ਼ੁਰੂ ਕਰੀਏ। MCP ਵਿੱਚ ਤੁਸੀਂ ਉਪਕਰਣ ਪਰਿਭਾਸ਼ਿਤ ਕਰਦੇ ਹੋ ਜੋ ਬੇਨਤੀਆਂ ਪ੍ਰਕਿਰਿਆ ਕਰਦੇ ਸਮੇਂ ਸੂਚਨਾਵਾਂ ਭੇਜ ਸਕਦੇ ਹਨ। ਸਰਵਰ ਸੰਦੇਸ਼ ਕਲਾਇੰਟ ਨੂੰ ਭੇਜਣ ਲਈ ਪ੍ਰਸੰਗ ਵਸਤੂ (ਅਕਸਰ `ctx`) ਦੀ ਵਰਤੋਂ ਕਰਦਾ ਹੈ।

#### ਪਾਇਥਨ

```python
@mcp.tool(description="A tool that sends progress notifications")
```

### ਕਲਾਇੰਟ-ਪਾਸਾ: ਸੂਚਨਾਵਾਂ ਪ੍ਰਾਪਤ ਕਰਨਾ

## ਪ੍ਰਗਤੀ ਸੂਚਨਾਵਾਂ ਅਤੇ ਸਥਿਤੀਆਂ

ਇਹ ਭਾਗ MCP ਵਿੱਚ ਪ੍ਰਗਤੀ ਸੂਚਨਾਵਾਂ ਦੀ ਧਾਰਣਾ, ਇਹ ਕਿਉਂ ਮਹੱਤਵਪੂਰਨ ਹੈ, ਅਤੇ ਸਟ੍ਰੀਮ ਕਰਨਯੋਗ HTTP ਦੀ ਵਰਤੋਂ ਨਾਲ ਇਨ੍ਹਾਂ ਨੂੰ ਕਿਵੇਂ ਲਾਗੂ ਕਰਨਾ ਹੈ ਇਹ ਸਮਝਾਉਂਦਾ ਹੈ। ਤੁਸੀਂ ਆਪਣੇ ਸਮਝ ਨੂੰ ਮਜ਼ਬੂਤ ਕਰਨ ਲਈ ਇਕ ਵਿਹਾਰਕ ਅਸਾਈਨਮੈਂਟ ਵੀ ਪਾਓਗੇ।

ਪ੍ਰਗਤੀ ਸੂਚਨਾਵਾਂ ਲੰਬੇ ਸਮੇਂ ਚੱਲਣ ਵਾਲੇ ਕਾਰਜਾਂ ਦੌਰਾਨ ਸਰਵਰ ਤੋਂ ਕਲਾਇੰਟ ਨੂੰ ਭੇਜੇ ਜਾਣ ਵਾਲੇ ਰੀਅਲ-ਟਾਈਮ ਸੁਨੇਹੇ ਹੁੰਦੇ ਹਨ। ਪੂਰੇ ਪ੍ਰਕਿਰਿਆ ਦੇ ਖਤਮ ਹੋਣ ਦੀ ਉਡੀਕ ਕਰਨ ਦੀ ਬਜਾਇ, ਸਰਵਰ ਕਲਾਇੰਟ ਨੂੰ ਵਰਤਮਾਨ ਸਥਿਤੀ ਬਾਰੇ ਅਪਡੇਟ ਜਾਰੀ ਰੱਖਦਾ ਹੈ। ਇਸ ਨਾਲ ਪਾਰਦਰਸ਼ਤਾ, ਉਪਭੋਗਤਾ ਅਨੁਭਵ ਬਿਹਤਰ ਹੁੰਦਾ ਹੈ ਅਤੇ ਡੀਬੱਗਿੰਗ ਆਸਾਨ ਹੁੰਦੀ ਹੈ।

### ਪ੍ਰਗਤੀ ਸੂਚਨਾਵਾਂ ਕਿਉਂ ਵਰਤੋਂ?

ਪ੍ਰਗਤੀ ਸੂਚਨਾਵਾਂ ਕਈ ਕਾਰਨਾਂ ਕਰਕੇ ਜ਼ਰੂਰੀ ਹਨ:

- **ਉਤਮ ਉਪਭੋਗਤਾ ਅਨੁਭਵ:** ਵਰਤੋਂਕਾਰ ਕੰਮ ਚੱਲਦਿਆਂ ਅਦਾਲਤੀ ਅਪਡੇਟ ਵੇਖਦੇ ਹਨ, ਸਿਰਫ ਅੰਤ ਵਿੱਚ ਨਹੀਂ।
- **ਰੀਅਲ-ਟਾਈਮ ਫੀਡਬੈਕ:** ਕਲਾਇੰਟ ਪ੍ਰਗਤੀ ਬਾਰ ਜਾਂ ਲਾਗ ਦਿਖਾ ਸਕਦੇ ਹਨ, ਜਿਨ੍ਹਾਂ ਨਾਲ ਐਪ ਜਵਾਬਦੀਹੀ ਮਹਿਸੂਸ ਹੁੰਦੀ ਹੈ।
- **ਅਸਾਨ ਡੀਬੱਗਿੰਗ ਅਤੇ ਨਿਗਰਾਨੀ:** ਵਿਕਾਸਕਾਰ ਅਤੇ ਵਰਤੋਂਕਾਰ ਦੇਖ ਸਕਦੇ ਹਨ ਕਿ ਕਾਰਜ ਕਿੱਥੇ ਸੁਸਤ ਜਾਂ ਅਟਕ ਰਿਹਾ ਹੈ।

### ਪ੍ਰਗਤੀ ਸੂਚਨਾਵਾਂ ਕਿਵੇਂ ਲਾਗੂ ਕਰਨੀ ਹੈ

- **ਸਰਵਰ ਤੇ:** ਹਰ ਆਈਟਮ ਪ੍ਰਕਿਰਿਆ ਕਰਨ ਦੌਰਾਨ `ctx.info()` ਜਾਂ `ctx.log()` ਨਾਲ ਸੂਚਨਾਵਾਂ ਭੇਜੋ। ਇਹ ਮੁੱਖ ਨਤੀਜੇ ਤੋਂ ਪਹਿਲਾਂ ਕਲਾਇੰਟ ਨੂੰ ਸੁਨੇਹਾ ਭੇਜਦਾ ਹੈ।
- **ਕਲਾਇੰਟ ਤੇ:** ਇੱਕ ਸੁਨੇਹਾ ਹੈਂਡਲਰ ਲਾਗੂ ਕਰੋ ਜੋ ਆਉਣ ਵਾਲੀਆਂ ਸੂਚਨਾਵਾਂ ਨੂੰ ਸੁਣਦਾ ਅਤੇ ਦਿਖਾਉਂਦਾ ਹੈ। ਇਹ ਹੈਂਡਲਰ ਸੂਚਨਾਵਾਂ ਅਤੇ ਅੰਤਿਮ ਨਤੀਜੇ ਵਿੱਚ ਫਰਕ ਕਰਦਾ ਹੈ।

## ਸੁਰੱਖਿਆ ਬਾਰੇ ਵਿਚਾਰ

ਜਦੋਂ MCP ਸਰਵਰਾਂ ਨੂੰ HTTP-ਅਧਾਰਿਤ ਟ੍ਰਾਂਸਪੋਰਟ ਨਾਲ ਲਾਗੂ ਕੀਤਾ ਜਾਂਦਾ ਹੈ, ਤਾਂ ਸੁਰੱਖਿਆ ਇੱਕ ਅਹੰਕਾਰਪੂਰਕ ਮੁੱਦਾ ਬਣ ਜਾਂਦੀ ਹੈ ਜਿਸ ਲਈ ਕਈ ਹਮਲਾਵਰ ਵੈਕਟਰਾਂ ਅਤੇ ਸੁਰੱਖਿਆ ਤਕਨੀਕਾਂ ਨੂੰ ਧਿਆਨ ਨਾਲ ਦੇਖਣ ਦੀ ਲੋੜ ਹੁੰਦੀ ਹੈ।

### ਸੰਖੇਪ

ਜਦੋਂ MCP ਸਰਵਰਾਂ ਨੂੰ HTTP ਰਾਹੀਂ ਖੋਲ੍ਹਿਆ ਜਾਂਦਾ ਹੈ ਤਾਂ ਸੁਰੱਖਿਆ ਬਹੁਤ ਜ਼ਰੂਰੀ ਹੈ। ਸਟ੍ਰੀਮੇਬਲ HTTP ਨਵੇਂ ਹਮਲੇ ਦੇ ਸਤਹ ਪੇਸ਼ ਕਰਦਾ ਹੈ ਅਤੇ ਧਿਆਨਪੂਰਵਕ ਸੰਰਚਨਾ ਦੀ ਲੋੜ ਹੈ।

### ਮੁੱਖ ਬਿੰਦੂ

- **ਅਸਲ ਸਿਰਲੇਖ ਦੀ ਜਾਂਚ**: DNS ਰੀਬਾਈਂਡਿੰਗ ਹਮਲਿਆਂ ਨੂੰ ਰੋਕਣ ਲਈ ਹਮੇਸ਼ਾ `Origin` ਸਿਰਲੇਖ ਦੀ ਜਾਂਚ ਕਰੋ।
- **ਲੋਕਲਹੋਸਟ ਬਿੰਡਿੰਗ**: ਸਥਾਨਕ ਵਿਕਾਸ ਲਈ ਸਰਵਰਾਂ ਨੂੰ `localhost` ਨਾਲ ਬਾਂਧੋ ਤਾਂ ਕਿ ਉਹ ਸਰਵਜਨਕ ਇੰਟਰਨੈੱਟ ਤੇ ਖੁੱਲ੍ਹੇ ਨਾ ਹੋਣ।
- **ਪਛਾਣ ਪ੍ਰਮਾਣਿਕਤਾ**: ਉਤਪਾਦਨ ਤਾਇਨਾਤੀ ਲਈ ਪ੍ਰਮਾਣਿਕਤਾ (ਜਿਵੇਂ API ਕੁੰਜੀਆਂ, OAuth) ਲਾਗੂ ਕਰੋ।
- **CORS**: ਪਹੁੰਚ ਨੂੰ ਸੀਮਤ ਕਰਨ ਲਈ ਕ੍ਰਾਸ-ਓਰਿਜਿਨ ਰਿਸੋਰਸ ਸਾਂਝਾ ਕਰਨ (CORS) ਨੀਤੀਆਂ ਸੈੱਟ ਕਰੋ।
- **HTTPS**: ਟ੍ਰੈਫਿਕ ਨੂੰ ਐਨਕ੍ਰਿਪਟ ਕਰਨ ਲਈ ਉਤਪਾਦਨ ਵਿੱਚ HTTPS ਵਰਤੋਂ।

### ਚੰਗੀਆਂ ਅਭਿਆਸ

- ਆਉਂਦੀਆਂ ਬੇਨਤੀਆਂ 'ਤੇ ਬਿਨਾਂ ਜਾਂਚ ਉੱਤੇ ਕਦੇ ਭਰੋਸਾ ਨਾ ਕਰੋ।
- ਸਾਰੀਆਂ ਪਹੁੰਚਾਂ ਅਤੇ ਗਲਤੀਆਂ ਨੂੰ ਲਾਗ ਕਰੋ ਅਤੇ ਨਿਗਰਾਨੀ ਕਰੋ।
- ਸੁਰੱਖਿਆ ਖਾਮੀਆਂ ਮਿਟਾਉਣ ਲਈ ਨਿਯਮਤ ਤੌਰ ਤੇ ਡਿਪੈਂਡੇਂਸੀਆਂ ਅਪਡੇਟ ਕਰੋ।

### ਚੁਣੌਤੀਆਂ

- ਵਿਕਾਸ ਦੀ ਸੌਖੀਅਤ ਨਾਲ ਸੁਰੱਖਿਆ ਵਿਚ ਸੰਤੁਲਨ ਬਣਾਉਣਾ
- ਵੱਖ-ਵੱਖ ਗਾਹਕ ਵਾਤਾਵਰਨਾਂ ਨਾਲ ਅਨੁਕੂਲਤਾ ਨੂੰ ਯਕੀਨੀ ਬਣਾਉਣਾ

## SSE ਤੋਂ ਸਟ੍ਰੀਮੇਬਲ HTTP ਨੂੰ ਅੱਪਗ੍ਰੇਡ ਕਰਨਾ

ਲਈ ਜੋ ਐਪਲੀਕੇਸ਼ਨ ਇਸ ਵੇਲੇ ਸਰਵਰ-ਸੇਂਟ ਇਵੈਂਟਸ (SSE) ਵਰਤ ਰਹੇ ਹਨ, ਸਟ੍ਰੀਮੇਬਲ HTTP ਨੂੰ ਮਾਈਗ੍ਰੇਟ ਕਰਨ ਨਾਲ ਤੁਹਾਡੇ MCP ਇੰਪਲੀਮੈਂਟੇਸ਼ਨਾਂ ਲਈ ਵਧੇਰੇ ਸਮਰੱਥਾ ਅਤੇ ਲੰਬੇ ਸਮੇਂ ਦਾ ਸਥਿਰਤਾ ਮਿਲਦੀ ਹੈ।

### ਇਹ ਕਿਉਂ ਅੱਪਗ੍ਰੇਡ ਕਰੀਏ?

SSE ਤੋਂ ਸਟ੍ਰੀਮੇਬਲ HTTP ਨੂੰ ਅੱਪਗ੍ਰੇਡ ਕਰਨ ਦੇ ਦੋ ਮੁੱਖ ਕਾਰਨ ਹਨ:

- ਸਟ੍ਰੀਮੇਬਲ HTTP ਬਿਹਤਰ ਸਕੇਲਯਬਿਲਟੀ, ਅਨੁਕੂਲਤਾ ਅਤੇ ਧਨਾਢੁ ਸੂਚਨਾ ਸਹਾਇਤਾ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ।
- ਇਹ ਨਵੇਂ MCP ਐਪਲੀਕੇਸ਼ਨਾਂ ਲਈ ਸਿਫਾਰਸ਼ੀ ਟ੍ਰਾਂਸਪੋਰਟ ਹੈ।

### ਮਾਈਗ੍ਰੇਸ਼ਨ ਕਦਮ

ਇੱਥੇ ਦੱਸਿਆ ਗਿਆ ਹੈ ਕਿ ਤੁਸੀਂ ਆਪਣੀਆਂ MCP ਐਪਲੀਕੇਸ਼ਨਾਂ ਵਿੱਚ SSE ਤੋਂ ਸਟ੍ਰੀਮੇਬਲ HTTP ਕਿਵੇਂ ਮਾਈਗ੍ਰੇਟ ਕਰ ਸਕਦੇ ਹੋ:

- **ਸਰਵਰ ਕੋਡ ਅਪਡੇਟ ਕਰੋ** ਤਾਂ ਜੋ `mcp.run()` ਵਿੱਚ `transport="streamable-http"` ਵਰਤੋਂ।
- **ਕਲਾਇੰਟ ਕੋਡ ਅਪਡੇਟ ਕਰੋ** ਤਾਂ ਜੋ SSE ਕਲਾਇੰਟ ਦੀ ਬਜਾਏ `streamablehttp_client` ਵਰਤੋਂ।
- **ਮੈसेਜ ਹੈਂਡਲਰ ਲਾਗੂ ਕਰੋ** ਕਲਾਇੰਟ ਵਿੱਚ ਸੂਚਨਾਵਾਂ ਪ੍ਰਕਿਰਿਆ ਲਈ।
- **ਮੌਜੂਦਾ ਟੂਲਾਂ ਅਤੇ ਵਰਕਫਲੋਜ਼ ਨਾਲ ਸੁਮੇਲਤਾ ਦੀ ਜਾਂਚ ਕਰੋ।**

### ਅਨੁਕੂਲਤਾ ਬਣਾਈ ਰੱਖਣਾ

ਮਾਈਗ੍ਰੇਸ਼ਨ ਦੌਰਾਨ ਮੌਜੂਦਾ SSE ਕਲਾਇੰਟਾਂ ਨਾਲ ਅਨੁਕੂਲਤਾ ਬਣਾਈ ਰੱਖਣ ਦੀ ਸਿਫਾਰਸ਼ ਕੀਤੀ ਜਾਂਦੀ ਹੈ। ਇੱਥੇ ਕੁਝ ਤਰਕ ਅਹਿਰੀ ਹਨ:

- ਤੁਸੀਂ ਦੋਹਾਂ SSE ਅਤੇ ਸਟ੍ਰੀਮੇਬਲ HTTP ਦਾ ਸਮਰਥਨ ਵੱਖ-ਵੱਖ ਮੁੱਖ ਬਿੰਦੂਆਂ 'ਤੇ ਦੋਹਾਂ ਟ੍ਰਾਂਸਪੋਰਟ ਚਲਾਕੇ ਕਰ ਸਕਦੇ ਹੋ।
- ਗਾਹਕਾਂ ਨੂੰ ਨਵੇਂ ਟ੍ਰਾਂਸਪੋਰਟ 'ਤੇ ਹੌਲੀ-ਹੌਲੀ ਲਿਜਾਓ।

### ਚੁਣੌਤੀਆਂ

ਮਾਈਗ੍ਰੇਸ਼ਨ ਦੌਰਾਨ ਹੇਠਾਂ ਦਿੱਤੀਆਂ ਚੁਣੌਤੀਆਂ ਨੂੰ ਹੱਲ ਕਰੋ:

- ਇਹ ਯਕੀਨੀ ਬਣਾਉਣਾ ਕਿ ਸਾਰੇ ਗਾਹਕ ਅਪਡੇਟ ਹੋ ਚੁੱਕੇ ਹਨ
- ਸੂਚਨਾ ਪ੍ਰਦਾਨੀ ਵਿੱਚ ਫਰਕਾਂ ਨੂੰ ਹਮਭਾਲਣਾ

## ਸੁਰੱਖਿਆ ਬਾਰੇ ਵਿਚਾਰ

ਕਿਸੇ ਵੀ ਸਰਵਰ, ਖਾਸ ਕਰਕੇ MCP ਵਿੱਚ ਸਟ੍ਰੀਮੇਬਲ HTTP ਵਰਗੇ HTTP-ਅਧਾਰਿਤ ਟ੍ਰਾਂਸਪੋਰਟ ਦੀ ਵਰਤੋਂ ਕਰਦਿਆਂ ਸੁਰੱਖਿਆ ਸਿਖਰ ਦੀ ਪਹਿਲਾਰ ਹੋਣੀ ਚਾਹੀਦੀ ਹੈ।

ਜਦ MCP ਸਰਵਰਾਂ ਨੂੰ HTTP-ਅਧਾਰਿਤ ਟ੍ਰਾਂਸਪੋਰਟ ਨਾਲ ਲਾਗੂ ਕੀਤਾ ਜਾਂਦਾ ਹੈ, ਤਾਂ ਸੁਰੱਖਿਆ ਇੱਕ ਅਹੰਕਾਰਪੂਰਕ ਮੁੱਦਾ ਬਣ ਜਾਂਦੀ ਹੈ ਜਿਸ ਲਈ ਕਈ ਹਮਲਾਵਰ ਵੈਕਟਰਾਂ ਅਤੇ ਸੁਰੱਖਿਆ ਤਕਨੀਕਾਂ ਨੂੰ ਧਿਆਨ ਨਾਲ ਦੇਖਣ ਦੀ ਲੋੜ ਹੁੰਦੀ ਹੈ।

### ਸੰਖੇਪ

ਜਦ MCP ਸਰਵਰਾਂ ਨੂੰ HTTP ਰਾਹੀਂ ਖੋਲ੍ਹਿਆ ਜਾਂਦਾ ਹੈ ਤਾਂ ਸੁਰੱਖਿਆ ਬਹੁਤ ਜ਼ਰੂਰੀ ਹੈ। ਸਟ੍ਰੀਮੇਬਲ HTTP ਨਵੇਂ ਹਮਲੇ ਦੇ ਸਤਹ ਪੇਸ਼ ਕਰਦਾ ਹੈ ਅਤੇ ਧਿਆਨਪੂਰਵਕ ਸੰਰਚਨਾ ਦੀ ਲੋੜ ਹੈ।

ਇੱਥੇ ਕੁਝ ਮੁੱਖ ਸੁਰੱਖਿਆ ਬਾਰੇ ਵਿਚਾਰ ਹਨ:

- **ਅਸਲ ਸਿਰਲੇਖ ਦੀ ਜਾਂਚ**: DNS ਰੀਬਾਈਂਡਿੰਗ ਹਮਲਿਆਂ ਨੂੰ ਰੋਕਣ ਲਈ ਹਮੇਸ਼ਾ `Origin` ਸਿਰਲੇਖ ਦੀ ਜਾਂਚ ਕਰੋ।
- **ਲੋਕਲਹੋਸਟ ਬਿੰਡਿੰਗ**: ਸਥਾਨਕ ਵਿਕਾਸ ਲਈ ਸਰਵਰਾਂ ਨੂੰ `localhost` ਨਾਲ ਬਾਂਧੋ ਤਾਂ ਕਿ ਉਹ ਸਰਵਜਨਕ ਇੰਟਰਨੈੱਟ ਤੇ ਖੁੱਲ੍ਹੇ ਨਾ ਹੋਣ।
- **ਪਛਾਣ ਪ੍ਰਮਾਣਿਕਤਾ**: ਉਤਪਾਦਨ ਤਾਇਨਾਤੀ ਲਈ ਪ੍ਰਮਾਣਿਕਤਾ (ਜਿਵੇਂ API ਕੁੰਜੀਆਂ, OAuth) ਲਾਗੂ ਕਰੋ।
- **CORS**: ਪਹੁੰਚ ਨੂੰ ਸੀਮਤ ਕਰਨ ਲਈ ਕ੍ਰਾਸ-ਓਰਿਜਿਨ ਰਿਸੋਰਸ ਸਾਂਝਾ ਕਰਨ (CORS) ਨੀਤੀਆਂ ਸੈੱਟ ਕਰੋ।
- **HTTPS**: ਟ੍ਰੈਫਿਕ ਨੂੰ ਐਨਕ੍ਰਿਪਟ ਕਰਨ ਲਈ ਉਤਪਾਦਨ ਵਿੱਚ HTTPS ਵਰਤੋਂ।

### ਚੰਗੀਆਂ ਅਭਿਆਸ

ਅਗਲੇ, ਆਪਣੇ MCP ਸਟ੍ਰੀਮਿੰਗ ਸਰਵਰ ਵਿੱਚ ਸੁਰੱਖਿਆ ਲਾਗੂ ਕਰਦੇ ਸਮੇਂ ਇਹ ਕੁਝ ਚੰਗੀਆਂ ਅਭਿਆਸ ਹਨ:

- ਆਉਂਦੀਆਂ ਬੇਨਤੀਆਂ 'ਤੇ ਬਿਨਾਂ ਜਾਂਚ ਉੱਤੇ ਕਦੇ ਭਰੋਸਾ ਨਾ ਕਰੋ।
- ਸਾਰੀਆਂ ਪਹੁੰਚਾਂ ਅਤੇ ਗਲਤੀਆਂ ਨੂੰ ਲਾਗ ਕਰੋ ਅਤੇ ਨਿਗਰਾਨੀ ਕਰੋ।
- ਸੁਰੱਖਿਆ ਖਾਮੀਆਂ ਮਿਟਾਉਣ ਲਈ ਨਿਯਮਤ ਤੌਰ ਤੇ ਡਿਪੈਂਡੇਂਸੀਆਂ ਅਪਡੇਟ ਕਰੋ।

### ਚੁਣੌਤੀਆਂ

MCP ਸਟ੍ਰੀਮਿੰਗ ਸਰਵਰਾਂ ਵਿੱਚ ਸੁਰੱਖਿਆ ਲਾਗੂ ਕਰਦੇ ਸਮੇਂ ਤੁਹਾਨੂੰ ਕੁਝ ਚੁਣੌਤੀਆਂ ਦਾ ਸਾਹਮਣਾ ਕਰਨਾ ਪਵੇਗਾ:

- ਵਿਕਾਸ ਦੀ ਸੌਖੀਅਤ ਨਾਲ ਸੁਰੱਖਿਆ ਵਿਚ ਸੰਤੁਲਨ ਬਣਾਉਣਾ
- ਵੱਖ-ਵੱਖ ਗਾਹਕ ਵਾਤਾਵਰਨਾਂ ਨਾਲ ਅਨੁਕੂਲਤਾ ਨੂੰ ਯਕੀਨੀ ਬਣਾਉਣਾ

### ਅਸਾਈਨਮੈਂਟ: ਆਪਣਾ ਸਵੈੰ MCP ਸਟ੍ਰੀਮਿੰਗ ਐਪ ਬਣਾਓ

**ਦ੍ਰਿਸ਼ਟੀਕੋਣ:**
ਇੱਕ MCP ਸਰਵਰ ਅਤੇ ਕਲਾਇੰਟ ਬਣਾਓ ਜਿੱਥੇ ਸਰਵਰ ਇੱਕ ਆਈਟਮਾਂ ਦੀ ਸੂਚੀ (ਜਿਵੇਂ ਫਾਇਲਾਂ ਜਾਂ ਦਸਤਾਵੇਜ਼) ਨੂੰ ਪ੍ਰਕਿਰਿਆ ਕਰਦਾ ਹੈ ਅਤੇ ਹਰ ਪ੍ਰਕਿਰਿਆ ਕੀਤੇ ਆਈਟਮ ਲਈ ਸੂਚਨਾ ਭੇਜਦਾ ਹੈ। ਕਲਾਇੰਟ ਹਰ ਸੂਚਨਾ ਨੂੰ ਜਿਵੇਂ ਮਿਲਦੀ ਹੈ ਵਿਖਾਏ।

**ਕਦਮ:**

1. ਇੱਕ ਸਰਵਰ ਟੂਲ ਲਾਗੂ ਕਰੋ ਜੋ ਇੱਕ ਸੂਚੀ ਨੂੰ ਪ੍ਰਕਿਰਿਆ ਕਰਦਾ ਹੈ ਅਤੇ ਹਰ ਆਈਟਮ ਲਈ ਸੂਚਨਾ ਭੇਜਦਾ ਹੈ।
2. ਇੱਕ ਕਲਾਇੰਟ ਲਾਗੂ ਕਰੋ ਜਿਸ ਵਿੱਚ ਮੈਸੇਜ ਹੈਂਡਲਰ ਹੋਵੇ ਜੋ ਰੀਅਲ-ਟਾਈਮ ਵਿੱਚ ਸੂਚਨਾਵਾਂ ਦਿਖਾਏ।
3. ਸਰਵਰ ਅਤੇ ਕਲਾਇੰਟ ਦੋਹਾਂ ਨੂੰ ਚਲਾਕੇ ਆਪਣੇ ਲਾਗੂ ਕਰਨ ਦੀ ਜਾਂਚ ਕਰੋ ਅਤੇ ਸੂਚਨਾਵਾਂ ਦਾ ਨਿਰੀਖਣ ਕਰੋ।
