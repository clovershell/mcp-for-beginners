# การสตรีม HTTPS ด้วยโปรโตคอลบริบทโมเดล (MCP)

บทนี้ให้คำแนะนำอย่างละเอียดเกี่ยวกับการใช้งานสตรีมแบบปลอดภัย ขยายตัวได้ และเรียลไทม์ด้วยโปรโตคอลบริบทโมเดล (MCP) โดยใช้ HTTPS ครอบคลุมแรงจูงใจสำหรับการสตรีม กลไกการขนส่งที่มีอยู่ วิธีการใช้งาน HTTP สตรีมที่สามารถใช้ได้ใน MCP แนวทางปฏิบัติที่ดีที่สุดด้านความปลอดภัย การย้ายจาก SSE และคำแนะนำที่ใช้ได้จริงสำหรับการสร้างแอปพลิเคชัน MCP สตรีมของคุณเอง

> **มองไปข้างหน้า:** บทเรียนนี้อธิบาย Streamable HTTP ภายใต้ **MCP Specification 2025-11-25** ซึ่งเซสชันจะถูกสร้างขึ้นในระหว่าง `initialize` และผูกไว้ด้วยหัวข้อ `Mcp-Session-Id` การเปิดตัวรุ่นส่ง (`release candidate`)  `2026-07-28` จะยกเลิกการจับมือและรหัสเซสชันทั้งหมด ทำให้แต่ละคำขอมีความเป็นอิสระและสามารถส่งไปยังอินสแตนซ์เซิร์ฟเวอร์ใดก็ได้โดยไม่ต้องมีการจับเซสชันแบบติดหนึบ ดูรายละเอียดที่ [What's Changing in MCP: The 2026-07-28 Release Candidate](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md)

## กลไกการขนส่งและการสตรีมใน MCP

ส่วนนี้สำรวจกลไกการขนส่งต่าง ๆ ที่มีอยู่ใน MCP และบทบาทของพวกเขาในการเปิดใช้งานความสามารถในการสตรีมสำหรับการสื่อสารเรียลไทม์ระหว่างไคลเอนต์และเซิร์ฟเวอร์

### กลไกการขนส่งคืออะไร?

กลไกการขนส่งกำหนดวิธีการแลกเปลี่ยนข้อมูลระหว่างไคลเอนต์และเซิร์ฟเวอร์ MCP รองรับประเภทการขนส่งหลายแบบเพื่อเหมาะสมกับสภาพแวดล้อมและความต้องการที่แตกต่างกัน:

- **stdio**: อินพุต/เอาต์พุตมาตรฐาน เหมาะสำหรับเครื่องมือท้องถิ่นและ CLI เครื่องมือที่ใช้งานง่ายแต่ไม่เหมาะสำหรับเว็บหรือคลาวด์
- **SSE (Server-Sent Events)**: ช่วยให้เซิร์ฟเวอร์ส่งการอัปเดตเรียลไทม์ไปยังไคลเอนต์ผ่าน HTTP ดีสำหรับเว็บ UI แต่มีข้อจำกัดในเรื่องการขยายตัวและความยืดหยุ่น ตั้งแต่ MCP Specification 2025-06-18 การขนส่ง SSE แบบเดี่ยวถูกเลิกใช้และแทนที่ด้วย "Streamable HTTP"
- **Streamable HTTP**: การขนส่งสตรีมมิ่งที่ใช้ HTTP แบบสมัยใหม่ รองรับการแจ้งเตือนและขยายตัวได้ดีกว่า แนะนำสำหรับกรณีการใช้งานในผลิตภัณฑ์และคลาวด์เป็นส่วนใหญ่

### ตารางเปรียบเทียบ

ดูตารางเปรียบเทียบด้านล่างเพื่อเข้าใจความแตกต่างระหว่างกลไกการขนส่งเหล่านี้:

| การขนส่ง         | การอัปเดตเรียลไทม์ | การสตรีม | การขยายตัว | กรณีใช้งาน          |
|-------------------|------------------|-----------|-------------|-------------------------|
| stdio             | ไม่               | ไม่        | ต่ำ         | เครื่องมือ CLI ท้องถิ่น  |
| SSE               | ใช่               | ใช่       | ปานกลาง    | เว็บ, การอัปเดตเรียลไทม์  |
| Streamable HTTP   | ใช่               | ใช่       | สูง         | คลาวด์, หลายไคลเอนต์    |

> **เคล็ดลับ:** การเลือกกลไกการขนส่งที่เหมาะสมมีผลต่อประสิทธิภาพ การขยายตัว และประสบการณ์ผู้ใช้ **Streamable HTTP** แนะนำสำหรับแอปพลิเคชันที่ทันสมัย ขยายตัวได้ และพร้อมสำหรับคลาวด์

โปรดสังเกตว่ากลไกการขนส่ง stdio และ SSE ที่กล่าวถึงในบทก่อนหน้า และการสตรีม HTTP ที่สามารถใช้ได้ คือกลไกการขนส่งที่ครอบคลุมในบทนี้

## การสตรีม: แนวคิดและแรงจูงใจ

การเข้าใจแนวคิดพื้นฐานและแรงจูงใจเบื้องหลังการสตรีมเป็นสิ่งสำคัญสำหรับการสร้างระบบสื่อสารแบบเรียลไทม์ที่มีประสิทธิภาพ

**การสตรีม** เป็นเทคนิคในโปรแกรมเครือข่ายที่อนุญาตให้ส่งและรับข้อมูลเป็นชิ้นเล็ก ๆ ที่จัดการได้ หรือเป็นลำดับของเหตุการณ์ แทนการรอคอยให้ได้คำตอบทั้งหมด เทคนิคนี้มีประโยชน์อย่างยิ่งสำหรับ:

- ไฟล์หรือชุดข้อมูลขนาดใหญ่
- การอัปเดตเรียลไทม์ (เช่น แชท แถบแสดงความก้าวหน้า)
- การคำนวณระยะยาวที่ต้องการแจ้งให้ผู้ใช้ทราบสถานะ

นี่คือสิ่งที่คุณควรรู้เกี่ยวกับการสตรีมในระดับสูง:

- ข้อมูลถูกส่งเป็นลำดับทีละน้อย ไม่ใช่ทั้งชุดพร้อมกัน
- ไคลเอนต์สามารถประมวลผลข้อมูลเมื่อได้รับทันที
- ลดความหน่วงที่รับรู้ และปรับปรุงประสบการณ์ผู้ใช้

### ทำไมต้องใช้การสตรีม?

เหตุผลในการใช้การสตรีมมีดังนี้:

- ผู้ใช้ได้รับข้อมูลตอบกลับทันที ไม่ใช่แค่ตอนจบ
- เปิดใช้งานแอปพลิเคชันเรียลไทม์และ UI ที่ตอบสนองเร็ว
- ใช้ทรัพยากรเครือข่ายและคอมพิวเตอร์ได้มีประสิทธิภาพมากขึ้น

### ตัวอย่างง่าย ๆ: เซิร์ฟเวอร์และไคลเอนต์ HTTP Streaming

นี่คือตัวอย่างง่าย ๆ ของการใช้งานการสตรีม:

#### Python

**เซิร์ฟเวอร์ (Python ใช้ FastAPI และ StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**ไคลเอนต์ (Python ใช้ requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

ตัวอย่างนี้แสดงให้เห็นเซิร์ฟเวอร์ส่งข้อความชุดหนึ่งไปยังไคลเอนต์เมื่อพร้อม แทนการรอคอยให้ข้อความทั้งหมดพร้อมก่อนส่ง

**วิธีทำงาน:**

- เซิร์ฟเวอร์ส่งข้อความแต่ละข้อความทันทีที่พร้อม
- ไคลเอนต์รับและแสดงข้อความแต่ละส่วนที่มาถึง

**ความต้องการ:**

- เซิร์ฟเวอร์ต้องใช้การตอบสนองแบบสตรีม (เช่น `StreamingResponse` ใน FastAPI)
- ไคลเอนต์ต้องประมวลผลการตอบสนองเป็นสตรีม (`stream=True` ใน requests)
- Content-Type มักเป็น `text/event-stream` หรือ `application/octet-stream`

#### Java

**เซิร์ฟเวอร์ (Java ใช้ Spring Boot และ Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**ไคลเอนต์ (Java ใช้ Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**บันทึกการใช้งาน Java:**

- ใช้สแต็กเชิงปฏิกิริยาของ Spring Boot พร้อม `Flux` สำหรับการสตรีม
- `ServerSentEvent` ให้สตรีมเหตุการณ์ที่มีโครงสร้างและประเภทเหตุการณ์
- `WebClient` กับ `bodyToFlux()` ช่วยให้รับสตรีมแบบปฏิกิริยา
- `delayElements()` จำลองเวลาประมวลผลระหว่างเหตุการณ์
- เหตุการณ์สามารถมีประเภท (`info`, `result`) เพื่อการจัดการที่ดีกว่า

### เปรียบเทียบ: การสตรีมแบบคลาสสิกกับการสตรีมใน MCP

ความแตกต่างระหว่างการทำงานของการสตรีมแบบ "คลาสสิก" กับการทำงานใน MCP สามารถอธิบายได้ดังนี้:

| คุณสมบัติ             | การสตรีม HTTP คลาสสิก            | การสตรีม MCP (การแจ้งเตือน)       |
|-----------------------|----------------------------------|-------------------------------------|
| ผลตอบกลับหลัก        | ถูกส่งเป็นชิ้นส่วน               | ส่งเพียงครั้งเดียวตอนจบ            |
| การอัปเดตก้าวหน้า   | ส่งเป็นชิ้นข้อมูล                | ส่งเป็นการแจ้งเตือน                |
| ความต้องการของไคลเอนต์ | ต้องประมวลผลสตรีม                | ต้องใช้ตัวจัดการข้อความ            |
| กรณีใช้งาน           | ไฟล์ขนาดใหญ่, สตรีมโทเค็น AI    | สถานะ ความคืบหน้า, บันทึก, ข้อเสนอแนะเรียลไทม์ |

### ความแตกต่างสำคัญที่สังเกตได้

นอกจากนี้ นี่ยังเป็นความแตกต่างสำคัญบางประการ:

- **รูปแบบการสื่อสาร:**
  - การสตรีม HTTP แบบคลาสสิก: ใช้การเข้ารหัสแบบ chunked เพื่อส่งข้อมูลเป็นชิ้น
  - การสตรีม MCP: ใช้ระบบการแจ้งเตือนแบบมีโครงสร้างด้วยโปรโตคอล JSON-RPC

- **รูปแบบข้อความ:**
  - HTTP คลาสสิก: ชิ้นข้อความเป็นข้อความธรรมดาแบ่งด้วย newline
  - MCP: วัตถุ LoggingMessageNotification ที่มีโครงสร้างพร้อมเมตาดาต้า

- **การใช้งานไคลเอนต์:**
  - HTTP คลาสสิก: ไคลเอนต์ง่าย ๆ ที่ประมวลผลการตอบสนองแบบสตรีม
  - MCP: ไคลเอนต์ซับซ้อนกว่าที่มีตัวจัดการข้อความเพื่อประมวลผลชนิดข้อความต่าง ๆ

- **การอัปเดตก้าวหน้า:**
  - HTTP คลาสสิก: ความก้าวหน้าเป็นส่วนหนึ่งของสตรีมผลตอบกลับหลัก
  - MCP: ความก้าวหน้าจะส่งแยกเป็นข้อความแจ้งเตือน ในขณะที่ผลลัพธ์หลักส่งตอนสิ้นสุด

### คำแนะนำ

มีบางสิ่งที่เราขอแนะนำเมื่อเลือกใช้งานระหว่างการสตรีมแบบคลาสสิก (เช่น จุดเชื่อมต่อที่ใช้ `/stream` ที่เราแสดงข้างต้น) กับการสตรีมผ่าน MCP

- **สำหรับความต้องการสตรีมง่าย ๆ:** การสตรีม HTTP แบบคลาสสิกง่ายต่อการใช้งานและเพียงพอสำหรับความต้องการสตรีมพื้นฐาน

- **สำหรับแอปที่ซับซ้อนและโต้ตอบได้:** การสตรีม MCP ให้แนวทางมีโครงสร้างมากขึ้น พร้อมเมตาดาต้ารายละเอียดและแยกระหว่างการแจ้งเตือนกับผลลัพธ์สุดท้าย

- **สำหรับแอป AI:** ระบบการแจ้งเตือนของ MCP มีประโยชน์อย่างยิ่งสำหรับงาน AI ที่ใช้เวลานานซึ่งต้องการแจ้งผู้ใช้ถึงความคืบหน้า

## การสตรีมใน MCP

โอเค ตอนนี้คุณเห็นคำแนะนำและการเปรียบเทียบระหว่างการสตรีมแบบคลาสสิกกับการสตรีมใน MCP แล้ว เรามาดูรายละเอียดว่าคุณจะใช้การสตรีมใน MCP ได้อย่างไร

การเข้าใจการทำงานของการสตรีมภายในเฟรมเวิร์ก MCP เป็นสิ่งสำคัญสำหรับการสร้างแอปพลิเคชันที่ตอบสนองเร็วและให้ข้อมูลย้อนกลับแบบเรียลไทม์แก่ผู้ใช้ในระหว่างการทำงานที่ใช้เวลานาน

ใน MCP การสตรีมไม่ใช่การส่งผลตอบกลับหลักเป็นชิ้น ๆ แต่เป็นการส่ง **การแจ้งเตือน (notifications)** ไปยังไคลเอนต์ระหว่างที่เครื่องมือกำลังประมวลผลคำขอ การแจ้งเตือนเหล่านี้อาจรวมถึงการอัปเดตก้าวหน้า บันทึก หรือเหตุการณ์อื่น ๆ

### วิธีการทำงาน

ผลลัพธ์หลักยังคงถูกส่งเป็นคำตอบเดียว อย่างไรก็ตาม การแจ้งเตือนสามารถส่งเป็นข้อความแยกต่างหากระหว่างการประมวลผลและอัปเดตไคลเอนต์แบบเรียลไทม์ ไคลเอนต์ต้องสามารถจัดการและแสดงการแจ้งเตือนเหล่านี้ได้

## การแจ้งเตือนคืออะไร?

เราพูดถึง "การแจ้งเตือน" นั้นหมายความว่าอะไรในบริบทของ MCP?

การแจ้งเตือนคือข้อความที่ส่งจากเซิร์ฟเวอร์ไปยังไคลเอนต์เพื่อแจ้งเกี่ยวกับความคืบหน้า สถานะ หรือเหตุการณ์อื่น ๆ ระหว่างการทำงานที่ใช้เวลานาน การแจ้งเตือนช่วยเพิ่มความโปร่งใสและประสบการณ์ของผู้ใช้

ตัวอย่างเช่น ไคลเอนต์ควรส่งการแจ้งเตือนเมื่อทำการจับมือเริ่มต้นกับเซิร์ฟเวอร์สำเร็จ

การแจ้งเตือนเป็นข้อความ JSON ดังนี้:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

การแจ้งเตือนเป็นหัวข้อหนึ่งใน MCP ที่เรียกว่า ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging)

> **ประกาศเลิกใช้:** ตัวส่งรุ่น MCP `2026-07-28` ประกาศให้ Logging primitive เลิกใช้และแนะนำให้ใช้ `stderr` สำหรับการขนส่ง stdio และ OpenTelemetry สำหรับการสังเกตการณ์ที่มีโครงสร้าง Logging ยังคงใช้งานใน `2025-11-25` และอย่างน้อยหนึ่งปีหลังการเลิกใช้อย่างเป็นทางการ ดูรายละเอียดที่ [What's Changing in MCP: The 2026-07-28 Release Candidate](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md)

เพื่อให้ logging ทำงานได้ เซิร์ฟเวอร์ต้องเปิดใช้งานเป็นฟีเจอร์/ความสามารถดังนี้:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> ขึ้นอยู่กับ SDK ที่ใช้งาน logging อาจเปิดใช้งานเป็นค่าเริ่มต้น หรือคุณอาจต้องเปิดใช้งาน explicitly ในการกำหนดค่าของเซิร์ฟเวอร์

มีประเภทการแจ้งเตือนต่าง ๆ ดังนี้:

| ระดับ       | คำอธิบาย                     | ตัวอย่างการใช้งาน            |
|-------------|------------------------------|------------------------------|
| debug       | ข้อมูลดีเทลสำหรับดีบัก       | จุดเข้า/ออกฟังก์ชัน          |
| info        | ข้อความข้อมูลทั่วไป           | การอัปเดตก้าวหน้า            |
| notice      | เหตุการณ์ปกติแต่สำคัญ         | การเปลี่ยนแปลงการตั้งค่า     |
| warning     | สภาวะเตือน                  | การใช้ฟีเจอร์ที่เลิกใช้      |
| error       | สภาวะข้อผิดพลาด             | การประมวลผลล้มเหลว          |
| critical    | สภาวะวิกฤต                  | การล้มเหลวของส่วนประกอบระบบ  |
| alert       | ต้องดำเนินการทันที           | ตรวจพบการเสียหายของข้อมูล   |
| emergency   | ระบบใช้งานไม่ได้              | ระบบล้มเหลวอย่างสมบูรณ์     |

## การใช้งานการแจ้งเตือนใน MCP

ในการใช้งานการแจ้งเตือนใน MCP คุณต้องตั้งค่าทั้งฝั่งเซิร์ฟเวอร์และไคลเอนต์ให้สามารถจัดการกับการอัปเดตแบบเรียลไทม์ได้ ซึ่งช่วยให้แอปพลิเคชันของคุณส่งข้อมูลย้อนกลับทันทีให้ผู้ใช้ระหว่างการทำงานที่ใช้เวลานาน

### ฝั่งเซิร์ฟเวอร์: ส่งการแจ้งเตือน

มาเริ่มที่ฝั่งเซิร์ฟเวอร์กัน ใน MCP คุณกำหนดเครื่องมือที่สามารถส่งการแจ้งเตือนในระหว่างประมวลผลคำขอ เซิร์ฟเวอร์ใช้วัตถุบริบท (ปกติคือ `ctx`) ในการส่งข้อความไปยังไคลเอนต์

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

ในตัวอย่างก่อนหน้า เครื่องมือ `process_files` ส่งการแจ้งเตือนสามครั้งไปยังไคลเอนต์ขณะประมวลผลแต่ละไฟล์ เมธอด `ctx.info()` ใช้ส่งข้อความข้อมูล

นอกจากนี้ เพื่อเปิดใช้งานการแจ้งเตือน ให้แน่ใจว่าเซิร์ฟเวอร์ของคุณใช้การขนส่งแบบสตรีม (เช่น `streamable-http`) และไคลเอนต์ของคุณมีตัวจัดการข้อความในการประมวลผลการแจ้งเตือน ต่อไปนี้คือวิธีตั้งค่าเซิร์ฟเวอร์ให้ใช้การขนส่ง `streamable-http`:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

ในตัวอย่าง .NET นี้ เครื่องมือ `ProcessFiles` ถูกตกแต่งด้วยแอททริบิวต์ `Tool` และส่งการแจ้งเตือนสามครั้งไปยังไคลเอนต์ขณะประมวลผลแต่ละไฟล์ เมธอด `ctx.Info()` ใช้ส่งข้อความข้อมูล

เพื่อเปิดใช้งานการแจ้งเตือนในเซิร์ฟเวอร์ MCP .NET ของคุณ ให้แน่ใจว่าคุณใช้การขนส่งแบบสตรีม:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### ฝั่งไคลเอนต์: รับการแจ้งเตือน

ไคลเอนต์ต้องมีตัวจัดการข้อความเพื่อประมวลผลและแสดงการแจ้งเตือนเมื่อได้รับ

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

ในโค้ดก่อนหน้า ฟังก์ชัน `message_handler` ตรวจสอบว่าข้อความที่เข้ามาเป็นการแจ้งเตือนหรือไม่ หากใช่จะแสดงข้อความนั้น หากไม่ใช่จะประมวลผลเป็นข้อความเซิร์ฟเวอร์ปกติ นอกจากนี้โปรดสังเกตว่าคลาส `ClientSession` ถูกเริ่มต้นด้วยตัวจัดการข้อความ `message_handler` เพื่อจัดการการแจ้งเตือนที่เข้ามา

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

ในตัวอย่าง .NET นี้ ฟังก์ชัน `MessageHandler` ตรวจสอบว่าข้อความที่เข้ามาเป็นการแจ้งเตือนหรือไม่ หากใช่จะแสดงข้อความนั้น หากไม่ใช่จะประมวลผลเป็นข้อความเซิร์ฟเวอร์ปกติ `ClientSession` ถูกเริ่มต้นด้วยตัวจัดการข้อความผ่าน `ClientSessionOptions`

เพื่อเปิดใช้งานการแจ้งเตือน ให้แน่ใจว่าเซิร์ฟเวอร์ของคุณใช้การขนส่งแบบสตรีม (เช่น `streamable-http`) และไคลเอนต์ของคุณมีตัวจัดการข้อความในการประมวลผลการแจ้งเตือน

## การแจ้งเตือนความก้าวหน้าและสถานการณ์

ส่วนนี้อธิบายแนวคิดของการแจ้งเตือนความก้าวหน้าใน MCP ทำไมมันจึงสำคัญ และวิธีการใช้งานโดยใช้ Streamable HTTP คุณยังจะพบแบบฝึกหัดเพื่อเสริมความเข้าใจ

การแจ้งเตือนความก้าวหน้าคือข้อความแบบเรียลไทม์ที่ส่งจากเซิร์ฟเวอร์ไปยังไคลเอนต์ในระหว่างการดำเนินงานที่ใช้เวลานาน แทนที่จะรอจนจบกระบวนการทั้งหมด เซิร์ฟเวอร์จะแจ้งให้ไคลเอนต์ทราบสถานะปัจจุบัน ซึ่งช่วยเพิ่มความโปร่งใส ประสบการณ์ผู้ใช้ และทำให้ง่ายต่อการดีบัก

**ตัวอย่าง:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### ทำไมถึงต้องใช้การแจ้งเตือนความก้าวหน้า?

การแจ้งเตือนความก้าวหน้ามีความสำคัญสำหรับหลายเหตุผล:

- **ประสบการณ์ผู้ใช้ที่ดีกว่า:** ผู้ใช้เห็นการอัปเดตขณะดำเนินการ ไม่ใช่แค่ตอนจบ
- **ข้อมูลตอบกลับแบบเรียลไทม์:** ไคลเอนต์แสดงแถบความก้าวหน้าหรือบันทึก ทำให้แอปดูตอบสนองเร็วขึ้น
- **ง่ายต่อการดีบักและตรวจสอบ:** นักพัฒนาและผู้ใช้เห็นจุดที่กระบวนการอาจช้าหรือหยุดนิ่ง

### วิธีใช้งานการแจ้งเตือนความก้าวหน้า

นี่คือวิธีใช้งานการแจ้งเตือนความก้าวหน้าใน MCP:

- **ฝั่งเซิร์ฟเวอร์:** ใช้ `ctx.info()` หรือ `ctx.log()` ส่งการแจ้งเตือนเมื่อประมวลผลแต่ละรายการ ส่งข้อความไปยังไคลเอนต์ก่อนที่ผลลัพธ์หลักจะพร้อม
- **ฝั่งไคลเอนต์:** สร้างตัวจัดการข้อความที่คอยรับและแสดงการแจ้งเตือนเมื่อมาถึง ตัวจัดการนี้แยกแยะระหว่างการแจ้งเตือนกับผลลัพธ์สุดท้าย

**ตัวอย่างเซิร์ฟเวอร์:**


#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**ตัวอย่างลูกค้า:**

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## ข้อควรพิจารณาด้านความปลอดภัย

เมื่อทำการใช้งานเซิร์ฟเวอร์ MCP ด้วยการส่งข้อมูลผ่าน HTTP ความปลอดภัยจะเป็นประเด็นสำคัญสูงสุดที่ต้องให้ความใส่ใจอย่างรอบคอบต่อหลายช่องทางการโจมตีและกลไกการป้องกัน

### ภาพรวม

ความปลอดภัยมีความสำคัญอย่างยิ่งเมื่อให้บริการ MCP ผ่าน HTTP การใช้ Streamable HTTP ทำให้เกิดพื้นผิวการโจมตีใหม่ ๆ และต้องมีการตั้งค่าที่รอบคอบ

### ประเด็นสำคัญ

- **การตรวจสอบ Origin Header**: ควรตรวจสอบค่า `Origin` เสมอเพื่อป้องกันการโจมตีแบบ DNS rebinding
- **การผูกกับ Localhost**: สำหรับการพัฒนาในเครื่อง ควรผูกเซิร์ฟเวอร์กับ `localhost` เพื่อไม่ให้เปิดเผยต่ออินเทอร์เน็ตสาธารณะ
- **การตรวจสอบสิทธิ์**: ใช้การตรวจสอบสิทธิ์ (เช่น คีย์ API, OAuth) สำหรับการใช้งานจริง
- **CORS**: ตั้งค่านโยบาย Cross-Origin Resource Sharing (CORS) เพื่อจำกัดการเข้าถึง
- **HTTPS**: ใช้ HTTPS ในการใช้งานจริงเพื่อเข้ารหัสการสื่อสาร

### แนวทางปฏิบัติที่ดีที่สุด

- อย่าไว้ใจคำขอที่เข้ามาโดยไม่ได้รับการตรวจสอบ
- บันทึกและตรวจสอบการเข้าถึงและข้อผิดพลาดทั้งหมด
- อัปเดตไลบรารีและโปรแกรมเสริมเป็นประจำเพื่อแก้ไขช่องโหว่ความปลอดภัย

### ความท้าทาย

- การปรับสมดุลระหว่างความปลอดภัยกับความสะดวกในการพัฒนา
- การทำให้รองรับกับสภาพแวดล้อมลูกค้าต่าง ๆ

## การอัปเกรดจาก SSE เป็น Streamable HTTP

สำหรับแอปพลิเคชันที่ใช้ Server-Sent Events (SSE) อยู่ การย้ายไปใช้ Streamable HTTP จะช่วยเพิ่มความสามารถและความยั่งยืนในระยะยาวสำหรับการใช้งาน MCP

### เหตุผลในการอัปเกรด

มีเหตุผลสองประการที่ควรอัปเกรดจาก SSE เป็น Streamable HTTP:

- Streamable HTTP มีความสามารถในการปรับขนาดได้ดีกว่า รองรับได้หลากหลาย และรองรับการแจ้งเตือนที่หลากหลายกว่า SSE
- เป็นการส่งข้อมูลที่แนะนำสำหรับแอป MCP ใหม่

### ขั้นตอนการย้ายระบบ

นี่คือวิธีการย้ายจาก SSE ไปใช้ Streamable HTTP ในแอป MCP ของคุณ:

- **อัปเดตโค้ดเซิร์ฟเวอร์** ให้ใช้ `transport="streamable-http"` ใน `mcp.run()`
- **อัปเดตโค้ดลูกค้า** ให้ใช้ `streamablehttp_client` แทน SSE client
- **เพิ่มตัวจัดการข้อความ** ในฝั่งลูกค้าเพื่อประมวลผลการแจ้งเตือน
- **ทดสอบความเข้ากันได้** กับเครื่องมือและเวิร์กโฟลว์ที่มีอยู่

### การรักษาความเข้ากันได้

แนะนำให้รักษาความเข้ากันได้กับลูกค้า SSE เดิมในขณะย้ายระบบ นี่คือบางกลยุทธ์:

- คุณสามารถสนับสนุนทั้ง SSE และ Streamable HTTP โดยรันทั้งสองแบบบนจุดเชื่อมต่อที่แตกต่างกัน
- ค่อยๆ ย้ายลูกค้าไปยังระบบส่งข้อมูลแบบใหม่

### ความท้าทาย

ควรจัดการกับความท้าทายเหล่านี้ในระหว่างการย้ายระบบ:

- ตรวจสอบให้แน่ใจว่าลูกค้าทุกตัวได้รับการอัปเดต
- จัดการความแตกต่างในการส่งมอบการแจ้งเตือน

## ข้อควรพิจารณาด้านความปลอดภัย

ความปลอดภัยควรเป็นลำดับความสำคัญสูงสุดเมื่อทำการใช้งานเซิร์ฟเวอร์ใด ๆ โดยเฉพาะอย่างยิ่งเมื่อใช้การส่งข้อมูลผ่าน HTTP เช่น Streamable HTTP ใน MCP

เมื่อทำการใช้งานเซิร์ฟเวอร์ MCP ด้วยการส่งข้อมูลผ่าน HTTP ความปลอดภัยจะเป็นประเด็นสำคัญสูงสุดที่ต้องให้ความใส่ใจอย่างรอบคอบต่อหลายช่องทางการโจมตีและกลไกการป้องกัน

### ภาพรวม

ความปลอดภัยมีความสำคัญอย่างยิ่งเมื่อให้บริการ MCP ผ่าน HTTP การใช้ Streamable HTTP ทำให้เกิดพื้นผิวการโจมตีใหม่ ๆ และต้องมีการตั้งค่าที่รอบคอบ

นี่คือข้อควรพิจารณาด้านความปลอดภัยที่สำคัญบางประการ:

- **การตรวจสอบ Origin Header**: ควรตรวจสอบค่า `Origin` เสมอเพื่อป้องกันการโจมตีแบบ DNS rebinding
- **การผูกกับ Localhost**: สำหรับการพัฒนาในเครื่อง ควรผูกเซิร์ฟเวอร์กับ `localhost` เพื่อไม่ให้เปิดเผยต่ออินเทอร์เน็ตสาธารณะ
- **การตรวจสอบสิทธิ์**: ใช้การตรวจสอบสิทธิ์ (เช่น คีย์ API, OAuth) สำหรับการใช้งานจริง
- **CORS**: ตั้งค่านโยบาย Cross-Origin Resource Sharing (CORS) เพื่อจำกัดการเข้าถึง
- **HTTPS**: ใช้ HTTPS ในการใช้งานจริงเพื่อเข้ารหัสการสื่อสาร

### แนวทางปฏิบัติที่ดีที่สุด

นอกจากนี้ นี่คือแนวทางปฏิบัติที่ดีที่สุดที่ควรปฏิบัติตามเมื่อทำการรักษาความปลอดภัยในเซิร์ฟเวอร์สตรีม MCP ของคุณ:

- อย่าไว้ใจคำขอที่เข้ามาโดยไม่ได้รับการตรวจสอบ
- บันทึกและตรวจสอบการเข้าถึงและข้อผิดพลาดทั้งหมด
- อัปเดตไลบรารีและโปรแกรมเสริมเป็นประจำเพื่อแก้ไขช่องโหว่ความปลอดภัย

### ความท้าทาย

คุณจะเผชิญกับความท้าทายบางอย่างเมื่อทำการรักษาความปลอดภัยในเซิร์ฟเวอร์สตรีม MCP:

- การปรับสมดุลระหว่างความปลอดภัยกับความสะดวกในการพัฒนา
- การทำให้รองรับกับสภาพแวดล้อมลูกค้าต่าง ๆ

### งานประจำ: สร้างแอปสตรีม MCP ของตัวคุณเอง

**สถานการณ์:**
สร้างเซิร์ฟเวอร์และไคลเอนต์ MCP ที่เซิร์ฟเวอร์ทำงานกับรายการของรายการ (เช่น ไฟล์หรือเอกสาร) และส่งการแจ้งเตือนสำหรับแต่ละรายการที่ประมวลผล ไคลเอนต์ควรแสดงแต่ละการแจ้งเตือนทันทีที่มาถึง

**ขั้นตอน:**

1. สร้างเครื่องมือเซิร์ฟเวอร์ที่ประมวลผลรายการและส่งการแจ้งเตือนสำหรับแต่ละรายการ
2. สร้างไคลเอนต์ที่มีตัวจัดการข้อความเพื่อนำเสนอการแจ้งเตือนแบบเรียลไทม์
3. ทดสอบการทำงานโดยรันทั้งเซิร์ฟเวอร์และไคลเอนต์ แล้วดูการแจ้งเตือน

### ต่อไป?

- ลองสร้างเครื่องมือ MCP ที่ซับซ้อนกว่าโดยใช้การสตรีมสำหรับการวิเคราะห์แบบเรียลไทม์, แชท หรือการแก้ไขร่วมกัน
- สำรวจการผนวก MCP สตรีมมิ่งกับเฟรมเวิร์กหน้าเว็บ (React, Vue, ฯลฯ) เพื่ออัปเดต UI แบบสด
