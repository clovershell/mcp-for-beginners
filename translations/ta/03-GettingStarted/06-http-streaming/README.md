# மாடல் Kontext Protocol (MCP) உடன் HTTPS ஸ்ட்ரீமிங்

இந்த அத்தியாயம் HTTPS பயன்படுத்தி மாடல் Kontext Protocol (MCP) உடன் பாதுகாப்பான, பரிமாணக்கூடிய மற்றும் நேரலை ஸ்ட்ரீமிங் செயல்படுத்துவதற்கான முழுமையான வழிகாட்டியை வழங்குகிறது. இது ஸ்ட்ரீமிங்கின் தூண்டலுக்கு, கிடைக்கும் போக்குவரத்துக் கருவிகளுக்கு, MCP இல் ஸ்ட்ரீமிங் HTTP ஐ எவ்வாறு செயல்படுத்துவது, பாதுகாப்பு சிறந்த நடைமுறைகள், SSE இலிருந்து மாற்றுதல் மற்றும் உங்கள் சொந்த ஸ்ட்ரீமிங் MCP செயலிகளைக் கட்டமைக்கும் நடைமுறைகளைக் கொண்டுள்ளது.

> **எதிர்நோக்கிக் கண்டு:** இந்த பாடம் **MCP விவரக்குறிப்பு 2025-11-25** இன் கீழ் ஸ்ட்ரீமிங் HTTP ஐ விவரிக்கிறது, இதில் ஒரு அமர்வு `initialize` போது உருவாக்கப்பட்டு `Mcp-Session-Id` தலைப்புத்தளத்துடன் பொருத்தப்படுகிறது. `2026-07-28` வெளியீட்டு வேடிக்கை கைப்பிடியை மற்றும் அமர்வு ஐடியை முற்றிலும் முடக்கும், இதனால் ஒவ்வொரு கோரிக்கையும் தானாக இருக்கிறது மற்றும் ஒட்டுமொத்த அமர்வுகள் இல்லாமல் எந்த சேவையகம் பிணையிலும் பாதுக்கதிக்கபடும். விரிவுகளுக்கு [MCP இல் என்ன மாற்றமாகிறது: 2026-07-28 வெளியீட்டு வேடிக்கை](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) பாருங்கள்.

## MCP இல் போக்குவரத்து கருவிகள் மற்றும் ஸ்ட்ரீமிங்

இந்தப் பகுதி MCP இல் கிடைக்கும் பல போக்குவரத்து கருவிகளை ஆராய்கிறது மற்றும் அவற்றின் பங்கு வாடிக்கையாளர்கள் மற்றும் சேவையகங்கள் இடையேயான நேரலை தொடர்புக்கான ஸ்ட்ரீமிங் திறன்களை இயல்பாக்குவதில் விளக்குகிறது.

### போக்குவரத்து கருவி란 என்ன?

ஒரு போக்குவரத்து கருவி என்பது வாடிக்கையாளர் மற்றும் சேவையகம் இடையே தரவு மாற்றமடையும் முறையை வரையறுக்கிறது. MCP பல்வேறு சூழல் மற்றும் தேவைகளுக்கு ஏற்ப பல போக்குவரத்து வகைகளை ஆதரிக்கிறது:

- **stdio**: ஸ்டான்டர்ட் உள்ளீடு/வெளியீடு, உள்ளூர் மற்றும் CLI அடிப்படையிலான கருவிகளுக்கு ஏற்றது. எளிது ஆனால் வலை அல்லது மேகத்துக்கு பொருத்தமல்ல.
- **SSE (சேவையகம் அனுப்பும் நிகழ்வுகள்)**: HTTP மூலம் சேவையகங்கள் வாடிக்கையாளர்களுக்கு நேரலை இற்றைப்படுத்தல்களை அனுப்ப அனுமதிக்கிறது. வலை UI களுக்கு சிறந்தது, ஆனால் பரிமாணம் மற்றும் நெகிழ்வுத்தன்மை குறைவாக உள்ளது. MCP விவரக்குறிப்பு 2025-06-18 இற்குப்பிறகு தனிப்படியாக SSE போக்குவரத்து "Streamable HTTP" போக்குவரத்தால் மாற்றப்பட்டது.
- **Streamable HTTP**: நவீன HTTP அடிப்படையிலான ஸ்ட்ரீமிங் போக்குவரத்து, அறிவிப்புகள் மற்றும் சிறந்த பரிமாண ஆதரவு கொண்டது. பெரும்பாலான உற்பத்தி மற்றும் மேக சூழலுக்கு பரிந்துரைக்கப்படுகிறது.

### ஒப்பிடும் அட்டவணை

இந்தப் போக்குவரத்து கருவிகளின் வேறுபாடுகளைப் புரிந்து கொள்ள கீழ்காணும் ஒப்பிடும் அட்டவணைக்குக் கவனம் செலுத்தவும்:

| போக்குவரத்து         | நேரலை இற்றைப்படுத்தல்கள் | ஸ்ட்ரீமிங் | பரிமாணம் | பயன்பாட்டு உள்கட்டமைப்பு          |
|-------------------|------------------|-----------|-------------|-------------------------|
| stdio             | இல்லை               | இல்லை        | குறைந்தது         | உள்ளூர் CLI கருவிகள்         |
| SSE               | ஆம்               | ஆம்       | நடுத்தரம்      | வலை, நேரலை இற்றைப்படுத்தல்கள்  |
| Streamable HTTP   | ஆம்               | ஆம்       | உயர்ந்தது        | மேகம், பல வாடிக்கையாளர்    |

> **குறிப்பு:** சரியான போக்குவரத்தைத் தேர்ந்தெடுப்பது செயல்திறன், பரிமாணம் மற்றும் பயனர் அனுபவத்தை பாதிக்கிறது. **Streamable HTTP** நவீன, பரிமாணக்கூடிய மற்றும் மேகம் தயாராக உள்ள செயலிகளுக்கு பரிந்துரைக்கப்படுகிறது.

முன்னுள்ள அத்தியாயங்களில் நீங்கள் பார்த்த stdio மற்றும் SSE போக்குவரத்தை நினைவில் வைக்கவும், இந்த அத்தியாயத்தில் ஸ்ட்ரீமிங் HTTP போக்குவரத்தைப் பற்றிப் பேசப்படுகிறது.

## ஸ்ட்ரீமிங்: கருத்துகள் மற்றும் தூண்டுதல்

ஸ்ட்ரீமிங்கின் அடிப்படை கருத்துகள் மற்றும் தூண்டுதல்களை புரிந்து கொள்வது, விளைவான நேரலை தொடர்பு அமைப்புகளை உருவாக்க அடிப்படையாகும்.

**ஸ்ட்ரீமிங்** என்பது ஒரு நெட்வொர்க் நிரலாக்க தொழில்நுட்பமாகும், இது முழு பதிலை காத்திருப்பதற்கு பதிலாக சிறிய, கையாளக்கூடிய துண்டுகளாக அல்லது நிகழ்வுகளின் தொடராக தரவை அனுப்பவும் பெறவும் அனுமதிக்கும். இது குறிப்பாக பயன்படும்:

- பெரிய கோப்புகள் அல்லது தரவுத்தொகுப்புகள்.
- நேரலை இற்றைப்படுத்தல்கள் (உதா: பேசுதல், முன்னேற்ற பட்டைகள்).
- நீண்டகால கணக்கிடல்கள், பயனரை தொடர்ந்து அறிய உறவுக் கொண்டிருக்க வேண்டும்.

ஸ்ட்ரீமிங் பற்றி உங்களுக்கு தெரிந்துகொள்ள வேண்டியவை:

- தரவு поступக்கமாக வழங்கப்படுகிறது, அனைத்தும் ஒரே நேரத்தில் கிடையாது.
- வாடிக்கையாளர் தரவை வந்தவுடன் செயலாற்ற முடியும்.
- உணரப்படும் தாமதத்தை குறைத்து, பயனர் அனுபவத்தை மேம்படுத்துகிறது.

### ஏன் ஸ்ட்ரீமிங்கைப் பயன்படுத்த வேண்டும்?

ஸ்ட்ரீமிங் பயன்படுத்துவதற்கான காரணங்கள் பின்வருமாறு:

- பயனர்கள் உடனடியாக பின்னூட்டம் பெறுவர், முடிவில் மட்டுமே அல்ல.
- நேரலை செயலிகள் மற்றும் பதிலளிக்கும் UI கள் இயல்பாக்கப்படுகிறது.
- நெட்வொர்க் மற்றும் கணினி வளங்களை திறமையாகப் பயன்படுத்துகிறது.

### எளிய உதாரணம்: HTTP ஸ்ட்ரீமிங் சேவையகம் & வாடிக்கையாளர்

ஸ்ட்ரீமிங் எப்படி செயல்படுத்தப்பட வேண்டும் என்பதற்கான எளிய உதாரணம்:

#### Python

**சேவையகம் (Python, FastAPI மற்றும் StreamingResponse பயன்படுத்தி):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**வாடிக்கையாளர் (Python, requests பயன்படுத்தி):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

இந்த உதாரணம், அனைத்து செய்திகளும் தயார் ஆக காத்திருப்பதற்கு பதிலாக, சேவையகம் தரவை கிடைக்கும் போதெல்லாம் வாடிக்கையாளருக்குத் தருவது எப்படி என்பதைக் காட்டுகிறது.

**எப்படி செயல்படுகிறது:**

- சேவையகம் ஒவ்வொரு செய்தியையும் தயார் ஆகும்போது அளிக்கிறது.
- வாடிக்கையாளர் வரும் துண்டுகளை பெறவும் அச்சிடவும் செய்கிறது.

**தேவைகள்:**

- சேவையகம் ஸ்ட்ரீமிங் பதிலுக்குப் பயன்படுத்த வேண்டும் (உதா: FastAPI யில் `StreamingResponse`).
- வாடிக்கையாளர் பதிலை ஸ்ட்ரீமாக செயலாக்க வேண்டும் (`stream=True` requests இல்).
- உள்ளடக்க வகை பொதுவாக `text/event-stream` அல்லது `application/octet-stream`.

#### Java

**சேவையகம் (Java, Spring Boot மற்றும் Server-Sent Events பயன்படுத்தி):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**வாடிக்கையாளர் (Java, Spring WebFlux WebClient பயன்படுத்தி):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Java செயல்படுத்தும் குறிப்புகள்:**

- ஸ்ட்ரீமிங்கிற்காக Spring Boot இன் எதிர்வினை ஸ்டாக் `Flux` க்காக பயன்படுத்தப்படுகிறது
- `ServerSentEvent` நிகழ்வு வகைகளுடன் கட்டமைக்கப்பட்ட நிகழ்வு ஸ்ட்ரீமிங்கை வழங்குகிறது
- `WebClient` உடன் `bodyToFlux()` எதிர்வினை ஸ்ட்ரீமிங் உணர்தல் உயர்த்துகிறது
- `delayElements()` நிகழ்வுகளுக்கிடையேயான செயலாக்க நேரத்தை பிம்பிக்கிறது
- `info`, `result` போன்ற வகைகள் நிகழ்வுகளுக்கு வேறுபாடு அளிக்க பயன்படும்

### ஒப்பீடு: பாரம்பரிய ஸ்ட்ரீமிங் மற்றும் MCP ஸ்ட்ரீமிங்

"பாரம்பரிய" முறையில் ஸ்ட்ரீமிங் எப்படி இயங்குகிறது மற்றும் MCP இல் அது எப்படி வேலை செய்கிறது என்பதற்கான வேறுபாடுகள்:

| அம்சம்                | பாரம்பரிய HTTP ஸ்ட்ரீமிங்         | MCP ஸ்ட்ரீமிங் (அறிவிப்புகள்)      |
|------------------------|-------------------------------|-------------------------------------|
| முக்கிய பதில்          | துண்டுபடுத்தப்பட்டது                       | ஒரே, முடிவில்                      |
| முன்னேற்றம் இற்றைப்படுத்தல்கள்       | தரவு துண்டுகளாக அனுப்பப்பட்டது           | அறிவிப்புகளாக அனுப்பப்பட்டது               |
| வாடிக்கையாளர் தேவைகள்    | ஸ்ட்ரீம் செயலாக்கம் செய்ய வேண்டும்           | செய்தி கையாள்பவரை செயலாக்க வேண்டும்      |
| பயன்பாடு               | பெரிய கோப்புகள், AI டோக்கன் ஸ்ட்ரீம்கள் | முன்னேற்றம், குறிப்புகள், நேரலை பின்னூட்டம்  |

### காணப்பட்ட முக்கிய வேறுபாடுகள்

மேலும, முக்கிய சில வேறுபாடுகள்:

- **தொடர்புக் கண்ணோட்டம்:**
  - பாரம்பரிய HTTP ஸ்ட்ரீமிங்: தரவை துண்டுகளாக அனுப்ப எளிய துண்டுபடுத்தப்பட்ட இடமாற்றக்கூறு குறியீடு பயன்படுத்துகிறது
  - MCP ஸ்ட்ரீமிங்: JSON-RPC நெறிமுறையுடன் கட்டமைக்கப்பட்ட அறிவிப்பு அமைப்பை பயன்படுத்துகிறது

- **செய்தி வடிவம்:**
  - பாரம்பரிய HTTP: பLAIN உரை துண்டுக்கள் மற்றும் புதிய வரிகளுடன்
  - MCP: கட்டமைக்கப்பட்ட LoggingMessageNotification பொருட்கள் மற்றும் மெட்டாடேட்டாவுடன்

- **வாடிக்கையாளர் செயல்படுத்தல்:**
  - பாரம்பரிய HTTP: ஸ்ட்ரீமிங் பதில்களை செயலாக்கம் செய்யும் எளிய வாடிக்கையாளர்
  - MCP: வெவ்வேறு வகை செய்திகளை செயலாக்க ஒரு செய்தி கையாள்பவர் கொண்ட மேம்பட்ட வாடிக்கையாளர்

- **முன்னேற்றம் இற்றைப்படுத்தல்கள்:**
  - பாரம்பரிய HTTP: முன்னேற்றம் முக்கிய பதிலின் ஒரு பகுதி
  - MCP: முன்னேற்றம் தனித்தனியான அறிவிப்பு செய்திகள் மூலம் அனுப்பப்படுகிறது, முக்கிய பதில் முடிவில் வருகிறது

### பரிந்துரைகள்

பாரம்பரிய ஸ்ட்ரீமிங் (மேலே /stream எனும் இடமாற்ற பாடத்தில் காட்டியது) அல்லது MCP வழியாக ஸ்ட்ரீமிங் செய்ய தேர்வு செய்வதில் சில பரிந்துரைகள் உள்ளன.

- **எளிய ஸ்ட்ரீமிங் தேவைகளுக்காக:** பாரம்பரிய HTTP ஸ்ட்ரீமிங் செயல்படுத்த எளிது மற்றும் அடிப்படையிலான தேவைகளுக்கு போதுமானது.

- **சிக்கலான, தொடர்புடைய செயலிகளுக்காக:** MCP ஸ்ட்ரீமிங் அறிவிப்புகள் மற்றும் இறுதி முடிவுகளுக்கிடையேயான பிரிவினைகளை கொண்ட கட்டமைக்கப்பட்ட அணுகுமுறையை வழங்குகிறது.

- **AI செயலிகளுக்காக:** நீண்டகால AI பணிகளுக்குப் பயனர்களுக்கு முன்னேற்றத்தைத் தெரிவிக்க MCP அறிவிப்பு அமைப்பு சிறப்பாக பயன்படும்.

## MCP இல் ஸ்ட்ரீமிங்

சரி, நீங்கள் இன்றுவரை பாரம்பரிய ஸ்ட்ரீமிங் மற்றும் MCP ஸ்ட்ரீமிங் வேறுபாடுகளில் சில பரிந்துரைகள் மற்றும் ஒப்பீடுகளைப் பார்த்தீர்கள். இப்போது MCP இல் ஸ்ட்ரீமிங்கை எவ்வாறு பயன்படுத்துவது என்பதில் விரிவாகப் பார்க்கலாம்.

MCP கட்டமைப்புக்குள் ஸ்ட்ரீமிங் எப்படி இயங்கி, நீண்டகால செயல்பாடுகளின் போது பயனர்களுக்கு நேரலை பின்னூட்டங்களை வழங்கும் பதிலளிக்கும் செயலிகளை கட்டமைக்க இது அவசியமானது.

MCP இல் ஸ்ட்ரீமிங் என்பது முக்கிய பதிலை துண்டுகளாக அனுப்புவது அல்ல, ஒரு கருவி கோரிக்கையை செயலாக்கும் போது வாடிக்கையாளருக்கு **அறிவிப்புகள்** அனுப்புவதே ஆகும். இந்த அறிவிப்புகள் முன்னேற்றம், குறிப்புகள் அல்லது பிற நிகழ்வுகளை உள்ளடக்கலாம்.

### இது எப்படி இயங்குகிறது

முக்கிய முடிவு இன்னும் ஒரே பதிலாக அனுப்பப்படுகிறது. ஆனால், செயலாக்கத்தின் போது அறிவிப்புகள் தனித்தனியான செய்திகளாக அனுப்பப்படலாம் மற்றும் அதன் மூலம் வாடிக்கையாளரை நேரலை புதுப்பிக்க முடியும். வாடிக்கையாளர் இந்த அறிவிப்புக்களை கையாளவும் காட்சிப்படுத்தவும் தயாராக இருக்க வேண்டும்.

## அறிவிப்பு란 என்ன?

நாம் "அறிவிப்பு" என்று கூறினோம், அது MCP சூழலில் என்ன பொருள்?

அறிவிப்பு என்பது நீண்டகால செயல்பாட்டின் போது முன்னேற்றம், நிலை அல்லது பிற நிகழ்வுகள் பற்றி தொழிலாளரிலிருந்து வாடிக்கையாளருக்கு அனுப்பப்படும் செய்தி ஆகும். அறிவிப்புகள் வெளிப்படைத்தன்மை மற்றும் பயனர் அனுபவத்தை மேம்படுத்துகின்றன.

உதாரணமாக, வாடிக்கையாளர் முதன்மை கைமாறுதலைச் செய்து கொண்டபின் அறிவிப்புகளை அனுப்ப வேண்டும்.

அறிவிப்பு கீழ்காணும் JSON செய்தி போன்று இருக்கும்:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

அறிவிப்புகள் MCP இல் ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) என்ற தலைப்புக்குள் சொந்தமானது.

> **பழையமைப்பு அறிவிப்பு:** `2026-07-28` MCP விவரக்குறிப்பு வெளியீட்டு வேடிக்கை Logging முதன்மையை stdio போக்குவரத்துக்கான `stderr` மற்றும் கட்டமைக்கப்பட்ட கவனிப்புக்கு OpenTelemetryக்கு மாற்றுகிறது. Logging `2025-11-25` மற்றும் அதிகாரப்பூர்வ பழையமைப்பு பிறகு ஒரு வருடம் வரை வேலை செய்யும். மேலும் விவரங்களுக்கு [MCP இல் என்ன மாற்றமாகிறது: 2026-07-28 வெளியீட்டு வேடிக்கை](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) பாருங்கள்.

Logging இயங்கச் செய்ய சேவையகம் இதை பின்வற்ற வேண்டும்:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> பயன்படுத்தப்படும் SDK இன் அடிப்படையில், பதிவு இயல்பாகவே இயக்கப்பட்டிருக்கலாம் அல்லது சேவையக கட்டமைப்பில் வெளிப்படையாக இயக்க வேண்டியிருக்கலாம்.

அறிவிப்புகளுக்கு பல வகைகள் உள்ளன:

| நிலை     | விளக்கம்                    | பயன்பாட்டு உதாரணம்                |
|-----------|-------------------------------|---------------------------------|
| debug     | விரிவான பிழைத்திருத்த தகவல் | செயல்பாட்டில் நுழைவு/வெளியேற்ற இடங்கள்      |
| info      | பொதுவான தகவல் செய்திகள் | செயல்பாட்டு முன்னேற்றம் இற்றைப்படுத்தல்கள்      |
| notice    | சாதாரண ஆனால் முக்கிய நிகழ்வுகள்  | கட்டமைப்பு மாற்றங்கள்           |
| warning   | எச்சரிக்கை நிலைகள்             | பழையமைப்பு அம்ச பயன்பாடு        |
| error     | பிழை நிலைகள்               | செயல்பாடு தோல்விகள்              |

| alert     | உடனடி நடவடிக்கை தேவை | தரவுdezaramachineனறிக்கப்படல்      |
| emergency | அமைப்பு பயன்படுத்த முடியாது             | முழுமையான அமைப்பு தோல்வி         |

## MCP இல் அறிவிப்புகளை செயல்படுத்துதல்

MCP இல் அறிவிப்புகளை செயல்படுத்த, நீங்கள் சேவையகம் மற்றும் வாடிக்கையாளர் இரண்டையும் நேரலை இற்றைப்படுத்தல்களை கையாள வசதியாக அமைக்க வேண்டும். இதனால் நீண்டகால செயல்பாடுகளில் உங்கள் செயலி பயனர்களுக்கு உடனடி பின்னூட்டம் தர முடியும்.

### சேவையக பக்கம்: அறிவிப்புகளை அனுப்புதல்

சேவையக பக்கத்திலிருந்து தொடங்குவோம். MCP இல், நீங்கள் கோரிக்கைகளை செயலாக்கும் போது அறிவிப்புகளை அனுப்பக்கூடிய கருவிகளை வரையறுக்கிறீர்கள். சேவையகம் விபரம் பொருளை (இருப்பாக `ctx`) பயன்படுத்தி வாடிக்கையாளருக்கு செய்திகளை அனுப்புகிறது.

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

மேலே உள்ள உதாரணத்தில், `process_files` கருவி ஒவ்வொரு கோப்பையும் செயலாக்கும் போது வாடிக்கையாளருக்கு மூன்று அறிவிப்புகளை அனுப்புகிறது. `ctx.info()` முறை தகவல்தொடர்பு செய்திகளை அனுப்ப பயன்படுகிறது.

மேலும, அறிவிப்புகளை இயக்கு, உங்கள் சேவையகம் ஸ்ட்ரீமிங் போக்குவரத்தை (உதா: `streamable-http`) பயன்படுத்த வேண்டும் மற்றும் உங்கள் வாடிக்கையாளர் அறிவிப்புகளை செயலாக்க ஒரு செய்தி கையாள்பவரை செயல்படுத்த வேண்டும். இங்கே சேவையகத்தை `streamable-http` போக்குவரத்தைப் பயன்படுத்த அமைக்கும் விதம்:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

இந்த .NET உதாரணத்தில், `ProcessFiles` கருவி `Tool` பண்புடன் அழைக்கப்பட்டு, ஒவ்வொரு கோப்பையும் செயலாக்கும் போது வாடிக்கையாளருக்கு மூன்று அறிவிப்புகளை அனுப்புகிறது. `ctx.Info()` முறை தகவல் செய்திகளை அனுப்ப பயன்படுகிறது.

உங்கள் .NET MCP சேவையகத்தில் அறிவிப்புகளை இயக்கு, ஸ்ட்ரீமிங் போக்குவரத்தை பயன்படுத்துகிறீர்களா என உறுதிப்படுத்துங்கள்:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### வாடிக்கையாளர் பக்கம்: அறிவிப்புகளைப் பெறுதல்

வாடிக்கையாளர் அறிவிப்புகளை வரும் போது கையாளவும் காட்சிப்படுத்தவும் ஒரு செய்தி கையாள்பவரை செயல்படுத்த வேண்டும்.

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

மேலே உள்ள குறியீட்டில், `message_handler` செயல்பாடு வரும் செய்தி ஒரு அறிவிப்பா என்று ஆய்வு செய்கிறது. அது என்றால் அச்சிடுகிறது; இல்லை என்றால் சாதாரண சேவையக செய்தியாக அதனை செயலாக்குகிறது. மேலும், `ClientSession` அறிவிப்புகளை கையாள `message_handler` உடன் ஆரம்பிக்கப்படுகிறது.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

இந்த .NET உதாரணத்தில், `MessageHandler` செயல்பாடு வரும் செய்தி அறிவிப்பா என்று ஆய்வு செய்கிறது. அது என்றால் அச்சிடுகிறது; இல்லை என்றால் சாதாரண சேவையக செய்தியாக செயலாக்குகிறது. `ClientSession` செய்திகள் கையாள்பவருடன் `ClientSessionOptions` மூலம் ஆரம்பிக்கப்படுகிறது.

அறிவிப்புகளை இயக்கு, உங்கள் சேவையகம் ஸ்ட்ரீமிங் போக்குவரத்தைப் பயன்படுத்துகிறது மற்றும் வாடிக்கையாளர் அறிவிப்புகளை செயலாக்கும் செய்தி கையாள்பவரைக் கொண்டிருக்க வேண்டும் என்பதை உறுதி செய்யுங்கள்.

## முன்னேற்ற அறிவிப்புகள் & சூழல்கள்

இந்தப் பகுதி MCP இல் முன்னேற்ற அறிவிப்புகளின் கருத்தை, அவை ஏன் முக்கியம் என்பதை மற்றும் Streamable HTTP உதவி கொண்டு அவற்றை எவ்வாறு செயல்படுத்துவது என்பதை விளக்குகிறது. உங்கள் புரிதலை உறுதி செய்ய ஒரு நடைமுறைப் பணியும் உள்ளது.

முன்னேற்ற அறிவிப்புகள் என்பது நீண்டகால செயல்பாடுகளின் போது சேவையகத்திலிருந்து வாடிக்கையாளருக்கு அனுப்பப்படும் நேரலை செய்திகள் ஆகும். முழு செயல்முறையின் முடிவை காத்திருப்பதற்கு பதிலாக, சேவையகம் தற்போதைய நிலையை வாடிக்கையாளருக்கு தொடர்ந்து தகவல் தருகிறது. இது வெளிப்படைத்தன்மையை, பயனர் அனுபவத்தை மேம்படுத்துகிறது மற்றும் பிழைத்திருத்தத்தைக் குறைக்கும்.

**உதாரணம்:**

```text

"Processing document 2/10"
...
"Processing complete!"

```

### ஏன் முன்னேற்ற அறிவிப்புகளை பயன்படுத்த வேண்டும்?

முன்னேற்ற அறிவிப்புகள் பல காரணங்களுக்காக அவசியம்:

- **மேன்மையான பயனர் அனுபவம்:** வேலை முன்னேறும் போதெல்லாம் பயனர்கள் புதுப்பிப்புகளை காணிறார்கள், முடிவில் மட்டும் அல்ல.
- **நேரலை பின்னூட்டம்:** வாடிக்கையாளர்கள் முன்னேற்ற பட்டைகள் அல்லது குறிப்புகளை காட்சிப்படுத்த முடியும், செயலி பதிலளிக்கிறது போல இருக்கும்.
- **எளிமையான பிழைத்திருத்தம் மற்றும் கண்காணிப்பு:** உருவாக்குநர்கள் மற்றும் பயனர்கள் செயல்முறை எங்கே மெதுவாகவோ ஜாமாகவோ இருக்கிறதோ என்பதை காணலாம்.

### முன்னேற்ற அறிவிப்புகளை எப்படி செயல்படுத்துவது

இனி MCP இல் முன்னேற்ற அறிவிப்புகளை செயல்படுத்தும் விதம்:

- **சேவையகத்தில்:** ஒவ்வொரு உருப்படியையும் செயலாக்கும்போது `ctx.info()` அல்லது `ctx.log()` மூலம் அறிவிப்புகளை அனுப்பு. முக்கிய முடிவு தயார் ஆகுவதற்கு முன்னர் வாடிக்கையாளருக்கு செய்தி அனுப்பப்படுகிறது.
- **வாடிக்கையாளருக்கு:** அறிவிப்புகளை வரும்போது கேட்கும் மற்றும் காட்சிப்படுத்தும் செய்தி கையாள்வாளர் செயல்படுத்துங்கள். இந்த கையாள்பவர் அறிவிப்புகளும் இறுதி முடிவும் வேறுபாட்டால் அறிய முடியும்.

## பாதுகாப்பு கருத்துக்கள்

HTTP அடிப்படையிலான பரிமாற்றங்களுடன் MCP சர்வர்களை செயல்படுத்தும்போது, பாதுகாப்பு அதிக முக்கியத்துவம் பெறும் ஒரு விஷயம் ஆகும், இது பல தாக்குதல் வழிகள் மற்றும் பாதுகாப்பு முறைமைகளை கவனமாகப் பார்த்து கடைபிடிக்க வேண்டும்.

### சுருக்கம்

MCP சர்வர்களை HTTP மூலம் வெளியிடும் போது பாதுகாப்பு மிகவும் அவசியம். ஸ்ட்ரீமபிள் HTTP புதிய தாக்குதல் மண்டலங்களை அறிமுகப்படுத்துகிறது மற்றும் கவனமாக அமைக்க வேண்டும்.

### முக்கிய அம்சங்கள்

- **Origin ஹெடர் சரிபார்ப்பு**: DNS ரிபைண்டிங் தாக்குதல்களைத் தடுக்கும் வகையில் `Origin` ஹெடரை எப்போதும் சரிபார்க்கவும்.
- **Localhost பைண்டிங்**: உள்ளூர் கூர்மையில், சர்வர்களை `localhost` க்கு பைண்ட் செய்து பொதுச்சட்ட இணையத்தில் வெளிப்படாது உறுதி செய்யவும்.
- **அங்கீகாரம்**: உற்பத்தி பயன்பாட்டிற்கு அங்கீகாரம் (உதாரணமாக API விசைகள், OAuth) செயல்படுத்தவும்.
- **CORS**: அணுகலை கட்டுப்படுத்த Cross-Origin Resource Sharing (CORS) கொள்கைகளை அமைக்கவும்.
- **HTTPS**: தகவல்தொடர்பை குறியாக்கம் செய்ய உற்பத்தியில் HTTPS ஐப் பயன்படுத்தவும்.

### சிறந்த நடைமுறைகள்

- எந்தவொரு கோரிக்கையையும் சரிபார்ப்பின்றி நம்பவும் வேண்டாம்.
- அனைத்து அணுகல் மற்றும் பிழைகள் பதிவும் கண்காணிக்கவும்.
- பாதுகாப்பு குறைகளை சரி செய்ய இணைப்புகளை அடிக்கடி புதுப்பிக்கவும்.

### சவால்கள்

- பாதுகாப்புக்கும் வளர்ச்சி சுலபத்துக்கும் இடையில் சமநிலை பெறுதல்
- பல்வேறு கிளையன்ட் சூழல்களை உடையிணைப்பு உறுதி செய்தல்

## SSE இலிருந்து ஸ்ட்ரீமபிள் HTTP க்கு மேம்படுத்தல்

தற்போது Server-Sent Events (SSE) பயன்படுத்தும் பயன்பாடுகளுக்கு, ஸ்ட்ரீமபிள் HTTP க்கு மாற்றுவது மேம்பட்ட திறன்களை மற்றும் உங்கள் MCP செயலாக்கங்களுக்கு நல்ல நீண்டகால தாங்குதலை வழங்கும்.

### ஏன் மேம்படுத்த வேண்டும்?

SSE இலிருந்து ஸ்ட்ரீமபிள் HTTP க்கு மேம்படுத்துவதற்கான இரண்டு முக்கிய காரணங்கள் உள்ளன:

- ஸ்ட்ரீமபிள் HTTP SSE க்கு மேல் சிறந்த அளவீடு, உடையிணைப்பு மற்றும் நிறைவான அறிவிப்பு ஆதரவை வழங்குகிறது.
- இது புதிய MCP பயன்பாடுகளுக்கான பரிந்துரைக்கப்பட்ட பரிமாற்றமாகும்.

### இடமாற்ற படிகள்

உங்கள் MCP பயன்பாடுகளில் SSE இலிருந்து ஸ்ட்ரீமபிள் HTTP க்கு இடமாற்றம் செய்ய இதோ வழி:

- **சர்வர் கோட்டை புதுப்பிக்கவும்** `mcp.run()` இல் `transport="streamable-http"` பயன்படுத்தவும்.
- **கிளையன்ட் கோட்டை புதுப்பிக்கவும்** SSE கிளையன்ட் பதிலாக `streamablehttp_client` பயன்படுத்தவும்.
- குறும்பட அறிவிப்புகளை செயலாக்க கிளையன்டில் ஒரு செய்தி கையாள்பவரை அமல்படுத்தவும்.
- இருப்பு கருவிகள் மற்றும் செயல்முறைகளுடன் உடையிணைப்பு சோதனை செய்யவும்.

### உடையிணைப்பை பரப்புதல்

இடமாற்றத்தின் போது பரிந்துரைக்கப்படும் விடயம் உள்ளது, தற்போதைய SSE கிளையன்ட்களுடன் உடையிணைப்பை பராமரிக்க வேண்டும். சில முறைகள் இங்கே:

- SSE மற்றும் ஸ்ட்ரீமபிள் HTTP இரண்டையும் வேறுபட்ட இடைகளில் ஒரே நேரத்தில் இயக்கலாம்.
- புதிய பரிமாற்றத்திற்குத் தனிமனிதர்களை படிப்படியாக மாற்றவும்.

### சவால்கள்

இடமாற்றத்தின் போது பின்வரும் சவால்களை கவனிக்கவும்:

- அனைத்து கிளையன்ட்களும் புதுப்பிக்கப்பட்டுள்ளதை உறுதி செய்தல்
- அறிவிப்புகளின் விநியோகத்தில் வித்தியாசங்களை கையாளுதல்

## பாதுகாப்பு கருத்துக்கள்

எந்த சர்வரையும் செயல்படுத்தும் போது பாதுகாப்பு மிக முக்கியமானது, குறிப்பாக MCP இல் ஸ்ட்ரீமபிள் HTTP போன்ற HTTP அடிப்படையிலான பரிமாற்றங்களை பயன்படுத்தும் போது.

HTTP அடிப்படையிலான பரிமாற்றங்களுடன் MCP சர்வர்களை செயல்படுத்தும்போது, பாதுகாப்பு முக்கிய கவனிக்கக்கூடியது மற்றும் பல தாக்குதல் வழிகள் மற்றும் பாதுகாப்பு முறைமைகளை கவனிக்க வேண்டும்.

### சுருக்கம்

MCP சர்வர்களை HTTP மூலம் வெளியிடும் போது பாதுகாப்பு மிகவும் அவசியம். ஸ்ட்ரீமபிள் HTTP புதிய தாக்குதல் மண்டலங்களை அறிமுகப்படுத்துகிறது மற்றும் கவனமாக அமைக்க வேண்டும்.

சில முக்கிய பாதுகாப்பு கருத்துக்கள் இங்கே உள்ளன:

- **Origin ஹெடர் சரிபார்ப்பு**: DNS ரிபைண்டிங் தாக்குதல்களைத் தடுக்கும் வகையில் `Origin` ஹெடரை எப்போதும் சரிபார்க்கவும்.
- **Localhost பைண்டிங்**: உள்ளூர் கூர்மையில், சர்வர்களை `localhost` க்கு பைண்ட் செய்து பொதுச்சட்ட இணையத்தில் வெளிப்படாது உறுதி செய்யவும்.
- **அங்கீகாரம்**: உற்பத்தி பயன்பாட்டிற்கு அங்கீகாரம் (உதாரணமாக API விசைகள், OAuth) செயல்படுத்தவும்.
- **CORS**: அணுகலை கட்டுப்படுத்த Cross-Origin Resource Sharing (CORS) கொள்கைகளை அமைக்கவும்.
- **HTTPS**: தகவல்தொடர்பை குறியாக்கம் செய்ய உற்பத்தியில் HTTPS ஐப் பயன்படுத்தவும்.

### சிறந்த நடைமுறைகள்

மேலும், உங்கள் MCP ஸ்ட்ரீமிங் சர்வர் பாதுகாப்பைச் செயல்படுத்தும்போது பின்பற்ற வேண்டிய சிறந்த நடைமுறைகள்:

- எந்தவொரு கோரிக்கையையும் சரிபார்ப்பின்றி நம்பவும் வேண்டாம்.
- அனைத்து அணுகல் மற்றும் பிழைகள் பதிவும் கண்காணிக்கவும்.
- பாதுகாப்பு குறைகளை சரி செய்ய இணைப்புகளை அடிக்கடி புதுப்பிக்கவும்.

### சவால்கள்

MCP ஸ்ட்ரீமிங் சர்வர்களில் பாதுகாப்பை செயல்படுத்தும் போது நீங்கள் எதிர்கொள்ளும் சில சவால்கள்:

- பாதுகாப்புக்கும் வளர்ச்சி சுலபத்துக்கும் இடையில் சமநிலை பெறுதல்
- பல்வேறு கிளையன்ட் சூழல்களை உடையிணைப்பு உறுதி செய்தல்

### பணிதலை: உங்கள் சொந்த ஸ்ட்ரீமிங் MCP பயன்பாட்டை உருவாக்குங்கள்

**காட்சி:**
ஒரு பட்டியலை (எ.கா., கோப்புகள் அல்லது ஆவணங்கள்) செயலாக்கும் MCP சர்வர் மற்றும் கிளையன்டை உருவாக்கி, செயலாக்கப்படும் ஒவ்வொரு பொருளுக்கும் அறிவிப்பை அனுப்பும். கிளையன்ட் வரும் உடனே ஒவ்வொரு அறிவிப்பையும் காட்ட வேண்டும்.

**படிகள்:**

1. ஒரு சர்வர் கருவியை செயல்படுத்தி பட்டியலை செயலாக்கி ஒவ்வொரு பொருளுக்கும் அறிவிப்புகளை அனுப்பவும்.
2. நேரடி அறிவிப்புகளை காட்ட ஒரு செய்தி கையாள்பவருடன் கிளையன்டை உருவாக்கவும்.
3. உங்கள் செயலாக்கத்தை சர்வர் மற்றும் கிளையண்டைப் ப இயக்கி சோதனை செய்து அறிவிப்புகளை கவனிக்கவும்.
