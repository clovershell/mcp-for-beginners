# HTTPS потокова передача з протоколом Model Context Protocol (MCP)

Цей розділ надає всебічний посібник з реалізації безпечної, масштабованої та потокової передачі в реальному часі за допомогою Model Context Protocol (MCP) через HTTPS. Розглядаються мотивація для потокової передачі, доступні транспортні механізми, як реалізувати потік HTTP у MCP, найкращі практики безпеки, міграція з SSE та практичні рекомендації для створення власних потокових MCP-додатків.

> **Подивимося вперед:** цей урок описує Streamable HTTP згідно з **MCP Specification 2025-11-25**, де сесія встановлюється під час `initialize` і фіксується заголовком `Mcp-Session-Id`. Кандидат на реліз `2026-07-28` повністю вилучає рукопотискання і ID сесії, роблячи кожен запит автономним і маршрутизованим до будь-якого серверного екземпляру без прив’язки до сесії. Деталі дивіться у [What's Changing in MCP: The 2026-07-28 Release Candidate](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md).

## Транспортні механізми та потокова передача в MCP

У цьому розділі розглядаються різні транспортні механізми, доступні в MCP, та їхня роль у забезпеченні можливостей потокової передачі для комунікації в реальному часі між клієнтами та серверами.

### Що таке транспортний механізм?

Транспортний механізм визначає, як обмінюються дані між клієнтом і сервером. MCP підтримує кілька типів транспорту, щоб відповідати різним середовищам і вимогам:

- **stdio**: Стандартний ввід/вивід, підходить для локальних і CLI-інструментів. Простий, але не призначений для вебу або хмари.
- **SSE (Server-Sent Events)**: Дозволяє серверам надсилати клієнтам оновлення в реальному часі через HTTP. Хороший варіант для веб-інтерфейсів, але має обмеження за масштабованістю і гнучкістю. Згідно з MCP Specification 2025-06-18, автономний транспорт SSE (Server-Sent Events) застарілий і замінений транспортом «Streamable HTTP».
- **Streamable HTTP**: Сучасний транспорт потокової передачі на базі HTTP, що підтримує повідомлення та кращу масштабованість. Рекомендовано для більшості виробничих і хмарних сценаріїв.

### Таблиця порівнянь

Ознайомтеся з наведеним нижче порівняльним таблицею, щоб зрозуміти відмінності між цими транспортними механізмами:

| Транспорт        | Оновлення в реальному часі | Потокова передача | Масштабованість | Сценарій використання     |
|------------------|----------------------------|-------------------|-----------------|--------------------------|
| stdio            | Ні                         | Ні                | Низька         | Локальні CLI-інструменти |
| SSE              | Так                        | Так               | Середня        | Веб, оновлення в реальному часі |
| Streamable HTTP  | Так                        | Так               | Висока         | Хмара, мультиклієнтські застосунки |

> **Порада:** Вибір правильного транспорту впливає на продуктивність, масштабованість і досвід користувача. **Streamable HTTP** рекомендований для сучасних, масштабованих та готових до хмарних додатків.

Зверніть увагу на транспорти stdio і SSE, показані у попередніх розділах, і те, що в цьому розділі розглядається транспорт Streamable HTTP.

## Потокова передача: концепції та мотивація

Розуміння базових концепцій і мотивації потокової передачі є важливим для впровадження ефективних систем комунікації в реальному часі.

**Потокова передача** — це техніка в мережевому програмуванні, яка дозволяє надсилати і отримувати дані малими, керованими частинами або як послідовність подій, замість очікування готовності всієї відповіді. Це особливо корисно для:

- Великих файлів або наборів даних.
- Оновлень у реальному часі (наприклад, чат, індикатори прогресу).
- Тривалих обчислень, коли потрібно інформувати користувача.

Ось що потрібно знати про потокову передачу на високому рівні:

- Дані доставляються поступово, не одразу всі разом.
- Клієнт може обробляти дані по мірі їх надходження.
- Знижує сприйману затримку та покращує користувацький досвід.

### Чому варто використовувати потокову передачу?

Ось причини для використання потокової передачі:

- Користувачі отримують зворотний зв’язок негайно, а не лише в кінці
- Дозволяє створювати реальні у часі додатки та чуйні UI
- Ефективніше використовує мережеві та обчислювальні ресурси

### Простий приклад: HTTP потоковий сервер та клієнт

Ось простий приклад реалізації потокової передачі:

#### Python

**Сервер (Python, використовуючи FastAPI і StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**Клієнт (Python, використовуючи requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

Цей приклад демонструє сервер, який надсилає серію повідомлень клієнту по мірі їх готовності, замість очікування готовності всіх повідомлень.

**Як це працює:**

- Сервер повертає кожне повідомлення, як тільки воно готове.
- Клієнт отримує та виводить кожну частину по мірі надходження.

**Вимоги:**

- Сервер повинен використовувати потокову відповідь (наприклад, `StreamingResponse` у FastAPI).
- Клієнт повинен обробляти відповідь як потік (`stream=True` у requests).
- Тип вмісту зазвичай `text/event-stream` або `application/octet-stream`.

#### Java

**Сервер (Java, використовуючи Spring Boot і Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**Клієнт (Java, використовуючи Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Примітки щодо реалізації Java:**

- Використовує реактивний стек Spring Boot з `Flux` для потокової передачі
- `ServerSentEvent` забезпечує структуровану передачу подій з типами подій
- `WebClient` з `bodyToFlux()` дозволяє реактивне споживання потоку
- `delayElements()` імітує обробку часу між подіями
- Події можуть мати типи (`info`, `result`) для кращої обробки клієнтом

### Порівняння: Класична потокова передача та потокова передача MCP

Відмінності між класичним способом потокової передачі та MCP можна зобразити таким чином:

| Особливість            | Класичний HTTP Streaming       | MCP Streaming (повідомлення)      |
|-----------------------|-------------------------------|-----------------------------------|
| Основна відповідь      | Частинами (chunked)            | Одна, в кінці                     |
| Оновлення прогресу     | Відправляються як частини даних| Відправляються у вигляді повідомлень |
| Вимоги до клієнта      | Має обробляти потік            | Має реалізувати обробник повідомлень  |
| Сценарій використання  | Великі файли, потоки токенів AI| Прогрес, журнали, зворотній зв’язок у реальному часі |

### Ключові спостережувані відмінності

Також ось кілька ключових відмінностей:

- **Патерн комунікації:**
  - Класична HTTP потокова передача: Використовує просте кодування із частинами для відправки даних
  - MCP потокова передача: Використовує структуровану систему повідомлень із протоколом JSON-RPC

- **Формат повідомлень:**
  - Класичний HTTP: Прості текстові частини з переносами рядків
  - MCP: Структуровані об’єкти LoggingMessageNotification з метаданими

- **Реалізація клієнта:**
  - Класичний HTTP: Простий клієнт, який обробляє потокові відповіді
  - MCP: Складніший клієнт з обробником повідомлень для обробки різних типів повідомлень

- **Оновлення прогресу:**
  - Класичний HTTP: Прогрес є частиною основного потоку відповіді
  - MCP: Прогрес надсилається окремими повідомленнями-повідомленнями, тоді як основна відповідь надходить наприкінці

### Рекомендації

Ось кілька порад щодо вибору між класичною потоковою передачею (через endpoint, який ми показали вище з `/stream`) та потоковою передачею через MCP.

- **Для простих потреб потокової передачі:** Класична HTTP потокова передача простіша у реалізації і достатня для базових задач.

- **Для складних, інтерактивних застосунків:** MCP надає більш структурований підхід з багатшими метаданими і розділенням між повідомленнями і кінцевими результатами.

- **Для AI-застосунків:** Система повідомлень MCP особливо корисна для довготривалих AI-задач, де потрібно постійно інформувати користувачів про прогрес.

## Потокова передача в MCP

Отже, ви вже ознайомилися з рекомендаціями та порівняннями класичної потокової передачі та MCP. Тепер докладно розглянемо, як саме можна використовувати потокову передачу в MCP.

Розуміння того, як працює потокова передача в рамках MCP, є необхідним для створення відзивчивих додатків, які надають зворотний зв’язок користувачам під час тривалих операцій.

У MCP потокова передача — це не про відправлення основної відповіді частинами, а про відсилання **повідомлень** клієнту під час опрацювання запиту інструментом. Ці повідомлення можуть містити оновлення прогресу, журнали або інші події.

### Як це працює

Основний результат надсилається як одна відповідь. Однак повідомлення можуть надсилатися окремо під час обробки і таким чином оновлюють клієнта в реальному часі. Клієнт має вміти обробляти і відображати ці повідомлення.

## Що таке повідомлення (Notification)?

Ми сказали "повідомлення", що це означає у контексті MCP?

Повідомлення — це повідомлення від сервера клієнту, яке інформує про прогрес, стан чи інші події під час довготривалої операції. Повідомлення покращують прозорість і користувацький досвід.

Наприклад, клієнт повинен надіслати повідомлення, щойно відбудеться початкове рукопотискання з сервером.

Повідомлення виглядає як JSON повідомлення так:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

Повідомлення належать до теми в MCP, яка називається ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging).

> **Оголошення про застарілість:** кандидат на реліз MCP `2026-07-28` позначає примітив Logging як застарілий на користь `stderr` для транспорту stdio та OpenTelemetry для структурованої спостережності. Logging буде працювати у версії `2025-11-25` і принаймні рік після будь-якого офіційного оголошення про застарілість. Див. [What's Changing in MCP: The 2026-07-28 Release Candidate](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md).

Щоб увімкнути логування, сервер повинен активувати цю функцію/можливість таким чином:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> Залежно від SDK, логування може бути ввімкнене за замовчуванням, або потрібно явне увімкнення у конфігурації сервера.

Існують різні типи повідомлень:

| Рівень    | Опис                         | Приклад використання           |
|----------|------------------------------|-------------------------------|
| debug    | Детальна інформація для налагодження | Вхід/вихід функції            |
| info     | Загальні інформаційні повідомлення   | Оновлення прогресу операції    |
| notice   | Звичайні, але важливі події           | Зміни конфігурації             |
| warning  | Стан попередження                    | Використання застарілої функції |
| error    | Стан помилки                       | Збій операції                  |
| critical | Критичні стани                    | Збої компонентів системи       |
| alert    | Необхідно негайно вжити заходів  | Виявлено корупцію даних        |
| emergency| Система непридатна до роботи    | Повний збій системи            |

## Реалізація повідомлень у MCP

Щоб реалізувати повідомлення у MCP, потрібно налаштувати як серверну, так і клієнтську частини для обробки оновлень у реальному часі. Це дозволяє вашому додатку надавати миттєвий зворотний зв’язок користувачам під час тривалих операцій.

### Серверна частина: надсилання повідомлень

Почнемо з серверної сторони. У MCP ви визначаєте інструменти, які можуть надсилати повідомлення під час обробки запитів. Сервер використовує об’єкт контексту (зазвичай `ctx`) для відправки повідомлень клієнту.

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

У наведеному прикладі інструмент `process_files` надсилає три повідомлення клієнту під час обробки кожного файлу. Метод `ctx.info()` використовується для відправки інформаційних повідомлень.

Крім того, щоб увімкнути повідомлення, переконайтеся, що сервер використовує потоковий транспорт (наприклад, `streamable-http`), а клієнт реалізує обробник повідомлень для обробки повідомлень. Ось як можна налаштувати сервер для використання транспорту `streamable-http`:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

У цьому прикладі .NET інструмент `ProcessFiles` позначений атрибутом `Tool` і надсилає три повідомлення клієнту під час обробки кожного файлу. Метод `ctx.Info()` використовується для відправки інформаційних повідомлень.

Щоб увімкнути повідомлення у вашому MCP сервері .NET, переконайтеся, що використовується потоковий транспорт:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### Клієнтська частина: отримання повідомлень

Клієнт повинен реалізувати обробник повідомлень для обробки та відображення повідомлень під час їх надходження.

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

У наведеному коді функція `message_handler` перевіряє, чи є вхідне повідомлення повідомленням (notification). Якщо так, воно виводиться; інакше обробляється як звичайне серверне повідомлення. Також зверніть увагу, як `ClientSession` ініціалізується з `message_handler` для обробки вхідних повідомлень.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

У цьому прикладі .NET функція `MessageHandler` перевіряє, чи є вхідне повідомлення повідомленням. Якщо так, воно друкується; інакше обробляється як звичайне повідомлення сервера. `ClientSession` ініціалізується з обробником повідомлень через `ClientSessionOptions`.

Щоб увімкнути повідомлення, переконайтеся, що сервер використовує потоковий транспорт (наприклад, `streamable-http`), а клієнт реалізує обробник повідомлень для обробки повідомлень.

## Повідомлення про прогрес і сценарії

У цьому розділі пояснюється концепція повідомлень про прогрес у MCP, чому вони важливі та як їх реалізувати за допомогою Streamable HTTP. Також ви знайдете практичне завдання для закріплення знань.

Повідомлення про прогрес — це повідомлення в реальному часі, які сервер надсилає клієнту під час тривалих операцій. Замість очікування завершення процесу сервер постійно інформує клієнта про поточний стан. Це підвищує прозорість, користувацький досвід і полегшує налагодження.

**Приклад:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### Чому варто використовувати повідомлення про прогрес?

Повідомлення про прогрес важливі з кількох причин:

- **Кращий користувацький досвід:** Користувачі бачать оновлення під час роботи, а не лише в кінці.
- **Зворотний зв’язок у реальному часі:** Клієнти можуть показувати індикатори прогресу або журнали, роблячи додаток більш відзивчивим.
- **Простіше налагодження та моніторинг:** Розробники і користувачі можуть бачити, де процес може сповільнюватися або зависати.

### Як реалізувати повідомлення про прогрес

Ось як можна реалізувати повідомлення про прогрес у MCP:

- **На сервері:** Використовуйте `ctx.info()` або `ctx.log()` для надсилання повідомлень під час обробки кожного елемента. Це надсилає повідомлення клієнту до того, як буде готовий основний результат.
- **На клієнті:** Реалізуйте обробник повідомлень, який слухає та відображає повідомлення в міру їх надходження. Цей обробник розрізняє повідомлення та кінцевий результат.

**Приклад сервера:**


#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**Приклад клієнта:**

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## Міркування про безпеку

При впровадженні серверів MCP з HTTP-базованими транспортами безпека стає першочерговим питанням, яке вимагає ретельної уваги до кількох векторів атак та механізмів захисту.

### Огляд

Безпека є критичною під час публічного доступу сервера MCP через HTTP. Стрімінговий HTTP вводить нові поверхні для атак і вимагає ретельного налаштування.

### Ключові моменти

- **Перевірка заголовка Origin**: Завжди перевіряйте заголовок `Origin`, щоб запобігти атакам DNS rebinding.
- **Прив’язка до localhost**: Для локальної розробки підключайте сервери до `localhost`, щоб уникнути їх відкриття у публічному інтернеті.
- **Аутентифікація**: Запровадьте аутентифікацію (наприклад, API-ключі, OAuth) для промислового застосування.
- **CORS**: Налаштуйте політики Cross-Origin Resource Sharing (CORS) для обмеження доступу.
- **HTTPS**: Використовуйте HTTPS у виробничому середовищі для шифрування трафіку.

### Найкращі практики

- Ніколи не довіряйте вхідним запитам без перевірки.
- Логування та моніторинг усіх доступів та помилок.
- Регулярно оновлюйте залежності для усунення вразливостей безпеки.

### Виклики

- Балансування безпеки та зручності розробки
- Забезпечення сумісності з різними клієнтськими середовищами

## Оновлення з SSE до Streamable HTTP

Для додатків, які зараз використовують Server-Sent Events (SSE), перехід на Streamable HTTP забезпечує розширені можливості та кращу довгострокову підтримку для ваших реалізацій MCP.

### Чому варто оновитися?

Є дві переконливі причини перейти з SSE на Streamable HTTP:

- Streamable HTTP пропонує кращу масштабованість, сумісність і багатший функціонал сповіщень, ніж SSE.
- Це рекомендований транспорт для нових MCP-додатків.

### Кроки міграції

Ось як ви можете перейти з SSE на Streamable HTTP у своїх MCP-додатках:

- **Оновіть серверний код** для використання `transport="streamable-http"` у `mcp.run()`.
- **Оновіть клієнтський код** для використання `streamablehttp_client` замість клієнта SSE.
- **Реалізуйте обробник повідомлень** у клієнта для обробки сповіщень.
- **Перевірте сумісність** з існуючими інструментами та робочими процесами.

### Підтримка сумісності

Рекомендується підтримувати сумісність з існуючими клієнтами SSE під час міграції. Ось деякі стратегії:

- Ви можете підтримувати обидва протоколи SSE та Streamable HTTP, запускаючи їх на різних кінцевих точках.
- Поступово переводьте клієнтів на новий транспорт.

### Виклики

Під час міграції обов’язково врахуйте наступні виклики:

- Забезпечення оновлення всіх клієнтів
- Обробка різниць у доставці сповіщень
