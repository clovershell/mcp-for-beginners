# मोडल कन्टेक्स्ट प्रोटोकल (MCP) सँग HTTPS स्ट्रिमिङ

यस अध्यायले HTTPS प्रयोग गरी मोडल कन्टेक्स्ट प्रोटोकल (MCP) सँग सुरक्षित, स्केलेबल, र वास्तविक-समय स्ट्रिमिङ लागू गर्ने पूर्ण मार्गदर्शन प्रदान गर्दछ। यसले स्ट्रिमिङको प्रेरणा, उपलब्ध ट्रान्सपोर्ट मेकानिजमहरू, MCP मा स्ट्रिमेबल HTTP कसरी लागू गर्ने, सुरक्षा उत्तम अभ्यासहरू, SSE बाट माईग्रेशन, र तपाईंका आफ्नै स्ट्रिमिङ MCP अनुप्रयोगहरू निर्माण गर्ने व्यावहारिक मार्गदर्शन समेटेको छ। 

> **अगाडि हेर्दै:** यो पाठले **MCP विनिर्देशन 2025-11-25** अन्तर्गत स्ट्रिमेबल HTTP वर्णन गर्दछ, जहाँ `initialize` गर्दा सत्र स्थापना हुन्छ र `Mcp-Session-Id` हेडरले पिन गरिएको हुन्छ। `2026-07-28` रिलिज क्यान्डिडेटले ह्यान्डशेक र सत्र ID पूर्ण रूपमा हटाउँछ, जसले प्रत्येक अनुरोधलाई आत्मनिर्भर र कुनै पनि सर्भर उदाहरणमा राउटेबल बनाउँछ, बिना स्टिकी सत्र। विस्तृत विवरणका लागि [MCP मा के परिवर्तन हुँदैछ: 2026-07-28 रिलिज क्यान्डिडेट](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) हेर्नुहोस्।

## MCP मा ट्रान्सपोर्ट मेकानिजम र स्ट्रिमिङ

यो खण्डले MCP मा उपलब्ध फरक-फरक ट्रान्सपोर्ट मेकानिजमहरू र क्लाइन्ट र सर्भर बीच वास्तविक-समय संवाद सक्षम गर्ने स्ट्रिमिङ क्षमता समेट्ने तिनीहरूको भूमिका विश्लेषण गर्दछ।

### ट्रान्सपोर्ट मेकानिजम के हो?

ट्रान्सपोर्ट मेकानिजमले कसरी क्लाइन्ट र सर्भर बीच डेटा आदान प्रदान हुन्छ भन्ने निर्धारण गर्छ। MCP विभिन्न वातावरण र आवश्यकता अनुसार धेरै ट्रान्सपोर्ट प्रकारहरू समर्थन गर्छ:

- **stdio**: मानक इनपुट/आउटपुट, स्थानीय र CLI-आधारित उपकरणहरूका लागि उपयुक्त। सरल तर वेब वा क्लाउडका लागि उपयुक्त छैन।
- **SSE (सर्भर-पठाइएको घटनाहरू)**: HTTP मार्फत सर्भरहरूले क्लाइन्टहरूलाई वास्तविक-समय अपडेटहरू पठाउन अनुमति दिन्छ। वेब UI का लागि राम्रो, तर स्केलेबिलिटी र लचकता मा सीमित। MCP विनिर्देशन 2025-06-18 बाट, स्वतन्त्र SSE ट्रान्सपोर्ट अवैध गरिएको छ र "Streamable HTTP" ट्रान्सपोर्टले प्रतिस्थापन गरेको छ।
- **Streamable HTTP**: आधुनिक HTTP-आधारित स्ट्रिमिङ ट्रान्सपोर्ट, सूचना र उन्नत स्केलेबिलिटी समर्थन गर्दछ। अधिकांश उत्पादन र क्लाउड परिदृश्यका लागि सिफारिस गरिएको।

### तुलना तालिका

यी ट्रान्सपोर्ट मेकानिजमहरू बीचको भिन्नता बुझ्न तलको तुलना तालिका हेरौं:

| ट्रान्सपोर्ट    | वास्तविक-समय अपडेटहरू | स्ट्रिमिङ | स्केलेबिलिटी | प्रयोग केस              |
|----------------|-----------------------|-----------|--------------|-------------------------|
| stdio          | छैन                    | छैन       | कम           | स्थानीय CLI उपकरणहरू    |
| SSE            | छ                     | छ        | मध्यम        | वेब, वास्तविक-समय अपडेट|
| Streamable HTTP| छ                     | छ        | उच्च         | क्लाउड, बहु क्लाइन्ट   |

> **सुझाव:** सहि ट्रान्सपोर्ट छान्नु प्रदर्शन, स्केलेबिलिटी, र प्रयोगकर्ता अनुभवलाई प्रभाव पार्छ। **Streamable HTTP** आधुनिक, स्केलेबल, र क्लाउड-तैयार अनुप्रयोगहरूको लागि सिफारिस गरिएको छ।

तपाईंले अघिल्लो अध्यायहरूमा देख्नुभएको stdio र SSE ट्रान्सपोर्टहरूलाई ध्यान दिनुहोस् र यस अध्यायमा समेटिएको स्ट्रिमेबल HTTP ट्रान्सपोर्ट कस्तो छ।

## स्ट्रिमिङ: अवधारणाहरू र प्रेरणा

स्ट्रिमिङको मौलिक अवधारणाहरू र प्रेरणाहरू बुझ्नु प्रभावकारी वास्तविक-समय संवाद प्रणालीहरू लागू गर्न अत्यावश्यक छ।

**स्ट्रिमिङ** नेटवर्क प्रोग्रामिङको एउटा प्रविधि हो जसले पूर्ण प्रतिक्रिया तयार हुन कुर्नुको सट्टामा सानो सानो टुक्रा वा घटनाहरूको श्रृंखला रूपमा डेटा पठाउन र प्राप्त गर्न अनुमति दिन्छ। यो विशेष गरी उपयोगी छ:

- ठूला फाइलहरू वा डाटासेटहरू।
- वास्तविक-समय अपडेटहरू (जस्तै, च्याट, प्रगति पट्टीहरू)।
- लामो समय चल्ने गणनाहरू जहाँ तपाईं प्रयोगकर्तालाई सूचित गर्न चाहनुहुन्छ।

प्रस्तुतिमा तपाईंले माथिल्लो स्तरमा स्ट्रिमिङको बारेमा जान्नुपर्ने कुरा:

- डेटा क्रमिक रूपमा वितरण हुन्छ, सबै एकै पटक होइन।
- क्लाइन्ट डेटा आउँदा प्रक्रियाकरण गर्न सक्छ।
- अनुभूति गरिएको लेटेन्सी घटाउँछ र प्रयोगकर्ता अनुभव सुधार गर्दछ।

### किन स्ट्रिमिङ प्रयोग गर्ने?

स्ट्रिमिङ प्रयोग गर्ने कारणहरू निम्न छन्:

- प्रयोगकर्ताहरू तुरुन्त प्रतिक्रिया पाउँछन्, मात्र अन्त्यमा होइन
- वास्तविक-समय अनुप्रयोगहरू र प्रतिक्रियाशील UI सक्षम पार्दछ
- नेटवर्क र कम्प्युट स्रोतहरू अझ प्रभावकारी उपयोग

### सरल उदाहरण: HTTP स्ट्रिमिङ सर्भर र क्लाइन्ट

यहाँ कसरी स्ट्रिमिङ लागू गर्न सकिन्छ भन्ने सरल उदाहरण छ:

#### पाइथन

**सर्भर (पाइथन, FastAPI र StreamingResponse प्रयोग गरी):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**क्लाइन्ट (पाइथन, requests प्रयोग गरी):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

यो उदाहरणले एक सर्भरले सबै सन्देश तयार हुने कुर्नुको सट्टा उपलब्ध हुँदा क्लाइन्टलाई सन्देशहरूको श्रृंखला कसरी पठाउँछ भनेर देखाउँछ।

**यसले कसरी काम गर्छ:**

- सर्भर प्रत्येक सन्देश तयार हुँदा त्यसलाई पठाउँछ।
- क्लाइन्ट प्रत्येक टुक्रा प्राप्त गरी छाप्छ।

**आवश्यकताहरू:**

- सर्भरले स्ट्रिमिङ प्रतिक्रिया चाहिन्छ (जस्तै FastAPI मा `StreamingResponse`)।
- क्लाइन्टले प्रतिक्रियालाई स्ट्रिमको रूपमा प्रक्रिया गर्नुपर्छ (`stream=True` requests मा)।
- सामग्री प्रकार प्रायः `text/event-stream` वा `application/octet-stream` हुन्छ।

#### जाभा

**सर्भर (जाभा, Spring Boot र Server-Sent Events प्रयोग गरी):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**क्लाइन्ट (जाभा, Spring WebFlux WebClient प्रयोग गरी):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**जाभा कार्यान्वयन नोटहरू:**

- Spring Boot को प्रतिक्रिया स्ट्याक `Flux` मार्फत स्ट्रिमिङ गर्न प्रयोग हुन्छ
- `ServerSentEvent` ले घटनाहरूको प्रकार सहित संरचित घटना स्ट्रिमिङ दिन्छ
- `WebClient` को `bodyToFlux()` ले प्रतिक्रिया स्ट्रिमिङ उपभोग सक्षम पार्छ
- `delayElements()` ले घटनाहरू बीचमा प्रक्रिया समय अनुकरण गर्छ
- घटनाहरूले प्रकारहरू (जस्तै `info`, `result`) राख्न सक्छन् जसले क्लाइन्टलाई सजिलो बनाउँछ

### तुलना: क्लासिक स्ट्रिमिङ बनाम MCP स्ट्रिमिङ

"क्लासिक" तरिका र MCP स्ट्रिमिङ कसरी फरक हुन्छन् त अझै स्पष्ट पार्न तल देखाइएको छ:

| विशेषता               | क्लासिक HTTP स्ट्रिमिङ      | MCP स्ट्रिमिङ (सूचना)            |
|-----------------------|-----------------------------|---------------------------------|
| मुख्य प्रतिक्रिया      | टुक्राहरूमा पठाइन्छ          | एकै पटक, अन्त्यमा               |
| प्रगति अपडेटहरू       | डेटाको टुक्रा रूपमा पठाइन्छ | सूचनाको रूपमा पठाइन्छ            |
| क्लाइन्ट आवश्यकताहरू   | स्ट्रिम प्रक्रिया गर्नुपर्छ  | सन्देश ह्यान्डलर लागू गर्नुपर्छ  |
| प्रयोग केस             | ठूला फाइल, AI टोकन स्ट्रिम | प्रगति, लगहरू, वास्तविक-समय प्रतिक्रिया |

### मुख्य भिन्नताहरू

साथै, केहि मुख्य भिन्नताहरू:

- **संवाद ढाँचा:**
  - क्लासिक HTTP स्ट्रिमिङ: साधारण टुक्रा ट्रान्सफर इन्कोडिङ प्रयोग गरी डेटा टुक्राहरूमा पठाउँछ
  - MCP स्ट्रिमिङ: JSON-RPC प्रोटोकल सहित संरचित सूचना प्रणाली प्रयोग गर्छ

- **सन्देश ढाँचा:**
  - क्लासिक HTTP: सामान्य पाठ टुक्राहरू नयाँ पंक्तिसहित
  - MCP: मेटाडाटा सहित संरचित LoggingMessageNotification वस्तुहरू

- **क्लाइन्ट कार्यान्वयन:**
  - क्लासिक HTTP: स्ट्रिमिङ प्रतिक्रिया प्रक्रिया गर्ने साधारण क्लाइन्ट
  - MCP: विभिन्न प्रकारका सन्देशहरू प्रक्रिया गर्ने सन्देश ह्यान्डलरसहित अधिक परिष्कृत क्लाइन्ट

- **प्रगति अपडेटहरू:**
  - क्लासिक HTTP: प्रगति मुख्य प्रतिक्रिया स्ट्रिमको हिस्सा हो
  - MCP: प्रगति छुट्टै सूचना सन्देशमार्फत पठाइन्छ भने मुख्य परिणाम अन्त्यमा आउँछ

### सिफारिसहरू

क्लासिक स्ट्रिमिङ (/stream मार्फत देखाइएको) र MCP मार्फत स्ट्रिमिङ चयन गर्ने सन्दर्भमा हामीले केही सिफारिसहरू दिएका छौं।

- **सरल स्ट्रिमिङ आवश्यकताहरूका लागि:** क्लासिक HTTP स्ट्रिमिङ लागू गर्न सजिलो र आधारभूत आवश्यकताका लागि पर्याप्त छ।

- **जटिल, अन्तरक्रियात्मक अनुप्रयोगहरूको लागि:** MCP स्ट्रिमिङ थप संरचित दृष्टिकोण दिन्छ, धनी मेटाडाटा र सूचना तथा अन्तिम परिणाम बीच पृथक्करणसहित।

- **AI अनुप्रयोगहरूको लागि:** MCP को सूचना प्रणाली लामो समय सम्म चल्ने AI कार्यहरूको लागि विशेष उपयोगी छ जहाँ प्रगतिका बारे प्रयोगकर्तालाई सूचित राख्नुपर्छ।

## MCP मा स्ट्रिमिङ

तपाईंले अहिलेसम्म क्लासिक स्ट्रिमिङ र MCP स्ट्रिमिङ बीच केही सिफारिस र तुलना देख्नु भएको छ। अब MCP मा स्ट्रिमिङलाई कसरी प्रयोग गर्नेमा विशेष विवरणमा जाऔं।

MCP ढाँचाभित्र स्ट्रिमिङ कसरी काम गर्छ भन्ने बुझ्नु प्रतिक्रियाशील अनुप्रयोगहरू निर्माणका लागि आवश्यक छ जुन लामो समय चल्ने प्रक्रियाहरूमा प्रयोगकर्तालाई वास्तविक-समय प्रतिक्रिया दिन्छ।

MCP मा, स्ट्रिमिङ मुख्य प्रतिक्रिया टुक्रामा पठाउने कुरा होइन, तर अनुरोध प्रक्रिया हुँदै गर्दा क्लाइन्टलाई **सूचना** पठाउने कुरा हो। यी सूचनामा प्रगति अपडेटहरू, लगहरू, वा अन्य घटनाहरू समावेश हुन सक्छन्।

### यसले कसरी काम गर्छ

मुख्य परिणाम अझै एकल प्रतिक्रियाका रूपमा पठाइन्छ। यद्यपि प्रक्रिया भइरहेका बेला सूचनाहरू छुट्टै सन्देशका रूपमा पठाउन सकिन्छ र यसले क्लाइन्टलाई वास्तविक-समयमा अपडेट गर्दा मद्दत गर्दछ। क्लाइन्टले ती सूचनाहरू व्यवस्थापन र प्रदर्शन गर्न सक्षम हुनुपर्छ।

## सूचना के हो?

हामीले "सूचना" भनेका छौं, MCP सन्दर्भमा यसको अर्थ के हो?

सूचना भनेको सर्भरबाट क्लाइन्टलाई पठाइने सन्देश हो जुन प्रगति, स्थिती, वा अन्य घटनाहरूको बारेमा जानकारी दिन्छ। सूचनाले पारदर्शिता र प्रयोगकर्ता अनुभव सुधार गर्दछ।

उदाहरणका लागि, क्लाइन्टले सूचना पठाउनुपर्छ जब प्रारम्भिक ह्यान्डशेक सर्भरसँग सम्पन्न हुन्छ।

सूचना JSON सन्देशको रूपमा यस्तो देखिन्छ:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

सूचनाहरू MCP मा ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) शीर्षक अन्तर्गत पर्छन्।

> **अवैधता सूचना:** `2026-07-28` MCP विनिर्देशन रिलिज क्यान्डिडेटले Logging प्रिमिटिभलाई stdio ट्रान्सपोर्टहरूका लागि `stderr` र संरचित अवलोकनका लागि OpenTelemetry को पक्षमा अवैध ठान्दछ। Logging ले `2025-11-25` मा र औपचारिक अवैधतापछि कम्तीमा एक वर्षसम्म काम गर्नेछ। विस्तृत जानकारीका लागि [MCP मा के परिवर्तन हुँदैछ: 2026-07-28 रिलिज क्यान्डिडेट](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) हेर्नुहोस्।

Logging कार्यान्वयन गर्न सर्भरले यसलाई सुविधा/सक्षम बनाउनु पर्छ जस्तै:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> प्रयोग गरिएको SDK अनुसार, logging डिफल्टमा सक्षम हुन सक्छ वा तपाईंले यसलाई सर्भर कन्फिगरेसनमा स्पष्ट रूपमा सक्षम पार्नुपर्ने हुनसक्छ।

विभिन्न प्रकारका सूचनाहरू हुन्छन्:

| स्तर       | विवरण                      | उदाहरण प्रयोग केस              |
|------------|-----------------------------|--------------------------------|
| debug      | विस्तृत डिबगिङ जानकारी     | कार्य/फंक्शन सुरु/समाप्त स्थल |
| info       | सामान्य सूचना सन्देशहरू     | सञ्चालन प्रगति अपडेट          |
| notice     | सामान्य तर महत्वपूर्ण घटना | कन्फिगरेसन परिवर्तन             |
| warning    | चेतावनी स्थितिहरू          | अवैध सुविधा प्रयोग            |
| error      | त्रुटि स्थितिहरू           | सञ्चालन असफलता                 |
| critical   | गम्भीर स्थितिहरू           | प्रणालीका कम्पोनेन्ट विफलता     |
| alert      | तुरुन्त कारबाही आवश्यक       | डाटा क्षति पत्ता लगियो          |
| emergency  | प्रणाली अनुपयोगी            | पूर्ण प्रणाली विफलता            |

## MCP मा सूचना कार्यान्वयन

MCP मा सूचनाहरू कार्यान्वयन गर्न, सर्भर र क्लाइन्ट दुवै तर्फ वास्तविक-समय अपडेटहरू ह्यान्डल गर्न तयार हुनु पर्दछ। यसले लामो समय तक चल्ने प्रक्रियाहरूमा प्रयोगकर्तालाई तुरुन्त प्रतिक्रिया दिने अनुमति दिन्छ।

### सर्भर-पक्ष: सूचनाहरू पठाउने

सर्भर पक्षबाट सुरु गरौं। MCP मा, तपाईंले अनुरोध प्रक्रिया गर्दै गर्दा सूचना पठाउन सक्ने उपकरणहरू परिभाषित गर्नुहुन्छ। सर्भरले सन्देश पठाउन सामान्यतया सन्दर्भ वस्तु (`ctx`) प्रयोग गर्दछ।

#### पाइथन

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

माथिको उदाहरणमा, `process_files` टूलले प्रत्येक फाइल प्रक्रियाकरण गर्दा तीनवटा सूचनाहरू क्लाइन्टलाई पठाउँछ। `ctx.info()` विधिले सूचना सन्देश पठाउन प्रयोग गरिएको छ।

थप रूपमा, सूचनाहरू सक्षम गर्न, तपाईंको सर्भरले स्ट्रिमिङ ट्रान्सपोर्ट (जस्तै `streamable-http`) प्रयोग गर्नुपर्ने हुन्छ र क्लाइन्टले सूचना प्रक्रिया गर्न सन्देश ह्यान्डलर कार्यान्वयन गर्नुपर्छ। तपाईंले सर्भरलाई `streamable-http` ट्रान्सपोर्ट प्रयोग गर्न यसरी सेट अप गर्न सक्नुहुन्छ:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

यो .NET उदाहरणमा, `ProcessFiles` टूल `Tool` एट्रिब्युटसहित सजाइएको छ र प्रत्येक फाइल प्रक्रियाकरण गर्दा तीनवटा सूचनाहरू क्लाइन्टलाई पठाउँछ। `ctx.Info()` विधिले सूचना सन्देश पठाउन प्रयोग गरिएको छ।

तपाईंको .NET MCP सर्भरमा सूचनाहरू सक्षम गर्न, स्ट्रिमिङ ट्रान्सपोर्ट प्रयोग गर्दै हुनुहुन्छ कि छैन भनि सुनिश्चित गर्नुहोस्:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### क्लाइन्ट-पक्ष: सूचनाहरू प्राप्त गर्ने

क्लाइन्टले आउने सूचनाहरू प्रक्रिया र प्रदर्शन गर्न सन्देश ह्यान्डलर कार्यान्वयन गर्नुपर्छ।

#### पाइथन

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

माथिको कोडमा, `message_handler` फंक्शनले भन्छ कि यदि आउने सन्देश सूचना हो भने त्यसलाई छाप्छ; नभए सामान्य सर्भर सन्देश रूपमा प्रक्रिया गर्छ। साथै `ClientSession` लाई सूचनाहरू ह्यान्डल गर्न `message_handler`सहित सुरु गरिएको छ।

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

यस .NET उदाहरणमा, `MessageHandler` फंक्शनले भन्छ कि यदि आउने सन्देश सूचना हो भने त्यसलाई छाप्छ; नभए सामान्य सर्भर सन्देशको रूपमा प्रक्रिया गर्छ। `ClientSession` सन्देश ह्यान्डलरमार्फत `ClientSessionOptions` प्रयोग गरी सुरु गरिएको छ।

सूचनाहरू सक्षम गर्न, सर्भरले स्ट्रिमिङ ट्रान्सपोर्ट (जस्तै `streamable-http`) प्रयोग गर्नुपर्ने र क्लाइन्टले सूचनाहरू प्रक्रिया गर्न सन्देश ह्यान्डलर कार्यान्वयन गर्नुपर्ने कुरा सुनिश्चित गर्नुहोस्।

## प्रगति सूचनाहरू र प्रयोग परिदृश्यहरू

यस खण्डले MCP मा प्रगति सूचनाको अवधारणा, यसको महत्व, र Streamable HTTP प्रयोग गरी कसरी यिनलाई लागू गर्ने व्याख्या गर्दछ। तपाईंलाई बुझाइ थप बलियो बनाउन व्यावहारिक काम पनि पाउनुहुनेछ।

प्रगति सूचनाहरू लामो समय चल्ने प्रक्रिया हुँदै गर्दा सर्भरबाट क्लाइन्टलाई पठाइने वास्तविक-समय सन्देशहरू हुन्। पूर्ण प्रक्रिया सकिन कुर्नुको सट्टा सर्भरले क्लाइन्टलाई हालको स्थितिबारे अपडेट राख्छ। यसले पारदर्शिता, प्रयोगकर्ता अनुभव सुधार र डिबगिंग सजिलो बनाउँछ।

**उदाहरण:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### किन प्रगति सूचनाहरू प्रयोग गर्ने?

प्रगति सूचनाहरूका धेरै कारणहरू छन्:

- **राम्रो प्रयोगकर्ता अनुभव:** प्रयोगकर्ताहरू काम अघि बढ्दै गर्दा अपडेटहरू देख्छन्, मात्र अन्त्यमा होइन।
- **वास्तविक-समय प्रतिक्रिया:** क्लाइन्टहरूले प्रगति बार वा लग देखाउन सक्छन्, जसले अनुप्रयोगलाई प्रतिक्रियाशील बनाउँछ।
- **डिबगिंग र अनुगमन सजिलो:** विकासकर्ता र प्रयोगकर्ताहरू प्रक्रिया कहाँ ढिलो वा अड्किएको छ देख्न सक्छन्।

### प्रगति सूचनाहरू कसरी लागू गर्ने

यहाँ MCP मा प्रगति सूचनाहरू लागू गर्ने तरिका छ:

- **सर्भरमा:** प्रत्येक वस्तु प्रक्रियाकरण हुँदा `ctx.info()` वा `ctx.log()` मार्फत सूचनाहरू पठाउनुहोस्। यसले मुख्य परिणाम तयार हुनु अघि क्लाइन्टलाई सन्देश पठाउँछ।
- **क्लाइन्टमा:** एउटा सन्देश ह्यान्डलर लागू गर्नुहोस् जसले आउने सूचनाहरू सुन्छ र प्रदर्शन गर्छ। यस ह्यान्डलरले सूचनाहरू र अन्तिम परिणामबीच भेद गर्दछ।

**सर्भर उदाहरण:**


#### पाइथन

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**ग्राहक उदाहरण:**

#### पाइथन

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## सुरक्षा विचारहरू

HTTP-आधारित ट्रान्सपोर्टहरूसँग MCP सर्भरहरू कार्यान्वयन गर्दा, सुरक्षा एक प्रमुख चिन्ता बन्छ जसले धेरै आक्रमण मार्गहरू र सुरक्षा उपायहरूमा सावधानीपूर्वक ध्यान आवश्यक पर्छ।

### अवलोकन

MCP सर्भरहरूलाई HTTP मा प्रदर्शन गर्दा सुरक्षा अत्यन्त महत्त्वपूर्ण हुन्छ। स्ट्रिमेबल HTTP ले नयाँ आक्रमण सतहहरू ल्याउँछ र सावधानीपूर्वक कन्फिगरेसन आवश्यक छ।

### मुख्य बुँदाहरू

- **Origin हेडर प्रमाणीकरण**: DNS रिबाइन्डिंग आक्रमण रोक्न सधैं `Origin` हेडर प्रमाणीकरण गर्नुहोस्।
- **Localhost बाइन्डिङ**: स्थानीय विकासको लागि सर्भरहरूलाई `localhost` मा बाइन्ड गर्नुहोस् ताकि तिनीहरूलाई सार्वजनिक इन्टरनेटमा प्रदर्शन नहोस्।
- **प्रमाणीकरण**: उत्पादन डिप्लोयमेन्टहरूका लागि प्रमाणीकरण (जस्तै API कुञ्जीहरू, OAuth) लागू गर्नुहोस्।
- **CORS**: पहुँच सीमित गर्न क्रस-ओरिजिन रिसोर्स शेयरिङ (CORS) नीतिहरू कन्फिगर गर्नुहोस्।
- **HTTPS**: ट्राफिक इन्क्रिप्ट गर्न उत्पादनमा HTTPS प्रयोग गर्नुहोस्।

### सर्वोत्तम अभ्यासहरू

- प्रमाणीकरणबिना आउने अनुरोधहरूमा कहिल्यै विश्वास नगर्नुहोस्।
- सबै पहुँच र त्रुटिहरूलाई लगिङ र अनुगमन गर्नुहोस्।
- सुरक्षा कमजोरिहरू ठेक्का गर्न नियमित रूपमा निर्भरताहरू अपडेट गर्नुहोस्।

### चुनौतीहरू

- विकास सजिलो हुनु र सुरक्षा सन्तुलन गर्नु
- विभिन्न ग्राहक वातावरणहरूसँग कम्प्याटिबिलिटी सुनिश्चित गर्नु

## SSE बाट स्ट्रिमेबल HTTP मा अपग्रेड

हाल Server-Sent Events (SSE) प्रयोग गरिरहेका अनुप्रयोगहरूको लागि, स्ट्रिमेबल HTTP मा माइग्रेट गर्दा तपाईँका MCP कार्यान्वयनहरूमा बढेको क्षमता र दीगो दीर्घकालीन सम्भावना प्राप्त हुन्छ।

### किन अपग्रेड गर्ने?

SSE बाट स्ट्रिमेबल HTTP मा अपग्रेड गर्न दुई कारणहरू छन्:

- स्ट्रिमेबल HTTP ले SSE भन्दा राम्रो स्केलेबिलिटी, कम्प्याटिबिलिटी र धनी सूचना समर्थन प्रदान गर्दछ।
- यो नयाँ MCP अनुप्रयोगहरूको लागि सिफारिस गरिएको ट्रान्सपोर्ट हो।

### माइग्रेशन चरणहरू

यहाँ कसरी तपाईंले SSE बाट स्ट्रिमेबल HTTP मा माइग्रेट गर्न सक्नुहुन्छ:

- `mcp.run()` मा `transport="streamable-http"` प्रयोग गर्न सर्भर कोड अपडेट गर्नुहोस्।
- SSE ग्राहकको सट्टामा `streamablehttp_client` प्रयोग गर्न ग्राहक कोड अपडेट गर्नुहोस्।
- सूचनाहरू प्रशोधन गर्न ग्राहकमा मेसेज ह्यान्डलर लागू गर्नुहोस्।
- विद्यमान उपकरणहरू र कार्यप्रवाहहरूसँग कम्प्याटिबिलिटी परीक्षण गर्नुहोस्।

### कम्प्याटिबिलिटी कायम राख्नु

माइग्रेशनको क्रममा विद्यमान SSE ग्राहकहरूसँग कम्प्याटिबिलिटी कायम राख्न सिफारिस गरिन्छ। केही रणनीतिहरू यस्ता छन्:

- SSE र स्ट्रिमेबल HTTP दुवैलाई विभिन्न अन्त बिन्दुहरूमा सञ्चालन गरेर समर्थन गर्न सकिन्छ।
- क्रमिक रूपमा ग्राहकहरूलाई नयाँ ट्रान्सपोर्टतर्फ माइग्रेट गर्नुहोस्।

### चुनौतीहरू

माइग्रेशनको क्रममा निम्न चुनौतीहरू समाधान गर्नुपर्छ:

- सबै ग्राहकहरू अपडेट गरिएको सुनिश्चित गर्नु
- सूचना प्रदानमा भिन्नताहरूको व्यवस्थापन गर्नु

## सुरक्षा विचारहरू

HTTP-आधारित ट्रान्सपोर्टहरू, विशेष गरी MCP मा स्ट्रिमेबल HTTP प्रयोग गर्दा सुरक्षा उच्च प्राथमिकता हुनुपर्छ।

HTTP-आधारित ट्रान्सपोर्टहरूसँग MCP सर्भरहरू कार्यान्वयन गर्दा, सुरक्षा एक प्रमुख चिन्ता बन्छ जसले धेरै आक्रमण मार्गहरू र सुरक्षा उपायहरूमा सावधानीपूर्वक ध्यान आवश्यक पर्छ।

### अवलोकन

MCP सर्भरहरूलाई HTTP मा प्रदर्शन गर्दा सुरक्षा अत्यन्त महत्त्वपूर्ण हुन्छ। स्ट्रिमेबल HTTP ले नयाँ आक्रमण सतहहरू ल्याउँछ र सावधानीपूर्वक कन्फिगरेसन आवश्यक छ।

यहाँ केही प्रमुख सुरक्षा विचारहरू छन्:

- **Origin हेडर प्रमाणीकरण**: DNS रिबाइन्डिंग आक्रमण रोक्न सधैं `Origin` हेडर प्रमाणीकरण गर्नुहोस्।
- **Localhost बाइन्डिङ**: स्थानीय विकासको लागि सर्भरहरूलाई `localhost` मा बाइन्ड गर्नुहोस् ताकि तिनीहरूलाई सार्वजनिक इन्टरनेटमा प्रदर्शन नहोस्।
- **प्रमाणीकरण**: उत्पादन डिप्लोयमेन्टहरूका लागि प्रमाणीकरण (जस्तै API कुञ्जीहरू, OAuth) लागू गर्नुहोस्।
- **CORS**: पहुँच सीमित गर्न क्रस-ओरिजिन रिसोर्स शेयरिङ (CORS) नीतिहरू कन्फिगर गर्नुहोस्।
- **HTTPS**: ट्राफिक इन्क्रिप्ट गर्न उत्पादनमा HTTPS प्रयोग गर्नुहोस्।

### सर्वोत्तम अभ्यासहरू

तपाईंको MCP स्ट्रिमिङ सर्भरमा सुरक्षा कार्यान्वयन गर्दा, अनुसरण गर्न केही सर्वोत्तम अभ्यासहरू यहाँ छन्:

- प्रमाणीकरणबिना आउने अनुरोधहरूमा कहिल्यै विश्वास नगर्नुहोस्।
- सबै पहुँच र त्रुटिहरूलाई लगिङ र अनुगमन गर्नुहोस्।
- सुरक्षा कमजोरिहरू ठेक्का गर्न नियमित रूपमा निर्भरताहरू अपडेट गर्नुहोस्।

### चुनौतीहरू

MCP स्ट्रिमिङ सर्भरमा सुरक्षा कार्यान्वयन गर्दा केही चुनौतीहरू सामना गर्नु पर्नेछ:

- विकास सजिलो हुनु र सुरक्षा सन्तुलन गर्नु
- विभिन्न ग्राहक वातावरणहरूसँग कम्प्याटिबिलिटी सुनिश्चित गर्नु

### कार्य: तपाईँको आफ्नै स्ट्रिमिङ MCP अनुप्रयोग बनाउनुहोस्

**परिदृश्य:**
एउटा MCP सर्भर र ग्राहक बनाउनुहोस् जहाँ सर्भरले वस्तुहरूको सूची (जस्तै, फाइलहरू वा कागजातहरू) प्रक्रिया गर्छ र हरेक प्रक्रियाकृत वस्तुको लागि सूचना पठाउँछ। ग्राहकले प्रत्येक सूचना जति बेला आउँछ देखाउनु पर्छ।

**चरणहरू:**

1. एउटा सर्भर उपकरण कार्यान्वयन गर्नुहोस् जसले सूची प्रोसेस गर्छ र हरेक वस्तुको लागि सूचना पठाउँछ।
2. एउटा ग्राहक कार्यान्वयन गर्नुहोस् जसमा मेसेज ह्यान्डलर हुन्छ र सूचना वास्तविक समयमा देखाउँछ।
3. दुवै सर्भर र ग्राहक चलाएर आफ्नो कार्यान्वयन परीक्षण गर्नुहोस् र सूचनाहरू अवलोकन गर्नुहोस्।
