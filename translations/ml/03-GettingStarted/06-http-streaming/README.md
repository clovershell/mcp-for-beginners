# മോഡല്‍ കോൺടെക്സ്റ്റ് പ്രോട്ടോക്കോള്‍ (MCP) ഉപയോഗിച്ച് HTTPS സ്ട്രീമിംഗ്

ഈ അധ്യായം HTTPS ഉപയോഗിച്ച് മോഡല്‍ കോൺടെയ്ന്‍റ് പ്രോട്ടോക്കോള്‍ (MCP) വഴി സുരക്ഷിതവും സ്കെയിലബിളുമുള്ള റിയല്‍ ടൈം സ്ട്രീമിംഗ് നടപ്പിലാക്കുന്നതിനുള്ള സമഗ്രമായ മാര്‍ഗനിര്‍ദേശമാണ്. ഇത് സ്ട്രീമിംഗ് നടത്താനുള്ള പ്രേരണ, ലഭ്യമായ ട്രാന്‍സ്‌പോര്‍ട്ട് മേക്കാനിസം, MCP-യില്‍ സ്ട്രീമബിള്‍ HTTP എങ്ങനെ നടപ്പാക്കാം, സുരക്ഷാ മികച്ച രീതികള്‍, SSE-യില്‍ നിന്നുള്ള മൈഗ്രേഷന്‍, നിങ്ങളുടെ സ്വന്തം സ്ട്രീമിംഗ് MCP അപ്ലിക്കേഷനുകള്‍ നിര്‍മ്മിക്കുന്നതിന് പ്രായോഗിക മാർഗനിർദേശങ്ങൾ എന്നിവ ആനുകൂല്യപ്പെടുന്നു.

> **നേര്‍ത്തുനോക്കുമ്പോള്‍:** ഈ പാഠം **MCP സ്പെസിഫിക്കേഷന്‍ 2025-11-25** പ്രകാരം സ്ട്രീമബിള്‍ HTTP-യെ വിശദീകരിക്കുന്നു, ഇവിടെ സെഷന്‍ `initialize` സമയത്ത് സ്ഥാപിച്ച് `Mcp-Session-Id` ഹെഡർ ഉപയോഗിച്ച് പിന്‍ചൂറ്റി നിര്‍ത്തുന്നു. `2026-07-28` റിലീസ് കണ്ഹിഡേറ്റ് പൂർണ്ണമായും ഹാന്‍ഡ്‌ഷേക്ക്, സെഷന്‍ ഐഡി നീക്കംചെയ്യും, ഇതിലൂടെ ഓരോ അഭ്യര്‍ത്ഥനയും സ്വയം പ്രധാനമായും കൈകാര്യം ചെയ്‌തും സ്റ്റിക്കി സെഷനുകള്‍ ഇല്ലാതെ ഏതു സര്‍വര്‍ ഇന്‍സ്റ്റന്‍സിലേക്കും റൂട്ടുചെയ്യാവുന്നതായിരിക്കും. വിശദാംശങ്ങള്‍ക്കായി [MCP-യിലെ മാറ്റങ്ങള്‍: 2026-07-28 റിലീസ് കണ്ഹിഡേറ്റ്](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) കാണുക.

## MCP-യിലെ ട്രാന്‍സ്‌പോര്‍ട്ട് മേക്കാനിസങ്ങളും സ്ട്രീമിങ്ങും

MCP-യിൽ ലഭ്യമായ വ്യത്യസ്ത ട്രാൻസ്പോർട്ട് മേക്കാനിസങ്ങളെയും അവയുടെ ജോലി ഗുരുത്വവും റിയൽ ടൈം ക്ലയന്റ്-സർവർ സ്ട്രീമിംഗ് കഴിവുകൾ സജ്ജമാക്കുന്നതിൽ അവയുടെ പങ്ക് ഇവിടെ വിശദീകരിക്കുന്നു.

### ട്രാന്‍സ്‌പോര്‍ട്ട് മേക്കാനിസ്‌ം എന്താണ്?

ട്രാന്‍സ്‌പോര്‍ട്ട് മേക്കാനിസം ക്ലയന്റും സര്‍വറും എങ്ങനെ ഡാറ്റ കൈമാറുന്നു എന്ന് നിർവചിക്കുന്നു. MCP വിവിധ പരിമിതികളോടും ആവശ്യങ്ങളിൽ പൊരുത്തപ്പെടുന്ന നിരവധി ട്രാൻസ്പോർട്ട് തരം പിന്തുണയ്ക്കുന്നു:

- **stdio**: സ്റ്റാൻഡേർഡ് ഇൻപുട്ട്/ഔട്ട്പുട്ട്, ലൊക്കൽ CLI അടിസ്ഥാനമുള്ള ഉപകരണങ്ങൾക്ക് അനുയോജ്യം. ലളിതമായതു, പക്ഷേ വെബ് അല്ലെങ്കിൽ ക്ലൗഡിന് അനുയോജ്യമല്ല.
- **SSE (Server-Sent Events)**: HTTP വഴിയായി സെർവർകൾ റിയൽ-ടൈം അപ്‌ഡേറ്റുകൾ ക്ലയന്റുകളിലേക്ക് പുഷ് ചെയ്യാം. വെബ് UIs-ക്ക് നല്ലത്, പക്ഷേ സ്കെയിലബിലിറ്റി, ഫ്‌ളെക്സിബിലിറ്റിയിൽ പരിമിതമാണ്. MCP സ്പെസിഫിക്കേഷൻ 2025-06-18-ന് സമകാലികമായി, സ്റ്റാന്‍‍ഡലോണ്‍ SSE (Server-Sent Events) ട്രാന്‍സ്‌പോർട്ട് ഓഫ് ചെയ്യപ്പെട്ടു, അതിനെ "Streamable HTTP" ട്രാന്‍സ്‌പോർട്ട് മാറ്റി.
- **Streamable HTTP**: ആധുനിക HTTP അടിസ്ഥാനമായ സ്ട്രീമിംഗ് ട്രാൻസ്പോർട്ട്, അറിയിപ്പുകൾക്കും മെച്ചപ്പെട്ട സ്കെയിൽ ചെയ്യലിനും പിന്തുണ നൽകുന്നു. ബഹുഭൂരിപക്ഷ ഊത്പാദനവും ക്ലൗഡ് സാഹചര്യങ്ങൾക്കുമായി ശുപാർശ ചെയ്യുന്നു.

### താരതമ്യ പട്ടിക

ഈ ടേബിൾ കാണുക, ഇവ ട്രാന്‍സ്‌പോര്‍ട്ടുകൾ തമ്മിലുള്ള വ്യത്യാസങ്ങൾ മനസിലാക്കാൻ സഹായിക്കും:

| ട്രാന്‍സ്‌പോര്‍ട്ട്         | റിയല്‍-ടൈം അപ്‌ഡേറ്റുകൾ | സ്ട്രീമിംഗ് | സ്കെയിലബിലിറ്റി | ഉപയോഗം                |
|--------------------------|---------------------|-----------|------------|------------------------|
| stdio                    | ഇല്ല                | ഇല്ല       | താഴ്ന്നത്  | ലൊക്കൽ CLI ഉപകരണങ്ങൾ |
| SSE                      | ഉണ്ട്               | ഉണ്ട്      | മധ്ധ്യമം | വെബ്, റിയല്‍-ടൈം അപ്‌ഡേറ്റ് |
| Streamable HTTP          | ഉണ്ട്               | ഉണ്ട്      | ഉയർന്നത്  | ക്ലൗഡ്, ബഹు-ക്ലയന്റ്   |

> **ടിപ്:** ശരിയായ ട്രാന്‍സ്‌പോര്‍ട്ട് തെരഞ്ഞെടുക്കുന്നത് പ്രകടനവും സ്കെയിലബിലിറ്റിയും ഉപയോക്തൃ അനുഭവവും ബാധിക്കും. **Streamable HTTP** ആധുനികവും സ്കെയിലബിൾയും ക്ലൗഡ്-പ്രദാന ആപ്ലിക്കേഷനുകൾക്കായി ശുപാർശ ചെയ്യുന്നു.

മുമ്പത്തെ അധ്യായങ്ങളിൽ കാണിച്ച stdio, SSE ട്രാന്‍സ്‌പോ‌ര്‍ട്ടുകൾ ശ്രദ്ധിക്കുക, ഈ അധ്യായത്തിൽ സ്വതന്ത്രമായ സ്ട്രീമബിള്‍ HTTP ആണ് വിശദീകരിച്ചിരിക്കുന്നത്.

## സ്ട്രീമിംഗ്: ആശയങ്ങൾക്കും പ്രേരണയ്ക്കും

സ്ട്രീമിംഗിന്റെ അടിസ്ഥാന ആശയങ്ങളും പ്രേരണകളും മനസ്സിലാക്കൽ പ്രത്യാശാപൂർവമായി പ്രവർത്തിക്കുന്ന റിയൽ-ടൈം കമ്മ്യൂണിക്കേഷൻ സിസ്റ്റങ്ങൾ നടപ്പിലാക്കാൻ അനിവാര്യമാണ്.

**സ്ട്രീമിംഗ്** നെറ്റ്‌വർക്ക് പ്രോഗ്രാമിങ്ങിൽ ഒരു സാങ്കേതിക വിദ്യയാണ്, മുഴുവൻ പ്രതികരണം തയ്യാറാകുന്നതിന് കാത്തിരിക്കുന്നതിന് പകരം ഡാറ്റ ചെറിയ സാമ്പിളുകളായി അല്ലെങ്കിൽ സംഭവങ്ങളായി ക്രമത്തിലും തയ്യാറാക്കിയതായാണ് അയയ്ക്കുന്നത്. ഇത് പ്രത്യേകിച്ച് താഴെ കണ്ട സാഹചര്യങ്ങളിൽ അനുയോജ്യമാണ്:

- വലുതായ ഫയലുകൾ അല്ലെങ്കിൽ ഡാറ്റാസെറ്റുകൾ.
- റിയൽ-ടൈം അപ്‌ഡേറ്റുകൾ (ഉദാഹരണം: ചാറ്റ്, പ്രോഗ്രസ് ബാറുകൾ).
- ദീർഘനേരം പ്രവർത്തിക്കുന്ന കംപ്യൂട്ടേഷൻവുകൾ, ഉപയോക്താവിനെ വിവരമറിയിക്കാൻ.

സ്ട്രീമിംഗിനെക്കുറിച്ച് നമുക്ക് അറിയേണ്ടതെന്തെന്ന് താഴെ കൊടുക്കുന്നു:

- ഡാറ്റ ക്രമത്തിലാണ് എത്തുന്നത്, മുഴുവൻ ഒരുമിച്ച് അല്ല.
- ക്ലയന്റ് ഡാറ്റ എത്തുന്ന പോലെ പ്രോസസ് ചെയ്യാം.
- കണക്കുവഞ്ചന കുറയ്ക്കുകയും ഉപയോക്തൃ അനുഭവം മെച്ചപ്പെടുത്തുകയും ചെയ്യുന്നു.

### സ്ട്രീമിംഗ് ഉപയോഗിക്കാനുള്ള കാരണം?

സ്ട്രീമിംഗ് ഉപയോഗിക്കാനുള്ള കാരണങ്ങൾ പിന്വരുന്നു:

- ഉപയോക്താക്കൾക്ക് ഉടൻ ഫീഡ്ബാക്ക് ലഭിക്കുന്നു, അവസാനത്തിലും മാത്രം അല്ല.
- റിയൽ-ടൈം ആപ്ലിക്കേഷനും പ്രതികരിക്കുന്ന UI-നുമായി സഹായിക്കുന്നു.
- നെറ്റ്വർക്ക്, കംപ്യൂട്ട് വിഭവങ്ങളുടെ കാര്യക്ഷമമായ ഉപയോഗം.

### ലളിതം ഉദാഹരണം: HTTP സ്ട്രീമിംഗ് സെർവർ & ക്ലയന്റ്

സ്ട്രീമിംഗ് എങ്ങനെ നടപ്പിലാക്കാവുന്നുവെന്ന് ലളിതമായ ഉദാഹരണം ഇവിടെ കാണാം:

#### Python

**സെർവർ (Python, FastAPI, StreamingResponse ഉപയോഗിച്ച്):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**ക്ലയന്റ് (Python, requests ഉപയോഗിച്ച്):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

ഈ ഉദാഹരണം എല്ലാ മെസ്സേജുകളും ഒരുമിച്ച് തിരിച്ചറിയാൻ കാത്തിരിക്കാതെ, സെർവർ മെസ്സേജുകൾ ലഭ്യമായ പോലെ ക്ലയന്റിലേക്ക് അയയ്ക്കുന്നു.

**എങ്ങനെ പ്രവർത്തിക്കുന്നു:**

- സെർവർ ഓരോ സന്ദേശവും ഉണ്ടാകുമ്പോൾ ഇടുന്നുണ്ട്.
- ക്ലയന്റ് ഓരോ ചങ്കും എത്തുമ്പോൾ സ്വീകരിച്ച് പ്രിന്റ് ചെയ്യുന്നു.

**ആവശ്യകതകൾ:**

- സെർവർ സ്ട്രീമിംഗ് മറുപടി ഉപയോഗിക്കണം (ഉദാ: FastAPI-ലെ `StreamingResponse`).
- ക്ലയന്റ് മറുപടിയെ സ്ട്രീമായി പ്രോസസ് ചെയ്യണം (`stream=True` requests-ൽ).
- Content-Type സാധാരണയായി `text/event-stream` അല്ലെങ്കിൽ `application/octet-stream` ആയിരിക്കും.

#### Java

**സെർവർ (Java, Spring Boot, Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

```java
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**ജാവ ഇൻപ്ലിമെന്റേഷന്‍ കുറിപ്പുകൾ:**

- ‘Flux’ ഉപയോഗിച്ച് Spring Boot-ന്റെ റിയാക്ടീവ് സ്റ്റാക്ക് ഉപയോഗിക്കുന്നു
- `ServerSentEvent` ഇവന്റിന്റെ തരം ഉൾപ്പെടെ ഘടനയുള്ള ഇവന്റ് സ്ട്രീമിംഗ് നൽകുന്നു
- `WebClient` `bodyToFlux()` ഉപയോഗിച്ച് റിയാക്ടീവ് സ്ട്രീമിംഗ് ഉപഭോഗം സജ്ജമാക്കുന്നു
- ‘delayElements()’ ഇവന്റുകൾക്കിടയിലെ പ്രോസസ്സിംഗ് സമയം അനുകരിക്കുന്നു
- ഇവന്റുകൾക്ക് `<info>`, `<result>` പോലുള്ള തരങ്ങൾ ക്ലയന്റ് പ്രോസസ്സിംഗിനു സഹായിക്കുന്നു

### താരതമ്യം: ക്ലാസിക് സ്ട്രീമിംഗ് vs MCP സ്ട്രീമിംഗ്

ക്ലാസിക് രീതിയിലുള്ള സ്ട്രീമിംഗും MCP സ്ട്രീമിംഗും തമ്മിലുള്ള വ്യത്യാസങ്ങൾ ഇപ്രകാരമാണ്:

| സവിശേഷത        | ക്ലാസിക് HTTP സ്ട്രീമിംഗ് | MCP സ്ട്രീമിംഗ് (അറിയിപ്പുകൾ) |
|------------------|---------------------------|------------------------------|
| പ്രധാന മറുപടി     | ചങ്കുകളായി              | ഒറ്റത്തവണ, അവസാനം               |
| പ്രോഗ്രസ് അപ്‌ഡേറ്റുകൾ | ഡാറ്റ ചങ്കുകളായി അയയ്ക്കുന്നു  | അറിയിപ്പുകളായി അയയ്ക്കുന്നു           |
| ക്ലയന്റ് ആവശ്യങ്ങൾ   | സ്ട്രീം പ്രോസസ് ചെയ്യണം        | മെസ്സേജ് ഹാന്ഡ്ലർ നടപ്പിലാക്കണം        |
| ഉപയോഗം            | വലിയ ഫയലുകൾ, AI ടോക്കണ്‍ സ്ട്രീമുകൾ | പ്രോഗ്രസ്, ലോഗുകൾ, റിയൽ-ടൈം ഫീഡ്ബാക്ക്  |

### പ്രധാന വ്യത്യാസങ്ങൾ

കൂടാതെ, പ്രധാന വ്യത്യാസങ്ങൾ ഇങ്ങനെയാണ്:

- **കമ്മ്യൂണിക്കേഷൻ മാതൃക:**
  - ക്ലാസിക് HTTP സ്ട്രീമിംഗ്: ലളിതമായ ചങ്ക് ട്രാൻസ്ഫർ എൻകോഡിംഗ് ഉപയോഗിച്ച് ഡാറ്റ അയയയ്ക്കുന്നു
  - MCP സ്ട്രീമിംഗ്: JSON-RPC പ്രോട്ടോക്കോളിൽ ഘടനാപരമായ അറിയിപ്പ് സിസ്റ്റം ഉപയോഗിക്കുന്നു

- **മെസ്സേജ് ഫോർമാറ്റ്:**
  - ക്ലാസിക് HTTP: പുതിയ വരികളുള്ള പ്ലെയിൻ ടെക്സ്റ്റ് ചങ്കുകൾ
  - MCP: മെടാഡേറ്റായുള്ള ഘടനാപരമായ LoggingMessageNotification ഒബ്ജക്ടുകൾ

- **ക്ലയന്റ് ഇൻപ്ലിമെന്റേഷൻ:**
  - ക്ലാസിക് HTTP: ലളിതമായ ക്ലയന്റ് സ്ട്രീമിംഗ് മറുപടി പ്രോസസ്സ് ചെയ്യുന്നു
  - MCP: വ്യത്യസ്ത മെസ്സേജ് തരങ്ങൾ കൈകാര്യം ചെയ്യുന്നതിനുള്ള മെസ്സേജ് ഹാൻഡ്ലറുള്ള കൂടുതൽ പ്രഗത്ഭ ക്ലയന്റ്

- **പ്രോഗ്രസ് അപ്‌ഡേറ്റുകൾ:**
  - ക്ലാസിക് HTTP: പ്രോഗ്രസ് മുഖ്യ മറുപടി സ്ട്രീമിന്റെ ഭാഗമാണ്
  - MCP: പ്രധാന മറുപടി അവസാനം അയയ്ക്കുമ്പോൾ പ്രോഗ്രസ് വേറെ അറിയിപ്പുകളായി അയയ്ക്കുന്നു

### ശുപാർശകൾ

ക്ലാസിക് സ്ട്രീമിംഗ് (മുകളില്‍ കാണിച്ച `/stream` എന്റ്പോയിന്റ്‌) നടപ്പിലാക്കുന്നതിനും MCP വഴി സ്ട്രീമിംഗ് തിരഞ്ഞെടുക്കുന്നതിനും ഇടയിലെ ചില നിര്‍ദ്ദേശങ്ങള്‍:

- **സിമ്പിള്‍ സ്ട്രീമിംഗിന്:** ക്ലാസിക് HTTP സ്ട്രീമിംഗ് ലളിതമായി നടപ്പിലാക്കാവുന്നതും അടിസ്ഥാന സ്ട്രീമിംഗ് ആവശ്യങ്ങള്‍ക്ക് പോരായ്മയില്ലാത്തതുമാണ്.

- **സങ്കീർണ്ണ, ഇന്ററാക്ടീവ് ആപ്ലിക്കേഷനുകൾക്കായി:** MCP സ്ട്രീമിംഗ് മെറ്റാഡേറ്റയും അറിയിപ്പുകളും മൈനായി ഒരു ഘടനയുള്ള സമീപനം നൽകുന്നു.

- **AI ആപ്ലിക്കേഷനുകൾക്കായി:** MCP-യുടെ അറിയിപ്പ് സിസ്റ്റം ദീർഘകാല AI ടاس്കുകളിലെ പുരോഗതിയെ ഉപയോക്താക്കളെ കോടതിയിലേക്ക് അറിയിക്കാൻ പ്രത്യേകമായി സഹായിക്കുന്നു.

## MCP-യില്‍ സ്ട്രീമിംഗ്

ഇതുവരെയും ക്ലാസിക്ക് സ്ട്രീമിംഗിന്റെ MCP-സ്ട്രീമിംഗുമായി താരതമ്യവും ശുപാർശകളും കണ്ടു. MCP-യില്‍ സ്ട്രീമിംഗ് എങ്ങനെ ഉപയോഗിക്കാമെന്ന് വിശദമായി പഠിക്കാം.

MCP ഫ്രെയിംവർക്കിനുള്ളിൽ സ്ട്രീമിംഗ് എങ്ങനെ പ്രവർത്തിക്കുന്നു എന്നത് മനസ്സിലാക്കുന്നത്, ദീർഘനേരം പ്രവർത്തന സമയത്ത് ഉപയോക്താക്കൾക്ക് റിയൽ-ടൈം ഫീഡ്ബാക്ക് നൽകുന്ന പ്രതികരണാപരമായ ആപ്ലിക്കേഷനുകൾ നിർമ്മിക്കാൻ അനിവാര്യമാണ്.

MCP-യിൽ സ്ട്രീമിംഗ് പ്രധാന മറുപടിയെ ചങ്കുകളായി അയയ്ക്കുന്നതല്ല, മറിച്ച് ടൂൾ അഭ്യർത്ഥന പ്രോസസ്സ് ചെയ്യുന്ന സമയത്ത് ഉപയോക്താവിനെ അറിയിപ്പുകൾ അയക്കുന്നതാണ്. ഇവയിൽ പ്രോഗ്രസ് അപ്‌ഡേറ്റുകൾ, ലോഗുകൾ, മറ്റു സംഭവങ്ങൾ അടങ്ങിയിരിക്കും.

### എങ്ങനെ പ്രവർത്തിക്കുന്നു

പ്രധാന ഫലം ഒറ്റ മറുപടി ആയി തന്നെയാണ് അയക്കുന്നത്. എന്നാൽ പ്രോസസ്സിംഗ് സമയത്ത് വ്യത്യസ്ത സന്ദേശങ്ങളായി അറിയിപ്പുകൾ അയക്കാം, അതുവഴി ഉപയോക്താവിന് റിയൽ-ടൈം അപ്‌ഡേറ്റുകൾ ലഭിക്കും. ക്ലയന്റ് ഇവ കൈകാര്യം ചെയ്ത് പ്രദർശിപ്പിക്കണം.

## അറിയിപ്പ് എന്താണ്?

ഞങ്ങൾ "അറിയിപ്പ്" എന്ന് പറഞ്ഞപ്പോള്‍, MCP സന്ധർഭത്തിൽ അതിന്റെ അർത്ഥം എന്താണ്?

അറിയിപ്പ് എന്നാണ് സെര്‍വര്‍ ഉപയോക്താവിനെ ഒരു ദീർഘനേര പ്രവര്‍ത്തനത്തിലെ പുരോഗതി, നില, മറ്റ് സംഭവങ്ങൾ എന്നിവയെക്കുറിച്ച് അറിയിക്കുന്ന സന്ദേശം. ഇത് സുതാര്യതയും ഉപയോക്തൃ അനുഭവവും മെച്ചപ്പെടുത്തുന്നു.

ഉദാഹരണത്തിന്, സെര്‍വർ അംഗീകാരം പൂർത്തിയാക്കിയ ശേഷം ക്ലയന്റ് അറിയിപ്പ് അയക്കണം.

അറിയിപ്പ് ഇങ്ങനെ ഒരു JSON സന്ദേശം ആണ്:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

MCP-യില്‍ അറിയിപ്പുകൾ ഒരു വിഷയം ആയി ഉള്ളതാണ് ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging).

> **പഴയതാക്കല്‍ അറിയിപ്പ്:** `2026-07-28` MCP സ്പെസിഫിക്കേഷൻ റിലീസ് കണ്ഹിഡേറ്റ് ലോഗിംഗ് പ്രിമിറ്റീവ് പഴയതാക്കലായി അടയാളപ്പെടുത്തുന്നു, stdio ട്രാന്‍സ്‌പോർട്ടുകൾക്ക് `stderr` ഉപയോഗിച്ച്, ഘടനാപരമായ നിരീക്ഷണത്തിനായി OpenTelemetry ഉപയോഗിച്ച് മാറ്റം വരുത്തുന്നു. 2025-11-25 സ്പെസിഫിക്കേഷനിലും ഔദ്യോഗിക പഴയതാക്കലിനും ശേഷം ഒരു വർഷത്തോളം ഇത് പ്രവർത്തിക്കും. വിശദാംശങ്ങൾക്കായി [MCP-നിൽ മാറ്റങ്ങൾ: 2026-07-28 റിലീസ് കണ്ഹിഡേറ്റ്](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) കാണുക.

ലോഗിംഗ് പ്രവർത്തിപ്പിക്കാൻ സെർവർക്ക് ഫീച്ചർ/സാധ്യതയായി സജ്ജമാക്കേണ്ടതുണ്ട് ഇങ്ങനെ:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> ഉപയോഗിക്കുന്ന SDK അനുസരിച്ച്, ലോഗിംഗ് ഡിഫോൾട്ടായി പ്രവർത്തനസജ്ജമാവാം അല്ലെങ്കിൽ സെർവർ കോൺഫിഗറേഷനിൽ വ്യക്തമായി ഓൺ ചെയ്‌തിരിക്കേണ്ടതുണ്ടാകും.

അറിയിപ്പുകൾക്ക് വിവിധ തരങ്ങളുണ്ട്:

| നില       | വിവരണം                      | ഉദാഹരണം ഉപയോഗം              |
|-----------|------------------------------|-----------------------------|
| debug     | വിശദമായ ഡീബഗ്ഗിംഗ് വിവരങ്ങൾ   | ഫംഗ്ഷൻ പ്രവേശനം/നിർഗമനം    |
| info      | സാധാരണ വിവരസൂചനകൾ         | പ്രവർത്തന പുരോഗതി അപ്‌ഡേറ്റുകൾ  |
| notice    | സാധാരണയായതിന്റെയും പ്രധാനപ്പെട്ട സംഭവങ്ങൾ | കോൺഫിഗറേഷൻ മാറ്റങ്ങൾ          |
| warning   | മുന്നറിയിപ്പ് അവസ്ഥകൾ          | പഴയ സ്‌പെസിഫിക്കേഷൻ ഉപയോഗം  |
| error     | പിശക് അവസ്ഥകൾ                 | പ്രവർത്തന പരാജയങ്ങൾ           |
| critical  | ഗുരുതര അവസ്ഥകൾ               | സിസ്റ്റം ഘടക പരാജയങ്ങൾ         |
| alert     | നേർതായി നടപടി വേണം          | ഡാറ്റ തകരാറു గుర్తിച്ചത്        |
| emergency | സിസ്റ്റം ഉപയോഗയോഗ്യമല്ല         | മുഴുവൻ സിസ്റ്റം പരാജയം          |

## MCP-യിൽ അറിയിപ്പുകൾ നടപ്പിലാക്കല്‍

MCP-ല്‍ അറിയിപ്പുകൾ നടപ്പിലാക്കാൻ, സെർവർ, ക്ലയന്റ് എങ്കിലും റിയൽ-ടൈം അപ്‌ഡേറ്റുകൾ കൈകാര്യം ചെയ്യുവാൻ സജ്ജമാക്കണം. ഇത് ദീർഘനേര പ്രവര്‍ത്തനങ്ങളിൽ ഉപയോക്താക്കൾക്ക് ഉടനെ ഫീഡ്ബാക്ക് നൽകാൻ സഹായിക്കുന്നു.

### സെർവർ-വശം: അറിയിപ്പുകൾ അയയ്ക്കൽ

സെർവർ വശത്ത് തുടങ്ങാം. MCP-യിൽ, ടൂൾസ് അഭ്യർത്ഥന പ്രോസസ് ചെയ്യുന്നിടത്ത് അറിയിപ്പുകൾ അയക്കാൻ കഴിയും. സെർവർ സാധാരണയായി `ctx` എന്ന കോൺടെക്സ്റ്റ് ഒബ്ജക്റ്റ് ഉപയോഗിച്ച് ക്ലയന്റിലേക്ക് സന്ദേശങ്ങൾ അയയ്ക്കുന്നു.

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

കാമ്പിള്‍ കോഡിലെ `process_files` ടൂളും പ്രോട്ടസസ്സിംഗ് സമയത്ത് ഓരോ ഫയലിനും മൂന്ന് അറിയിപ്പുകൾ അയക്കുന്നു. വിവരസന്ദേശങ്ങൾ നൽകുവാൻ `ctx.info()` രീതിയാണ് ഉപയോഗിക്കുന്നത്.

കൂടാതെ, അറിയിപ്പുകൾ സജ്ജമാകാൻ, സെർവർ സ്ട്രീമിംഗിന് അനുയോജ്യമായ ട്രാൻസ്പോർട്ട് ഉപയോഗിക്കണമെന്നും (ഉദാ: `streamable-http`), ക്ലയന്റ് അറിയിപ്പുകൾ കൈകാര്യം ചെയ്യുവാൻ മെസ്സേജ് ഹാൻഡ്ല്‍ർ നടപ്പിലാക്കണമെന്നും ഉറപ്പാക്കണം. സെർവർ `streamable-http` ട്രാൻസ്പോർട്ട് ഉപയോഗിക്കുന്നതിന് ഉദാഹരണമാണ്:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

ഈ .NET ഉദാഹരണത്തിൽ, `ProcessFiles` ടൂള്‍ `Tool` അട്രിബ്യൂട്ട് കൊണ്ട് അലങ്കരിച്ചിട്ട് മൂന്ന് അറിയിപ്പുകൾ ക്ലയന്റിന് അയയ്ക്കുന്നു. വിവര സന്ദേശങ്ങൾ അയയ്ക്കാൻ `ctx.Info()` രീതിയാണ് ഉപയോഗിക്കുന്നത്.

.NET MCP സെർവറിൽ അറിയിപ്പുകൾ സജ്ജമാക്കാൻ, സ്ട്രീമിംഗ് ട്രീന്‌സ്‌പോർട്ട് ഉപയോഗിക്കുന്നുവെന്ന് ഉറപ്പാക്കുക:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### ക്ലയന്റ്-വശം: അറിയിപ്പുകൾ സ്വീകരിക്കൽ

klയന്റ് വരുന്നതു പോലെ അറിയിപ്പുകൾ അവലോകനം ചെയ്ത് പ്രദർശിപ്പിക്കാൻ മെസ്സേജ് ഹാൻഡ്ലർ നടപ്പിലാക്കണം.

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

മുകളിൽ കാണിച്ച കോഡിൽ, `message_handler` ഫങ്ഷൻ വരുന്ന സന്ദേശം അറിയിപ്പാണോ എന്ന് പരിശോധിക്കുന്നു. അറിയിപ്പാണെങ്കിൽ പ്രിന്റ് ചെയ്യുന്നു, അല്ലെങ്കിൽ സാധാരണ സെർവർ മെസ്സേജ് ആയി പ്രോസസ് ചെയ്യുന്നു. കൂടാതെ `ClientSession` `message_handler` ഉപയോഗിച്ച് ആരംഭിച്ചിട്ടുള്ളതാണ്.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

ഈ .NET ഉദാഹരണത്തിൽ, `MessageHandler` ഫങ്ഷൻ വരുന്ന മെസ്സേജ് അറിയിപ്പാണോ എന്ന് പരിശോധിക്കുന്നു. അറിയിപ്പാണെങ്കിൽ പ്രിന്റ് ചെയ്യുന്നു, അല്ലെങ്കിൽ സാധാരണ സെർവർ മെസ്സേജായി പ്രോസസ് ചെയ്യുന്നു. `ClientSession` `ClientSessionOptions` വഴി മെസ്സേജ് ഹാൻഡ്ലർ ഉപയോഗിച്ച് ആരംഭിച്ചിട്ടുള്ളതാണ്.

അറിയിപ്പുകൾ സജ്ജമാക്കാൻ, സെർവർ സ്ട്രീമിംഗ് ട്രാൻസ്പോർട്ട് (ഉദാ: streamable-http) ഉപയോഗിക്കുകയും ക്ലയന്റ് മെസ്സേജ് ഹാൻഡ്ലർ നടപ്പിലാക്കുകയും വേണം.

## പ്രോഗ്രസ് അറിയിപ്പുകളും സാഹചര്യങ്ങളും

MCP-ൽ പ്രോഗ്രസ് അറിയിപ്പുകളെക്കുറിച്ച്, അവയുണ്ടാക്കേണ്ട ആവശ്യകത, Streamable HTTP ഉപയോഗിച്ചുള്ള നടപ്പിലാക്കൽ എന്നിവ ഇവിടെ വിശദീകരിക്കുന്നു. കൂടാതെ മനസ്സിലാക്കൽ ശക്തിപ്പെടുത്താനുളള പ്രായോഗിക അസൈന്‍മെന്റ് ഉണ്ട്.

പ്രോഗ്രസ് അറിയിപ്പുകൾ ദീർഘനേരം പ്രവർത്തനങ്ങളിൽ സെർവർ നിന്നു ക്ലയന്റിലേക്ക് അയക്കുന്ന റിയൽ-ടൈം സന്ദേശങ്ങളാണ്. പ്രക്രിയ മുഴുവൻ പൂർത്തിയായതിന് കാത്തിരിക്കാതെയാണ് നിലവിലെ നില ഉപയോക്താവിന് അറിയിക്കുന്നതു. ഇത് സുതാര്യത, ഉപയോക്തൃ അനുഭവം മെച്ചപ്പെടുത്തുകയും ഡീബഗ്ഗിംഗും എളുപ്പമാക്കുകയും ചെയ്യുന്നു.

**ഉദാഹരണം:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### പ്രോഗ്രസ് അറിയിപ്പുകൾ ഉപയോഗിക്കേണ്ടതെന്തിന്?

പ്രോഗ്രസ് അറിയിപ്പുകൾ താഴെക്കാരണം പ്രധാനമാണ്:

- **നല്ല ഉപയോക്തൃ അനുഭവം:** പ്രവർത്തനം പുരോഗമിക്കുന്നതുവരെ ഉപയോക്താക്കൾക്ക് അപ്‌ഡേറ്റുകൾ കാണാൻ കഴിയണം, അവസാനം മാത്രമല്ല.
- **റിയൽ-ടൈം ഫീഡ്ബാക്ക്:** ക്ലയന്റുകൾ പ്രോഗ്രസ്സ് ബാറുകൾ അല്ലെങ്കിൽ ലോഗുകൾ കാണിച്ച് ആപ്പ് പ്രതികരിക്കുന്നതുപോലെ തോന്നിക്കും.
- **ഡീബഗ്ഗിംഗ്, നിരീക്ഷണം എളുപ്പമാക്കൽ:** വികസനക്കാരും ഉപയോക്താക്കളും പ്രോസസ് എവിടെയാണ് മന്ദഗതിയിലോ തടസം വന്നിരിക്കുകയോ ചെയ്യാമെന്ന് കണ്ടു പിടിക്കാം.

### പ്രോഗ്രസ് അറിയിപ്പുകൾ എങ്ങനെ നടപ്പിലാക്കാം

MCP-യിൽ പ്രോഗ്രസ് അറിയിപ്പ് നടപ്പിലാക്കുന്നത് ഇങ്ങനെ ചെയ്യാം:

- **സെർവർ-വശം:** ഓരോ ഇനം പ്രോസസ് ചെയ്തശേഷം `ctx.info()` അല്ലെങ്കിൽ `ctx.log()` ഉപയോഗിച്ച് അറിയിപ്പുകൾ അയയ്ക്കുക. ഇതു മുഖ്യ ഫലം തയ്യാറാകുന്നതിന് മുൻപ് ക്ലയന്റിന് സന്ദേശം അയയ്ക്കുന്നു.
- **ക്ലയന്റ്-വശം:** മെസ്സേജ് ഹാൻഡ്ലർ നടപ്പിലാക്കി അറിയിപ്പുകൾ സ്വീകരിക്കയും പ്രദർശിപ്പിക്കയും ചെയ്യുക. ഈ ഹാൻഡ്ലർ അറിയിപ്പുകളും അന്തിമ ഫലവും വേർതിരിക്കും.

**സെർവർ ഉദാഹരണം:**


#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

## സുരക്ഷാ പരിഗണനകൾ

HTTP-അധിഷ്ഠിത ട്രാൻസ്പോർട്ടുകൾ ഉപയോഗിച്ച് MCP സെർവറുകൾ നടപ്പിലാക്കുമ്പോൾ, നിരവധി ആക്രമണ വക്ടറുകൾക്കും സംരക്ഷണ ഘടകങ്ങൾക്കും പ്രത്യേക ശ്രദ്ധ ആവശ്യമുള്ള ഏറ്റവും പ്രധാനപ്പെട്ട വിഷയം സുരക്ഷയാണ്.

### അവലോകനം

MCP സെർവറുകൾ HTTP മുഖേന പുറത്തുവച്ചപ്പോൾ സുരക്ഷ വളരെ പ്രധാനമാണ്. സ്ട്രീമബിള് HTTP പുതിയ ആക്രമണ സാധ്യതകൾ അവതരിപ്പിക്കുകയും കംഫിഗറേഷൻ സൂക്ഷ്മത ആവശ്യപ്പെടുകയും ചെയ്യുന്നു.

### പ്രധാനപ്പെട്ട കാര്യങ്ങൾ

- **ഓറിജിൻ ഹെഡർ പരിശോധന**: DNS റീബൈൻഡ് ആക്രമണങ്ങൾ തടയാൻ `Origin` ഹെഡർ എല്ലായ്പ്പോഴും പരിശോധിക്കുക.
- **ലോകൽഹോസ്റ്റ് ബൈൻഡിംഗ്**: ലോക്കൽ ഡെവലപ്പ്മെന്റിനായി സെർവറുകൾ `localhost`-ലേയ്ക്ക് ബൈൻഡ് ചെയ്യുക, പൊതുമരമ്പിലേക്ക് അവ പുറത്തു വിടാതിരിക്കാൻ.
- **ഓത്തന്റിക്കേഷൻ**: പ്രൊഡക്ഷൻ ഡിപ്ലോയ്മെന്റുകൾക്ക് ഓത്തന്റിക്കേഷൻ (ഉദാ. API കീകൾ, OAuth) നടപ്പിലാക്കുക.
- **CORS**: ആക്‌സസ് നിയന്ത്രിക്കാൻ ക്രോസ്-ഓറിജിൻ റീസോഴ്സ് ഷെയറിംഗ് (CORS) നയം ക്രമീകരിക്കുക.
- **HTTPS**: ട്രാഫിക് എൻക്രിപ്റ്റ് ചെയ്യാൻ പ്രൊഡക്ഷനിൽ HTTPS ഉപയോഗിക്കുക.

### മികച്ച പ്രവർത്തന രീതി

- വരുമാന വിശ്വാസം ഇല്ലാതെ ആവശ്യകത പരിശോധന ഇല്ലാതെ വിശ്വസിക്കരുത്.
- എല്ലാ ആക്‌സസുകളും പിശകുകളും ലോഗ് ചെയ്യുകയും നിരീക്ഷിക്കുകയും ചെയ്യുക.
- സുരക്ഷാ ദുർബലതകൾ പിച്ചിംഗ് വേണ്ടി ഡിപ്പെൻഡൻസികൾ നിരന്തരം അപ്‌ഡേറ്റ് ചെയ്യുക.

### വെല്ലുവിളികൾ

- സുഖമുള്ള ഡെവലപ്പ്മെന്റ് അനുഭവത്തോടൊപ്പം സുരക്ഷ നിലനിർത്തൽ
- വിവിധ ക്ലയന്റ് പരിസ്ഥിതികളുടെ അനുഭവക്രമം ഉറപ്പാക്കൽ

## SSE-ൽ നിന്ന് Streamable HTTP-ലേക്ക് അപ്ഡേറ്റ് ചെയ്യൽ

നിലവിൽ Server-Sent Events (SSE) ഉപയോഗിക്കുന്ന ആപ്ലിക്കേഷനുകൾക്ക് Streamable HTTP-യിലേക്ക് മാറ്റം ചെയ്യുന്നത് നിങ്ങളുടെ MCP നടപ്പാക്കലുകൾക്കായി മെച്ചപ്പെട്ട ശേഷിയും ദീർഘകാല സുസ്ഥിരതയും നൽകുന്നു.

### എന്തുകൊണ്ട് അപ്ഡേറ്റ് ചെയ്യണം?

SSE-യിൽ നിന്ന് Streamable HTTP-ലേക്ക് അപ്ഡേറ്റ് ചെയ്യാനുള്ള രണ്ട് പ്രധാന കാരണങ്ങളുണ്ട്:

- Streamable HTTP SSE-യുടെ കാൾ ഷെയർ വർത്തിക്കിലും മെച്ചപ്പെട്ട സ്കെയിലബിലിറ്റിയും സമ്പൂർണ്ണ നോട്ടിഫിക്കേഷൻ പിന്തുണയും നൽകുന്നു.
- പുതിയ MCP ആപ്ലിക്കേഷനുകൾക്ക് ഇത് ശുപാർശചെയ്യപ്പെടുന്ന ട്രാൻസ്പോർട്ടാണ്.

### മൈഗ്രേഷൻ ഘട്ടങ്ങൾ

നിങ്ങളുടെ MCP ആപ്ലിക്കേഷനുകളിൽ SSE-യിൽ നിന്ന് Streamable HTTP-ലേക്ക് മാറ്റം ചെയ്യാനുള്ള വിധം:

- **സെർവർ കോഡ് അപ്‌ഡേറ്റ് ചെയ്യുക** `mcp.run()`-ൽ `transport="streamable-http"` ഉപയോഗിച്ച്.
- **ക്ലയന്റ് കോഡ് അപ്‌ഡേറ്റ് ചെയ്യുക** SSE ക്ലയന്റിന്റെ പകരം `streamablehttp_client` ഉപയോഗിക്കുക.
- **ക്ലയന്റിൽ ഒരു മെസ്സേജ് ഹാൻഡ്ലർ നടപ്പിലാക്കുക** നോട്ടിഫിക്കേഷനുകൾ പ്രോസസ് ചെയ്യാൻ.
- ** നിലവിലുള്ള ഉപകരണങ്ങളും വർക്ക്‌ഫ്ലോകളുമായി പാസാക്കലിനായി** ടെസ്റ്റ് ചെയ്യുക.

### പൊരുത്തം നിലനിർത്തൽ

മൈഗ്രേഷൻ പ്രക്രിയയിൽ നിലവിലുള്ള SSE ക്ലയന്റുകളുമായി പൊരുത്തം നിലനിർത്തുന്നത് ശുപാർശ ചെയ്യപ്പെടുന്നു. കുറെ മാർഗ്ഗങ്ങൾ:

- വിവിധ എൻഡ്‌പോയിന്റുകളിൽ SSEയും Streamable HTTPയും അടുത്തുചേർത്ത് പ്രവർത്തിപ്പിച്ച് രണ്ട് ട്രാൻസ്പോർട്ടുകൾക്കും പിന്തുണ നൽകാം.
- ക്ലയന്റുകൾ പുതിയ ട്രാൻസ്പോർട്ടിലേക്ക് ക്രമേണ മൈഗ്രേറ്റ് ചെയ്യുക.

### വെല്ലുവിളികൾ

മൈഗ്രേഷൻ സമയത്ത് താഴെപ്പറയുന്ന വെല്ലുവിളികൾ പരിഹരിക്കണമെന്ന് ഉറപ്പാക്കുക:

- എല്ലാ ക്ലയന്റുകളും അപ്‌ഡേറ്റ് ചെയ്യുന്നത് ഉറപ്പാക്കുക
- നോട്ടിഫിക്കേഷൻ ഡെലിവറിയിലെ വ്യത്യാസങ്ങൾ കൈകാര്യം ചെയ്യൽ

## സുരക്ഷാ പരിഗണനകൾ

MCP-യിൽ Streamable HTTP പോലുള്ള HTTP-അധിഷ്ഠിത ട്രാൻസ്പോർട്ടുകൾ ഉപയോഗിക്കുമ്പോൾ സെർവർ നടപ്പിലാക്കുമ്പോൾ സുരക്ഷ മികച്ച പരിപാലനമായിരിക്കണം.

HTTP-അധിഷ്ഠിത ട്രാൻസ്പോർട്ടുകൾ ഉപയോഗിച്ച് MCP സെർവറുകൾ നടപ്പിലാക്കുമ്പോൾ, കൂടുതൽ ആക്രമണ മാർഗ്ഗങ്ങളും സംരക്ഷണ ഘടകങ്ങളും ശ്രദ്ധിക്കേണ്ട ഏറ്റവും വലിയ വിഷയം സുരക്ഷയാണ്.

### അവലോകനം

MCP സെർവറുകൾ HTTP മുഖേന പുറത്തുവച്ചപ്പോൾ സുരക്ഷ വളരെ നിർണ്ണായകമാണ്. സ്ട്രീമബിള് HTTP പുതിയ ആക്രമണ സാധ്യതകൾ നിർമിക്കുകയും സൂക്ഷ്മമായ കംഫിഗറേറ്റ്ഷൻ ആവശ്യമുണ്ടാക്കുകയും ചെയ്യുന്നു.

حفاظتی برخی نکات کلیدی عبارتند از:

- **ഓറിജിൻ ഹെഡർ പരിശോധന**: DNS റീബൈൻഡ് ആക്രമണങ്ങൾ തടയാൻ `Origin` ഹെഡർ എല്ലായ്പ്പോഴും പരിശോധിക്കുക.
- **ലോകൽഹോസ്റ്റ് ബൈൻഡിംഗ്**: ലോക്കൽ ഡെവലപ്പ്മെന്റിനായി സെർവറുകൾ `localhost`-ലേയ്ക്ക് ബൈൻഡ് ചെയ്യുക, പൊതുമരമ്പിലേക്ക് അവ പുറത്തു വിടാതിരിക്കാൻ.
- **ഓത്തന്റിക്കേഷൻ**: പ്രൊഡക്ഷൻ ഡിപ്ലോയ്മെന്റുകൾക്ക് ഓത്തന്റിക്കേഷൻ (ഉദാ. API കീകൾ, OAuth) നടപ്പിലാക്കുക.
- **CORS**: ആക്‌സസ് നിയന്ത്രിക്കാൻ ക്രോസ്-ഓറിജിൻ റീസോഴ്സ് ഷെയറിംഗ് (CORS) നയം ക്രമീകരിക്കുക.
- **HTTPS**: ട്രാഫിക് എൻക്രിപ്റ്റ് ചെയ്യാൻ പ്രൊഡക്ഷനിൽ HTTPS ഉപയോഗിക്കുക.

### മികച്ച പ്രവർത്തന രീതി

MCP സ്ട്രീമിംഗ് സെർവറിൽ സുരക്ഷയാണ് നടപ്പിലാക്കുമ്പോൾ അനുയോജ്യമായി പാലിക്കേണ്ട ചില മികച്ച പ്രവർത്തന രീതി:

- വരുമാന വിശ്വാസം ഇല്ലാതെ ആവശ്യകത പരിശോധന ഇല്ലാതെ വിശ്വസിക്കരുത്.
- എല്ലാ ആക്‌സസുകളും പിശകുകളും ലോഗ് ചെയ്യുകയും നിരീക്ഷിക്കുകയും ചെയ്യുക.
- സുരക്ഷാ ദുർബലതകൾ പിച്ചിംഗ് വേണ്ടി ഡിപ്പെൻഡൻസികൾ നിരന്തരം അപ്‌ഡേറ്റ് ചെയ്യുക.

### വെല്ലുവിളികൾ

MCP സ്ട്രീമിംഗ് സെർവറുകളിൽ സുരക്ഷ നടപ്പിലാക്കുമ്പോൾ നിങ്ങൾക്ക് ഏതാനും വെല്ലുവിളികൾ നേരിടേണ്ടി വരും:

- സുഖമുള്ള ഡെവലപ്പ്മെന്റ് അനുഭവത്തോടൊപ്പം സുരക്ഷ നിലനിർത്തൽ
- വിവിധ ക്ലയന്റ് പരിസ്ഥിതികളുടെ അനുഭവക്രമം ഉറപ്പാക്കൽ

### അസൈൻമെന്റ്: നിങ്ങളുടെ സ്വന്തം MCP സ്ട്രീമിംഗ് ആപ്പ് നിർമ്മിക്കുക

**ഘടിബന്ധം:**
ഒരു MCP സെർവർ, ക്ലയന്റ് നിർമ്മിക്കുക, സെർവർ ഒരു ലിസ്റ്റ് (ഉദാ. ഫയലുകൾ അല്ലെങ്കിൽ ഡോക്യുമെന്റുകൾ) പ്രോസസ് ചെയ്ത് ഓരോ പ്രോസസ്സുചെയ്ത ഇനത്തിനും ഒരു നോട്ടിഫിക്കേഷൻ അയയ്ക്കുക. ക്ലയന്റ് ഈ നോട്ടിഫിക്കേഷനുകൾ ഉണ്ടാകത്തേൽ തന്നെ പ്രദർശിപ്പിക്കണം.

**ചടങ്ങുകൾ:**

1. ലിസ്റ്റ് പ്രോസസ് ചെയ്ത് ഓരോ ഇനത്തിനും നോട്ടിഫിക്കേഷനുകൾ അയക്കുന്ന ഒരു സെർവർ ടൂൾ നടപ്പിലാക്കുക.
2. നോട്ടിഫിക്കേഷനുകൾ തത്സമയം പ്രദർശിപ്പിക്കാൻ മെസ്സേജ് ഹാൻഡ്ലർ അടങ്ങിയ ക്ലയന്റ് നടപ്പിലാക്കുക.
3. സെർവർ, ക്ലയന്റ് രണ്ടും പ്രവർത്തിപ്പിച്ച് നിങ്ങളുടെ നടപ്പിലാക്കൽ പരിശോധന നടത്തുക, നോട്ടിഫിക്കേഷനുകൾ നിരീക്ഷിക്കുക.
