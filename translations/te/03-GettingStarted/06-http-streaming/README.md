# మోడల్ కాంటెక్ట్స్ ప్రోటోకాల్ (MCP)తో HTTPS స్ట్రీమింగ్

ఈ అధ్యాయం HTTPS ఉపయోగించి మోడల్ కాంటెక్ట్స్ ప్రోటోకాల్ (MCP)తో సురక్షితమైన, స్కేలబుల్ మరియు రియల్-టైమ్ స్ట్రీమింగ్ ఎలా అమలు చేయాలో సమగ్ర మార్గదర్శకాన్ని అందిస్తుంది. ఇది స్ట్రీమింగ్ యొక్క ప్రేరణ, అందుబాటులో ఉన్న ట్రాన్స్‌పోర్ట్ యాంత్రిక ప్రమాణాలు, MCPలో స్ట్రీమబుల్ HTTP ఎలా అమలు చేయాలో, భద్రత ఆచారాలు, SSE నుండి మార్పిడి, మరియు మీ స్వంత స్ట్రీమింగ్ MCP అనువర్తనాలు నిర్మించడం గురించి ఆచరణాత్మక సూచనలను కవర్ చేస్తుంది.

> **భవిష్యత్తు కోసం చూద్దాం:** ఈ పాఠం **MCP స్పెసిఫికేషన్ 2025-11-25** కింద స్ట్రీమబుల్ HTTPను వివరించిందే, ఇక్కడ సెషన్ `initialize` సమయంలో ఏర్పాటు చేయబడుతుంది మరియు `Mcp-Session-Id` హెడర్‌తో పిన్ చేయబడుతుంది. `2026-07-28` విడుదల అభ్యర్థి పూర్తి హ్యాండ్‌షేక్ మరియు సెషన్ IDని తొలగించి, ప్రతి అభ్యర్థనను స్వయం ఉంటుంది, మరియు స్టికీ సెషన్ల లేని ఏ సర్వర్ ఇన్‌స్టాన్సుకి రూటు చేయగలదు. వివరాలకు [MCPలో ఏమి మారుతోంది: 2026-07-28 విడుదల అభ్యర్థి](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) చూడండి.

## MCPలో ట్రాన్స్‌పోర్ట్ యాంత్రిక ప్రమాణాలు మరియు స్ట్రీమింగ్

ఈ భాగం MCPలో అందుబాటులో ఉన్న వివిధ ట్రాన్స్‌పోర్ట్ యాంత్రిక ప్రమాణాలను మరియు క్లయింట్లు మరియు సర్వర్ల మధ్య రియల్-టైమ్ కమ్యూనికేషన్ కోసం స్ట్రీమింగ్ సామర్థ్యాలను ఎలా సాధించాలో పరిశీలిస్తుంది.

### ట్రాన్స్‌పోర్ట్ యాంత్రిక ప్రమాణం అంటే ఏమిటి?

ట్రాన్స్‌పోర్ట్ యాంత్రిక ప్రమాణం క్లయింట్ మరియు సర్వర్ మధ్య డేటా ఎలా మార్పిడి అవుతుందో నిర్వచిస్తుంది. MCP వివిధ వాతావరణాల మరియు అవసరాలకు అనుగుణంగా పలు ట్రాన్స్‌పోర్ట్ రకాలను మద్దతు ఇస్తుంది:

- **stdio**: సాంప్రదాయ ఇన్పుట్/అవుట్పుట్, స్థానిక మరియు CLI ఆధారిత టూల్స్‌కు అనుకూలం. సులభమైనది కానీ వెబ్ లేదా క్లౌడ్‌కు అనుకూలం కాదు.
- **SSE (సర్వర్-సెంట్ ఈవెంట్స్)**: సర్వర్లు HTTP పై రియల్-టైమ్ నవీకరణలను క్లయింట్లకు పంపటం కోసం అనుమతిస్తుంది. వెబ్ UIలకు మంచి, కానీ స్కేలబిలిటీ మరియు ఫ్లెక్సిబిలిటీ పరిమితముగా ఉంటుంది. MCP స్పెసిఫికేషన్ 2025-06-18 నుండి, స్వతంత్ర SSE ట్రాన్స్‌పోర్ట్ డిప్రికేటెడ్ గా Streamable HTTP ట్రాన్స్‌పోర్ట్ ద్వారా ప్రత్యామ్నాయంగా మార్చబడింది.
- **స్ట్రీమబుల్ HTTP**: ఆధునిక HTTP ఆధారిత స్ట్రీమింగ్ ట్రాన్స్‌పోర్ట్, నోటిఫికేషన్లను మరియు మెరుగైన స్కేలబిలిటీలను మద్దతు ఇస్తుంది. అతి ఎక్కువగా ప్రొడక్షన్ మరియు క్లౌడ్ వాతావరణాలలో సిఫార్సు చేయబడుతుంది.

### సరిపోలిక పట్టిక

ఈ ట్రాన్స్‌పోర్ట్ మెకానిజమ్ల మధ్య తేడాలను అర్థం చేసుకోవడానికి కింది సరిపోలిక పట్టికను చూడండి:

| ట్రాన్స్‌పోర్ట్         | రియల్-టైమ్ నవీకరణలు | స్ట్రీమింగ్ | స్కేలబిలిటీ | ఉపయోగం  |
|-----------------------|---------------------|-----------|------------|-----------|
| stdio                 | లేదు                | లేదు      | తక్కువ    | స్థానిక CLI టూల్స్ |
| SSE                   | ఉంది                | ఉంది      | మాధ్యమం  | వెబ్, రియల్-టైమ్ నవీకరణలు |
| స్ట్రీమబుల్ HTTP       | ఉంది                | ఉంది      | అధికం    | క్లౌడ్, బహు-క్లయింట్ |

> **సూచన:** సరైన ట్రాన్స్‌పోర్ట్ ఎంపిక పనితనం, స్కేలబిలిటీ, మరియు వినియోగదారు అనుభవం మొత్తానికి ప్రభావం చూపుతుంది. ఆధునిక, స్కేలబుల్, మరియు క్లౌడ్-తయారైన అనువర్తనాలకు **స్ట్రీమబుల్ HTTP**ను సిఫార్సు చేయబడుతుంది.

గత అధ్యాయాలలో మీరు చూసిన stdio మరియు SSE ట్రాన్స్‌పోర్ట్‌లను గమనించండి, మరియు ఈ అధ్యాయం కవర్ చేయడం స్ట్రీమబుల్ HTTP ట్రాన్స్‌పోర్ట్ అని గమనించండి.

## స్ట్రీమింగ్: సాంకేతిక సూత్రాలు మరియు ప్రేరణ

వ్యూహం మరియు ప్రేరణ సార్థకమైన రియల్-టైమ్ కమ్యూనికేషన్ వ్యవస్థలను అమలు చేయడంలో అనివార్యం.

**స్ట్రీమింగ్** అనేది నెట్‌వర్క్ ప్రోగ్రామింగ్ లో ఒక సాంకేతికత, ఇది పూర్తి ప్రతిఫలాన్ని అందుకు ఎదురు చూపకుండా చిన్న, నిర్వహించదగిన భాగాలుగా లేదా ఈవెంట్స్ క్రమంగా డేటాను పంపడం మరియు స్వీకరించడానికి అనుమతిస్తుంది. ఇది ముఖ్యంగా ఉపయోగపడుతుంది:

- పెద్ద ఫైళ్ల లేదా డేటాసెట్ల కోసం.
- రియల్-టైమ్ నవీకరణలు (ఉదా., చాట్, ప్రోగ్రెస్ బార్లు).
- యూజర్‌కు సమాచారాన్ని అందిస్తూ నిడివి గల గణనీయమైన లెక్కల సమయంలో.

స్ట్రీమింగ్ గురించి మీకు అవసరమైన సమగ్ర సమాచారం:

- డేటా కొనసాగుతున్నప్పుడు ప్రోగ్రెసివ్‌గా అందుతుంది, అన్ని ఒక్కసారి కాదు.
- క్లయింట్ డేటాను వచ్చిన వెంటనే ప్రాసెస్ చేయగలడు.
- అనుభూతి గల అక్షేపార్ధతను తగ్గిస్తుంది మరియు వినియోగదారు అనుభవాన్ని మెరుగుపరుస్తుంది.

### స్ట్రీమింగ్ ఎందుకు ఉపయోగించాలి?

స్ట్రీమింగ్ ఉపయోగించాల్సిన కారణాలు:

- యూజర్లు వెంటనే స్పందన పొందుతారు, కేవలం చివర్లో కాకుండా
- రియల్-టైమ్ అప్లికేషన్లు మరియు ప్రతిస్పందన UIలను సాధ్యపడచేస్తుంది
- నెట్‌వర్క్ మరియు గణన వనరులను మరింత సమర్థవంతంగా ఉపయోగిస్తుంది

### సులభ ఉదాహరణ: HTTP స్ట్రీమింగ్ సర్వర్ & క్లయింట్

ఇక్కడ స్ట్రీమింగ్ ఎలా అమలు చేయబడవచ్చు అనే ఒక సులభ ఉదాహరణ ఉంది:

#### Python

**సర్వర్ (Python, FastAPI మరియు StreamingResponse ఉపయోగించి):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**క్లయింట్ (Python, requests ఉపయోగించి):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

ఈ ఉదాహరణలో సర్వర్ అందుబాటులో ఉన్నంతవరకు క్లయింట్‌కు పత్రాల శ్రేణిని పంపుతుంది, ఒకేసారి అన్ని సందేశాల కోసం ఎదురు చూడదు.

**ఇది ఎలా పనిచేస్తుంది:**

- సర్వర్ ప్రతి సందేశం సిద్ధంగా ఉన్న వెంటనే అందిస్తుంది.
- క్లయింట్ ప్రతి భాగాన్ని స్వీకరించి ప్రింట్ చేస్తుంది.

**అవసరాలు:**

- సర్వర్ స్ట్రీమింగ్ స్పందన ఉపయోగించాలి (ఉదాహరణకు FastAPIలో `StreamingResponse`).
- క్లయింట్ స్పందనను స్ట్రీమ్‌గా ప్రాసెస్ చేయాలి (`stream=True` requestsలో).
- కంటెంట్-టైప్ సాధారణంగా `text/event-stream` లేదా `application/octet-stream`గా ఉంటుంది.

#### Java

**సర్వర్ (Java, Spring Boot మరియు Server-Sent Events ఉపయోగించి):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**క్లయింట్ (Java, Spring WebFlux WebClient ఉపయోగించి):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Java అమలు గమనికలు:**

- స్ట్రీమింగ్ కోసం Spring Boot రియాక్టివ్ స్టాక్ `Flux` ఉపయోగిస్తుంది
- `ServerSentEvent` ఈవెంట్లతో నిర్మించబడిన ఈవెంట్ స్ట్రీమింగ్ అందిస్తుంది
- `WebClient` `bodyToFlux()`తో రియాక్టివ్ స్ట్రీమింగ్ వినియోగం సాధ్యం చేస్తుంది
- `delayElements()` ఈవెంట్ల మధ్య ప్రాసెసింగ్ సమయాన్ని అధిగమిస్తుంది
- ఈవెంట్లు క్లయింట్ నిర్వహణకు మెరుగైనట్లుగా రకాలను కలిగి ఉండవచ్చు (`info`, `result`)

### సరిపోలిక: క్లాసిక్ స్ట్రీమింగ్ వర్సస్ MCP స్ట్రీమింగ్

"క్లాసిక్" విధానంలో స్ట్రీమింగ్ ఎలా ఉంటుంది మరియు MCPలో ఎలా ఉంటుంది తేడాలు ఇలాగే ఉన్నాయి:

| లక్షణం                | క్లాసిక్ HTTP స్ట్రీమింగ్     | MCP స్ట్రీమింగ్ (నోటిఫికేషన్లు)  |
|-----------------------|----------------------------|----------------------------------|
| ప్రధాన ప్రతిస్పందన     | భాగాలుగా పంపబడుతుంది      | ఒకటిగా, చివరలో                       |
| ప్రోగ్రెస్ నవీకరణలు   | డేటా భాగాలుగా పంపబడతాయి | నోటిఫికేషన్స్‌గా పంపబడతాయి       |
| క్లయింట్ అవసరాలు      | స్ట్రీమ్ ప్రాసెస్ చేయాలి     | సందేశ హ్యాండ్లర్ అమలుచేయాలి        |
| ఉపయోగం                | పెద్ద ఫైళ్లు, AI టోకెన్ స్ట్రీమ్స్ | ప్రోగ్రెస్, లోగ్స్, రియల్-టైమ్ ఫీడ్బ్యాక్ |

### తేలికపడిన ముఖ్య తేడాలు

అదనంగా, కింది ప్రధాన తేడాలు ఉన్నాయి:

- **కమ్యూనికేషన్ ప్యాటర్న్:**
  - క్లాసిక్ HTTP స్ట్రీమింగ్: డేటాను భాగాలుగా పంపడానికి సరళమైన చంక్డ్ ట్రాన్స్‌ఫర్ ఎన్‌కోడింగ్ ఉపయోగిస్తుంది
  - MCP స్ట్రీమింగ్: JSON-RPC ప్రోటోకాల్‌తో నిర్మాణబద్ధమైన నోటిఫికేషన్ వ్యవస్థ ఉపయోగిస్తుంది

- **సందేశ ఫార్మాట్:**
  - క్లాసిక్ HTTP: కొత్త లైన్స్ తో సాధారణ టెక్స్ట్ భాగాలు
  - MCP: మెటాడేటాతో నిర్మాణాత్మక LoggingMessageNotification ఆబ్జెక్ట్స్

- **క్లయింట్ అమలు:**
  - క్లాసిక్ HTTP: స్ట్రీమింగ్ స్పందనలను ప్రాసెస్ చేసే సరళమైన క్లయింట్
  - MCP: వివిధ రకాల సందేశాలను ప్రాసెస్ చేయడానికి మెసేజ్ హ్యాండ్లర్ ఉన్న మరింత పరిజ్ఞానంతో కూడిన క్లయింట్

- **ప్రోగ్రెస్ నవీకరణలు:**
  - క్లాసిక్ HTTP: ప్రోగ్రెస్ ప్రధాన స్పందన స్ట్రీమ్ భాగం
  - MCP: ప్రోగ్రెస్ వేర్వేరు నోటిఫికేషన్ సందేశాల ద్వారా పంపబడుతుంది, ప్రధాన స్పందన చివరలో వస్తుంది

### సిఫార్సులు

క్లాసిక్ స్ట్రీమింగ్ (/stream ఉపయోగంతో ముందుగా చూపిన విధానం) మరియు MCP స్ట్రీమింగ్ మధ్య ఎంపికపై కొన్ని సిఫార్సులు:

- **సాధారణ స్ట్రీమింగ్ అవసరాలకు:** క్లాసిక్ HTTP స్ట్రీమింగ్ అమలు చేయడానికి సులభం మరియు ప్రాథమిక అవసరాలకు సరిపోతుంది.

- **సంక్లిష్ట, ఇంటరాక్టివ్ అప్లికేషన్లకు:** MCP స్ట్రీమింగ్ నిర్మాణబద్ధమైన పథకాన్ని అందిస్తుంది, మెటాడేటాతో కూడిన నోటిఫికేషన్లు మరియు తుది ఫలితాల వేరిపరచు సహా.

- **AI అప్లికేషన్ల కోసం:** MCP నోటిఫికేషన్ వ్యవస్థ దీర్ఘకాలిక AI పనులలో యూజర్లకు ప్రగతి వివరాలను అందించడానికి చాలా ఉపయోగకరం.

## MCPలో స్ట్రీమింగ్

ఇప్పటి వరకు క్లాసిక్ స్ట్రీమింగ్ మరియు MCPలో స్ట్రీమింగ్ మధ్య తేడాలపై కొన్ని సిఫార్సులు మరియు సరిపోలికలు చూశాం. ఇప్పుడు MCPలో స్ట్రీమింగ్‌ను ఎలా ఉపయోగించవచ్చో వివరంగా తెలుస్తుందాం.

MCP ఫ్రేమ్‌వర్క్‌లో స్ట్రీమింగ్ ఎలా పనిచేస్తుందో అర్థం చేసుకోవటం, దీర్ఘకాలిక కార్యకలాపాల సమయంలో యూజర్లకు రియల్-టైమ్ ఫీడ్బ్యాక్ అందించే ప్రతిస్పందన అనువర్తనాలు నిర్మించడానికి అవసరం.

MCPలో స్ట్రీమింగ్ అనగా ప్రధాన ప్రతిస్పందనను భాగాలుగా పంపడం కాదు, కానీ టూల్ అభ్యర్థన ప్రాసెసింగ్ చేస్తున్న సమయంలో క్లయింట్‌కి **నోటిఫికేషన్లు** పంపడం. ఈ నోటిఫికేషన్లు ప్రగతి నవీకరణలు, లోగ్స్, లేదా ఇతర ఈవెంట్లను కలిగి ఉండవచ్చు.

### ఇది ఎలా పనిచేస్తుంది

ప్రధాన ఫలితం ఒకే ప్రతిస్పందనగా పంపబడుతుంది. అయితే, ప్రాసెసింగ్ సమయంలో వేరే సందేశాలుగా నోటిఫికేషన్లు పంపబడతాయి మరియు క్లయింట్‌ను రియల్-టైమ్‌లో నవీకరిస్తాయి. క్లయింట్ ఈ నోటిఫికేషన్లను స్వీకరించి ప్రదర్శించగలగాలి.

## నోటిఫికేషన్ అంటే ఏమిటి?

"నోటిఫికేషన్" అంటే MCP సందర్భంలో ఏమిటి?

నోటిఫికేషన్ అనేది సర్వర్ నుండి క్లయింట్‌కు పంపబడే సందేశం, దీర్ఘకాలిక కార్యాచరణలో ప్రగతి, స్థితి లేదా ఇతర ఈవెంట్ల గురించి సమాచారం ఇవ్వడం. నోటిఫికేషన్లు పారదర్శకత మరియు వినియోగదారు అనుభవాన్ని మెరుగుపరుస్తాయి.

ఉదాహరణకి, సర్వర్‌తో మొదటి హ్యాండ్‌షేక్ పూర్తి అయిన వెంటనే క్లయింట్ నోటిఫికేషన్ పంపాలి.

ఒక నోటిఫికేషన్ JSON సందేశంగా ఇలా ఉంటుందని గుర్తించవచ్చు:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

నోటిఫికేషన్లు MCPలో ["లాగింగ్"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) అనే టాపిక్‌కి చెందినవి.

> **డిప్రికేషన్ నోటిస్:** `2026-07-28` MCP స్పెసిఫికేషన్ విడుదల అభ్యర్థి లాగింగ్ ప్రిమిటివ్‌ను stdio ట్రాన్స్‌పోర్ట్స్ కోసం `stderr` మరియు నిర్మాణాత్మక పరిశీలన కోసం OpenTelemetryకి మారుస్తోంది. 2025-11-25లో ఇది పని చేస్తూనే ఉంటుంది మరియు ఏ ఫార్మల్ డిప్రికేషన్ తర్వాత కనీసం ఒక సంవత్సరం. వివరాల కోసం [MCPలో ఏమి మారుతోంది: 2026-07-28 విడుదల అభ్యర్థి](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) చూడండి.

లాగ్ చేయడానికి, సర్వర్ దీన్ని ఒక ఫీచర్/క్యాపబిలిటీగా యాక్టివేట్ చేయాలి:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> ఉపయోగిస్తున్న SDK మీద ఆధారపడి, లాగింగ్ డిఫాల్ట్‌గా సక్రియమవ్వచ్చు లేదా మీరు సర్వర్ కాన్ఫిగరేషన్‌లో దీన్ని స్పష్టంగా యాక్టివేట్ చేయాల్సి ఉంటుంది.

| అత్యవసరం | సిస్టమ్ ఉపయోగించలేని స్థితి  | పూర్తి సిస్టమ్ వైఫల్యం         |

## MCPలో నోటిఫికేషన్లు అమలు చేయడం

MCPలో నోటిఫికేషన్లు అమలు చేయడానికి, రియల్-టైమ్ నవీకరణలను హ్యాండిల్ చేయడానికి సర్వర్ మరియు క్లయింట్ వైపులా సిద్ధం చేసుకోవాలి. ఇది దీర్ఘకాలిక కార్యకలాపాలలో వినియోగదారులకు వెంటనే ప్రతిస్పందనను అందిస్తుంది.

### సర్వర్-వైపు: నోటిఫికేషన్లు పంపడం

సర్వర్ వైపుతో ప్రారంభిద్దాం. MCPలో, మీరు అభ్యర్థనలను ప్రాసెస్ చేస్తూ నోటిఫికేషన్లు పంపగల టూల్స్‌ను నిర్వచిస్తారు. సర్వర్ సాధారణంగా `ctx` అనే కాంటెక్స్ట్ ఆబ్జెక్టును ఉపయోగించి సందేశాలను క్లయింట్‌కు పంపుతుంది.

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

పై ఉదాహరణలో, `process_files` టూల్ ప్రతీ ఫైల్ ప్రాసెసింగ్ సమయంలో క్లయింట్‌కు మూడు నోటిఫికేషన్లు పంపుతుంది. ఫార్మేట్ చేయడానికి `ctx.info()` పద్ధతి ఉపయోగించబడింది.

అదనంగా, నోటిఫికేషన్లు పంపడానికి, మీ సర్వర్ స్ట్రీమబుల్-http వంటి స్ట్రీమింగ్ ట్రాన్స్‌పోర్ట్ ఉపయోగించి ఉండాలి, మరియు క్లయింట్ నోటిఫికేషన్లను ప్రాసెస్ చేయడానికి మెసేజ్ హ్యాండ్లర్ అమలు చేయాలి. దిగువ మీ సర్వర్ ఎలా `streamable-http` ట్రాన్స్‌పోర్ట్ ఉపయోగించవచ్చో చూపుతుంది:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

ఈ .NET ఉదాహరణలో, `ProcessFiles` టూల్ `Tool` అట్రిబ్యూట్‌తో అలంకరించబడింది మరియు ప్రతీ ఫైల్ ప్రాసెసింగ్ సమయంలో మూడు నోటిఫికేషన్లు పంపుతోంది. `ctx.Info()` పద్ధతి సమాచార సందేశాలు పంపడానికి ఉపయోగించబడుతుంది.

మీ .NET MCP సర్వర్‌లో నోటిఫికేషన్లు యాక్టివేట్ చేయడానికి, మీరు స్ట్రీమింగ్ ట్రాన్స్‌పోర్ట్ ఉపయోగిస్తున్నారని నిర్ధారించుకోండి:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### క్లయింట్-వైపు: నోటిఫికేషన్లు స్వీకరించడం

క్లయింట్ కొత్త వచ్చిన నోటిఫికేషన్లను ప్రాసెస్ చేసి ప్రదర్శించే మెసేజ్ హ్యాండ్లర్ అమలు చేయాలి.

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

పైన కోడ్‌లో, `message_handler` ఫంక్షన్ వచ్చిన మెసేజ్ నోటిఫికేషన్ అయితే దానిని ప్రింట్ చేస్తుంది; కాకపోతే సాధారణ సర్వర్ సందేశంగా ప్రాసెస్ చేస్తుంది. అలాగే `ClientSession`ను `message_handler`తో ప్రారంభించడం గమనించండి.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

ఈ .NET ఉదాహరణలో, `MessageHandler` ఫంక్షన్ వచ్చిన మెసేజ్ నోటిఫికేషన్ అయితే దానిని ప్రింట్ చేస్తుంది; లేకపోతే సాధారణ సర్వర్ సందేశంగా ప్రాసెస్ చేస్తుంది. `ClientSession` ని మెసేజ్ హ్యాండ్లర్‌తో `ClientSessionOptions` ద్వారా ప్రారంభిస్తారు.

నోటిఫికేషన్లు సక్రియం చేయడానికి, మీ సర్వర్ స్ట్రీమబుల్-http వంటి స్ట్రీమింగ్ ట్రాన్స్‌పోర్ట్ ఉపయోగించాలని మరియు క్లయింట్ నోటిఫికేషన్ల ప్రాసెసింగ్‌కు మెసేజ్ హ్యాండ్లర్ అమలు చేసుకోవాలి.

## ప్రోగ్రెస్ నోటిఫికేషన్లు & పరిస్థితులు

ఈ విభాగం MCPలో ప్రోగ్రెస్ నోటిఫికేషన్ల భావన, వారందించిన ప్రాధాన్యాలు, మరియు Streamable HTTP ఉపయోగించి వాటిని ఎలా అమలు చేయాలో వివరిస్తుంది. మీరు మీ అర్థాన్ని బలపరచడానికి ఒక ఆచరణాత్మక అసైన్‌మెంట్ కూడా పొందుతారు.

ప్రోగ్రెస్ నోటిఫికేషన్లు అనేవి సర్వర్ నుండి క్లయింట్‌కు తక్షణ సందేశాలు, దీర్ఘకాలిక కార్యాచరణలో. మొత్తం ప్రక్రియ పూర్తి అయ్యే వరకు ఎదురు చూడకుండా, సర్వర్ వినియోగదారుని ప్రస్తుత స్థితిపై నవీకరిస్తూ ఉంటుంది. ఇది పారదర్శకతను, వినియోగదారు అనుభవాన్ని మెరుగుపరుస్తుంది, మరియు డీబగ్గింగ్ సులభతరం చేస్తుంది.

**ఉదాహరణ:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### ప్రోగ్రెస్ నోటిఫికేషన్లను ఎందుకు ఉపయోగించాలి?

ప్రోగ్రెస్ నోటిఫికేషన్లు అనేక కారణాల వల్ల అవసరం:

- **మంచి వినియోగదారు అనుభవం:** పని సాగుతూ ఉన్నప్పుడు నవీకరణలు గమనిస్తారు, కేవలం చివరలో కాదు.
- **రియల్-టైమ్ ఫీడ్బ్యాక్:** క్లయింట్లు ప్రోగ్రెస్ బార్లు లేదా లోగ్స్ ప్రదర్శించగలవు, అప్లికేషన్ ప్రతిస్పందన గలవాటిగా మారుతుంది.
- **డీబగ్గింగ్ మరియు పరిశీలనలో సౌలభ్యం:** డెవలపర్లు మరియు వినియోగదారులు ప్రక్రియ ఎక్కడ నెమ్మదిగా లేదా నిలిచిపోయిందో పరిశీలించగలరు.

### ప్రోగ్రెస్ నోటిఫికేషన్లు ఎలా అమలు చేయాలి

ఈ విధంగా MCPలో ప్రోగ్రెస్ నోటిఫికేషన్లను అమలు చేయవచ్చు:

- **సర్వర్‌లో:** ప్రతీ వస్తువును ప్రాసెస్ చేసే రోజు `ctx.info()` లేదా `ctx.log()` ఉపయోగించి నోటిఫికేషన్లు పంపండి. ఇది ప్రధాన ఫలితం సిద్ధమయ్యే ముందుగానే క్లయింట్‌కు సందేశం ఇస్తుంది.
- **క్లయింట్‌లో:** వచ్చిన నోటిఫికేషన్లను వినే, ప్రదర్శించే మెసేజ్ హ్యాండ్లర్ అమలు చేయండి. ఈ హ్యాండ్లర్ నోటిఫికేషన్లు మరియు తుది ఫలితం మధ్య తేడా చేస్తుంది.

**సర్వర్ ఉదాహరణ:**


#### పైథాన్

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**క్లయింట్ ఉదాహరణ:**

#### పైథాన్

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## భద్రతా పరిగణనలు

HTTP ఆధారిత ట్రాన్స్‌పోర్ట్‌లతో MCP సర్వర్‌లను అమలులో పడేటప్పుడు, భద్రత అనేది ముఖ్యమైన అంశంగా మారి అనేక దాడి మార్గాలు మరియు రక్షణ చర్యలపై జాగ్రత్తగా శ్రద్ధ పెట్టాలి.

### అవలోకనం

MCP సర్వర్‌లను HTTP ద్వారా అందిస్తున్నప్పుడు భద్రత చాలా కీలకం. స్ట్రీమబుల్ HTTP కొత్త దాడి ఉపరితలాలను పరిచయపరుస్తుంది మరియు జాగ్రత్తగా కాన్ఫిగర్ చేయడం అవసరం.

### ముఖ్యమైన అంశాలు

- **Origin హెడ్డర్ నిర్థారణ**: DNS రీబైండింగ్ దాడులను నివారించేందుకు ఎప్పుడూ `Origin` హెడ్డర్‌ను ధృవీకరించండి.
- **Localhost బైండింగ్**: స్థానిక అభివృద్ధి కోసం సర్వర్లు `localhost` కు బైండ్ చేసి వాటిని పబ్లిక్ ఇంటర్నెట్‌కు బయటకు ఉంచకండి.
- **ఆథెంటికేషన్**: ఉత్పత్తి వినియోగాల్లో ఆథెంటికేషన్ (ఉదా: API కీలు, OAuth) అమలుచేయండి.
- **CORS**: యాక్సెస్ పరిమితి కోసం క్రాస్-ఒరిజిన్ రిసోర్స్ షేరింగ్ (CORS) విధానాలను కాన్ఫిగర్ చేయండి.
- **HTTPS**: ఉత్పత్తిలో ట్రాఫిక్‌ను ఎన్‌క్రిప్ట్ చేయడానికి HTTPS ఉపయోగించండి.

### ఉత్తమ అభ్యాసాలు

- ధృవీకరణ లేని ఇన్‌కమింగ్ అభ్యర్థనలను ఎప్పుడూ నమ్మకండి.
- అన్ని యాక్సెస్లు మరియు తప్పిదాలను లాగ్ చేసి పరిశీలించండి.
- భద్రతా లోపాలను కోసం డిపెండెన్సీలను నిరంతరం పునఃసమీక్షించి నవీకరించండి.

### సవాళ్ళు

- భద్రత మరియు అభివృద్ధి సౌలభ్యం మధ్య సమతౌల్యం
- వివిధ క్లయింట్ పరిసరాల సరిపోలికను నిర్ధారించడం

## SSE నుండి స్ట్రీమబుల్ HTTP కి అప్‌గ్రేడ్

ప్రస్తుతం Server-Sent Events (SSE) ఉపయోగిస్తున్న అనువర్తనాల కోసం, స్ట్రీమబుల్ HTTP కి మారడం మీ MCP అమలులకు మెరుగైన సామర్థ్యం మరియు దీర్ఘకాలిక స్థిరత్వం అందిస్తుంది.

### ఎందుకు అప్‌గ్రేడ్ చేయాలి?

SSE నుండి స్ట్రీమబుల్ HTTP కి అప్‌గ్రేడ్ చేయడానికి రెండు ప్రధాన కారణాలు ఉన్నాయి:

- స్ట్రీమబుల్ HTTP SSE కంటే మెరుగైన స్కేలబిలిటీ, సరిపోలిక మరియు సంపూర్ణ సమాచారం మద్దతు అందిస్తుంది.
- ఇది కొత్త MCP అనువర్తనాలకు సిఫార్సు చేయబడిన ట్రాన్స్‌పోర్ట్.

### మార్చే దశలు

మీ MCP అనువర్తనాల్లో SSE నుండి స్ట్రీమబుల్ HTTP కి ఎలా మారాలో ఇక్కడ ఉంది:

- సర్వర్ కోడ్‌ను `mcp.run()` లో `transport="streamable-http"` ఉపయోగించ도록 నవీకరించండి.
- SSE క్లయింట్ బదులు `streamablehttp_client` ఉపయోగించేలా క్లయింట్ కోడ్‌ను నవీకరించండి.
- నోటిఫికేషన్‌లను ప్రాసెస్ చేయడానికి క్లయింట్‌లో సందేశ హ్యాండ్లర్‌ను అమలు చేయండి.
- ఉన్న సాధనాలు మరియు వర్క్‌ఫ్లోలతో సరిపోలుతుందో లేదో పరీక్షించండి.

### సరిపోలికను నిర్వహించడం

మార్చే ప్రక్రియలో ఉన్న SSE క్లయింట్లతో సరిపోలికను కొనసాగించడం సిఫారసు చేయబడింది. కొన్ని రణనీతులు ఇక్కడ ఉన్నాయి:

- SSE మరియు స్ట్రీమబుల్ HTTP రెండు ట్రాన్స్‌పోర్ట్‌లను వేర్వేరు ఎండ్‌పాయింట్‌లపై నడుపుతూ రెండింటినీ మద్దతు ఇవ్వవచ్చు.
- క్లయింట్లను కొత్త ట్రాన్స్‌పోర్ట్‌కి స్థిరంగా మార్చండి.

### సవాళ్ళు

మార్చే సమయంలో ఈ సవాళ్లకు సమాధానం ఇవ్వండి:

- అన్ని క్లయింట్లు నవీకరించబడి ఉండటం
- నోటిఫికేషన్‌ల పంపిణీలో తేడాలు నిర్వహించడం

## భద్రతా పరిగణనలు

Streamable HTTP లాంటి HTTP ఆధారిత ట్రాన్స్‌పోర్ట్‌లను ఉపయోగించే MCP సర్వర్‌లను అమలులో పెట్టేటప్పుడు భద్రత ప్రాధాన్యత కావాలి.

HTTP ఆధారిత ట్రాన్స్‌పోర్ట్‌లతో MCP సర్వర్‌లను అమలులో పడేటప్పుడు, భద్రత అనేది ముఖ్యమైన అంశంగా మారి అనేక దాడి మార్గాలు మరియు రక్షణ చర్యలపై జాగ్రత్తగా శ్రద్ధ పెట్టాలి.

### అవలోకనం

MCP సర్వర్‌లను HTTP ద్వారా అందిస్తున్నప్పుడు భద్రత చాలా కీలకం. స్ట్రీమబుల్ HTTP కొత్త దాడి ఉపరితలాలను పరిచయపరుస్తుంది మరియు జాగ్రత్తగా కాన్ఫిగర్ చేయడం అవసరం.

కొన్ని ముఖ్య భద్రతా పరిగణనలివి:

- **Origin హెడ్డర్ నిర్థారణ**: DNS రీబైండింగ్ దాడులను నివారించేందుకు ఎప్పుడూ `Origin` హెడ్డర్‌ను ధృవీకరించండి.
- **Localhost బైండింగ్**: స్థానిక అభివృద్ధి కోసం సర్వర్లు `localhost` కు బైండ్ చేసి వాటిని పబ్లిక్ ఇంటర్నెట్‌కు బయటకు ఉంచకండి.
- **ఆథెంటికేషన్**: ఉత్పత్తి వినియోగాల్లో ఆథెంటికేషన్ (ఉదా: API కీలు, OAuth) అమలుచేయండి.
- **CORS**: యాక్సెస్ పరిమితి కోసం క్రాస్-ఒరిజిన్ రిసోర్స్ షేరింగ్ (CORS) విధానాలను కాన్ఫిగర్ చేయండి.
- **HTTPS**: ఉత్పత్తిలో ట్రాఫిక్‌ను ఎన్‌క్రిప్ట్ చేయడానికి HTTPS ఉపయోగించండి.

### ఉత్తమ అభ్యాసాలు

MCP స్ట్రీమింగ్ సర్వర్‌లో భద్రతను అమలులో పెట్టేటప్పుడు అనుసరించాల్సిన కొన్ని ఉత్తమ అభ్యాసాలు ఇక్కడ ఉన్నాయి:

- ధృవీకరణ లేని ఇన్‌కమింగ్ అభ్యర్థనలను ఎప్పుడూ నమ్మకండి.
- అన్ని యాక్సెస్లు మరియు తప్పిదాలను లాగ్ చేసి పరిశీలించండి.
- భద్రతా లోపాలను కోసం డిపెండెన్సీలను నిరంతరం పునఃసమీక్షించి నవీకరించండి.

### సవాళ్ళు

MCP స్ట్రీమింగ్ సర్వర్‌లలో భద్రతను అమలులో పెట్టేటప్పుడు కొన్ని సవాళ్ళను ఎదుర్కొనవలసి ఉంటుంది:

- భద్రత మరియు అభివృద్ధి సౌలభ్యం మధ్య సమతౌల్యం
- వివిధ క్లయింట్ పరిసరాల సరిపోలికను నిర్ధారించడం

### అసైన్‌మెంట్: మీ సొంత స్ట్రీమింగ్ MCP అనువర్తనం రూపొందించండి

**సిద్ధాంతం:**
ఒక MCP సర్వర్ మరియు క్లయింట్‌ను రూపొందించండి, ఇక్కడ సర్వర్ ఐటెమ్‌ల (ఉదా: ఫైళ్లు లేదా డాక్యుమెంట్లు) జాబితాను ప్రాసెస్ చేసి ప్రతి ప్రాసెస్ అయ్యే ఐటెమ్‌కు నోటిఫికేషన్ పంపుతుంది. క్లయింట్ ప్రతి నోటిఫికేషన్‌ను వచ్చే సరికి ప్రదర్శించాలి.

**దశలు:**

1. జాబితాను ప్రాసెస్ చేసి ప్రతి ఐటెమ్‌కు నోటిఫికేషన్ పంపే సర్వర్ సాధనాన్ని అమలు చేయండి.
2. రియల్ టైమ్‌లో నోటిఫికేషన్‌లను ప్రదర్శించడానికి సందేశ హ్యాండ్లర్ ఉన్న క్లయింట్‌ను అమలు చేయండి.
3. సర్వర్ మరియు క్లయింట్ రెండింటినీ నడిపించి, నోటిఫికేషన్‌లను గమనించండి మరియు పరీక్షించండి.

## తదుపరి పఠనం & తరువాత ఏమి?

MCP స్ట్రీమింగ్ పయనం కొనసాగించడానికి మరియు మీ జ్ఞానం విస్తరించడానికి, ఈ విభాగంలో అదనపు వనరులు మరియు మరింత అభివృద్ధి చెందిన అనువర్తనాలను రూపొందించడానికి సూచించిన తదుపరి దశలను అందిస్తున్నాం.

- [Microsoft: Server-Sent Events (SSE)](https://learn.microsoft.com/azure/application-gateway/for-containers/server-sent-events?tabs=server-sent-events-gateway-api&WT.mc_id=%3Fwt.mc_id%3DMVP_452430)

- [Python requests: Streaming Requests](https://requests.readthedocs.io/en/latest/user/advanced/#streaming-requests)

### తదుపరి ఏమి?

- స్ట్రీమింగ్‌ను ఉపయోగించి రియల్ టైమ్ విశ్లేషణ, చాట్ లేదా సహకార ఎడిటింగ్ కోసం మరింత అభివృద్ధి చెందిన MCP సాధనాలను రూపొందించడానికి ప్రయత్నించండి.
- MCP స్ట్రీమింగ్‌ను ఫ్రంట్‌ఎండ్ ఫ్రేమ్‌వర్క్‌లతో (React, Vue మరియు ఇతరులు) సహకరించి ప్రత్యక్ష UI నవీకరణలను అన్వేషించండి.
- తదుపరి: [VSCode కోసం AI టూల్‌కిట్ ఉపయోగించడం](../07-aitk/README.md)
