# मॉडल कॉन्टेक्स्ट प्रोटोकॉल (MCP) के साथ HTTPS स्ट्रीमिंग

यह अध्याय मॉडल कॉन्टेक्स्ट प्रोटोकॉल (MCP) का उपयोग करके सुरक्षित, स्केलेबल और वास्तविक-समय स्ट्रीमिंग को लागू करने के लिए एक व्यापक मार्गदर्शिका प्रदान करता है। इसमें स्ट्रीमिंग के प्रेरणा, उपलब्ध ट्रांसपोर्ट मैकेनिज्म, MCP में स्ट्रीमेबल HTTP को कैसे लागू किया जाए, सुरक्षा सर्वोत्तम प्रथाएं, SSE से माइग्रेशन, और अपने खुद के स्ट्रीमिंग MCP एप्लिकेशन बनाने के लिए व्यावहारिक मार्गदर्शन शामिल है।

> **आगे देखते हुए:** यह पाठ MCP विनिर्देशन 2025-11-25 के तहत स्ट्रीमेबल HTTP को वर्णित करता है, जहाँ एक सत्र `initialize` के दौरान स्थापित होता है और एक `Mcp-Session-Id` हेडर के साथ पिन किया जाता है। `2026-07-28` रिलीज़ उम्मीदवार हैंडशेक और सत्र आईडी को पूरी तरह हटा देता है, जिससे प्रत्येक अनुरोध स्व-निहित होता है और किसी भी सर्वर इंस्टेंस पर बिना स्टिकी सेशंस के मार्गित किया जा सकता है। विवरण के लिए [MCP में क्या बदल रहा है: 2026-07-28 रिलीज़ उम्मीदवार](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) देखें।

## MCP में ट्रांसपोर्ट मैकेनिज्म और स्ट्रीमिंग

यह अनुभाग MCP में उपलब्ध विभिन्न ट्रांसपोर्ट मैकेनिज्म और ग्राहकों और सर्वरों के बीच वास्तविक-समय संचार के लिए स्ट्रीमिंग क्षमताएँ सक्षम करने में उनकी भूमिका का अन्वेषण करता है।

### ट्रांसपोर्ट मैकेनिज्म क्या है?

एक ट्रांसपोर्ट मैकेनिज्म यह परिभाषित करता है कि डेटा ग्राहक और सर्वर के बीच कैसे आदान-प्रदान किया जाता है। MCP विभिन्न परिवेशों और आवश्यकताओं के अनुसार कई ट्रांसपोर्ट प्रकारों का समर्थन करता है:

- **stdio**: मानक इनपुट/आउटपुट, स्थानीय और CLI-आधारित उपकरणों के लिए उपयुक्त। सरल है लेकिन वेब या क्लाउड के लिए उपयुक्त नहीं।
- **SSE (सर्वर-सेंट इवेंट्स)**: सर्वर को HTTP के माध्यम से ग्राहकों को वास्तविक-समय अपडेट भेजने की अनुमति देता है। वेब UIs के लिए अच्छा, लेकिन स्केलेबिलिटी और लचीलापन सीमित। MCP विनिर्देशन 2025-06-18 के अनुसार, स्टैंडअलोन SSE ट्रांसपोर्ट को हटा दिया गया है और इसे "स्ट्रीमेबल HTTP" ट्रांसपोर्ट से प्रतिस्थापित किया गया है।
- **स्ट्रीमेबल HTTP**: आधुनिक HTTP-आधारित स्ट्रीमिंग ट्रांसपोर्ट, अधिसूचनाओं और बेहतर स्केलेबिलिटी का समर्थन करता है। अधिकांश उत्पादन और क्लाउड परिदृश्यों के लिए अनुशंसित।

### तुलना तालिका

इन ट्रांसपोर्ट मैकेनिज्म के बीच अंतर समझने के लिए नीचे तुलना तालिका देखें:

| ट्रांसपोर्ट         | वास्तविक-समय अपडेट्स | स्ट्रीमिंग | स्केलेबिलिटी | उपयोग मामला          |
|-------------------|------------------|-----------|-------------|---------------------|
| stdio             | नहीं              | नहीं      | कम          | स्थानीय CLI टूल्स   |
| SSE               | हाँ               | हाँ       | मध्यम       | वेब, वास्तविक-समय अपडेट्स |
| स्ट्रीमेबल HTTP   | हाँ               | हाँ       | उच्च        | क्लाउड, मल्टी-क्लाइंट|

> **टिप:** सही ट्रांसपोर्ट चुनने से प्रदर्शन, स्केलेबिलिटी और उपयोगकर्ता अनुभव प्रभावित होता है। **स्ट्रीमेबल HTTP** आधुनिक, स्केलेबल और क्लाउड-तैयार एप्लिकेशन के लिए अनुशंसित है।

ध्यान दें कि पिछले अध्यायों में जो stdio और SSE ट्रांसपोर्ट दिखाए गए थे, और इस अध्याय में कवर किया गया स्ट्रीमेबल HTTP ट्रांसपोर्ट।

## स्ट्रीमिंग: अवधारणाएँ और प्रेरणा

स्ट्रीमिंग के मूलभूत सिद्धांतों और प्रेरणाओं को समझना प्रभावी वास्तविक-समय संचार प्रणाली को लागू करने के लिए आवश्यक है।

**स्ट्रीमिंग** नेटवर्क प्रोग्रामिंग में एक तकनीक है जो डेटा को छोटी, प्रबंधनीय खंडों या घटनाओं की श्रृंखला के रूप में भेजने और प्राप्त करने की अनुमति देती है, बजाय पूरे प्रतिक्रिया के तैयार होने का इंतजार करने के। यह विशेष रूप से उपयोगी है:

- बड़े फ़ाइलों या डेटा सेट के लिए।
- वास्तविक-समय अपडेट्स के लिए (जैसे, चैट, प्रोग्रेस बार)।
- लंबे समय तक चलने वाली गणनाओं के लिए जहाँ उपयोगकर्ता को सूचित रखना हो।

यहाँ स्ट्रीमिंग के बारे में उच्च स्तरीय जानकारी दी गई है:

- डेटा धीरे-धीरे दिया जाता है, एकसाथ नहीं।
- ग्राहक डेटा के आते ही उसे संसाधित कर सकता है।
- प्रत्याशित विलंब को कम करता है और उपयोगकर्ता अनुभव को बेहतर बनाता है।

### स्ट्रीमिंग क्यों उपयोग करें?

स्ट्रीमिंग उपयोग करने के कारण निम्नलिखित हैं:

- उपयोगकर्ताओं को तुरंत प्रतिक्रिया मिलती है, केवल अंत में नहीं।
- वास्तविक-समय एप्लिकेशन और उत्तरदायी यूआई सक्षम करता है।
- नेटवर्क और कंप्यूट संसाधनों का अधिक कुशल उपयोग।

### साधारण उदाहरण: HTTP स्ट्रीमिंग सर्वर और क्लाइंट

यहाँ एक सरल उदाहरण है कि स्ट्रीमिंग को कैसे लागू किया जा सकता है:

#### पायथन

**सर्वर (पायथन, FastAPI और StreamingResponse का उपयोग करते हुए):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**क्लाइंट (पायथन, requests का उपयोग करते हुए):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

यह उदाहरण दिखाता है कि सर्वर कैसे संदेशों की एक श्रृंखला ग्राहक को भेजता है जैसे-जैसे वे उपलब्ध होते हैं, बजाय सभी संदेशों के तैयार होने का इंतजार करने के।

**यह कैसे काम करता है:**

- सर्वर प्रत्येक संदेश को जैसे ही तैयार होता है यील्ड करता है।
- ग्राहक प्रत्येक खंड को प्राप्त करता है और आते ही प्रिंट करता है।

**आवश्यकताएं:**

- सर्वर को स्ट्रीमिंग प्रतिक्रिया का उपयोग करना चाहिए (जैसे, FastAPI में `StreamingResponse`)।
- ग्राहक को प्रतिक्रिया को स्ट्रीम के रूप में संसाधित करना चाहिए (`requests` में `stream=True`)।
- कंटेंट-टाइप आमतौर पर `text/event-stream` या `application/octet-stream` होता है।

#### जावा

**सर्वर (जावा, Spring Boot और Server-Sent Events का उपयोग करते हुए):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**क्लाइंट (जावा, Spring WebFlux WebClient का उपयोग करते हुए):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**जावा कार्यान्वयन नोट्स:**

- स्ट्रीमिंग के लिए Spring Boot के रिएक्टिव स्टैक का उपयोग करता है `Flux` के साथ
- `ServerSentEvent` संरचित इवेंट स्ट्रीमिंग प्रदान करता है जिसमें इवेंट प्रकार होते हैं
- `WebClient` `bodyToFlux()` के साथ रिएक्टिव स्ट्रीमिंग उपभोग सक्षम करता है
- `delayElements()` इवेंट्स के बीच प्रसंस्करण समय का अनुकरण करता है
- इवेंट्स के टाइप हो सकते हैं (`info`, `result`) बेहतर क्लाइंट हैंडलिंग के लिए

### तुलना: पारंपरिक स्ट्रीमिंग बनाम MCP स्ट्रीमिंग

"क्लासिक" तरीके से स्ट्रीमिंग कैसे काम करती है उसके मुकाबले MCP में यह कैसे काम करती है के बीच अंतर इस प्रकार हो सकते हैं:

| फीचर                  | क्लासिक HTTP स्ट्रीमिंग          | MCP स्ट्रीमिंग (अधिसूचनाएं)      |
|------------------------|---------------------------------|-----------------------------------|
| मुख्य प्रतिक्रिया       | खंडित                          | एकल, अंत में                     |
| प्रगति अपडेट           | डेटा खंड के रूप में भेजा गया    | अधिसूचनाओं के रूप में भेजा गया |
| ग्राहक आवश्यकताएं      | स्ट्रीम को संसाधित करना चाहिए   | संदेश हैंडलर लागू करना चाहिए    |
| उपयोग मामला            | बड़ी फ़ाइलें, AI टोकन स्ट्रीम्स | प्रगति, लॉग, वास्तविक-समय प्रतिक्रिया|

### प्रमुख अंतर देखे गए

इसके अतिरिक्त, यहाँ कुछ प्रमुख अंतर हैं:

- **संचार पैटर्न:**
  - क्लासिक HTTP स्ट्रीमिंग: डेटा को खंडों में भेजने के लिए सरल खंडित ट्रांसफर एन्कोडिंग का उपयोग करता है
  - MCP स्ट्रीमिंग: JSON-RPC प्रोटोकॉल के साथ एक संरचित अधिसूचना प्रणाली का उपयोग करता है

- **संदेश प्रारूप:**
  - क्लासिक HTTP: नई लाइनों के साथ साधारण टेक्स्ट खंड
  - MCP: मेटाडेटा के साथ संरचित LoggingMessageNotification वस्तुएं

- **ग्राहक कार्यान्वयन:**
  - क्लासिक HTTP: स्ट्रीमिंग प्रतिक्रियाओं को संसाधित करने वाला साधारण क्लाइंट
  - MCP: विभिन्न प्रकार के संदेशों को संसाधित करने के लिए एक संदेश हैंडलर के साथ अधिक परिष्कृत क्लाइंट

- **प्रगति अद्यतन:**
  - क्लासिक HTTP: प्रगति मुख्य प्रतिक्रिया स्ट्रीम का हिस्सा होती है
  - MCP: प्रगति मुख्य प्रतिक्रिया के बजाय अलग अधिसूचना संदेशों के माध्यम से भेजी जाती है

### सिफारिशें

क्लासिक स्ट्रीमिंग (जैसे हमने `/stream` एपीआई पर दिखाया) बनाम MCP के माध्यम से स्ट्रीमिंग चुनने के संबंध में कुछ सलाह हम देते हैं।

- **सरल स्ट्रीमिंग आवश्यकताओं के लिए:** क्लासिक HTTP स्ट्रीमिंग लागू करना सरल है और बुनियादी स्ट्रीमिंग आवश्यकताओं के लिए पर्याप्त है।

- **जटिल, इंटरैक्टिव ऐप्लिकेशनों के लिए:** MCP स्ट्रीमिंग अधिक संरचित दृष्टिकोण प्रदान करता है जिसमें समृद्ध मेटाडेटा और अधिसूचनाओं व अंतिम परिणामों के बीच पृथक्करण होता है।

- **AI एप्लिकेशनों के लिए:** MCP की अधिसूचना प्रणाली लंबे समय तक चलने वाले AI कार्यों में विशेष रूप से उपयोगी है जहाँ आप उपयोगकर्ताओं को प्रगति से सूचित रखना चाहते हैं।

## MCP में स्ट्रीमिंग

ठीक है, आपने अब तक क्लासिक स्ट्रीमिंग और MCP में स्ट्रीमिंग के बीच अंतर और सुझाव देख लिए हैं। अब विस्तार से जानते हैं कि आप MCP में स्ट्रीमिंग का उपयोग कैसे कर सकते हैं।

MCP फ्रेमवर्क के भीतर स्ट्रीमिंग कैसे काम करता है, यह समझना उत्तरदायी एप्लिकेशन बनाने के लिए आवश्यक है जो लंबे समय तक चलने वाले कार्यों के दौरान उपयोगकर्ताओं को वास्तविक-समय प्रतिक्रिया प्रदान करे।

MCP में, स्ट्रीमिंग मुख्य प्रतिक्रिया को खंडों में भेजने के बारे में नहीं है, बल्कि एक टूल के अनुरोध को संसाधित करते समय क्लाइंट को **अधिसूचनाएं** भेजने के बारे में है। ये अधिसूचनाएं प्रगति अपडेट्स, लॉग्स या अन्य घटनाओं को शामिल कर सकती हैं।

### यह कैसे काम करता है

मुख्य परिणाम अभी भी एकल प्रतिक्रिया के रूप में भेजा जाता है। हालांकि, अधिसूचनाएं संसाधन के दौरान पृथक संदेशों के रूप में भेजी जा सकती हैं और इस प्रकार ग्राहक को वास्तविक समय में अपडेट करती हैं। ग्राहक को इन अधिसूचनाओं को संभालने और प्रदर्शित करने में सक्षम होना चाहिए।

## अधिसूचना क्या है?

हमने "अधिसूचना" कहा, MCP के संदर्भ में इसका क्या मतलब है?

अधिसूचना सर्वर से ग्राहक को भेजा गया संदेश है जो एक लंबे समय तक चलने वाले ऑपरेशन के दौरान प्रगति, स्थिति, या अन्य घटनाओं के बारे में सूचित करता है। अधिसूचनाएं पारदर्शिता और उपयोगकर्ता अनुभव को बेहतर बनाती हैं।

उदाहरण के लिए, एक ग्राहक को अधिसूचना भेजनी चाहिए जब सर्वर के साथ प्रारंभिक हैंडशेक पूरा हो जाता है।

एक अधिसूचना JSON संदेश के रूप में इस प्रकार दिखती है:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

अधिसूचनाएं MCP में एक विषय से संबंधित हैं जिसे ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) कहा जाता है।

> **डिप्रिकेट नोटिस:** `2026-07-28` MCP विनिर्देशन रिलीज़ उम्मीदवार लॉगिंग प्रिमिटिव को stdio ट्रांसपोर्ट के लिए `stderr` और संरचित अवलोकन के लिए OpenTelemetry के पक्ष में डिप्रिकेट घोषित करता है। लॉगिंग `2025-11-25` और किसी भी आधिकारिक डिप्रेशन के कम से कम एक वर्ष बाद तक काम करता रहेगा। देखें [MCP में क्या बदल रहा है: 2026-07-28 रिलीज़ उम्मीदवार](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md)।

लॉगिंग काम करने के लिए, सर्वर को इसे फीचर/क्षमता के रूप में सक्षम करना होता है, इस प्रकार:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> उपयोग किए गए SDK के आधार पर, लॉगिंग डिफ़ॉल्ट रूप से सक्षम हो सकता है, या आपको इसे स्पष्ट रूप से अपने सर्वर कॉन्फ़िगरेशन में सक्षम करना पड़ सकता है।

विभिन्न प्रकार के अधिसूचनाएं होती हैं:

| स्तर        | वर्णन                          | उपयोग का उदाहरण               |
|-------------|-------------------------------|-------------------------------|
| debug       | विस्तृत डिबगिंग जानकारी          | फ़ंक्शन में प्रवेश/निकास बिंदु   |
| info        | सामान्य जानकारीपूर्ण संदेश        | संचालन प्रगति अपडेट्स            |
| notice      | सामान्य लेकिन महत्वपूर्ण घटनाएं   | कॉन्फ़िगरेशन परिवर्तन            |
| warning     | चेतावनी स्थितियां               | डिप्रिकेटेड फीचर का उपयोग      |
| error       | त्रुटि स्थितियां               | संचालन विफलताएं                 |
| critical    | गंभीर स्थितियां                | सिस्टम घटक विफलताएं             |
| alert       | तुरंत कार्रवाई आवश्यक है        | डेटा भ्रष्टाचार पाया गया        |
| emergency   | सिस्टम अनुपयोगी है             | पूर्ण सिस्टम विफलता              |

## MCP में अधिसूचनाएं लागू करना

MCP में अधिसूचनाएं लागू करने के लिए, आपको दोनों सर्वर और क्लाइंट पक्ष को वास्तविक-समय अपडेट्स संभालने के लिए सेटअप करना होगा। यह आपके एप्लिकेशन को लंबे समय तक चलने वाले कार्यों के दौरान उपयोगकर्ताओं को तत्काल प्रतिक्रिया देने की अनुमति देता है।

### सर्वर-पक्ष: अधिसूचनाएं भेजना

सर्वर पक्ष से शुरू करते हैं। MCP में, आप ऐसे टूल परिभाषित करते हैं जो अनुरोधों को संसाधित करते हुए अधिसूचनाएं भेज सकते हैं। सर्वर संदर्भ ऑब्जेक्ट (`ctx` सामान्यतः) का उपयोग संदेश भेजने के लिए करता है।

#### पायथन

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

पिछले उदाहरण में, `process_files` टूल हर फ़ाइल के संसाधित होते समय ग्राहक को तीन अधिसूचनाएं भेजता है। `ctx.info()` विधि का उपयोग सूचना संदेश भेजने के लिए किया जाता है।

इसके अतिरिक्त, अधिसूचनाएं सक्षम करने के लिए सुनिश्चित करें कि आपका सर्वर स्ट्रीमिंग ट्रांसपोर्ट (जैसे `streamable-http`) का उपयोग कर रहा है और आपका क्लाइंट अधिसूचनाओं को संसाधित करने के लिए संदेश हैंडलर लागू करता है। यहाँ बताया गया है कि सर्वर को `streamable-http` ट्रांसपोर्ट का उपयोग कैसे सेटअप करें:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

इस .NET उदाहरण में, `ProcessFiles` टूल `Tool` विशेषता के साथ सजाया गया है और प्रत्येक फ़ाइल संसाधित करते समय ग्राहक को तीन अधिसूचनाएं भेजता है। `ctx.Info()` विधि का उपयोग सूचना संदेश भेजने के लिए किया जाता है।

आपके .NET MCP सर्वर में अधिसूचनाएं सक्षम करने के लिए, सुनिश्चित करें कि आप स्ट्रीमिंग ट्रांसपोर्ट का उपयोग कर रहे हैं:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### क्लाइंट-पक्ष: अधिसूचनाएं प्राप्त करना

क्लाइंट को संदेश हैंडलर लागू करना चाहिए जो अधिसूचनाओं को प्राप्त करे और प्रदर्शित करे जैसे ही वे आते हैं।

#### पायथन

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

पिछले कोड में, `message_handler` फ़ंक्शन जांचता है कि क्या आने वाला संदेश एक अधिसूचना है। यदि हाँ, तो यह अधिसूचना प्रदर्शित करता है; अन्यथा, इसे नियमित सर्वर संदेश के रूप में संसाधित करता है। साथ ही, ध्यान दें कि `ClientSession` को इनकमिंग अधिसूचनाओं को संभालने के लिए `message_handler` के साथ और इनिशियलाइज़ किया गया है।

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

इस .NET उदाहरण में, `MessageHandler` फ़ंक्शन जांचता है कि क्या आने वाला संदेश एक अधिसूचना है। यदि हाँ, तो यह प्रदर्शित करता है; अन्यथा, इसे सामान्य सर्वर संदेश के रूप में संसाधित करता है। `ClientSession` को `ClientSessionOptions` के माध्यम से संदेश हैंडलर के साथ इनिशियलाइज़ किया गया है।

अधिसूचनाएं सक्षम करने के लिए, सुनिश्चित करें कि आपके सर्वर द्वारा स्ट्रीमिंग ट्रांसपोर्ट (जैसे `streamable-http`) का उपयोग किया जा रहा है और आपका क्लाइंट अधिसूचनाओं को संसाधित करने वाला संदेश हैंडलर लागू करता है।

## प्रगति अधिसूचनाएं और परिदृश्य

यह अनुभाग MCP में प्रगति अधिसूचनाओं की अवधारणा, उनका महत्व, और स्ट्रीमेबल HTTP के उपयोग से उन्हें कैसे लागू करें, समझाता है। आप अपने समझ को मजबूत करने के लिए एक व्यावहारिक कार्य भी पाएंगे।

प्रगति अधिसूचनाएं वे वास्तविक-समय संदेश होते हैं जो लंबे चलने वाले ऑपरेशन के दौरान सर्वर से ग्राहक को भेजे जाते हैं। पूरी प्रक्रिया खत्म होने का इंतजार करने के बजाय, सर्वर ग्राहक को चालू स्थिति के बारे में अपडेट करता रहता है। इससे पारदर्शिता, उपयोगकर्ता अनुभव बेहतर होता है और डिबगिंग आसान हो जाती है।

**उदाहरण:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### प्रगति अधिसूचनाएं क्यों उपयोग करें?

प्रगति अधिसूचनाएं कई कारणों से आवश्यक हैं:

- **बेहतर उपयोगकर्ता अनुभव:** उपयोगकर्ता काम की प्रगति को अंतिम में ही नहीं, करते हुए भी देख सकते हैं।
- **वास्तविक-समय प्रतिक्रिया:** ग्राहक प्रगति बार या लॉग दिखा सकते हैं, जिससे ऐप उत्तरदायी महसूस होता है।
- **सरल डिबगिंग और निगरानी:** डेवलपर्स और उपयोगकर्ता देख सकते हैं कि प्रक्रिया कहाँ धीमी या अटकी हुई है।

### प्रगति अधिसूचनाएं कैसे लागू करें

यहाँ MCP में प्रगति अधिसूचनाएं लागू करने का तरीका बताया गया है:

- **सर्वर पर:** `ctx.info()` या `ctx.log()` का उपयोग करें ताकि प्रत्येक आइटम संसाधित होते समय अधिसूचनाएं भेजी जा सकें। इससे मुख्य परिणाम तैयार होने से पहले ग्राहक को संदेश भेजा जाता है।
- **क्लाइंट पर:** एक ऐसा संदेश हैंडलर लागू करें जो अधिसूचनाओं को सुनता और प्रदर्शित करता है। यह हैंडलर अधिसूचनाओं और अंतिम परिणाम के बीच अंतर करता है।

**सर्वर उदाहरण:**


#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**क्लाइंट उदाहरण:**

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## सुरक्षा विचार

जब HTTP-आधारित ट्रांसपोर्ट के साथ MCP सर्वर लागू किए जाते हैं, तो सुरक्षा एक प्रमुख चिंता बन जाती है जिसे कई हमले के तरीकों और सुरक्षा तंत्रों पर सावधानीपूर्वक ध्यान देने की आवश्यकता होती है।

### अवलोकन

MCP सर्वरों को HTTP पर एक्सपोज़ करने पर सुरक्षा अत्यंत महत्वपूर्ण होती है। स्ट्रीमेबल HTTP नए हमले के सतहें पेश करता है और सावधानीपूर्वक कॉन्फ़िगरेशन की आवश्यकता होती है।

### मुख्य बिंदु

- **Origin हेडर सत्यापन**: DNS रिबाइंडिंग हमलों को रोकने के लिए हमेशा `Origin` हेडर का सत्यापन करें।
- **Localhost बाइंडिंग**: स्थानीय विकास के लिए, सर्वरों को `localhost` से बांधें ताकि वे सार्वजनिक इंटरनेट पर एक्सपोज़ न हों।
- **प्रमाणीकरण**: उत्पादन डिप्लॉयमेंट के लिए प्रमाणीकरण लागू करें (जैसे API कुंजी, OAuth)।
- **CORS**: एक्सेस को सीमित करने के लिए Cross-Origin Resource Sharing (CORS) नीतियां कॉन्फ़िगर करें।
- **HTTPS**: ट्रैफ़िक को एन्क्रिप्ट करने के लिए उत्पादन में HTTPS का उपयोग करें।

### सर्वोत्तम अभ्यास

- बिना सत्यापन के इनकमिंग अनुरोधों पर कभी भरोसा न करें।
- सभी एक्सेस और त्रुटियों को लॉग और मॉनिटर करें।
- सुरक्षा कमजोरियों को ठीक करने के लिए नियमित रूप से निर्भरता अपडेट करें।

### चुनौतियाँ

- विकास की सुविधा और सुरक्षा के बीच संतुलन बनाना
- विभिन्न क्लाइंट वातावरणों के साथ संगतता सुनिश्चित करना

## SSE से Streamable HTTP में उन्नयन

जो एप्लिकेशन वर्तमान में Server-Sent Events (SSE) का उपयोग कर रहे हैं, उनके लिए Streamable HTTP पर माइग्रेट करना बेहतर क्षमताएं और दीर्घकालिक स्थिरता प्रदान करता है।

### क्यों अपग्रेड करें?

SSE से Streamable HTTP में अपग्रेड करने के दो मुख्य कारण हैं:

- Streamable HTTP बेहतर स्केलेबिलिटी, संगतता और समृद्ध नोटिफिकेशन समर्थन प्रदान करता है।
- यह नए MCP एप्लिकेशन के लिए अनुशंसित ट्रांसपोर्ट है।

### माइग्रेशन चरण

अपने MCP एप्लिकेशन में SSE से Streamable HTTP पर माइग्रेट करने के तरीके:

- `mcp.run()` में `transport="streamable-http"` का उपयोग करने के लिए सर्वर कोड अपडेट करें।
- SSE क्लाइंट के बजाय `streamablehttp_client` का उपयोग करने के लिए क्लाइंट कोड अपडेट करें।
- नोटिफिकेशन संसाधित करने के लिए क्लाइंट में एक संदेश हैंडलर लागू करें।
- मौजूदा उपकरणों और वर्कफ़्लोज़ के साथ संगतता के लिए परीक्षण करें।

### संगतता बनाए रखना

माइग्रेशन प्रक्रिया के दौरान मौजूदा SSE क्लाइंट के साथ संगतता बनाए रखना अनुशंसित है। यहां कुछ रणनीतियाँ हैं:

- आप दोनों ट्रांसपोर्ट्स को अलग-अलग एंडपॉइंट्स पर चलाकर SSE और Streamable HTTP दोनों समर्थित कर सकते हैं।
- क्लाइंट को धीरे-धीरे नए ट्रांसपोर्ट पर माइग्रेट करें।

### चुनौतियाँ

माइग्रेशन के दौरान निम्न चुनौतियों को संबोधित करना आवश्यक है:

- सभी क्लाइंट को अपडेट करना सुनिश्चित करना
- नोटिफिकेशन डिलीवरी में भिन्नताओं को संभालना

## सुरक्षा विचार

किसी भी सर्वर को लागू करते समय सुरक्षा एक शीर्ष प्राथमिकता होनी चाहिए, विशेष रूप से MCP में Streamable HTTP जैसे HTTP-आधारित ट्रांसपोर्ट का उपयोग करते समय।

जब HTTP-आधारित ट्रांसपोर्ट के साथ MCP सर्वर लागू किए जाते हैं, तो सुरक्षा एक प्रमुख चिंता बन जाती है जिसे कई हमले के तरीकों और सुरक्षा तंत्रों पर सावधानीपूर्वक ध्यान देने की आवश्यकता होती है।

### अवलोकन

MCP सर्वरों को HTTP पर एक्सपोज़ करने पर सुरक्षा अत्यंत महत्वपूर्ण होती है। स्ट्रीमेबल HTTP नए हमले के सतहें पेश करता है और सावधानीपूर्वक कॉन्फ़िगरेशन की आवश्यकता होती है।

यहाँ कुछ प्रमुख सुरक्षा विचार हैं:

- **Origin हेडर सत्यापन**: DNS रिबाइंडिंग हमलों को रोकने के लिए हमेशा `Origin` हेडर का सत्यापन करें।
- **Localhost बाइंडिंग**: स्थानीय विकास के लिए, सर्वरों को `localhost` से बांधें ताकि वे सार्वजनिक इंटरनेट पर एक्सपोज़ न हों।
- **प्रमाणीकरण**: उत्पादन डिप्लॉयमेंट के लिए प्रमाणीकरण लागू करें (जैसे API कुंजी, OAuth)।
- **CORS**: एक्सेस को सीमित करने के लिए Cross-Origin Resource Sharing (CORS) नीतियां कॉन्फ़िगर करें।
- **HTTPS**: ट्रैफ़िक को एन्क्रिप्ट करने के लिए उत्पादन में HTTPS का उपयोग करें।

### सर्वोत्तम अभ्यास

इसके अतिरिक्त, अपने MCP स्ट्रीमिंग सर्वर में सुरक्षा लागू करते समय निम्न सर्वोत्तम अभ्यासों का पालन करें:

- बिना सत्यापन के इनकमिंग अनुरोधों पर कभी भरोसा न करें।
- सभी एक्सेस और त्रुटियों को लॉग और मॉनिटर करें।
- सुरक्षा कमजोरियों को ठीक करने के लिए नियमित रूप से निर्भरता अपडेट करें।

### चुनौतियाँ

MCP स्ट्रीमिंग सर्वरों में सुरक्षा लागू करते समय आपको कुछ चुनौतियों का सामना करना पड़ेगा:

- विकास की सुविधा और सुरक्षा के बीच संतुलन बनाना
- विभिन्न क्लाइंट वातावरणों के साथ संगतता सुनिश्चित करना

### असाइनमेंट: अपना स्वयं का स्ट्रीमिंग MCP एप्लिकेशन बनाएं

**परिदृश्य:**
एक MCP सर्वर और क्लाइंट बनाएं जहाँ सर्वर आइटमों की एक सूची (जैसे, फ़ाइलें या दस्तावेज़) संसाधित करता है और प्रत्येक संसाधित आइटम के लिए एक नोटिफिकेशन भेजता है। क्लाइंट को प्रत्येक नोटिफिकेशन जैसे ही आता है, प्रदर्शित करना चाहिए।

**चरण:**

1. एक सर्वर टूल लागू करें जो एक सूची संसाधित करता हो और प्रत्येक आइटम के लिए नोटिफिकेशन भेजे।
2. एक क्लाइंट लागू करें जिसमें रीअल टाइम में नोटिफिकेशन प्रदर्शित करने के लिए संदेश हैंडलर हो।
3. अपने कार्यान्वयन का परीक्षण करने के लिए सर्वर और क्लाइंट दोनों चलाएं और नोटिफिकेशन देखें।
